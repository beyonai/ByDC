# 本体调用优化设计 - 问题修复总结

## 问题描述

用户查询"查询管理网格的清单"时返回空结果，经排查发现以下问题：

1. **OqlRouter 未初始化**：导致知识注入中间件无法构建本体 Schema
2. **工具调用日志不完整**：前端只显示工具名称，不显示入参和出参
3. **Agent 递归调用**：Agent 持续用相同错误参数调用 query_objects，触发递归限制
4. **参数序列化问题**：LangChain 将列表参数序列化为 JSON 字符串
5. **Executor.run() 调用错误**：缺少 request_id 参数且未使用 await

## 修复方案

### 1. 初始化 OqlRouter 及依赖组件

**文件**: `packages/datacloud-analysis/src/datacloud_analysis/bootstrap.py`

**修改内容**:

```python
# 1. 创建 OntologyLoader（注册表）
# 注意：OntologyLoader 的数据源配置来自 OWL 文件（dbsource.owl），不需要手动配置
ontology_loader = OntologyLoader()

# 2. 创建 DataSourceConfig（供 SqlExecutor 使用）
# 这是独立的数据源配置，用于 SQL 执行器
if settings.db.type.lower() in ("postgresql", "opengauss"):
    jdbc_url = f"jdbc:postgresql://{settings.db.host}:{settings.db.port}/{settings.db.name}"
elif settings.db.type.lower() == "mysql":
    jdbc_url = f"jdbc:mysql://{settings.db.host}:{settings.db.port}/{settings.db.name}"
else:
    jdbc_url = f"jdbc:{settings.db.type}://{settings.db.host}:{settings.db.port}/{settings.db.name}"

ds_config = DataSourceConfig(
    alias="default",
    db_type=settings.db.type.upper(),
    jdbc_url=jdbc_url,
    user=settings.db.user,
    password=settings.db.password,
)

# 4. 创建 DataSourceManager
ds_manager = DataSourceManager(configs={"default": ds_config})

# 5. 创建 SqlExecutor
sql_executor = SqlExecutor(ds_manager=ds_manager)

# 6. 创建 Executor
executor = Executor(sql_executor=sql_executor)

# 7. 创建 OqlRouter（使用 OntologyLoader 作为 registry）
oql_router = OqlRouter(registry=ontology_loader)

# 8. 初始化全局依赖
init_dependencies(
    oql_router=oql_router,
    executor=executor,
    datasource_registry=ConnectorRegistry,
)
```

**关键点**:
- OntologyLoader 需要配置数据源（字典格式）
- DataSourceConfig 需要 JDBC URL 格式
- OqlRouter 使用 OntologyLoader 作为 registry 参数
- 通过 init_dependencies 注册全局单例

### 5. 修复场景路径解析

**文件**: `packages/datacloud-analysis/src/datacloud_analysis/middlewares/knowledge_injection.py`

**修改**: 使 _resolve_scene_path() 与 Worker 逻辑一致

```python
def _resolve_scene_path(self, scene_path: str) -> Path | None:
    """解析场景路径，支持相对路径和绝对路径"""
    p = Path(scene_path)
    if p.is_absolute() and p.exists():
        return p

    # 相对路径：从当前工作目录查找
    cwd_path = Path.cwd() / scene_path
    if cwd_path.exists():
        return cwd_path

    return None
```

```python
@tool
def query_objects(
    object_type: str,
    # ... 其他参数
    config: RunnableConfig = None,
) -> dict[str, Any]:
    # 生成 request_id
    import uuid
    import asyncio
    request_id = (
        ((config or {}).get("configurable") or {}).get("thread_id")
        or uuid.uuid4().hex[:12]
    )

    # 获取或创建事件循环
    try:
        loop = asyncio.get_event_loop()
    except RuntimeError:
        loop = asyncio.new_event_loop()
        asyncio.set_event_loop(loop)

    # 调用 OqlRouter（使用 run_until_complete 包装异步调用）
    records = loop.run_until_complete(router.route(
        oql_params=oql_params,
        term_resolver=term_resolver,
        executor=executor,
        datasource_registry=datasource_registry,
        request_id=request_id,
    ))
```

1. **OqlRouter 初始化**: 检查日志中是否有 "OqlRouter initialized for knowledge injection"
2. **本体加载**: 检查知识注入中间件是否成功加载本体定义
3. **工具调用**: 检查日志中是否打印完整的工具入参和出参
4. **查询执行**: 验证 query_objects 能否正确解析参数并返回结果

## 架构说明

```
bootstrap.setup()
├── 创建 OntologyLoader (registry)
│   └── 数据源配置来自 OWL 文件（dbsource.owl）
├── 创建 DataSourceManager
│   └── 使用 DataSourceConfig (JDBC URL 格式，来自环境变量)
├── 创建 SqlExecutor
│   └── 使用 DataSourceManager
├── 创建 Executor
│   └── 使用 SqlExecutor
├── 创建 OqlRouter
│   └── 使用 OntologyLoader 作为 registry
└── 初始化全局依赖
    └── init_dependencies(oql_router, executor, datasource_registry)

知识注入中间件
├── 从环境变量读取 scene_path
├── 调用 OqlRouter.registry.load_from_owl_directory(scene_path)
│   ├── 解析 dbsource.owl 中的数据源定义
│   └── 通过 _dict_to_config() 转换为 DataSourceConfig
└── 将本体 Schema 注入到 LLM system message

数据源配置流程
├── OWL 文件定义（dbsource.owl）
│   └── dbParams: {"jdbc_url": "...", "user": "...", "password": "..."}
├── OWL 解析器（OwlParser）
│   └── _parse_database_definition() 解析为 datasource_configs
├── OntologyLoader
│   └── load_from_owl_directory() 调用 _dict_to_config() 转换
└── SqlExecutor/Executor
    └── 使用独立的 DataSourceConfig（来自环境变量）

异步调用链
query_objects (async tool)
└── await router.route(request_id)
    ├── await router.execute_single_step(request_id)
    │   └── await executor.run([task], request_id)
    └── await cross_source_executor.execute(request_id)
        ├── await executor.run([task], request_id)
        └── await _fetch_sub_records_batched(request_id)
            └── await executor.run([task], request_id)
```

## 注意事项

1. **数据源配置来源**:
   - **OntologyLoader 的数据源**：从 OWL 文件（dbsource.owl）加载，包含 jdbc_url 等完整配置
   - **SqlExecutor 的数据源**：从环境变量构建，用于执行 SQL 查询
   - 两者是独立的配置，不要混淆

2. **参数类型兼容性**:
   - 工具函数签名使用 Union[list, str] 接受两种格式
   - 函数体开头立即解析 JSON 字符串为原生类型

3. **环境变量命名**:
   - 使用 validation_alias 避免 Pydantic 字段名冲突
   - DB_SCHEMA 映射到 db_schema 字段

4. **日志级别**:
   - 初始化过程使用 INFO 级别
   - 工具调用详情使用 INFO 级别（便于调试）

5. **异步调用**:
   - 所有调用 executor.run() 的方法都必须是 async
   - 必须使用 await 调用 executor.run()
   - 必须传入 request_id 参数（用于追踪和日志）
   - LangChain 工具必须是同步函数，使用 loop.run_until_complete() 包装异步调用
