## 一、概述

开源项目不仅是代码的共享，更是一种社区驱动的开发模式。一个成功的开源项目需要：

- **清晰的代码结构** - 便于理解和贡献（参考 OpenClaw 的模块化设计）
- **完善的文档** - 降低上手门槛（参考 OpenAI Codex 的开发者文档）
- **规范的运营** - 维护社区健康发展（参考 Google Gemini CLI 的社区治理）
- **持续的迭代** - 保证项目活力

## 二、开源项目通用规范

### 2.1 核心文件清单

一个成熟的开源项目，其根目录必须具备“自解释性”，让外部开发者一眼看懂项目架构并知道如何参与。通用根目录结构如下：

*   **`.github/` (GitHub 自动化与生态目录)**
    *   `workflows/`：存放CI/CD流水线脚本（如自动化测试、构建镜像、发布Release）。
    *   `ISSUE_TEMPLATE/`：Issue提交模板（分Bug反馈、需求建议等，规范外部提问格式）。
    *   `PULL_REQUEST_TEMPLATE.md`：PR模板，**强制要求勾选是否包含测试用例、是否更新了文档**。
*   **`docs/` (官方文档目录)**
    *   推荐采用 Markdown 编写。按模块分为 `architecture/`（架构设计）、`api/`（接口文档）、`quick-start/`（快速入门）。
*   **`scripts/` (自动化与辅助脚本目录)**
    *   存放环境搭建、依赖打包、一键部署脚本。
    *   **特色集成**：存放用于**调用大模型校验测试用例**的Python/Shell脚本。
*   **`tests/` (全局测试目录)**
    *   严格划分为 `unit/` (单元测试) 和 `integration/` (集成测试)，约定全局统一的测试用例存放位置。
*   **核心规范文件 (根目录下)**
    *   `README.md`：项目门面（包含简介、架构图、快速开始、徽章）。
    *   `CONTRIBUTING.md`：**贡献指南（详细定义PR规范、代码规范，外部参与必读）**。
    *   `CHANGELOG.md`：版本变更日志，配合里程碑使用。
    *   `LICENSE`：开源许可证（如Apache 2.0, MIT）。
    *   `CODE_OF_CONDUCT.md`：社区行为准则。

### 2.2 许可证选择

**主流许可证对比：**

| 许可证 | 特点 | 适用场景 |
|--------|------|----------|
| **MIT** | 简单宽松，允许商用 | 适合大多数项目 |
| **Apache 2.0** | 允许专利授权 | 商业友好的开源项目 |
| **GPL v3** | 要求开源衍生作品 | 强制开源的项目 |
| **BSD 3-Clause** | 类似 MIT，含专利条款 | 商业项目 |
| **AGPL v3** | 强制网络版开源 | SaaS 项目 |

**推荐：Apache 2.0（商业友好）**

### 2.3 README.md 编写指南

README 是项目的门面，决定了访客的第一印象。

**必备结构：**
- 项目名称和一句话描述
- 徽章（Build、Version、License）
- 核心特性列表
- 安装说明
- 快速开始代码示例
- 依赖说明
- 文档链接
- 贡献指南链接
- 许可证说明

### 2.4 CONTRIBUTING.md 贡献指南

贡献指南降低贡献门槛，规范代码提交。

**必备内容：**
- 开发环境设置步骤
- 代码规范说明
- Commit 规范（推荐 Conventional Commits）
- Pull Request 流程：任何一个 PR 提交，必须同时包含：**业务代码变更 + 对应的测试用例 + 对应 docs/ 目录下的文档更新**。
- 代码审查：至少需要 1 名核心 Maintainer Review 通过后才可合并。
- 测试要求和测试文档要求

### 2.6 SECURITY.md 安全策略

指导用户如何报告安全漏洞。

### 2.7 CHANGELOG.md 版本日志

记录每个版本的变更，遵循 [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) 格式。

### 2.8 依赖管理规范
#### 2.8.1 模块之间依赖
*   **禁止源码级强依赖**：跨产品模块之间（如“门户”与“底层引擎”）绝不能在代码层直接互相引用源码。
*   **二进制/镜像交付**：引擎层代码变更后，通过 CI 自动构建出 Docker 镜像包或二进制包，推送到公共/私有仓库；门户层通过修改配置文件，以**版本号**（Tag）的方式引入引擎的镜像包。
#### 2.8.2 代码依赖工具
##### 前端项目

- **npm** - 官方包管理器
- **pnpm** - 高效磁盘存储（推荐）
- **yarn** - Facebook 出品

**最佳实践：**
- 使用 lock 文件锁定版本
- 定期使用 Dependabot 更新依赖
- 分离 devDependencies

##### Python 项目

- **uv** - 现代快速（推荐，FastAPI 在用）
- **poetry** - 优雅管理
- **pip** - 传统方式

**依赖分离：**
```

requirements-dev.txt  # 开发
requirements-test.txt # 测试
```

##### Java 项目

- **Maven** - 传统主流

### 4.2 Workflows 最佳实践
*   PR 提交 -> 触发 CI 自动化代码检查 (Lint) -> 自动执行所有单元测试 -> 大模型校验用例质量 -> 全部通过并被 Review 后合并。

#### 前端项目 CI

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build
```

#### Python 项目 CI

```yaml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        python-version: ['3.8', '3.9', '3.10', '3.11', '3.12']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: 'pip'
      - run: pip install -e ".[dev]"
      - run: ruff check .
      - run: mypy src
      - run: pytest
```

#### 自动发布


### 5.1 代码检查工具

| 语言 | 格式化 | Lint | 类型检查 |
|------|--------|------|----------|
| JavaScript/TypeScript | Prettier | ESLint | TypeScript |
| Python | Black | Ruff | MyPy |

### 5.2 测试规范

**测试金字塔：**
- 单元测试（底层，大量）
- 集成测试（中层）
- E2E 测试（顶层，少量）

**大模型自动校验**：
- 在 CI/CD 流水线中加入校验节点。当开发者提交 PR 时，自动触发 `scripts/` 下的大模型校验脚本，将新增代码与新增测试用例发送给 LLM（如本地部署的模型或 API），评估**测试用例的覆盖度、边界条件是否合理**，并在 PR 下方自动回复校验结果。

## 六、AI 驱动的开发闭环
### 6.1 事前：规范定义 - 让 AI 先懂“规则”
#### 6.1.1 脚手架文件定义（.ai/ 目录）
通过 `.skill`/`.cursor` 等结构化文件，提前定义大模型的编程规范，确保 AI 生成的代码从源头符合项目标准。

**示例：.ai/skill.json**
```json
{
  "language": "Python",
  "framework": "FastAPI",
  "code规范": {
    "命名规则": "变量使用蛇形命名，类使用大驼峰，函数使用小驼峰",
    "注释要求": "所有公共函数必须包含文档字符串，说明入参、出参、异常",
    "复杂度限制": "单个函数代码行数不超过 50 行，圈复杂度不超过 10",
    "依赖规范": "仅允许使用 requirements.txt 中声明的依赖，版本号需锁定"
  },
  "测试规范": {
    "覆盖率要求": "单元测试覆盖率不低于 80%",
    "用例规范": "必须包含正常场景、边界场景、异常场景"
  },
  "文档规范": {
    "API文档": "每个接口必须生成 OpenAPI 规范文档",
    "变更文档": "代码变更必须同步更新 CHANGELOG.md"
  }
}
```

**示例：.ai/cursor.json**
```json
{
  "生成约束": {
    "禁止硬编码": "所有配置项必须通过环境变量或配置文件读取",
    "安全要求": "用户输入必须做参数校验，防止注入攻击",
    "性能要求": "循环遍历需考虑大数据量场景，避免 O(n²) 复杂度"
  },
  "输出格式": {
    "代码结构": "先写注释，再写代码，最后写测试用例",
    "错误处理": "统一使用项目自定义的异常类，禁止裸抛 Exception"
  }
}
```

#### 6.1.2 AI 提示词模板预置
在 `.ai/prompt/` 目录预置不同场景的提示词模板，例如：
- `code-gen.md`：生成业务代码的提示词（关联 skill.json 规范）
- `test-gen.md`：生成单元测试的提示词
- `code-fix.md`：修正代码问题的提示词
- `doc-gen.md`：生成接口文档的提示词

#### 6.1.3 开发环境预置 AI 工具
要求所有开发者本地配置 AI 编程助手（如 Cursor、Claude、Copilot），并关联项目的 `.ai/` 规范文件，确保日常开发中 AI 辅助生成的代码符合项目标准。

### 6.2 事中：实时校验 - 本地提交的 AI 闭环
#### 6.2.1 Git Hook 触发自动化校验
在 `.git/hooks` 目录配置 pre-commit、pre-push 钩子，触发以下流程：
1. 自动运行 lint（代码格式检查）
2. 自动运行 format（代码格式化）
3. 自动运行单元测试
4. 调用 AI 脚本，校验代码是否符合 `.ai/skill.json` 定义的规范
5. 若任意步骤失败，触发 AI 自我反思修正流程

**示例：pre-commit 钩子脚本**
```bash
#!/bin/sh

# 1. 运行 lint
echo "运行代码格式检查..."
ruff check .
if [ $? -ne 0 ]; then
  echo "Lint 失败，触发 AI 分析修正..."
  python scripts/ai_code_fix.py --type lint
  exit 1
fi

# 2. 运行单元测试
echo "运行单元测试..."
pytest tests/unit/
if [ $? -ne 0 ]; then
  echo "测试失败，触发 AI 分析修正..."
  python scripts/ai_code_fix.py --type test
  exit 1
fi

# 3. AI 规范校验
echo "AI 规范校验中..."
python scripts/ai_validate.py --config .ai/skill.json
if [ $? -ne 0 ]; then
  echo "AI 校验失败，触发 AI 分析修正..."
  python scripts/ai_code_fix.py --type skill
  exit 1
fi

exit 0
```

#### 6.2.2 AI 自我反思修正流程
当本地校验失败时，`scripts/ai_code_fix.py` 脚本会：
1. 收集失败信息（lint 错误、测试失败日志、规范不符点）
2. 调用大模型 API，传入失败信息 + 项目 `.ai/` 规范
3. 要求大模型分析问题原因，并给出具体的修正方案
4. 将修正方案输出到控制台，或直接生成补丁文件
5. 提示开发者选择是否应用 AI 修正方案

**示例：ai_code_fix.py 核心逻辑**
```python
import openai
import json

def load_skill_config():
    with open(".ai/skill.json", "r") as f:
        return json.load(f)

def analyze_error(error_type, error_log):
    skill_config = load_skill_config()
    prompt = f"""
    你是一个遵循 {skill_config['framework']} 规范的资深开发者，请分析以下 {error_type} 错误，并给出修正方案：
    
    错误日志：
    {error_log}
    
    项目规范：
    {json.dumps(skill_config, ensure_ascii=False, indent=2)}
    
    要求：
    1. 明确指出问题所在
    2. 给出具体的代码修正示例
    3. 解释修正依据（关联项目规范）
    """
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

if __name__ == "__main__":
    import argparse
    parser = argparse.ArgumentParser()
    parser.add_argument("--type", required=True, choices=["lint", "test", "skill"])
    args = parser.parse_args()
    
    # 读取错误日志（实际场景需从标准输出/文件读取）
    with open(f".tmp/{args.type}_error.log", "r") as f:
        error_log = f.read()
    
    # AI 分析并输出修正方案
    fix_suggestion = analyze_error(args.type, error_log)
    print("=== AI 修正方案 ===\n")
    print(fix_suggestion)
```

#### 6.2.3 事中校验的容错机制
- 允许开发者手动跳过 AI 校验（需在 commit message 中注明原因，如 `[SKIP-AI] 紧急修复`）
- 跳过的校验会在 PR 提交时被 CI/CD 重点检查
- 累计跳过次数超过阈值，会触发 Maintainer 人工审核

### 6.3 事后：最终把控 - CI/CD 中的 AI 防线
#### 6.3.1 CI/CD 流水线增强（新增 AI 校验节点）
PR 合并前的最终校验流程：
```
PR 提交
    ↓
触发 CI 流水线
    ↓
1. 代码拉取 & 环境初始化
    ↓
2. 运行 Lint/Format/Test（与本地一致）
    ↓
3. AI 深度校验（比本地更严格）
   - 代码规范全量扫描（对比 .ai/skill.json）
   - 测试用例质量评估（覆盖度、边界场景）
   - 文档与代码一致性校验
    ↓
4. AI 生成校验报告（同步到 PR 评论区）
    ↓
5. 人工 Review（参考 AI 报告）
    ↓
6. 全部通过 → 合并；否则 → 驳回并附 AI 修正建议
```

#### 6.3.2 CI/CD 中 AI 校验示例（Python 项目）
```yaml
# .github/workflows/ci.yml
name: CI with AI Check

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

jobs:
  base-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
          cache: 'pip'
      - run: pip install -e ".[dev]"
      - run: ruff check .
      - run: mypy src
      - run: pytest --cov=src --cov-report=json > .tmp/test-report.json
  
  ai-check:
    needs: base-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install openai requests
      # 运行 AI 校验脚本
      - run: python scripts/ai_ci_check.py --skill .ai/skill.json --test-report .tmp/test-report.json > .tmp/ai-report.md
      # 将 AI 报告评论到 PR
      - name: Comment AI Report
        uses: marocchino/sticky-pull-request-comment@v2
        with:
          path: .tmp/ai-report.md
      # 校验不通过则失败
      - run: |
          if grep -q "校验不通过" .tmp/ai-report.md; then
            exit 1
          fi
```

#### 6.3.3 事后的问题沉淀
- CI/CD 中发现的共性问题，自动更新 `.ai/skill.json` 规范，避免重复出现
- 测试失败的场景，由 AI 自动生成新的测试用例，补充到 `tests/auto-gen/` 目录
- PR 驳回的原因，分类统计后优化事前的 `.ai/` 规范

### 8.3 用户反馈处理

#### 8.3.1 Issue 管理

**分类标签：**
- `bug` - Bug 报告（Something isn't working）
- `documentation` - 文档改进（Improvements or additions to documentation）
- `duplicate` - 重复问题（This issue or pull request already exists）
- `enhancement` - 功能增强（New feature or request）
- `good first issue` - 适合新手的任务（Good for newcomers）
- `help wanted` - 需要帮助（Extra attention is needed）
- `invalid` - 无效问题（This doesn't seem right）
- `question` - 问题咨询（Further information is requested）
- `wontfix` - 不予修复（This will not be worked on）


**处理流程：**

```
收到 Issue
    ↓
确认有效（1-2 天）
    ↓
分类标签
    ↓
分配优先级
    ↓
安排解决（Contribuor/Maintainer）
    ↓
验证关闭
```

#### 8.3.2 响应规范

**响应时间目标：**
| 类型 | 响应时间 | 解决时间 |
|------|----------|----------|
| Critical Bug | 24 小时 | 1 周 |
| Bug | 2 天 | 2 周 |
| Feature | 1 周 | 按计划 |
| Question | 3 天 | - |

**沟通原则：**
- 保持友好和专业
- 感谢用户的反馈
- 清晰解释决策原因
- 提供替代方案

#### 8.3.3 AI 驱动的 Issue 分类与响应
通过脚本自动处理 Issue，流程如下：
1. Issue 提交后，AI 自动分析内容，打上分类标签（bug/enhancement/question 等）
2. 根据标签触发不同的响应模板：
   - Bug 类：自动回复“已收到反馈，我们会在 X 小时内确认，如需快速解决可参考 [自查指南]”
   - 功能请求类：自动回复“感谢建议！我们已纳入版本规划评估，你可以关注 #XXX 里程碑”
   - 问题咨询类：AI 自动分析问题并给出初步解答，无法解答则分配给对应维护者
3. 自动关联相似 Issue，避免重复处理


**示例脚本：scripts/auto_issue_handler.py**
```python
import openai
import github
from github import Github

def classify_issue(issue_body):
    prompt = f"""
    请将以下 GitHub Issue 分类，返回唯一标签：bug、enhancement、question、documentation、invalid
    Issue 内容：{issue_body}
    """
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content.strip()

def auto_reply_issue(github_token, repo_name, issue_number):
    g = Github(github_token)
    repo = g.get_repo(repo_name)
    issue = repo.get_issue(issue_number)
    
    # 分类 Issue
    label = classify_issue(issue.body)
    issue.add_to_labels(label)
    
    # 根据标签回复
    reply_templates = {
        "bug": "感谢反馈！我们已标记为 Bug，会在 24 小时内确认问题。如需临时解决方案，可参考文档：[Bug 自查指南]",
        "enhancement": "感谢你的功能建议！我们会在本周的版本规划会议中评估，评估结果会同步到本 Issue。",
        "question": f"AI 初步解答：{get_ai_answer(issue.body)}\n如未解决，请补充更多信息，我们会尽快回复。"
    }
    if label in reply_templates:
        issue.create_comment(reply_templates[label])

def get_ai_answer(question):
    # 调用大模型生成问题解答
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": f"解答以下技术问题：{question}"}]
    )
    return response.choices[0].message.content

if __name__ == "__main__":
    import os
    github_token = os.getenv("GITHUB_TOKEN")
    repo_name = os.getenv("REPO_NAME")
    issue_number = int(os.getenv("ISSUE_NUMBER"))
    auto_reply_issue(github_token, repo_name, issue_number)
```

#### 8.3.4 Issue 与 PR/测试用例的自动关联
- 会议纪要中的待办事项，AI 自动提取并创建 Issue，关联到对应里程碑
- Issue 解决后，AI 自动检查是否有对应的 PR，若无则提示开发者创建
- 用户问答中发现的错误场景，AI 自动沉淀为测试用例，补充到 `tests/auto-gen/` 目录

#### 8.3.5 用户反馈的自动化沉淀：错误场景 → 测试用例
用户在 Issue/问答中反馈的错误，经 AI 分析后：
1. 提取错误的核心场景（输入、操作、环境）
2. 自动生成对应的单元测试/集成测试用例
3. 提交 PR 到 `tests/auto-gen/` 目录，由维护者审核后合并
4. 后续 CI/CD 会持续运行这些用例，防止问题复现
