# 项目技术架构与工程规范方案

基于《开源项目最佳实践汇报》的指导，并深入参考了 **OpenClaw**、**Gemini CLI** 以及 **Claude Code** 等先进 AI 开源项目的工程组织理念，现对 `whale_datacloud` 的代码工程架构、目录规范、质量检查以及自动化流程提出整体落地方案。

---

## 一、 核心架构与模块划分

为了使工程结构具备更强的“自解释性”，并符合开源社区的认知习惯，工程业务核心划分为以下几层，并明确单向依赖关系（严禁底层反向依赖上层）：

### 1. 应用展现层 (Application & Examples)
* **模块**：`examples/` (由原 `datacloud-apps` 与 `datacloud-mock` 整合)
* **职责**：面向最终用户的集成演示，包含业务组装逻辑。
* **说明**：将模拟数据和环境（Mock）直接内聚到示例应用中（如 `examples/crm_demo/mock_data/`），保证外部开发者在跑样例时，代码与环境数据能在一处自闭环，提升“开箱即用”体验。

### 2. 决策分析层 (Analysis & Planning)
* **模块**：`datacloud-analysis` (由原 `datacloud-agent` 更名)
* **职责**：作为顶层 AI 核心引擎，提供独立的 SDK，利用大模型进行决策、规划和分析。
* **依赖流向**：依赖 `datacloud-data`（外部数据）、`datacloud-knowledge`（领域约束）、`datacloud-memory`（历史上下文）。

### 3. 数据执行层 (Data Execution)
* **模块**：`datacloud-data` (由原 `datacloud-data-service` 更名)
* **职责**：提供统一的各类数据查询能力抽象（API、数据库、文档），将分析层的意图转化为具体执行并返回结果。
* **依赖流向**：依赖 `datacloud-knowledge`（如 Schema 映射）、`datacloud-memory`。

### 4. 基础设施层 (Infrastructure)
* **模块**：`datacloud-memory`、`datacloud-knowledge` (由原 `datacloud-knowledge-service` 更名)
* **职责**：最底层的基石。`memory` 提供上下文记忆存储能力；`knowledge` 提供领域术语、对象图谱等查询支持。
* **依赖流向**：**不依赖**其它业务模块，仅依赖外部第三方中间件。

## 二、 目录规范与工程骨架

为了实现“代码开源、核心测试闭源”的商业双轨制策略，整个工程骨架分为**主仓库（开源）**和**私有测试仓库（闭源）**两部分。

### 2.1 主仓库 (whale_datacloud - 开源部分)

基于单体仓库（Monorepo）模式，融合 AI 技能配置、国际化支持（i18n）与自动化规范，主仓库的标准目录树如下：

### 2.3 核心 SDK 内部规范 (以 datacloud-analysis 为例)

作为整个工程的 AI 大脑，`datacloud-analysis`（原 `datacloud-agent`）必须遵循清晰的领域驱动设计（DDD）与单一职责原则。其内部目录骨架规范如下：

```text
packages/datacloud-analysis/
├── src/
│   └── datacloud_analysis/    # Python 标准包目录 (包名使用下划线)
│       ├── orchestration/     # 核心编排层：负责多步推理、意图识别与任务拆解 (如 DAG 生成、React Loop)
│       ├── message_handler/   # 消息处理层：负责对接外部消息、适配多端输入输出格式
│       ├── session/           # 会话管理层：处理上下文的组装、截断与状态持久化 (如对接 PG Checkpointer)
│       ├── tools/             # 工具层：定义可供大模型调用的 Tools (如对接 datacloud-data 和 knowledge 的胶水代码)
│       ├── workspace/         # 沙盒/工作区层：执行代码沙盒、动态挂载等环境隔离操作
│       ├── memory/            # 记忆层抽象：对接底层的 datacloud-memory SDK，实现长短期记忆的存取
│       ├── i18n/              # 内部多语言支持层 (负责加载全局 locales/ 下的 prompt 与文案)
│       └── config/            # 模块内专属配置解析 (如 Pydantic Settings, env 解析)
├── tests/                     # 当前分析模块专属的单元与功能测试 (Unit / Integration)
│   ├── unit/                  # 无外部依赖的极速单元测试 (如测试 Prompt 组装逻辑、纯函数的字符串匹配)
│   └── integration/           # 模块内部边界集成测试 (如测试当前 SDK 内部组件串联、或直接调用 LLM API 是否联通，绝不调用其他兄弟 SDK)
├── README.md                  # 本模块的接入使用文档 (供 backend 或其他 SDK 使用)
└── pyproject.toml             # 本模块的依赖声明文件 (仅声明本模块所需的库)
```
**设计要点**：
* 采用 `src/` 布局防范隐式路径注入。
* **内聚测试**：注意这里的 `tests/` 只负责当前 `datacloud-analysis` 自身的逻辑验证。涉及到跨越多个 SDK 的测试必须放到根目录的全局 `tests/` 中。
* 将大模型核心流转机制 (`orchestration`) 与外部消息输入 (`message_handler`)、历史上下文 (`session`) 完全解耦。
* 通过 `tools/` 统一收拢对其他 SDK（如 `datacloud-data`）的调用调用边界。

### 三、 开源与闭源测试用例划分标准

为了在“拥抱开源社区”与“保护核心商业资产”之间取得平衡，测试用例必须严格按照以下标准进行物理隔离。

#### 3.1 应该公开的测试用例 (放在 `whale_datacloud/tests/`)
这些用例旨在向开源社区**证明系统是可用的**，并帮助外部贡献者验证他们的 PR 没有破坏基础功能。
* **基础功能与单元测试**：针对 SDK 基础函数的测试（如字符串处理、基础 SQL 拼接逻辑、工具类的加解密）。
* **标准 API 契约测试**：针对公开暴露的接口（如 FastAPI 的 `/health`、`/query`）的入参出参格式测试，使用虚构的、脱敏的假数据（如 `test_user_1`）。
* **公共组件测试**：开源模型（如 Llama3、Qwen）对接的 Adapter 测试、Redis/PostgreSQL 的基础连接与 CRUD 测试。
* **教学性质的 E2E 测试**：使用 `examples/` 目录下完全公开的 mock 数据（如精简版的 CRM demo 数据），跑通一个极简的“提问 -> 规划 -> 查库 -> 返回”的快乐路径（Happy Path）。

#### 3.2 必须闭源的私有资产 (放在 `whale_datacloud_private/`)
这些内容是您的**商业护城河**。大模型应用的核心竞争力往往不在于框架代码（这些极易被模仿），而在于“评测集（Eval Set）”、“行业 Know-how（知识库）”和“复杂的 Bad Case 修复逻辑”。

* **行业领域知识库 (Ontology & Terms)**：您提到的“行业黑话”、指标计算公式（如“GMV去重逻辑”）、细分领域的私有实体字典。这些数据能让通用的分析框架瞬间拥有行业专家的业务 sense，是极具变现价值的数据资产。
* **高阶 Prompt 模板 (Advanced Prompts)**：开源库中可以放基础的 System Prompt，但那些经过千百次试错、能够完美绕过某些模型缺陷、甚至针对特定甲方业务深度定制的 Prompt 集合，必须闭源。
* **高阶决策分析技能 (Analytical Skills)**：这往往是数据产品的核心卖点。基础的查数（Text-to-SQL）开源，但像“异动归因分析”（为什么昨天销量下降了）、“销量预测模型”（通过历史数据推演未来）、“自动研报生成”（将数据图表组装成行研报告）这些极其复杂的、融合了统计学与大模型规划能力的复合型技能（Tools/Sub-Agents），必须放在闭源仓库的 `analytical_skills/` 目录下。
* **真实客户数据驱动的评测集 (Eval)**：任何包含甲方真实业务数据、脱敏不完全的生产数据、或具有极高行业壁垒的语料库及问答对。竞争对手如果拿到了这些，可以直接用来微调（SFT）他们自己的模型。
* **核心防御性测试用例 (Anti-Bad-Case)**：您团队花了大量人力成本发现的、导致 AI 幻觉或 SQL 注入的罕见极端边界情况（Edge Cases）。这些用例体现了系统的鲁棒性上限。
* **商业版特有功能的 TDD 用例**：如果不准备开源某些企业级功能（如复杂的细粒度数据权限改写、高并发调度），与之对应的 TDD 测试代码必须闭源。

## 四、 代码检查与质量保障 (Lint & Test)

代码质量的防线需通过工具固化，避免人为疏漏。统一采用 Python 现代工具链：

1. **格式化与静态检查 (Linter & Formatter)**
   * **工具**：采用 **Ruff** 替代传统的 Flake8/Black/Isort，提供极速的代码检查与自动格式化。
   * **类型检查**：引入 **MyPy**，在业务模块（尤其是核心 SDK 的 API 边界）强制要求完整的类型注解。
2. **测试规范 (Testing)**
   * **框架**：**Pytest** 驱动。
   * **测试分层策略 (极其重要)**：
     * **模块级内聚测试 (如 `packages/datacloud-analysis/tests/`)**：
       * **必要性**：这是保证开发者“心流”的关键。一个模块的代码改动后，开发者只需要在当前目录下运行 `pytest tests/`，通常几秒钟内就能知道有没有破坏基础逻辑。
       * **内容**：只包含**无外部依赖的极速单元测试**（如纯函数、状态机扭转逻辑）和**仅限当前模块内部的集成测试**（如 `analysis` 模块调用真实的 LLM 接口，但 mock 掉数据库查询）。**绝不允许**跨模块调用（如去调用 `datacloud-data`）。
     * **全局跨模块测试 (如 `whale_datacloud/tests/`)**：
       * **必要性**：验证整个系统像齿轮一样咬合在一起时的表现。它通常执行较慢，需要拉起真实的数据库、缓存和第三方服务。
       * **内容**：**E2E (端到端)** 测试和真实的**跨包集成测试**（比如 `analysis` 接收到一句话，真实调用 `data` 查出结果）。这种测试放在根目录统一管理，供 CI/CD 流水线做最终拦截。
     * **AI 自动化错题本**：根目录的 `tests/auto-gen/` 也是必不可少的。它负责接收云端 AI 脚本根据 Issue 自动生成的回归测试用例。
3. **本地拦截 (Pre-commit Hooks)**
   * 强制配置 `.pre-commit-config.yaml`。在开发者 `git commit` 时，自动运行 Ruff 格式化和 MyPy 校验，未通过的代码禁止提交。

5. **TDD 与私有测试集隔离策略 (Private Testing)**
   * **背景需求**：采用 TDD（测试驱动开发）模式，但在开源场景下，某些核心业务测试用例可能包含商业机密、敏感真实数据或不想公开的核心判断逻辑。详细划分标准见第三章。
   * **架构实现 (Git Submodule 双库模式)**：
     * **主仓库 (开源)**：`whale_datacloud/` 正常开源，其自带的 `tests/` 仅包含基础用例，供社区跑通基础 CI。
     * **私有仓库 (闭源)**：建立一个完全独立的私有 Git 仓库（如 `whale_datacloud_private`），专门存放核心 TDD 用例、真实评估数据以及**不公开的商业级 Demo（Examples）**。
     * **本地挂载 (重点)**：在内部员工的开发电脑上，我们希望只用打开一个 VS Code/Cursor 窗口就能同时改动开源代码和测试闭源逻辑。因此，在主仓库中建立一个空目录 `.private/`，通过 Git Submodule 的方式将私有仓库拉取并挂载到 `.private/` 目录下。同时在主仓库的 `.gitignore` 中必须忽略该目录，防止误提交私有代码到开源库。
   * **研发与 CI 机制**：
     * **开源社区**：外部开发者 clone 主仓库时，没有权限也无法拉取 `.private/` 的内容，仅执行 `pytest tests/`。
     * **内部研发**：内部员工 clone 主仓库后，执行 `git submodule update --init` 即可拉取私有业务与用例。内部 CI 流水线则会执行 `pytest tests/ .private/tests/` 进行全量商业级覆盖率测试。

## 五、 可观测性与日志规范 (Observability & Logging)

为了确保各类 DevOps Agent（运维助手）、监控系统以及 MCP Tools 能够高效、准确地从系统中读取并解析运行状态，必须实行严格的结构化日志规范：

1. **结构化输出 (Structured Logging)**
   * **要求**：所有核心业务模块（尤其是 `datacloud-analysis` 和 `datacloud-data`）必须支持并默认使用 **JSON 格式** 输出日志。避免纯文本的随意拼接，因为纯文本极难被 AI Agent 准确正则或解析。
   * **标准字段**：每条日志必须包含标准的元数据：`timestamp` (时间戳), `level` (级别), `trace_id` (追踪ID), `module` (产生日志的模块/文件), `message` (核心摘要), 以及 `extra_data` (附加的JSON上下文)。

2. **上下文追踪链路 (Traceability)**
   * **Trace ID 贯穿**：在整个处理链路中（从应用展现层入口 -> 决策分析层规划 -> 数据执行层调用 -> 基础设施层落盘），必须在上下文中携带并传递统一的 `trace_id`。
   * **作用**：运维 Agent 或查询工具在排障时，只需根据一个 `trace_id` 就能抓取该用户请求在全链路上的所有日志，快速还原 AI 的决策过程和报错根因。

3. **日志分级标准 (Log Levels)**
   * `DEBUG`：记录大模型原始的 Prompt/Response、详尽的中间状态等。仅在排障或本地开发时开启。
   * `INFO`：记录系统流转的关键节点（如“开始规划”、“SQL拼装完成”、“开始查询数据库”）。
   * `WARN`：记录发生重试、降级处理、或存在潜在风险但未阻断流程的异常。
   * `ERROR / FATAL`：记录导致整个流程阻断的严重错误。此时必须附带异常类型、具体的错误代码以及完整的异常堆栈 (Stacktrace)。

## 六、 AI 驱动与开发 Skills 集成 (AI-Native Integration)

借鉴 **Claude Code** 的 Plugin 机制、**OpenClaw** 的 Skills 体系以及 **Gemini CLI** 的 MCP 与 Context 注入，工程原生集成 AI 开发能力：

1. **项目全局上下文注入 (Context Injection)**
   * 根目录提供 `AGENTS.md` 文件，系统性定义当前项目的**架构全貌**、**开发守则**、**核心术语/名词解释**。AI 助手（如 Cursor/Claude/Gemini/VS Code Cline等）在回答或写代码前，应该被配置为默认读取此文件，以保证生成的代码风格与架构意图不偏离。
   * **与 README.md 的区别**：`README.md` 是写给**人类开发者**看的，侧重于“如何安装、如何启动、项目的商业价值”；而 `AGENTS.md` 是写给**AI 模型**看的，重点约束“工程边界、命名规范、被废弃的第三方库名（Anti-patterns）”。
2. **开发技能库 (Skills Registry)**
   * 在 `.ai/skills/` (兼容任意编辑器) 中沉淀标准化开发技能，如：
     * `brainstorming`: 需求分析与方案设计技能。
     * `test-driven-development`: TDD 编写测试用例技能。
     * `systematic-debugging`: 缺陷排查指南技能。
   * 强制流程：开发者要求 AI 编写代码前，必须先调用对应的 Skill 进行规划，参考“先规划再写代码” (Plan Before Code) 规则。
### 3. MCP 服务集成 (Model Context Protocol)
   * 预留 `mcps/` 目录。在需要引入外部工具（如数据库直连查验、Chrome 抓取等）时，通过 MCP 协议让大模型直接调用底层能力，极大扩展 AI 助手的操作边界。
   * **文档即服务**：将 `docs/` 目录中的使用手册、接口文档封装为 `mcps/project_docs/` 服务，利用 RAG 或全文检索将其作为资源(Resource) 或工具(Tool) 暴露。这样，新加入的开发者只需问 AI：“帮我查一下怎么部署”，AI 即可通过该 MCP 实时读取并返回最新文档内容。

### 4. 主流 AI IDE 适配落地 (以 Cursor 为例)
* **Rules 适配**：Cursor 默认识别 `.cursor/rules/*.mdc`。为了兼顾通用性，可以将规范放在 `.cursor/rules/` 中，其他工具（如 VS Code Cline）通过工程根目录的说明文件进行引用。
* **Skills 触发**：Cursor 目前没有自动的 "Skill" 拦截器。使用方式为：在对话框中输入 `@.ai/skills/brainstorming.md`，将标准的规划 Prompt 作为上下文显式喂给 Agent。
* **MCP 挂载**：Cursor 原生支持 MCP。需在 Cursor Settings -> Features -> MCP 中，新增配置指向本地的 MCP Server（例如 `command: python mcps/query_db/server.py`）。配置生效后，Cursor Agent 将自动获得 `query_db` 等工具能力。

## 七、 Git 规范与 CI/CD 自动化流程

社区化开源项目必须有极强的流程规范，依托 GitHub Actions 建设自动化流水线。

### 1. Git 与版本规范
* **Commit 规范**：强制遵循 **Conventional Commits** (如 `feat:`, `fix:`, `docs:`, `chore:`)。可使用 Commitizen 工具进行辅助和拦截。
* **版本管理**：遵循 **Semantic Versioning** 语义化版本号，结合 `CHANGELOG.md` 自动生成发版日志。
* **PR 提交要求**：所有 Feature/Bugfix 的 PR 必须关联 Issue。提交 PR 时，模板中必须勾选：“包含对应测试用例”及“已更新相应文档”。

### 2. CI/CD 自动化流水线 (.github/workflows)
* **CI 基础卡点 (Push / PR 触发)**：
  * 环境拉起 -> 运行 Ruff 检查 -> 运行 MyPy -> 运行全量 Pytest 并输出 Coverage 覆盖率。
  * **任何一步失败则阻塞 PR 合并**。
* **AI 辅助 Review (PR 触发)**：
  * 借鉴 Gemini CLI / 开源实践：配置 `run-gemini-cli` 或编写自定义的 Python 脚本（如 `scripts/ci_ai_review.py`）。
  * 该脚本的运行逻辑：在 PR 发起时，利用 GitHub API 获取本次 PR 的代码 `diff`；读取工程根目录 `.ai/rules/` 下的架构与编码规范；将两者拼接为 Prompt 请求大模型（如 Claude 3.5 或 GPT-4）；最后将大模型生成的 Review 建议与潜在风险，通过 GitHub API 自动作为 Comment 回复在 PR 下方。
* **Issue 自动分发与管理 (Issue 触发)**：
  * 当用户提交 Issue 时，AI 根据内容自动打上 `bug`, `enhancement`, `question` 等 Label，并可自动回复常见问题的文档链接。
* **自动发布 (Tag 触发)**：
  * 合并到 `master/main` 并打上 `vX.Y.Z` 标签时，自动打包 `packages/` 下的各个 SDK，发布至 PyPI，并自动生成 GitHub Release Notes。
