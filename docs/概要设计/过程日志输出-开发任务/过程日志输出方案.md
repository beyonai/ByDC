# 过程日志输出方案

## 背景

用户发起数据查询或分析请求后，后端需要经历意图理解、查询规划、SQL生成、工具调用、结果汇总等多个步骤，整体耗时可能达到数十秒。若前端无任何反馈，用户体验极差。

本方案目标：在不改变最终答案质量的前提下，通过"过程日志"实时告知用户当前进展，让用户感知到系统正在工作，并能理解每一步在做什么。

---

## 核心约束

| 指标 | 要求 |
|------|------|
| 首字节响应 | ≤ 1 秒 |
| 后续输出速率 | ≥ 20 字/秒 |
| 内容可读性 | 面向业务用户，无技术术语 |

---

## 现状分析

`worker.py` 中已有以下过程日志输出点：

```
⑤ 发送"开始推理"通知
   → REASONING_LOG_START / think_title  → "思考中..."
   → REASONING_LOG_START / think_text   → "已接收到用户消息，开始处理"   ← 首字节保障

⑦ 流式驱动图事件
   → on_tool_start  → TASK_CREATE / task_title   → "调用工具: {tool_name}"
   → on_tool_end    → STEP_COMPLETE / task_finished → "工具完成: {tool_name}"
```

**问题：**
1. `on_tool_start/end` 输出的是内部工具名，业务用户看不懂。
2. 工具调用的参数（如文件路径、查询关键词）未打印，用户不知道在处理什么。
3. 两个工具事件之间可能有较长的 LLM 推理静默期，超过数秒无输出。
4. 缺少阶段性进度提示（如"正在理解您的问题"、"正在制定查询方案"）。

---

## 方案设计

### 1. 首字节保障（已实现）

在 `process_command` 的步骤 ⑤，进入图执行前立即推送：

```python
# think_title（折叠标题）
await context.emit_chunk(
    StreamChunkEvent(content="思考中..."),
    event_type=EventType.REASONING_LOG_START.value,
    content_type=SseReasonMessageType.think_title.value,
)
# think_text（首条内容，保障首字节）
await context.emit_chunk(
    StreamChunkEvent(content="已接收到用户消息，开始处理"),
    event_type=EventType.REASONING_LOG_START.value,
    content_type=SseReasonMessageType.think_text.value,
)
```

此两条消息在图启动前同步发出，确保 ≤ 1 秒到达前端。

---

### 2. 工具名称业务化映射 + 参数打印

在 `worker.py` 中增加工具名到业务描述的映射表，替换 `on_tool_start/end` 的原始输出。

实际注册到图中的内置工具如下（经代码核实）：

```python
_TOOL_DISPLAY = {
    # 知识与记忆
    "search_knowledge":  ("正在查阅业务知识库", "业务知识查阅完成"),
    "recall_memory":     ("正在回忆相关经验",   "相关经验回忆完成"),
    # 代码执行
    "sbx_run_code":      ("正在执行数据分析",   "数据分析执行完成"),
    # 文件操作
    "sbx_read_file":     ("正在读取分析文件",   "分析文件读取完成"),
    "sbx_write_file":    ("正在保存分析结果",   "分析结果保存完成"),
    # 技能与报告
    "build_skill":       ("正在构建分析技能",   "分析技能构建完成"),
    "render_report":     ("正在生成分析报告",   "分析报告生成完成"),
    # ReAct 能力选择
    "choose_capability": ("正在决策下一步操作", ""),
}

def _tool_display(tool_name: str) -> tuple[str, str]:
    """返回 (开始描述, 完成描述)，未命中时生成通用描述。"""
    return _TOOL_DISPLAY.get(tool_name, ("正在处理中...", "处理完成"))
```

> 注：`search_memory` / `read_memory` 虽已定义，但未注册到内置能力列表，不会在执行中触发，故不列入。

在 `_stream_graph` 的事件处理中，`on_tool_start` 同时打印工具参数：

```python
elif kind == "on_tool_start":
    tool_name = event.get("name", "")
    tool_input = event.get("data", {}).get("input", {})
    start_desc, _ = _tool_display(tool_name)

    # 提取关键参数拼接到描述中，让用户知道在处理什么
    detail = _extract_tool_detail(tool_name, tool_input)
    display_text = f"{start_desc}：{detail}" if detail else start_desc

    await context.emit_chunk(
        StreamChunkEvent(content=display_text),
        event_type=EventType.TASK_CREATE.value,
        content_type=SseReasonMessageType.task_title.value,
    )
elif kind == "on_tool_end":
    tool_name = event.get("name", "")
    _, end_desc = _tool_display(tool_name)
    if end_desc:
        await context.emit_chunk(
            StreamChunkEvent(content=end_desc),
            event_type=EventType.STEP_COMPLETE.value,
            content_type=SseReasonMessageType.task_finished.value,
        )
```

参数提取函数，针对文件路径、查询关键词等关键信息做特殊处理：

```python
def _extract_tool_detail(tool_name: str, tool_input: Any) -> str:
    """从工具调用参数中提取对用户有意义的关键信息。"""
    if not isinstance(tool_input, dict):
        return ""
    if tool_name in ("sbx_read_file", "sbx_write_file"):
        path = tool_input.get("file_path") or tool_input.get("path", "")
        if path:
            # 只取文件名，不暴露完整路径
            return os.path.basename(str(path))
    if tool_name == "search_knowledge":
        query = tool_input.get("query", "")
        return str(query)[:30] if query else ""
    if tool_name == "recall_memory":
        query = tool_input.get("query", "")
        return str(query)[:30] if query else ""
    if tool_name == "sbx_run_code":
        # 代码内容不展示，只提示
        return ""
    return ""
```

### 3. 思考过程阶段划分（think_title）

整个处理过程划分为 **4 个阶段**，每个阶段推送一条 `think_title`，作为前端折叠区块的标题。

#### 阶段划分与触发时机

| 阶段 title | 对应节点 | 触发事件 & 条件 |
|-----------|---------|----------------|
| **问题理解** | `knowledge_enhance` + `planning` | `on_chain_start`，`event["name"] == "knowledge_enhance"` |
| **任务生成** | `planning` 内 LLM 调用完成后 | `on_chat_model_end`，`event["metadata"]["langgraph_node"] == "planning"` |
| **任务执行** | `execution`（可循环） | `on_chain_start`，`event["name"] == "execution"`，仅首次触发 |
| **结果生成** | `end` | `on_chain_start`，`event["name"] == "end"` |

> 说明：
> - "问题理解"在 `knowledge_enhance` 节点启动时推送，覆盖知识库查询 + planning 前半段（意图解析）。
> - "任务生成"在 `planning` 节点的 LLM 调用结束后推送，此时任务列表已生成，标题切换有实际意义。
> - "任务执行"的 `execution` 节点会循环多次，通过 `_phase_emitted` 集合去重，只在首次进入时推送。

#### worker.py 改动点 — 阶段标题推送

在 `_stream_graph` 方法开头初始化去重集合：

```python
_phase_emitted: set[str] = set()
```

在 `astream_events` 循环中新增以下事件处理分支：

```python
# ── 阶段标题：问题理解 / 任务执行 / 结果生成 ──
elif kind == "on_chain_start":
    node_name = event.get("name", "")
    phase = _NODE_PHASE_TITLE.get(node_name)
    if phase and phase not in _phase_emitted:
        _phase_emitted.add(phase)
        await context.emit_chunk(
            StreamChunkEvent(content=phase),
            event_type=EventType.REASONING_LOG_START.value,
            content_type=SseReasonMessageType.think_title.value,
        )

# ── 阶段标题：任务生成（planning LLM 调用结束后） ──
elif kind == "on_chat_model_end":
    node_name = event.get("metadata", {}).get("langgraph_node", "")
    if node_name == "planning" and "任务生成" not in _phase_emitted:
        _phase_emitted.add("任务生成")
        await context.emit_chunk(
            StreamChunkEvent(content="任务生成"),
            event_type=EventType.REASONING_LOG_START.value,
            content_type=SseReasonMessageType.think_title.value,
        )
```

阶段标题映射常量（`on_chain_start` 路径使用）：

```python
_NODE_PHASE_TITLE: dict[str, str] = {
    "knowledge_enhance": "问题理解",
    # planning 的标题由 on_chat_model_end 触发，不在此映射
    "execution":         "任务执行",
    "end":               "结果生成",
}
```

#### 完整阶段流转示意（用户视角）

```
[think_title] 问题理解
  ✦ 正在理解业务术语...          ← on_chat_model_start in knowledge_enhance
  ✦ 正在查阅业务知识库：品类 销售额 上个月   ← on_tool_start search_knowledge
  ✓ 业务知识查阅完成             ← on_tool_end search_knowledge
  ✦ 正在分解分析任务...          ← on_chat_model_start in planning

[think_title] 任务生成           ← on_chat_model_end in planning
  （已生成 2 个子任务）

[think_title] 任务执行           ← on_chain_start execution（首次）
  ✦ 正在决策执行步骤...          ← on_chat_model_start in execution
  ✦ 正在执行数据分析             ← on_tool_start sbx_run_code
  ✓ 数据分析执行完成             ← on_tool_end sbx_run_code
  ✦ 正在保存分析结果：result.csv ← on_tool_start sbx_write_file
  ✓ 分析结果保存完成             ← on_tool_end sbx_write_file

[think_title] 结果生成           ← on_chain_start end
  ✦ 正在生成分析结论...          ← on_chat_model_start in end
  ✦ 正在生成分析报告             ← on_tool_start render_report
  ✓ 分析报告生成完成             ← on_tool_end render_report
```

### 4. LLM 推理阶段进度提示（think_text）

LLM 推理期间（`on_chat_model_start` → `on_chat_model_end`）可能有数秒静默。
在 `on_chat_model_start` 时推送一条 `think_text`，告知用户当前在做什么。

#### worker.py 改动点

在 `astream_events` 循环中新增：

elif kind == "on_chat_model_start":

### 5. 输出速率保障（≥ 20 字/秒）

过程日志本身是短文本（10~20 字），每条推送间隔取决于图节点执行速度。
为避免长时间静默，在 `_stream_graph` 中增加心跳机制：

```python
_HEARTBEAT_INTERVAL = 3.0   # 秒，超过此时间无输出则发一条心跳

_HEARTBEAT_MESSAGES = [
    "数据量较大，正在处理中...",
    "查询较复杂，请耐心等待...",
    "正在整合多维度数据...",
    "即将完成，请稍候...",
]
```

实现思路：用 `asyncio.wait_for` 包装每个事件的等待，超时则推送心跳消息，循环直到流结束。

```python
last_emit_time = asyncio.get_event_loop().time()
heartbeat_idx = 0

async for event in target_graph.astream_events(...):
    now = asyncio.get_event_loop().time()
    if now - last_emit_time > _HEARTBEAT_INTERVAL:
        msg = _HEARTBEAT_MESSAGES[heartbeat_idx % len(_HEARTBEAT_MESSAGES)]
        await context.emit_chunk(
            StreamChunkEvent(content=msg),
            event_type=EventType.REASONING_LOG_START.value,
            content_type=SseReasonMessageType.think_text.value,
        )
        heartbeat_idx += 1
        last_emit_time = now
    # ... 正常事件处理 ...
    if kind in ("on_tool_start", "on_tool_end", "on_chat_model_start"):
        last_emit_time = asyncio.get_event_loop().time()
```

## 注意事项

- `think_text` 类型消息在前端以折叠形式展示，不影响最终答案区域。
- 心跳消息应避免重复，使用轮转索引。
- 工具名映射表需随工具集扩展同步维护，建议放在独立常量文件中。
- `on_chat_model_stream` token 流由 `insight_node` 自行推送，`worker.py` 不重复转发（现有注释已说明）。
