# 架构边界重构方案

> 版本：v2.1
> 日期：2026-04-11
> 目标：完成 `dataCloud` 与 `byclaw-data` 的架构边界解耦重构

---

## 一、背景与目标

### 1.1 现状问题

当前架构存在以下耦合问题：

| 问题域 | 现状 | 影响 |
|--------|------|------|
| 依赖耦合 | `datacloud-analysis` 在 `pyproject.toml` 中硬编码依赖 `by-framework>=0.1.2` | 无法在非 gateway 场景（API 服务、MCP 服务）下复用 |
| 依赖耦合 | `datacloud-data` 的 `object.py` / `view.py` / `plan_agent.py` 通过 `get_gateway_context()` 直接调用 `GatewayProgressReporter`，绕过已有的 `EventBus` | SDK 内部存在两条并行的事件通道，`EventBus` 形同虚设 |
| 上下文污染 | `InvocationContext`（`context.py`）的 `RequestContext` 中存储了 `gateway_context: Any` | gateway 运行时对象渗入 SDK 数据层，SDK 无法独立运行 |
| agent 初始化 | agent 所有配置散落在 `InitDataCloudDigitalEmployeePlugin` 硬编码中 | 多 agent、多租户场景无法通过配置驱动 |
| 服务形态单一 | 只能以 gateway worker 方式（`byclaw-data/main.py`）启动 | 无法支持标准 HTTP/OpenAI 协议或 MCP 协议接入 |
| 文件写入耦合 | 文件写入路径和行为无法在不同运行环境（gateway / API server / CLI）间切换 | gateway 场景和非 gateway 场景行为不一致 |

### 1.2 重构目标

1. `datacloud-analysis` / `datacloud-data` 作为纯 SDK，零 gateway 运行时依赖，可独立安装使用。
2. `datacloud-data` 内部统一走 **EventBus** 发布查询进度事件；`GatewayProgressReporter` 作为 EventBus 订阅者，移至 `byclaw-data` 层注册。
3. `datacloud-analysis` 的进度推送完全依赖 **LangGraph 原生 streaming**（`astream_events`），不引入额外接口层。
4. agent 配置文件化，以 `agentID` 为文件名，支持多 agent 配置驱动初始化。
5. 新增 API Server 和 MCP Server 服务形态示例。
6. 文件写入行为插件化，由宿主层注入，不在 agent 配置文件中声明。

---

## 二、核心机制原理

> 本章说明重构方案依赖的两个核心机制：**EventBus** 和 **LangGraph Streaming**。
> 理解这两个机制是理解解耦方案的前提。

### 2.1 EventBus（事件总线）

#### 2.1.1 原理

EventBus 实现**发布-订阅（Pub/Sub）**模式：

- **发布者（Publisher）**：查询管线内部逻辑（`QueryObserver`），只负责"发生了什么"，不关心谁在监听。
- **订阅者（Subscriber）**：任意数量的处理器，在 Bus 上注册对特定事件类型的处理函数，与发布者完全解耦。
- **事件（Event）**：纯数据对象（`@dataclass`），描述一次管线状态变更，不携带行为。

```
查询管线（object.py）
    └─ QueryObserver.on_plan_generated()
            └─ EventBus.publish(QueryPlanGenerated(...))
                    ├─ [订阅者 A] GatewayProgressHandler → emit_state()  ← byclaw-data 注册
                    ├─ [订阅者 B] TraceLogger → 写日志                    ← 已有
                    └─ [订阅者 C] MetricsCollector → 指标上报             ← 未来扩展
```

项目中 `datacloud_data_sdk/events/` 目录下已有完整实现：

| 文件 | 职责 |
|------|------|
| `bus.py` | `EventBus`：`subscribe(event_type, handler)` / `publish(event)` |
| `events.py` | 领域事件定义：`QueryPlanGenerated`、`StepsExecuted`、`AggregationCompleted` 等 |
| `query_observer.py` | `QueryObserver`：查询管线调用此对象，内部 `publish` 到 bus |
| `handlers.py` | `register_query_handlers()`：批量注册通用处理器的工厂函数 |

#### 2.1.2 当前的问题

`object.py` 中当前存在**两条并行的事件通道**：

```python
# object.py — 现状（有问题）

# 通道 1：正确的做法，通过 QueryObserver 发布到 EventBus
observer = QueryObserver(config.event_bus, trace_id=trace_id)
await observer.on_plan_generated(request_id, plan)

# 通道 2：错误的做法，直接绕过 EventBus 调用 GatewayProgressReporter
gw_reporter = None
_gw_ctx = get_gateway_context()       # ← 从 contextvars 取 gateway 对象
if _gw_ctx is not None:
    gw_reporter = GatewayProgressReporter(_gw_ctx)
if gw_reporter:
    await gw_reporter.on_plan_generated(summary)  # ← 直接调 gateway
```

两条通道并存导致：EventBus 已经发布了事件，但 UI 进度显示依赖的是通道 2 的直调，使 EventBus 实际上未被充分利用。

#### 2.1.3 重构后的正确用法

重构后，`object.py` 只走通道 1，通道 2 完全删除：

```python
# object.py — 重构后（正确）

# 只通过 QueryObserver 发布事件，SDK 层不再感知 gateway
observer = QueryObserver(config.event_bus, trace_id=trace_id)
await observer.on_plan_generated(request_id, plan)
# ↑ EventBus 会将此事件分发给所有订阅者
```

`byclaw-data` 层在请求开始时，将 `GatewayProgressHandler` 注册为订阅者：

```python
# byclaw_data/plugins/observer_plugins/gateway_progress_handler.py（新增）

from datacloud_data_sdk.events.bus import EventBus
from datacloud_data_sdk.events.events import QueryPlanGenerated, StepsExecuted, AggregationCompleted

class GatewayProgressHandler:
    """将 datacloud-data 的 EventBus 事件桥接到 gateway emit_state/emit_chunk。"""

    def __init__(self, gateway_ctx) -> None:
        self._ctx = gateway_ctx

    def register(self, bus: EventBus) -> None:
        bus.subscribe(QueryPlanGenerated, self._on_plan_generated)
        bus.subscribe(StepsExecuted, self._on_steps_executed)
        bus.subscribe(AggregationCompleted, self._on_aggregation_completed)

    async def _on_plan_generated(self, event: QueryPlanGenerated) -> None:
        await self._ctx.emit_state(f"查询计划已生成（{len(event.plan.get('steps', []))} 步）", ...)

    async def _on_steps_executed(self, event: StepsExecuted) -> None:
        await self._ctx.emit_state("查询步骤执行完成", ...)

    async def _on_aggregation_completed(self, event: AggregationCompleted) -> None:
        await self._ctx.emit_state(f"结果聚合完成，共 {len(event.records)} 条", ...)
```

**好处**：
- `datacloud-data` SDK 完全不感知 gateway，可独立运行。
- 新增一个观察者（如指标采集、日志）只需订阅 EventBus，不改管线代码。
- 单元测试中直接断言 EventBus 上的事件，不需要 mock gateway。

### 2.2 LangGraph Streaming

#### 2.2.1 原理

LangGraph 编译后的图（`CompiledGraph`）提供两种流式输出接口：

**① `astream(input, stream_mode=...)`**：返回图执行过程中的**状态快照**流。

```python
async for state_snapshot in graph.astream(input, stream_mode="values"):
    # state_snapshot 是每次节点执行后的完整 State 字典
    messages = state_snapshot.get("messages", [])
    final_answer = state_snapshot.get("final_answer", "")
```

适合：需要知道"每一步之后整个状态是什么"的场景（嵌入式调用、调试）。

**② `astream_events(input, version="v2")`**：返回图内部的**细粒度事件流**，包含：

| 事件类型 | 触发时机 |
|----------|----------|
| `on_chat_model_start` | LLM 开始生成 |
| `on_chat_model_stream` | LLM 生成一个 token |
| `on_chat_model_end` | LLM 生成完毕 |
| `on_tool_start` | 工具调用开始 |
| `on_tool_end` | 工具调用结束 |
| `on_chain_start/end` | 节点（Node）开始/结束 |

```python
async for event in graph.astream_events(input, version="v2"):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        token = event["data"]["chunk"].content
        await context.emit_chunk(token)          # 流式输出 token
    elif kind == "on_tool_start":
        tool_name = event["name"]
        await context.emit_state(f"调用工具：{tool_name}")  # 推送工具调用进度
```

适合：需要流式推送 token 和节点进度的场景（gateway worker、SSE API）。

#### 2.2.2 当前项目的使用情况

`byclaw-data/worker.py` 的 `_stream_graph()` 方法**已经正确使用了 `astream_events`**：

```python
# byclaw_data/worker.py — _stream_graph()（已有，正确）

async for event in target_graph.astream_events(graph_input, config=config, version="v2"):
    kind = str(event["event"])

    if kind == "on_chat_model_end":
        node_name = (event.get("metadata") or {}).get("langgraph_node", "")
        if node_name == "planning":
            async with context.sub_step("任务生成"):   # ← 阶段标题推送
                pass

    elif kind == "on_tool_start":
        tool_name = event.get("name", "")
        async with context.sub_step(tool_name):        # ← 工具调用标题推送
            pass
```

`datacloud-analysis/orchestration/runner.py` 的 `run_agent()` 使用 `astream(stream_mode="values")` 用于嵌入式调用（非 gateway 场景）：

```python
# runner.py — run_agent()（已有，正确）

async for event in compiled.astream(input_payload, config=run_config, stream_mode="values"):
    yield event    # 调用方获取每步状态快照
```

#### 2.2.3 为什么不需要额外的 IProgressReporter 接口

由于 `astream_events` 已经把图内部的所有事件暴露出来，**宿主层只需消费这个 stream，根据事件类型做不同处理**：

```
astream_events 事件流
        │
        ├─ [gateway worker]    → context.emit_state() / emit_chunk()   ← 已有
        ├─ [API Server/SSE]    → 转换为 OpenAI SSE chunk               ← 新增示例
        └─ [MCP Server]        → 转换为 MCP CallToolResult stream      ← 新增示例
```

三种宿主层只是对同一个事件流做不同的翻译，不需要在 `datacloud-analysis` 内部再定义一个抽象接口。引入接口反而会造成：
1. 接口与 LangGraph stream 语义重叠，未来维护者不知用哪条路。
2. 每个宿主层都要实现同一个接口，增加样板代码。

**结论**：`datacloud-analysis` 层的进度推送不需要新接口，`astream_events` 就是那个接口。

## 三、架构边界全景

### 3.1 调用层次与职责边界

各层之间是**自上而下的调用**关系，下层不感知上层的存在：

```
┌──────────────────────────────────────────────────────────────────────┐
│         宿主层（byclaw-data / API Server / MCP Server）                │
│                                                                      │
│  职责：接收用户请求，驱动 agent 执行，将结果翻译为对应协议输出                │
│                                                                      │
│  · 调用 graph.astream_events() 并迭代事件流                             │
│  · 按事件类型翻译输出：emit_state / SSE chunk / MCP CallToolResult      │
│  · 在请求开始时向 EventBus 注册 GatewayProgressHandler                  │
│  · 在初始化时向 SDK 注入 IFileWriter 实现                               │
│                                                                      │
│  插件：worker_plugins/  observer_plugins/  file_writer_plugins/ ...   │
└────────────┬──────────────────────────────────────────┬──────────────┘
             │ ① 调用 astream_events()                  │ ③ 注入 IFileWriter
             │   并迭代返回的事件流                       │ ④ 注册 EventBus 订阅者
             ▼ ② 事件流（yield 返回给调用方）             │
┌────────────────────────────┐              ┌────────────▼─────────────┐
│   datacloud-analysis       │              │   datacloud-data          │
│   （纯 SDK，无 gateway 依赖）│              │   （纯 SDK，无 gateway 依赖）│
│                            │              │                           │
│   AgentConfigLoader        │   工具调用    │   EventBus + QueryObserver│
│   ← 读 agentID.yaml        │──────────────►   publish 领域事件         │
│                            │              │                          │
│   LangGraph CompiledGraph  │              │   IFileWriter（接口）     │
│   · astream_events() 是    │              │   LocalFileWriter（默认）  │
│     graph 上的方法，        │              │                           │
│     由宿主层调用，           │              │   不含 gateway 对象        │
│     graph 执行时 yield 事件  │              │                          │
│                            │              └──────────────────────────┘
│   不含 gateway 对象         │
└────────────────────────────┘
```

> `datacloud-analysis` 与 `datacloud-data` 之间的箭头是**工具调用**：
> LangGraph agent 执行 `query_Order` 等工具时，工具内部调用 `datacloud-data` SDK。
> 这与 `astream_events` 没有关系，是两条独立的路径。

---

### 3.2 astream_events 工作方式

`astream_events` 不是主动推送，而是一个 **async generator（异步生成器）**，宿主层主动驱动迭代：

```
宿主层（_stream_graph）                        LangGraph graph 内部执行

async for event in                  ←────────  graph.astream_events(input, version="v2")
    graph.astream_events(...):
                                                执行到某个动作时 yield 对应事件：
    kind = event["event"]                       ├─ LLM 生成一个 token
                                                │      → yield {"event": "on_chat_model_stream", ...}
    if kind == "on_chat_model_stream":          │
        token = event["data"]["chunk"].content  ├─ 工具调用开始
        emit_chunk(token)       ◄── 宿主层处理  │      → yield {"event": "on_tool_start", ...}
                                                │
    elif kind == "on_tool_start":               ├─ 工具调用结束
        tool_name = event["name"]               │      → yield {"event": "on_tool_end", ...}
        emit_state(tool_name)   ◄── 宿主层处理  │
                                                └─ 节点结束
                                                       → yield {"event": "on_chain_end", ...}
```

**关键点**：
- 宿主层用 `async for` **拉取**事件，graph 用 `yield` **生产**事件，控制权始终在宿主层
- `datacloud-analysis` SDK 内部不知道宿主层是 gateway、API Server 还是 MCP Server
- `datacloud-data` 与 `astream_events` **无关**，两者通过工具调用连接

---

### 3.3 两套事件机制的分工

方案中涉及两套独立的事件机制，职责不同，不要混淆：

| 机制 | 所在层 | 谁生产 | 谁消费 | 传递什么 |
|------|--------|--------|--------|----------|
| LangGraph `astream_events` | datacloud-analysis | LangGraph 执行引擎 | 宿主层（`_stream_graph`） | agent 执行过程的原始事件：节点切换、LLM token、工具调用 |
| EventBus `publish/subscribe` | datacloud-data | `QueryObserver` | `GatewayProgressHandler`（宿主层注册的订阅者） | 数据查询管线的业务语义事件：计划生成、步骤执行、聚合完成 |

两者共同构成宿主层的进度感知能力：
- `astream_events`：知道"agent 在调哪个工具、LLM 在输出什么 token"
- `EventBus`：知道"数据查询管线走到哪步了、查了多少条数据"

## 四、重构方案详细设计

### 4.1 Agent 配置文件化

#### 4.1.1 目标

将 agent 初始化参数从 `InitDataCloudDigitalEmployeePlugin` 中的硬编码逻辑抽取为 YAML 配置文件，以 `agentID` 为文件名，支持多 agent 并存。

**agent 配置描述 agent 的认知能力**（模型、工具、本体、技能），不描述部署拓扑（文件写到哪里、进度推送到哪里）——后者由宿主层注入。

#### 4.1.2 配置文件规范

**目录结构**

```
agents/
└── {agentID}.yaml          # 每个 agent 一个配置文件
```

**配置文件格式**

```yaml
# agents/datacloud_default.yaml
agent_id: datacloud_default
display_name: "DataCloud 数据分析助手"

# LLM 配置（可被环境变量覆盖）
llm:
  model: "Qwen/Qwen3-235B-A22B"
  temperature: 0.0
  max_tokens: 8192

# 本体 OWL 文件路径（支持 ${ENV_VAR} 占位符）
ontology:
  root_dir: "${WORKSPACE_DIR}/resource/knowledge/import_package_owl_latest"
  entry_file: "meta/library.owl"

# 工具开关
tools:
  enable_code_exec: true
  enable_file_io: true
  enable_knowledge: true
  enable_ask_user: true

# 技能加载路径
skills:
  paths: []

# 记忆配置
memory:
  enabled: true
  backend: "in_memory"    # in_memory | postgres

# 检查点配置（连接信息从环境变量读取）
checkpointer:
  backend: "postgres"     # none | postgres

# 工具钩子插件
tool_hook_plugins:
  - "datacloud_analysis.tool_hook_plugins.builtin.semantic_param_enhancer:SemanticParamEnhancer"
```

> **注意**：配置文件中不包含 `file_writer.backend` 等部署拓扑信息。文件写入实现由宿主层（`byclaw-data` 插件 / API Server 初始化代码）注入，与 agent 身份无关。

#### 4.1.3 新增文件

**`AgentConfig` 数据模型**

```python
# datacloud_analysis/config/agent_config.py（新增）
from pydantic import BaseModel, Field
from typing import Literal

class LLMConfig(BaseModel):
    model: str
    temperature: float = 0.0
    max_tokens: int = 8192

class OntologyConfig(BaseModel):
    root_dir: str
    entry_file: str = "meta/library.owl"

class MemoryConfig(BaseModel):
    enabled: bool = True
    backend: Literal["in_memory", "postgres"] = "in_memory"

class CheckpointerConfig(BaseModel):
    backend: Literal["none", "postgres"] = "none"

class AgentConfig(BaseModel):
    agent_id: str
    display_name: str = ""
    llm: LLMConfig
    ontology: OntologyConfig
    tools: dict = Field(default_factory=dict)
    skills: dict = Field(default_factory=dict)
    memory: MemoryConfig = Field(default_factory=MemoryConfig)
    checkpointer: CheckpointerConfig = Field(default_factory=CheckpointerConfig)
    tool_hook_plugins: list[str] = Field(default_factory=list)
```

**`AgentConfigLoader`**

```python
# datacloud_analysis/config/agent_config_loader.py（新增）
import os, yaml
from pathlib import Path
from datacloud_analysis.config.agent_config import AgentConfig

class AgentConfigLoader:
    def __init__(self, config_dir: str | Path):
        self._config_dir = Path(config_dir)

    def load(self, agent_id: str) -> AgentConfig:
        config_path = self._config_dir / f"{agent_id}.yaml"
        if not config_path.exists():
            raise FileNotFoundError(f"Agent config not found: {config_path}")
        raw = os.path.expandvars(config_path.read_text(encoding="utf-8"))
        return AgentConfig.model_validate(yaml.safe_load(raw))
```

### 4.3 依赖解耦：datacloud-data 去除 gateway 直调，统一走 EventBus

#### 4.3.1 问题根源

`object.py` / `view.py` / `plan_agent.py` 中存在以下模式：

```python
from datacloud_data_sdk.context import get_gateway_context
from datacloud_data_sdk.events.gateway_reporter import GatewayProgressReporter

_gw_ctx = get_gateway_context()
if _gw_ctx is not None:
    gw_reporter = GatewayProgressReporter(_gw_ctx)
    await gw_reporter.on_plan_generated(summary)
```

这是绕过 EventBus 的直接调用，造成：
- `InvocationContext.RequestContext` 被迫存储 `gateway_context: Any` 这个运行时对象。
- SDK 测试必须 mock gateway context。
- EventBus 上的同类事件与这里的直调存在语义重复。

#### 4.3.2 重构方案

**第一步：删除直调逻辑**

从 `object.py` / `view.py` / `plan_agent.py` 中删除所有 `get_gateway_context()` 和 `GatewayProgressReporter` 的调用。这些位置已有 `QueryObserver` 的对应调用，事件已经发布到 EventBus，直调是多余的。

**第二步：从 RequestContext 移除 gateway_context 字段**

```python
# context.py — 修改前
```

`GatewayProgressHandler` 在每次请求开始时，从 `InvocationContext` 拿到 `EventBus` 实例并注册订阅：

```python
# byclaw_data/plugins/observer_plugins/gateway_progress_handler.py（新增）

from datacloud_data_sdk.events.bus import EventBus
from datacloud_data_sdk.events.events import (
    QueryPlanGenerated, StepsExecuted, AggregationCompleted,
    PlanRetryRequested, PlanValidationFailed,
)

class GatewayProgressHandler:
    """将 datacloud-data EventBus 事件桥接到 gateway 的 emit_state/emit_chunk。"""

    def __init__(self, gateway_ctx) -> None:
        self._ctx = gateway_ctx

    def register(self, bus: EventBus) -> None:
        """在请求开始时调用，订阅本次请求的 EventBus。"""
        bus.subscribe(QueryPlanGenerated, self._on_plan_generated)
        bus.subscribe(PlanRetryRequested, self._on_plan_retry)
        bus.subscribe(StepsExecuted, self._on_steps_executed)
        bus.subscribe(AggregationCompleted, self._on_aggregation_completed)
        bus.subscribe(PlanValidationFailed, self._on_validation_failed)

    async def _on_plan_generated(self, event: QueryPlanGenerated) -> None:
        steps = event.plan.get("steps", [])
        await self._ctx.emit_state(f"查询计划已生成（共 {len(steps)} 步）", ...)

    async def _on_plan_retry(self, event: PlanRetryRequested) -> None:
        errors = "；".join(event.validation_errors[:3])
        await self._ctx.emit_state(f"计划校验失败，正在重试（第 {event.retry_count} 次）：{errors}", ...)

    async def _on_steps_executed(self, event: StepsExecuted) -> None:
        await self._ctx.emit_state(f"查询步骤执行完成（{len(event.step_results)} 步）", ...)

    async def _on_aggregation_completed(self, event: AggregationCompleted) -> None:
        await self._ctx.emit_state(f"结果聚合完成，共 {len(event.records)} 条记录", ...)

    async def _on_validation_failed(self, event: PlanValidationFailed) -> None:
        await self._ctx.emit_state(f"计划验证最终失败：{event.errors[:2]}", ...)
```

**注册时机**：在 `DataCloudWorker` 处理每次请求时，构建 `InvocationContext` 的同时，向当次请求的 `EventBus` 注册 handler。

#### 4.3.3 对 API Server / MCP Server 场景的影响

无 gateway context 时，EventBus 上不注册任何 `GatewayProgressHandler`，事件发布后静默忽略（无订阅者时 `EventBus.publish()` 为空操作）。无需注入 `NoOp` 实现，天然兼容。

### 4.4 Agent 服务形态扩展

#### 4.4.1 背景：ontology_query 模式下的两层 MCP 架构

在 `ONTOLOGY_LOAD_MODE=ontology_query` 模式下，**系统中存在两个层次的 MCP**，职责完全不同，不能混淆：

```
外部用户 / Claude Desktop
        ↓ 自然语言问题："本月各区域销售额是多少？"
┌───────────────────────────────────────────┐
│  外部服务入口（三种形态之一）               │
│  Gateway Worker / API Server / MCP Server  │  ← 本章新增示例的目标
└───────────────────────────────────────────┘
        ↓ 驱动 LangGraph agent
┌───────────────────────────────────────────┐
│  datacloud-analysis（LangGraph agent）     │
│  意图理解 → 工具调度 → 结果整合            │
│  工具：query_Order, compute_Order, ...     │
└───────────────────────────────────────────┘
        ↓ query_{ObjectCode} 工具调用
        ↓ 路由到 ONTOLOGY_MCP_ENDPOINT
┌───────────────────────────────────────────┐
│  内部 MCP（已有）                          │
│  datacloud-data 服务 /api/v1/mcp           │  ← mcp_handler.py，不是新增目标
│  执行具体数据查询，返回结构化数据           │
└───────────────────────────────────────────┘
        ↓
  数据库 / 数据源
```

- **内部 MCP**（`datacloud_data_service/api/mcp_handler.py`）：已有实现，对应 `.env` 中的 `ONTOLOGY_MCP_ENDPOINT`，供 analysis agent 内部调用数据查询工具。
- **外部服务入口**（新增 example）：接受用户的自然语言问题，驱动完整的 analysis agent 进行问答，最终返回答案。三种形态使用相同的 agent 内核，只是面向外部的协议不同。

> 因此，新增的 API Server 和 MCP Server 示例都包含完整的 LangGraph agent，不是仅暴露数据查询层。`DATACLOUD_DISABLE_ASK_USER_TOOL=1`（byclaw-data `.env` 已配置）确保 agent 不会向外部追问，适合包装为单轮问答形式。

#### 4.4.2 三种服务形态对比

| 服务形态 | 入口文件 | 面向协议 | 适用场景 |
|----------|----------|----------|----------|
| Gateway Worker（现有）| `byclaw-data/main.py` | by-framework 私有协议（Redis） | 生产部署，与 AI Factory 集成 |
| API Server（新增）| `examples/datacloud_api_server/` | OpenAI Chat Completions | 独立 HTTP 服务，第三方直接对接 |
| MCP Server（新增）| `examples/datacloud_mcp_server/` | MCP JSON-RPC 2.0 | Claude Desktop、Cursor 等 MCP 客户端 |

三种形态共同依赖：
- `datacloud-analysis` + `datacloud-data[service]`（SDK，零 gateway 依赖）
- `ONTOLOGY_LOAD_MODE=ontology_query`（工具命名模式）
- `ONTOLOGY_MCP_ENDPOINT`（agent 内部调用数据查询的地址）
- `LocalFileWriter`（文件写入，不走 gateway）

```python
async def stream_to_sse(graph, input_payload, run_config) -> AsyncIterator[str]:
    async for event in graph.astream_events(input_payload, config=run_config, version="v2"):
        kind = event["event"]
        if kind == "on_chat_model_stream":
            token = event["data"]["chunk"].content
            if token:
                yield make_openai_sse_chunk(token)   # 格式化为 OpenAI SSE chunk
    yield "data: [DONE]\n\n"
```

#### 4.4.4 MCP Server 示例

**定位**：将 `datacloud-analysis` 完整的自然语言问答能力包装为 MCP tool，供 Claude Desktop / Cursor 等 MCP 客户端调用。Claude 作为外部推理者，通过调用此 tool 获取业务数据，再组织最终回答。

> **为什么可以包装完整 agent？**
> 在 `ontology_query` 模式 + `DATACLOUD_DISABLE_ASK_USER_TOOL=1` 下，analysis agent 不会向用户追问，天然适合单轮调用。Claude 提问 → agent 查数据 → 返回结构化结果，Claude 再用数据写最终答案。这是 MCP 的正确用法：外部 LLM 通过 tool 获取数据/能力，不是两个 agent 相互嵌套推理。

### 4.5 文件写入插件化

#### 4.5.1 IFileWriter 接口（新增至 datacloud-data）

```python
# datacloud_data_sdk/storage/file_writer.py（新增）
from abc import ABC, abstractmethod
from collections.abc import AsyncIterator

class IFileWriter(ABC):
    @abstractmethod
    async def write(self, filename: str, content: bytes | str) -> str:
        """写入文件，返回可访问的文件路径或 URL。"""
        ...

    @abstractmethod
    async def write_stream(self, filename: str, chunks: AsyncIterator[bytes]) -> str:
        """流式写入文件。"""
        ...
```

#### 4.5.2 LocalFileWriter（默认实现）

```python
# datacloud_data_sdk/storage/local_file_writer.py（新增）
import aiofiles
from pathlib import Path
from datacloud_data_sdk.storage.file_writer import IFileWriter

class LocalFileWriter(IFileWriter):
    """本地文件系统写入，base_dir 从调用方传入（宿主层配置）。"""

    def __init__(self, base_dir: str | Path):
        self._base_dir = Path(base_dir)
        self._base_dir.mkdir(parents=True, exist_ok=True)

    async def write(self, filename: str, content: bytes | str) -> str:
        target = self._base_dir / filename
        mode, enc = ("wb", None) if isinstance(content, bytes) else ("w", "utf-8")
        async with aiofiles.open(target, mode=mode, encoding=enc) as f:
            await f.write(content)
        return str(target)

    async def write_stream(self, filename: str, chunks: AsyncIterator[bytes]) -> str:
        target = self._base_dir / filename
        async with aiofiles.open(target, "wb") as f:
            async for chunk in chunks:
                await f.write(chunk)
        return str(target)
```

#### 4.5.3 GatewayFileWriter（byclaw-data 层实现）

```python
# byclaw_data/plugins/file_writer_plugins/gateway_file_writer.py（新增）
from datacloud_data_sdk.storage.file_writer import IFileWriter
from by_framework import AgentContext

class GatewayFileWriter(IFileWriter):
    """通过 gateway AgentContext 上传文件，返回前端可访问的 URL。"""

    def __init__(self, ctx: AgentContext) -> None:
        self._ctx = ctx

    async def write(self, filename: str, content: bytes | str) -> str:
        if isinstance(content, str):
            content = content.encode("utf-8")
        return await self._ctx.upload_file(filename=filename, data=content)

    async def write_stream(self, filename: str, chunks) -> str:
        buf = b"".join([chunk async for chunk in chunks])
        return await self.write(filename, buf)
```

#### 4.5.4 注入方式

各宿主层在初始化时将对应的 `IFileWriter` 实例注入：

| 宿主层 | 注入的实现 | 注入时机 |
|--------|-----------|----------|
| byclaw-data gateway worker | `GatewayFileWriter(ctx)` | `InitDataCloudDigitalEmployeePlugin` 执行时 |
| API Server | `LocalFileWriter(os.environ["OUTPUT_DIR"])` | FastAPI app 启动时 |
| MCP Server | `LocalFileWriter(os.environ["OUTPUT_DIR"])` | Server 启动时 |
| 单元测试 | `InMemoryFileWriter()`（测试用） | 测试 fixture |

| `by-framework` 变为可选依赖 | `byclaw-data` 显式声明 `[gateway]` extra，现有部署不受影响 |

### 6.2 风险点

| 风险 | 缓解措施 |
|------|----------|
| EventBus 订阅者注册时机不对（请求级 bus vs 全局 bus） | 明确 `EventBus` 实例为请求级（每次 `InvocationContext` 创建新实例），`GatewayProgressHandler.register()` 在请求开始时调用，请求结束后 bus 随上下文销毁 |
| `astream_events` 事件结构在 LangGraph 版本升级时变化 | 在 worker 的 stream 消费代码中做防御性取值（`event.get("name", "")` 而非直接索引），并在升级 LangGraph 时回归 stream 事件覆盖测试 |
| MCP Server 无 agent 层，复杂分析需求无法满足 | 明确 MCP Server example 定位为"数据查询工具"，复杂分析场景引导用户走 API Server（内含完整 agent）或 gateway worker |
| `IFileWriter.write_stream()` 在 `GatewayFileWriter` 中需缓冲全量数据 | 已知限制，文档中说明；大文件场景建议走 `LocalFileWriter` 后再由 gateway 拉取文件 |

## 七、验收标准

- [ ] `datacloud-analysis` 在不安装 `by-framework` 的环境下可正常 `pip install` 并初始化 agent。
