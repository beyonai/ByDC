# G20 回滚预案

## 1. 触发条件
1. 发布后出现 P0/P1 线上故障。
2. 核心链路（闲聊短路、规划执行、中断恢复、子 agent 委托）任一不可用。
3. 数据一致性或幂等保障失效，且 15 分钟内无法热修复。

## 2. 回滚目标
将服务恢复到上一个稳定版本（发布前 commit/tag），并保证：
1. 会话服务可用。
2. 中断恢复链路可用。
3. 回归必测项可在回滚后通过。

## 3. 执行步骤
1. 冻结发布窗口，通知相关方（研发、测试、运维、产品）。
2. 记录当前版本信息：
   - 当前 commit/tag
   - 回滚目标 commit/tag
3. 执行回滚：
   - 代码回滚到目标版本并部署。
   - 清理与新版本耦合的临时配置（如插件开关）。
4. 执行最小验证：
   - `C1`、`C3`、`P1` 回归通过。
5. 发布回滚公告并进入故障复盘。

## 4. 验证命令（建议）
```bash
uv run pytest packages/datacloud-analysis/tests/dca/unit/test_worker_resume_regressions.py -k "ask_chitchat_short_circuits_without_graph_execution or resume" -q
uv run pytest packages/datacloud-analysis/tests/dca/unit/test_graph_builder_pipeline.py -q
```

## 5. 复盘输出
1. 故障根因。
2. 回滚耗时。
3. 数据/会话影响范围。
4. 永久修复计划与截止时间。
