# 亦庄-百应方案-虚拟动作改造详细设计

## 1. 文档目标

本文基于 [亦庄-百应方案](./亦庄-百应方案.md) 与当前 `datacloud-data` 实现，给出“虚拟动作”改造的详细设计方案。目标不是直接改代码，而是先明确：

- 当前项目里的虚拟动作到底是什么
- 它和亦庄方案的目标差距在哪里
- 应该如何重构元数据、生成逻辑、执行链路与技能暴露方式
- 后续实施时需要改哪些模块

---

## 2. 当前实现盘点

### 2.1 当前虚拟动作的真实含义

当前项目中的“虚拟动作”，本质上是运行时为 `DB` / `KNOWLEDGE_BASE` 对象自动注入的通用查询动作，命名为 `query_{object_code}`。

其核心链路如下：

| 环节 | 当前实现 | 说明 |
| --- | --- | --- |
| 注入时机 | `packages/datacloud-data/src/datacloud_data_service/api/routes.py` 启动时调用 `inject_virtual_actions()` | 服务启动后给所有 DB/KB 对象追加虚拟动作 |
| 注入逻辑 | `packages/datacloud-data/src/datacloud_data_service/tools/virtual_action_injector.py` | 只要对象是 `DB/KNOWLEDGE_BASE` 就追加一个 `query_xxx` |
| 动作生成 | `packages/datacloud-data/src/datacloud_data_service/tools/dynamic_query_tool_generator.py` | DB 对象生成 `filters + aggregates + group_by`，KB 对象生成 `query + filters` |
| 动作标识 | `packages/datacloud-data/src/datacloud_data_sdk/ontology/models.py` 中 `OntologyAction.is_virtual` | 只有一个布尔值，表达能力很弱 |
| 执行入口 | `packages/datacloud-data/src/datacloud_data_sdk/action.py` | `Action.execute()` 中 `is_virtual=True` 直接走 `_execute_virtual()` |
| 执行实现 | `packages/datacloud-data/src/datacloud_data_sdk/executor/dynamic_query_executor.py` | DB 走动态 SQL，KB 当前还是占位实现 |
| MCP 暴露 | `packages/datacloud-data/src/datacloud_data_service/tools/registry.py` | `per_object` 模式会把虚拟动作直接暴露给上层 |
| MCP 调用路由 | `packages/datacloud-data/src/datacloud_data_service/api/mcp_handler.py`、`mcp_sdk_handler.py` | 当前通过扫描 `cls.actions` 查找 `action_code -> object_code`，天然只支持对象级动作 |
| 规划器视图 | `packages/datacloud-data/src/datacloud_data_sdk/plan/object_view_builder.py` | 构建 ObjectView 时又把 `is_virtual` 动作过滤掉 |

### 2.2 当前虚拟动作的输入输出特征

当前 DB 类型虚拟动作的输入模式是：

```json
{
  "filters": {
    "field_code": {
      "op": "eq|in|gt|gte|lt|lte|like|is_null|is_not_null",
      "value": "..."
    }
  },
  "aggregates": [
    {
      "field": "字段编码",
      "func": "count|sum|avg|min|max",
      "as": "别名"
    }
  ],
  "group_by": ["字段编码"],
  "limit": 100,
  "offset": 0
}
```

当前 KB 类型虚拟动作的输入模式是：

```json
{
  "query": "检索词",
  "filters": {}
}
```

### 2.3 当前实现的关键特征

- 虚拟动作是“对象级”的，不是“视图级”的。
- 虚拟动作偏向“通用查询器”，不是“原子业务能力”。
- 规则主要根据 `field_type` 与是否绑定术语推断，没有体现亦庄方案中的“维度/度量/账期/指标”等语义。
- `is_virtual` 只区分“是不是虚拟动作”，不区分“明细查询 / 统计分析 / 下钻 / 检索”等动作族。
- MCP 暴露与规划器注入不一致：MCP 会暴露，ObjectViewBuilder 却跳过。
- MCP 调用路由只认对象动作：当前 `tools/call` 通过遍历 `loader.get_ontology_classes()` 下的 `cls.actions` 找归属对象，因此没有“视图级 action”执行入口。

## 3. 与亦庄方案的核心差距

### 3.1 与“生产模型不直接碰数据库”的差距

亦庄方案强调：

- 本体生产模型不直接操作数据库
- 应由很多原子方法组成，再通过技能进行组织

而当前 `query_{object_code}` 是一个比较大的通用查询入口，允许模型直接拼装过滤、聚合、分组，本质上仍然是“半结构化 SQL 能力透出”。

### 3.2 与“对象属性规则”的差距

亦庄方案对字段有清晰的分析语义约束：

- 维度 / 度量两大类
- 维度再细分为 ID、名称、时间、账期、数值
- 度量再细分为 ID、普通数值、指标数值
- 不同类型字段在分组、条件、聚合函数上规则不同
- 账期字段要求强制过滤
- 部分字段不允许 `count`

当前实现没有这些一等元数据，只是简单使用：

- `field_type`
- `term_set`
- `property_kind`

因此当前虚拟动作无法表达亦庄方案中的分析规则。

### 3.3 与“视图分析能力”的差距

亦庄方案中，统计能力明显是面向“视图”而不是单个对象：

- 员工对象
- 组织对象
- 城市对象
- 员工统计视图

当前虚拟动作仅对对象生成，不对视图生成，无法直接承载“跨对象统计视图”的分析动作。

### 3.4 与“技能组织原子动作”的差距

当前虚拟动作的问题不是“没有动态查询”，而是“粒度不合适”：

- 一个 `query_xxx` 同时承载明细查询、聚合查询、分组查询
- 没有区分“查询明细”和“统计分析”
- 没有区分“模型可见动作”和“仅供技能编排使用的内部动作”

这会导致：

- 模型理解成本高
- 门户编排难
- 技能包示例难写
- 权限与审计边界不清晰

### 3.5 与“统一规则驱动”的差距

当前规则散落在多个模块：

- 动态工具生成器决定 schema
- 动态执行器决定 SQL 能力
- 术语解析器单独决定过滤值翻译
- ObjectViewBuilder 又将虚拟动作隐藏

缺少一个统一的“虚拟动作规则模型”。

---

## 4. 改造目标

本次虚拟动作改造应达到以下目标：

1. 将当前“通用查询虚拟动作”改造成“语义清晰的原子能力动作”。
2. 把亦庄方案中的字段分析规则沉淀为一等元数据，而不是散落在生成器里推断。
3. 支持“对象级虚拟动作”和“视图级虚拟动作”并存。
4. 明确“技能可见”和“模型直连可见”的暴露策略。
5. 统一 MCP、技能包、规划器、执行器对虚拟动作的认知。
6. 为后续百应门户配置对象/视图/动作打下可治理基础。

---

## 5. 目标设计总览

### 5.1 设计原则

- 规则先于 SQL：先定义语义规则，再决定如何下推到 SQL 或其他执行器。
- 动作小而清晰：每个虚拟动作只承担一种清晰意图。
- 视图优先承载统计：对象偏明细，视图偏分析。
- 技能编排优先：生产态默认不把“自由查询器”直接暴露给大模型。
- 元数据单点治理：虚拟动作的生成、校验、暴露、执行共享同一份规则模型。

### 5.2 目标能力分层

改造后建议将虚拟动作拆成以下动作族：

| 动作族 | 适用范围 | 用途 | 是否首期必须 |
| --- | --- | --- | --- |
| `lookup` | 对象 / 视图 | 明细查询、列表浏览、条件过滤 | 是 |
| `analyze` | 视图优先，对象可选 | 分组统计、聚合分析 | 是 |
| `drilldown` | 视图 / 对象 | 从统计结果回钻明细 | 否，二期 |
| `search` | 知识库对象 | 检索文本与结构化过滤 | 是 |
| `resolve` | 术语/关联辅助 | 名称转标准编码、关联补全 | 否，可内置为辅助能力 |

首期不建议继续沿用一个 `query_{object_code}` 覆盖所有用途，而是至少拆成：

- `lookup_{scope_code}`
- `analyze_{scope_code}`
- `search_{scope_code}`

其中：

- `scope_code` 可以是对象编码，也可以是视图编码
- `lookup` 偏明细
- `analyze` 偏统计
- `search` 专用于 KB

### 5.3 对现有详细设计的合理性判断

结合当前代码，这份详细设计的大方向是合理的，尤其是以下几点应当保留：

- 将单一 `query_*` 拆分为 `lookup / analyze / search`，这能明显降低模型误用通用查询器的概率。
- 将“维度/度量 + 子类”沉淀为字段一等元数据，而不是继续在生成器里靠 `field_type` 弱推断。
- 把 `skill_only` 作为生产默认暴露策略，符合“生产模型不直接碰数据库”的原则。
- 将 JSON Schema、扩展元数据、服务端 Validator 分层，这和当前项目“schema 生成 + 执行器 + 规划器”三段式结构是兼容的。

但从当前实现出发，还需要补三条收敛原则，否则方案会显得过于理想化：

1. 不建议首期“一步到位移除 `is_virtual`”。
   当前 `Action.execute()`、`ToolRegistry`、`ObjectViewBuilder`、`mcp_handler` 都直接或间接依赖现有对象动作模型。首期应采用“新增字段 + 保留兼容布尔位”的加法改造，等新路由稳定后再逐步退掉 `is_virtual`。
2. 视图级虚拟动作不要绕开现有 `Action` 执行模型。
   当前对象动作已经形成 `obj.invoke_action() -> Action.execute()` 的稳定路径。视图级虚拟动作应补充 `view.invoke_action()`，让对象和视图都复用同一套 `OntologyAction / Action.execute()` 机制，而不是额外再造一条独立执行体系。
3. 高级分析函数不要作为首期默认能力。
   `median`、`topn` 等能力在当前执行器和多数据源方言层里都没有现成支撑，若首期直接写成默认暴露能力，文档会超前于实现。首期默认应收敛到可移植函数子集，高级函数由 profile 显式开启。

## 6. 元数据模型改造

### 6.1 字段元数据扩展

当前 `OntologyField` 只有 `field_type`、`term_set`、`property_kind` 等基础信息，不足以表达亦庄方案中的分析规则。建议扩展以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `analytic_role` | `dimension \| measure` | 分析语义角色 |
| `analytic_kind` | `id \| name \| time \| period \| number \| indicator` | 对应亦庄方案中的细分类 |
| `filter_ops` | `list[str]` | 允许的过滤操作符 |
| `group_ops` | `list[str]` | 允许的分组方式，如 `self` / `date` / `month` / `year` / `quarter` / `range` |
| `aggregate_ops` | `list[str]` | 允许的聚合函数 |
| `sort_enabled` | `bool` | 是否允许排序 |
| `term_resolve_ops` | `list[str]` | 哪些操作符下触发术语标准化，默认 `eq/in` |
| `required_filter_group` | `str \| None` | 强制过滤组，例如 `period_required` |
| `count_enabled` | `bool` | 是否允许 `count` / `count_distinct` |
| `view_alias` | `str \| None` | 视图场景下字段所属对象别名 |

> 说明：上表是“最容易落地的简化模型”。结合亦庄方案中的对象字段示例，后续推荐升级为“多角色能力模型”，因为同一个字段可能同时具备维度与度量能力，例如 `员工ID`、`年龄`。

### 6.1.1 结合当前 OWL 的落地调整

结合 `examples/ontology/objects/` 下的对象 OWL 实际样例，字段侧已经存在以下槽位，且 **`ext_property` 已承载了分析角色信息**：

| OWL 字段 | 说明 | 示例值 |
| --- | --- | --- |
| `property_category` | 字段大类 | `DIMENSION` / `MEASURE` |
| `property_group` | 存储分组 | `STORAGE` |
| `measurement_unit` | 度量单位 | `万元` |
| `ext_property` | 扩展属性 JSON | `{“property_role_rule”:{“property_role”:”DIMENSION_ATTR”,”rule_type”:”id”}}` |

**实际的 `ext_property` 结构（来自 OWL 样例文件）：**

```json
// 维度-ID 字段（如企业ID）
{“property_role_rule”: {“property_role”: “DIMENSION_ATTR”, “rule_type”: “id”}}

// 维度-名称 字段（如企业名称、网格名称）
{“property_role_rule”: {“property_role”: “DIMENSION_ATTR”, “rule_type”: “name”}}

// 维度-时间 字段（如统计日期）
{“property_role_rule”: {“property_role”: “DIMENSION_ATTR”, “rule_type”: “time”}}

// 维度-账期 字段（时间型账期，需强制过滤）
{“property_role_rule”: {“property_role”: “DIMENSION_ATTR”, “rule_type”: “period”}}

// 度量-普通数值 字段（如年龄、身高）
{“property_role_rule”: {“property_role”: “MEASURE”, “rule_type”: “numerical”}}

// 度量-指标数值 字段（如企业总营收）
{“property_role_rule”: {“property_role”: “MEASURE”, “rule_type”: “indicator”}}
```

> 注：`property_role` 枚举值为 `DIMENSION_ATTR`（维度）和 `MEASURE`（度量），对应设计文档中的 `dimension` 和 `measure`。`rule_type` 中 `numerical` 对应设计文档中的 `number`。

同时当前项目现状还有两个重要约束：

1. `datacloud-data` 的 OWL 解析器当前并不会把 `ext_property.property_role_rule` 读进 `OntologyField`，只解析基础字段和术语绑定信息。
2. `datacloud-knowledge` 的知识包预检目前明确禁止 `ontology/` 目录入库，因此分析角色信息不依赖知识库 `term.ext_attrs`。

因此设计以”对象 OWL `ext_property` 为唯一事实来源，OWL 解析器负责将 `property_role_rule` 映射到运行时模型”。

推荐 OWL → 运行时模型的映射规则：

#### OWL 解析 → 运行时字段映射

```python
def parse_analytic_role(ext_property_json: str) -> tuple[str | None, str | None]:
    “””
    从 OWL ext_property 中提取运行时分析角色
    返回 (analytic_role, analytic_kind)
    “””
    data = json.loads(ext_property_json) if ext_property_json else {}
    rule = data.get(“property_role_rule”, {})
    property_role = rule.get(“property_role”, “”)
    rule_type = rule.get(“rule_type”, “”)

    # property_role 映射
    role_map = {
        “DIMENSION_ATTR”: “dimension”,
        “MEASURE”: “measure”,
    }
    # rule_type 映射（numerical → number，其余保留原值）
    kind_map = {
        “id”: “id”,
        “name”: “name”,
        “time”: “time”,
        “period”: “period”,
        “numerical”: “number”,    # OWL 中叫 numerical，运行时叫 number
        “indicator”: “indicator”,
    }
    return role_map.get(property_role), kind_map.get(rule_type)
```

执行顺序：

1. OWL parser 解析 `ext_property` JSON 字符串为 `dict`
2. 提取 `property_role_rule.property_role` → `analytic_role`（`DIMENSION_ATTR` → `dimension`，`MEASURE` → `measure`）
3. 提取 `property_role_rule.rule_type` → `analytic_kind`（`numerical` → `number`，其余直接使用）
4. 规则引擎根据 `analytic_role + analytic_kind` 生成 `filter_ops / group_ops / aggregate_ops / required_filter_group`
5. 虚拟动作生成器优先使用这些运行时生成结果
6. 若 `ext_property` 缺失或解析失败，再按 `field_type` 和旧规则做兼容推断

#### 方案 B：运行时模型再映射为强类型字段

建议在 `datacloud-data` 运行时模型里新增：

- `OntologyField.analytic_role: str | None`（`dimension` / `measure`）
- `OntologyField.analytic_kind: str | None`（`id` / `name` / `time` / `period` / `number` / `indicator`）
- `OntologyField.filter_ops: list[str]`（由规则引擎派生）
- `OntologyField.group_ops: list[str]`（由规则引擎派生）
- `OntologyField.aggregate_ops: list[str]`（由规则引擎派生）
- `OntologyField.required_filter_group: str | None`（由规则引擎派生）

这比”在 OWL 中手填 filter_ops 等操作符列表”更安全，能保证规则的集中治理。

### 6.1.2 双角色字段的处理方式

亦庄方案中存在一类重要字段：

- `企业ID`：既可作为维度（用于分组/过滤），也可作为度量（用于计数）
- `年龄`：既可作为维度-数值（用于过滤），也可作为度量-普通数值（用于 SUM/AVG）

当前 OWL 的 `ext_property` **只存储一条 `property_role_rule`**（主要角色）。双角色能力不需要在 OWL 中显式声明，而是由规则引擎自动推断：

| 主要角色 (OWL) | 规则引擎自动推断的附加能力 |
| --- | --- |
| `DIMENSION_ATTR` + `id` | 附加 `MEASURE.id` 能力（允许 COUNT/COUNT_DISTINCT） |
| `DIMENSION_ATTR` + `numerical` | 附加 `MEASURE.numerical` 能力（允许 SUM/AVG/MAX/MIN） |
| `MEASURE` + `indicator` | 不附加维度能力（指标字段通常不做分组键） |

推荐运行时字段模型对双角色的表示：

```python
@dataclass
class OntologyField:
    field_code: str
    field_name: str
    # 主要分析角色（来自 OWL ext_property.property_role_rule）
    analytic_role: str | None      # “dimension” | “measure”
    analytic_kind: str | None      # “id” | “name” | “time” | “period” | “number” | “indicator”
    # 附加能力（由规则引擎从主角色推断）
    secondary_role: str | None     # 若非 None，表示该字段还具备此角色的统计能力
    # 运行时派生的操作符集合
    filter_ops: list[str]
    group_ops: list[str]
    aggregate_ops: list[str]
    required_filter_group: str | None
```

虚拟动作生成器按场景读取：

- `lookup`：读取 `analytic_role=dimension` 的 `filter_ops/group_ops`
- `analyze.dimensions`：读取 `analytic_role=dimension` 的 `group_ops`
- `analyze.metrics`：读取 `analytic_role=measure`（含 secondary_role 推断）的 `aggregate_ops`

推荐默认映射如下（`analytic_role` / `analytic_kind` 使用运行时规范名称）：

| analytic_role | analytic_kind | 默认 filter_ops | 默认 group_ops | 默认 aggregate_ops |
| --- | --- | --- | --- | --- |
| dimension | id | `eq,in,is_null,is_not_null` | `self` | — |
| dimension | name | `eq,in,like,is_null,is_not_null` | `self` | — |
| dimension | time | `eq,in,gt,gte,lt,lte,between` | `day,month,quarter,year` | — |
| dimension | period | `eq,in,gt,gte,lt,lte,between` | `month,quarter,year` | — |
| dimension | number | `eq,in,gt,gte,lt,lte` | — | — |
| measure | id | `eq,in` | `self` | `count,count_distinct` |
| measure | number | `eq,in,gt,gte,lt,lte` | `range` | `sum,avg,min,max` |
| measure | indicator | `eq,in,gt,gte,lt,lte` | `range` | `sum,count,count_distinct,avg,min,max,topn,median` |

> OWL 中 `rule_type=numerical` 对应运行时 `analytic_kind=number`；OWL 中 `DIMENSION_ATTR` 对应运行时 `analytic_role=dimension`；OWL 中 `MEASURE` 对应运行时 `analytic_role=measure`。

### 6.2 对象 / 视图元数据扩展

建议在对象或视图级别新增虚拟动作配置：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `virtual_action_profiles` | `list[VirtualActionProfile]` | 声明要生成哪些虚拟动作 |
| `default_exposure` | `direct \| skill_only \| hidden` | 默认暴露策略 |
| `default_limit` | `int` | 默认分页 |
| `max_limit` | `int` | 最大返回行数 |
| `required_filters` | `list[str]` | 强制过滤规则，如 `period` |
| `preferred_sort` | `list[SortSpec]` | 默认排序 |
| `analysis_subject` | `str \| None` | 统计主体，如“员工统计视图” |

### 6.3 动作元数据扩展

当前 `OntologyAction` 中只有 `action_type` 与 `is_virtual`，建议扩展：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `action_family` | `lookup \| analyze \| drilldown \| search \| operation` | 动作族 |
| `virtual_backend` | `db_lookup \| db_analyze \| kb_search \| api_proxy` | 虚拟动作执行后端 |
| `exposure_policy` | `direct \| skill_only \| hidden` | 是否直接出现在 MCP/技能中 |
| `scope_type` | `object \| view` | 动作挂载在对象还是视图 |
| `scope_code` | `str` | 动作所属对象编码或视图编码 |
| `rule_profile` | `str` | 引用规则模板 |
| `legacy_aliases` | `list[str]` | 兼容旧动作名，例如 `query_obj_emp` |
| `planner_visible` | `bool` | 是否进入规划器上下文 |

### 6.3.1 兼容期的动作模型建议

首期不建议把 `OntologyAction` 彻底重做为全新模型，而建议采用“兼容增强”：

- 保留 `is_virtual`，但把它降级为兼容位。
- 新增 `action_family / virtual_backend / exposure_policy / scope_type / scope_code / planner_visible`。
- 对对象级虚拟动作，继续实例化为 `OntologyAction`，保持 `obj.invoke_action() -> Action.execute()` 不变。
- 对视图级虚拟动作，同样使用 `OntologyAction` 作为动作定义，但挂载到 `View.actions`，新增 `view.invoke_action()` 复用 `Action.execute()`。
- MCP 路由层只负责先解析 `tool_name` 属于对象还是视图，再分发到 `obj.invoke_action()` 或 `view.invoke_action()`。

### 6.4 推荐新增数据结构

**VirtualActionProfile 不是必须的手工配置项。** 在亦庄场景下，规则足够简单：
- DB 对象 → 自动生成 `lookup_{object_code}`
- DB 对象（有度量字段）→ 额外生成 `analyze_{object_code}`
- DB 视图 → 自动生成 `lookup_{view_code}` + `analyze_{view_code}`
- KB 对象 → 自动生成 `search_{object_code}`

生成规则完全由字段元数据（`analytic_role/kind`）推导，不需要手工写 profile 配置。

`VirtualActionProfile` 作为**运行时生成结果的承载结构**而非输入配置，保留为内存对象：

```python
@dataclass
class VirtualActionProfile:
    action_family: str          # lookup / analyze / search
    scope_type: str             # object / view
    scope_code: str             # 对象编码或视图编码
    exposure_policy: str        # direct / skill_only / hidden
    default_limit: int = 100
    max_limit: int = 1000
    required_filters: list[str] = field(default_factory=list)   # 由字段元数据推导
    dimension_fields: list[str] = field(default_factory=list)   # analytic_role=dimension
    measure_fields: list[str] = field(default_factory=list)     # analytic_role=measure
```

推荐职责分工如下：

- `VirtualActionInjector`：遍历所有对象/视图，按规则推导并实例化 `VirtualActionProfile`
- `VirtualActionGenerator`：消费 profile，生成 `OntologyAction`（含 inputSchema）
- `OntologyAction`：继续承载对象级与视图级运行时动作
- `tool -> scope` 索引：仅用于 MCP / 门户 / 技能路由定位，不另起执行模型

## 7. 虚拟动作生成设计

### 7.1 生成策略调整

当前是”给每个 DB/KB 对象盲注入一个 `query_xxx`”。改造后改为”按字段元数据规则自动推导并生成多个虚拟动作”：

1. 加载对象/视图元数据。
2. 按以下规则自动推导应生成哪些动作：
   - DB 对象：必生成 `lookup_{object_code}`；若存在度量字段则同时生成 `analyze_{object_code}`
   - DB 视图：必生成 `lookup_{view_code}` + `analyze_{view_code}`
   - KB 对象：生成 `search_{object_code}`
3. 根据 `action_family` 生成不同 inputSchema。
4. 对象级动作写入 `OntologyClass.actions`；视图级动作写入 `View.actions`。
5. 注入过程必须幂等，避免重复追加。

> 说明：视图级动作不需要在 OWL 中声明任何特殊配置，生成器直接读取视图的 `objects`、`relations`、字段元数据即可推导出完整的 profile。

### 7.1.1 为什么要补 `view.invoke_action()`

这里需要按当前项目的执行习惯收口。现有稳定路径是：

- `obj.invoke_action(action_code, params)`
- `Action(action, loader).execute(params)`

视图层当前只有：

- `invoke_object_action(object_code, action_code, params)`
- 没有 `actions`
- 没有 `invoke_action()`

因此视图级虚拟动作的关键不是新增独立 executor，而是补齐视图动作入口：

- `View` 增加 `actions: list[OntologyAction]`
- `View` 增加 `_find_action(action_code)`
- `View` 增加 `invoke_action(action_code, params)`，内部同样走 `Action(action, loader).execute(params)`

这样对象级和视图级虚拟动作都共享：

- 同一个 `OntologyAction` 模型
- 同一个 `Action.get_schema()`
- 同一个 `Action.execute()`
- 同一套 Validator / TermResolver / ResultFormatter

### 7.2 推荐生成规则

| 场景 | 生成动作 |
| --- | --- |
| DB 对象 | `lookup_{object_code}` |
| DB 统计视图 | `lookup_{view_code}`、`analyze_{view_code}` |
| 纯统计对象 | `analyze_{object_code}` |
| KB 对象 | `search_{object_code}` |

### 7.3 不再推荐默认生成的动作

以下动作不建议继续作为默认形态保留：

- 单一 `query_{object_code}` 通用动作
- 不区分明细与统计的“混合型 schema”
- 无视图级别的虚拟分析动作

### 7.4 命名规范

| 动作族 | 命名格式 | 示例 |
| --- | --- | --- |
| lookup | `lookup_{scope_code}` | `lookup_obj_employee` |
| analyze | `analyze_{scope_code}` | `analyze_view_employee_stat` |
| drilldown | `drilldown_{scope_code}` | `drilldown_view_employee_stat` |
| search | `search_{scope_code}` | `search_kb_policy_doc` |

旧命名 `query_{object_code}` 可在过渡期作为 `legacy_alias` 保留。

## 8. 虚拟动作输入协议设计

### 8.1 `lookup` 动作输入协议

`lookup` 只负责明细查询，不承担聚合统计。

```json
{
  "select": ["员工ID", "员工名称", "组织名称"],
  "filters": [
    {"field": "组织名称", "op": "eq", "value": "市场部"},
    {"field": "员工名称", "op": "like", "value": "%张%"}
  ],
  "order_by": [
    {"field": "员工ID", "direction": "asc"}
  ],
  "limit": 100,
  "offset": 0
}
```

设计要点：

- `lookup` 不允许传 `metrics`
- `lookup` 不允许分组聚合
- `select` 为空时按配置返回默认字段
- `filters` 使用数组，方便后续支持逻辑组合与校验扩展

### 8.2 `analyze` 动作输入协议

`analyze` 只负责统计分析。

```json
{
  "dimensions": [
    {"field": "账期", "group_op": "month"},
    {"field": "组织名称", "group_op": "self"},
    {
      "field": "年龄",
      "group_op": "range",
      "buckets": [
        {"from": 0, "to": 18, "label": "未成年"},
        {"from": 18, "to": 60, "label": "成年"},
        {"from": 60, "to": null, "label": "老年"}
      ]
    }
  ],
  "metrics": [
    {"field": "组织收入总利润总数", "agg": "sum", "as": "利润汇总"},
    {"field": "员工ID", "agg": "count_distinct", "as": "员工数"}
  ],
  "filters": [
    {"field": "账期", "op": "between", "value": ["2026-01", "2026-03"]},
    {"field": "组织名称", "op": "in", "value": ["华北区", "华东区"]}
  ],
  "having": [
    {"field": "利润汇总", "op": "gt", "value": 100000}
  ],
  "order_by": [
    {"field": "利润汇总", "direction": "desc"}
  ],
  "limit": 100
}
```

设计要点：

- `dimensions` 只能引用 `analytic_role=dimension` 字段，或少量允许 `self` 分组的 measure-id 字段
- `metrics` 只能引用 `analytic_role=measure` 字段；若无聚合需求，应使用 `lookup` 而非空 `analyze`
- 时间类维度必须通过 `group_op` 明确粒度（`day/month/quarter/year`）
- 数值度量若使用 `range` 分组，必须提供 `buckets` 数组；最后一个桶的 `to` 可为 `null` 表示无上限
- `having` 用于聚合后的结果过滤，字段必须是 `metrics.as` 别名之一；不支持原始字段直接引用
- 若 profile 配置 `required_filters=["period"]`，则 `filters` 中必须出现账期字段
- `count_all` 是一个特殊内建指标，不需要 `field`，表示行数统计：`{"agg": "count_all", "as": "记录数"}`

### 8.3 `search` 动作输入协议

```json
{
  "query": "请假制度",
  "filters": [
    {"field": "文档类型", "op": "eq", "value": "制度"}
  ],
  "limit": 20
}
```

### 8.4 不建议继续保留的协议形态

以下输入形式建议废弃：

- `filters` 为“字段名 -> 条件对象”的 map 结构
- `aggregates + group_by` 混装在一个通用 schema 里
- DB 与 KB 共用一个“query_xxx”命名习惯

原因是：

- 难以做规则校验
- 不利于技能示例编写
- 不利于门户做可视化配置

## 9. 校验与约束设计

### 9.1 通用校验

所有虚拟动作在执行前必须经过统一校验器：

1. 动作族校验
2. 字段存在性校验
3. 字段权限校验
4. 字段能力校验
5. 强制过滤校验
6. 术语翻译校验
7. 返回行数限制校验

### 9.2 字段能力校验

校验规则示例：

| 规则 | 说明 |
| --- | --- |
| 维度-ID 只能 `self` 分组 | 不允许时间粒度与范围分组 |
| 维度-名称 不允许 `count` | 避免错误统计 |
| 维度-账期 必须出现在过滤条件中 | 防止全表扫描 |
| 维度-数值 不允许 `group_by` | 只能过滤 |
| 度量-普通数值 不允许 `count_distinct` 之外的非法聚合 | 由元数据控制 |
| 度量-指标数值 允许 `sum/avg/max/min/topn/median` | 受 profile 约束 |
| `like` 不触发术语标准化 | 对齐亦庄方案 |
| `eq/in` 触发术语标准化 | 对齐亦庄方案 |
| `range` 分组必须携带 `buckets` 数组 | 缺少 buckets 时直接拒绝，不允许默认分桶 |
| `having` 字段必须是 `metrics.as` 别名 | 不允许引用原始字段名，防止 SQL 注入风险 |
| `analyze` 中 `metrics` 为空时拒绝 | 纯维度枚举请使用 `lookup`，不要用空 metrics 的 analyze |

### 9.3 强制过滤校验

例如账期场景：

- `analyze_view_employee_stat` 若 profile 声明 `required_filters=["period"]`
- 执行时必须在过滤条件中出现 `analytic_kind=period` 字段
- 否则直接拒绝执行，并返回明确错误信息

### 9.4 暴露策略校验

若动作配置 `exposure_policy=skill_only`：

- 不直接出现在公共 MCP `tools/list`
- 可以出现在技能包内部
- 可以被门户编排成更高层技能

## 10. 执行链路改造设计

### 10.1 目标执行链

```mermaid
flowchart LR
    A[Skill / MCP Tool Call] --> B[ScopeResolver]
    B --> C[ActionInvoker]
    C --> D[Action.execute]
    D --> E[VirtualActionValidator]
    E --> F[TermResolver]
    F --> G[PermissionRewriter]
    G --> H[LogicalPlanBuilder]
    H --> I[BackendExecutor]
    I --> J[ResultFormatter]
```

### 10.1.1 ScopeResolver 的职责

建议在现有路由前新增 `ScopeResolver`，先解决“这个 tool name 到底属于谁”的问题：

- 若命中对象动作索引，则走 `obj.invoke_action()`
- 若命中视图级动作索引，则走 `view.invoke_action()`
- 若命中旧 `query_*` 兼容别名，则先转换为新 spec，再执行

这样可以保持统一的 `Action.execute()` 主执行模型，同时避免把对象动作和视图动作继续混在同一条查找路径里。

### 10.2 执行器拆分建议

当前 `DynamicQueryExecutor` 同时承担：

- DB 明细查询
- DB 聚合查询
- derived 字段拼装
- linked 字段 LEFT JOIN
- KB 占位检索

职责过于混杂，建议拆分为：

| 模块 | 职责 |
| --- | --- |
| `VirtualActionRouter` | 根据 `action_family` 选择执行器 |
| `LookupExecutor` | 执行明细查询 |
| `AnalyzeExecutor` | 执行统计分析 |
| `KbSearchExecutor` | 执行知识库检索 |
| `VirtualActionValidator` | 统一规则校验 |
| `VirtualSqlPlanner` | 逻辑计划转 SQL |

### 10.3 SQL 规划建议

`AnalyzeExecutor` 不应直接拿 `metrics/dimensions` 拼字符串，而应分两段：

1. 语义计划层
   - 解析维度
   - 解析度量
   - 应用规则
   - 应用权限

2. 物理执行层
   - 将合法的语义计划翻译成 SQL
   - 由方言层处理不同 DB 差异

这样后续可扩展：

- MySQL / PostgreSQL / SQLite / ClickHouse
- API 代理分析
- 非 SQL 引擎

### 10.3.1 数据源函数差异的处理原则

亦庄方案里的时间粒度、范围分组、统计函数，落到不同数据源时函数并不一致。例如：

- MySQL 常用 `DATE()`、`YEAR()`、`QUARTER()`
- PostgreSQL / openGauss 常用 `DATE_TRUNC()`
- ClickHouse 常用 `toStartOfMonth()`、`toStartOfQuarter()`
- Hive 与 SQLite 的日期函数能力更弱，表达方式也不同
- `median`、`topn` 等高级聚合函数并非所有数据源都原生支持

因此改造后必须遵循两个原则：

1. MCP 与虚拟动作对外暴露“逻辑函数”，不暴露底层数据库函数名。
2. 执行层通过“数据源函数能力注册表”将逻辑函数翻译为物理函数，必要时做降级或拒绝执行。

推荐将逻辑函数分为四类：

| 逻辑函数族 | 示例 | MCP 对外名称 |
| --- | --- | --- |
| 时间分组 | 按日/按月/按季/按年 | `day` / `month` / `quarter` / `year` |
| 数值分桶 | 范围分组 | `range` |
| 基础聚合 | 计数、求和、平均、最大、最小 | `count` / `count_distinct` / `sum` / `avg` / `max` / `min` |
| 高级聚合 | 中位数、TopN、分位数 | `median` / `topn` / `percentile` |

执行层的职责不是让模型知道 `DATE_TRUNC` 或 `toStartOfQuarter`，而是：

- 输入 `group_op=month`
- 根据 `db_type` 翻译成对应方言表达式
- 若当前数据源不支持，则走 fallback 或明确报错

### 10.3.2 数据源函数能力注册表

建议新增一层“函数能力注册表”：

```python
@dataclass
class DatasourceFunctionCapability:
    db_type: str
    supported_group_ops: list[str]
    supported_filter_ops: list[str]
    supported_agg_ops: list[str]
    expression_templates: dict[str, str]
    fallback_strategies: dict[str, str]
```

示例：

| db_type | `month` | `quarter` | `median` | `topn` |
| --- | --- | --- | --- | --- |
| MYSQL | 原生支持 | 原生支持 | 不支持或需自定义 | 不支持 |
| POSTGRESQL | 原生支持 | 原生支持 | 可通过 percentile 近似 | 不支持 |
| OPENGAUSS | 原生支持 | 原生支持 | 视版本而定 | 不支持 |
| CLICKHOUSE | 原生支持 | 原生支持 | 原生或近似支持 | 可通过排序/limit 实现 |
| HIVE | 支持但表达不同 | 支持但表达不同 | 一般不推荐 | 不支持 |
| SQLITE | 支持有限 | 需拼表达式 | 不支持 | 不支持 |

这里的“支持”分为三种级别：

- `native`：数据源原生支持
- `rewrite`：可通过表达式改写支持
- `engine_fallback`：需要回退到内存引擎或聚合层

### 10.3.3 生成器与 Schema 的联动策略

虚拟动作生成时，不能只看字段元数据，还要看当前 scope 的数据源能力：

1. 单对象且单数据源
   - 直接按该 `db_type` 生成 `group_op` / `agg` 的枚举
2. 单视图但同源
   - 取该数据源能力集
3. 多对象且跨数据源
   - 默认只暴露“可移植函数子集”
   - 若计划统一走 `SQLITE_MEM` / 本地聚合，则暴露该执行引擎支持的逻辑函数

推荐默认可移植子集：

- 时间分组：`day` / `month` / `quarter` / `year`
- 聚合：`count` / `count_distinct` / `sum` / `avg` / `max` / `min`
- 过滤：`eq` / `in` / `gt` / `gte` / `lt` / `lte` / `between` / `like`

高级函数如 `median` / `topn` 不要默认暴露，必须满足以下条件之一：

- 当前数据源 profile 明确支持
- 当前执行策略声明支持本地 fallback
- 对应动作 profile 显式开启

### 10.3.4 视图级虚拟动作的 SQL 执行全链路

视图由多个对象通过 Relation 的 `join_keys` 关联，`lookup` 和 `analyze` 的 SQL 都需要翻译为跨对象 JOIN。

#### 视图的物理模型

视图不是数据库里真实存在的 VIEW，而是运行时按关联关系动态拼接的 SQL。每个视图包含：

- **View OWL**（`*_view.owl`）：声明视图基本信息，`object_codes` 为参与对象列表，`relations` 为关系列表
- **Mapping OWL**（`*_mapping.owl`）：声明视图字段列表，每个 `Mapping` 包含：
  - `property_code`：视图字段编码
  - `property_name`：视图字段显示名
  - `source_object_code`：该字段来自哪个对象
  - `source_object_column_code`：对应源对象的哪个字段编码
  - `ext_property`：`{"property_role_rule":{"property_role":"...","rule_type":"..."}}` — **视图字段的分析角色直接从 mapping 的 `ext_property` 读取，无需再去源对象查**

这是与对象字段解析的关键差异：**视图字段的分析语义来自 mapping.ext_property，物理列来自 source_object_code + source_object_column_code 的联合查找**。

**Mapping OWL 样例：**

```xml
<!-- 企业ID字段映射 -->
<owl:NamedIndividual rdf:about="#enterprise_id_mapping">
    <rdf:type rdf:resource="#Mapping"/>
    <property_code>enterprise_id</property_code>
    <source_object_code>example_enterprise_analysis</source_object_code>
    <source_object_column_code>enterprise_id</source_object_column_code>
    <ext_property>{"property_role_rule":{"property_role":"DIMENSION_ATTR","rule_type":"id"}}</ext_property>
</owl:NamedIndividual>

<!-- 企业总营收字段映射 -->
<owl:NamedIndividual rdf:about="#total_revenue_mapping">
    <rdf:type rdf:resource="#Mapping"/>
    <property_code>total_revenue</property_code>
    <source_object_code>example_enterprise_analysis</source_object_code>
    <source_object_column_code>total_revenue</source_object_column_code>
    <ext_property>{"property_role_rule":{"property_role":"MEASURE","rule_type":"indicator"}}</ext_property>
</owl:NamedIndividual>
```

#### 视图 lookup SQL 生成

```sql
SELECT
    t0.emp_id        AS emp_id,
    t0.emp_name      AS emp_name,
    t1.org_name      AS org_name,
    t2.city_name     AS city_name
FROM employee t0
LEFT JOIN organization t1 ON t0.org_id = t1.org_id
LEFT JOIN city         t2 ON t0.city_id = t2.city_id
WHERE
    t0.org_name = '市场部'
    AND t0.period = '2026-03'   -- 强制账期
    AND t1.org_type IN (1, 2)   -- 权限注入
ORDER BY t0.emp_id ASC
LIMIT 100 OFFSET 0
```

生成规则：

1. **主表**：`view.objects[0]` 为驱动表，别名 `t0`；其余对象依次编号 `t1, t2, ...`
2. **JOIN 顺序**：按 relation 的 `source_object → target_object` 顺序，均使用 LEFT JOIN
3. **JOIN 条件**：从 `relation.join_keys` 取 `sourceField/targetField`，替换为对应表别名前缀
4. **字段前缀**：每个字段通过"遍历 `view.objects`，找到包含该字段的对象"确定所属别名
5. **WHERE**：`filters` 中字段解析后加表别名前缀；强制账期过滤在 Validator 校验后由执行器注入
6. **SELECT**：`select` 为空时取所有字段；非空时过滤后按别名前缀输出

#### 视图 analyze SQL 生成

```sql
SELECT
    DATE_FORMAT(t0.period, '%Y-%m')  AS period_month,  -- 维度账期按月分组
    t1.org_name                       AS org_name,       -- 维度名称自身分组
    SUM(t1.profit)                    AS 利润汇总,
    COUNT(DISTINCT t0.emp_id)         AS 员工数
FROM employee t0
LEFT JOIN organization t1 ON t0.org_id = t1.org_id
WHERE
    t0.period BETWEEN '2026-01' AND '2026-03'
    AND t1.org_name IN ('华北区', '华东区')
GROUP BY
    DATE_FORMAT(t0.period, '%Y-%m'),
    t1.org_name
HAVING
    SUM(t1.profit) > 100000           -- having 条件：别名替换为原始聚合表达式
ORDER BY SUM(t1.profit) DESC
LIMIT 100
```

生成规则：

1. **FROM + JOIN** 与 lookup 完全一致
2. **GROUP BY**：每个 dimension 按 `group_op` 翻译：
   - `self` → `{alias}.{col}`
   - `month` → `DATE_FORMAT({alias}.{col}, '%Y-%m')`（MySQL），其他库按函数 profile 翻译
   - `quarter` → `CONCAT(YEAR({alias}.{col}), '-Q', QUARTER({alias}.{col}))`（MySQL）
   - `range` → `CASE WHEN ... THEN '标签' END`，按 `buckets` 展开
3. **HAVING**：`having.field` 是 `metrics.as` 别名 → 用对应的聚合表达式替换（防 SQL 注入）
4. **WHERE** 只放 `filters`（聚合前过滤）；`having` 放 HAVING（聚合后过滤）

#### RANGE 分桶 SQL 翻译

```json
{"field": "age", "group_op": "range", "buckets": [
    {"from": 0, "to": 18, "label": "未成年"},
    {"from": 18, "to": 60, "label": "成年"},
    {"from": 60, "to": null, "label": "老年"}
]}
```

翻译为：

```sql
CASE
    WHEN t0.age >= 0  AND t0.age < 18 THEN '未成年'
    WHEN t0.age >= 18 AND t0.age < 60 THEN '成年'
    WHEN t0.age >= 60                  THEN '老年'
    ELSE '其他'
END AS age_range
```

#### 视图字段归属解析

```python
def resolve_field(view: View, field_code: str) -> tuple[str, str]:
    """返回 (table_alias, physical_column)，别名按 view.objects 顺序编号"""
    for idx, obj in enumerate(view.objects):
        for field in obj.cls.fields:
            if field.field_code == field_code:
                return f"t{idx}", field.source_column or field.field_code
    raise FieldNotFoundError(f"字段 {field_code} 在视图 {view.view_id} 中不存在")
```

#### 执行器职责划分

```
ViewLookupExecutor / ViewAnalyzeExecutor
    ↓
ViewSqlPlanner  （视图专用：字段归属解析、JOIN 链生成、逻辑函数翻译）
    ↓
VirtualSqlPlanner（共享：WHERE/HAVING/ORDER BY/LIMIT 拼接）
    ↓
DatasourceFunctionRegistry（共享：逻辑函数 → 方言 SQL）
    ↓
DataSourceManager（共享：执行 SQL）
```

| 能力 | 对象级 | 视图级 |
| --- | --- | --- |
| FROM 子句 | 单表 | 主表 + N 次 LEFT JOIN |
| 字段前缀 | 无前缀或单表别名 | `t{N}.` 按 JOIN 链编号 |
| 字段归属解析 | 直接查 `cls.fields` | 遍历 `view.objects` 找归属 |
| JOIN 条件 | 不需要 | 由 `relation.join_keys` 生成 |

### 10.4 权限回调接入点

亦庄方案中明确要求通过 Callback 做列级 / 行级权限控制。建议接入点如下：

- `lookup`：在 `select` 与 `filters` 校验后进行字段裁剪与行级重写
- `analyze`：在 `dimensions/metrics/filters` 校验后进行字段权限校验与过滤条件补写
- 执行器收到的必须是“权限重写后的逻辑计划”，而不是原始入参

## 11. 技能与 MCP 暴露设计

### 11.1 暴露策略

建议引入三种暴露级别：

| 策略 | MCP 公共列表 | 技能包 | 说明 |
| --- | --- | --- | --- |
| `direct` | 可见 | 可见 | 调试或专家模式 |
| `skill_only` | 不可见 | 可见 | 生产推荐 |
| `hidden` | 不可见 | 不可见 | 内部辅助动作 |

### 11.2 百应场景的推荐策略

在百应生产场景下，建议：

- `lookup_*` 默认 `skill_only`
- `analyze_*` 默认 `skill_only`
- `operation` 类动作按原有策略暴露
- 顶层大模型不要直接看到原始 DB 查询器
- 由门户将虚拟动作组织成“员工分析技能”“组织分析技能”“城市分析技能”等上层技能

### 11.3 规划器一致性要求

当前 ObjectViewBuilder 会直接跳过 `is_virtual` 动作，这会导致：

- MCP/技能能看到虚拟动作
- 规划器上下文却看不到

改造后需要统一：

1. ObjectViewBuilder 不再简单依据 `is_virtual` 过滤
2. 而是依据 `exposure_policy` 与 `planner_visible` 控制
3. 规划器看到的是“语义化后的虚拟动作描述”，不是底层 SQL 细节
4. 对于 `scope_type=view` 的动作，规划器应从 `View.actions` 读取，而不是假装它属于某个对象

## 14. 分阶段实施建议

### 第一阶段：元数据与协议落地

- 扩展 `OntologyField` / `OntologyAction`
- 支持对象 / 视图声明 `virtual_action_profiles`
- 明确 `lookup` / `analyze` / `search` 三类 schema
- 支持 `View.actions` 与 `view.invoke_action()`
- 先保留 `is_virtual` 兼容位

### 第二阶段：生成器与校验器改造

- 重构虚拟动作生成器
- 新增统一校验器
- 建立统一动作索引，让对象级与视图级都可被 MCP 路由
- 让 `query_{object_code}` 与新动作并行一段时间

### 第三阶段：执行器拆分

- 将当前 `DynamicQueryExecutor` 拆分
- 引入语义计划层
- 接入字段规则、账期强制过滤、权限重写

### 第四阶段：技能化暴露

- MCP 默认不暴露 `skill_only` 动作
- 门户以技能包方式组织虚拟动作
- 让百应场景通过技能调用，而非直接感知数据库型查询动作

### 第五阶段：清理旧能力

- 废弃 `query_{object_code}`
- 清理 `is_virtual` 单布尔逻辑
- 完成新旧协议迁移

## 15. 推荐结论

结合亦庄方案，当前项目里的虚拟动作不应继续停留在“对象级通用动态查询器”阶段，而应重构成：

1. 基于字段分析语义驱动的原子动作体系。
2. 对象与视图双层承载的虚拟能力体系。
3. 面向技能编排而不是直接给大模型暴露数据库能力的执行体系。

具体落地方向建议明确为：

- 用 `lookup/analyze/search` 取代单一 `query`
- 用字段规则元数据取代当前基于 `field_type` 的弱推断
- 当前对象 OWL 首期优先通过字段 `ext_property` 承载维度/度量扩展参数，再映射到运行时字段模型
- 用 `action_family + exposure_policy + virtual_backend` 增强并逐步取代 `is_virtual`
- 视图级虚拟动作通过 `View.actions + view.invoke_action()` 接入统一动作模型
- 用“技能组织虚拟动作”取代“模型直连对象通用查询器”

这套方案既能对齐亦庄-百应方案，也能最大程度复用当前项目已有的本体、动作、MCP、执行器基础设施。
