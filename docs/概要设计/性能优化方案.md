# DataCloud Agent 性能改进与架构优化方案

## 2. 节点上下文架构设计：严守 Prefix Cache 法则

本节的核心原则：**固定前缀优先，动态后缀置尾**。KV Cache 的命中条件是”请求中所有 token 的前缀与缓存完全相同”，只要 `SystemMessage` 里掺入任何随请求变化的动态内容，前缀立刻失效，缓存清零。

---

### 2.0 现有代码问题诊断

在对三个核心节点的代码进行审查后，发现以下具体问题：

| 文件 | 关键代码行 | 问题描述 | 缓存影响 |
| :--- | :--- | :--- | :--- |
| `intent.py:74-105` | `sys_prompt = f”...{last_user_msg}...{knowledge_text}...{tools_line}...”` | 用户问题、RAG 知识、工具名全部插值进 `SystemMessage` 字符串 | 每次请求 System Prompt 必然不同，**命中率 0%** |
| `intent.py:105` | `llm.ainvoke([SystemMessage(content=sys_prompt)])` | 整个 LLM 调用只有一条消息，**丢弃全部对话历史** | 上下文理解缺失（代词”那个”无法消解） |
| `dag.py:63-153` | `sys_prompt = f”...{intent}...{custom_tools_hint}...”` | 当轮意图字符串和动态工具提示插值进 `SystemMessage` | 同上，**命中率 0%** |
| `dag.py:155` | `llm.ainvoke([SystemMessage(content=sys_prompt)])` | 同样丢弃全部对话历史 | — |
| `insight.py:747,843` | `llm.astream(messages + [SystemMessage(content=sys_prompt)])` | `SystemMessage` 被**追加到消息列表末尾**，而非置于首位 | KV Cache 机制要求前缀固定；末尾的 System Prompt 永远无法命中缓存 |
| `insight.py:828-831` | `default_analysis_prompt` 中内嵌 `{data_summary}` | 动态数据摘要混入系统提示词 | 进一步加大缓存失效面积 |

> **insight_node 是当前最严重的 bug**：它是整个对话中调用次数最多、Token 消耗最大的节点，且当前实现中 `SystemMessage` 固定出现在所有历史消息的**最末端**，与阿里 Qwen 等 Prefix Caching 机制对”固定前缀”的要求完全相反。

### 2.1 四层上下文模型

我们将单个 Node 对 LLM 的请求上下文严格分为 **4 层（Layers）**：

| 上下文层级 | 内容与作用 | 是否可缓存 | 消息类型与位置 |
| :--- | :--- | :--- | :--- |
| **Layer 0: 静态约束层** | 纯静态内容：角色定义、节点任务规则、JSON 返回 Schema、代码沙箱约定。**绝对不能包含任何变量插值。** | ✅ **100% 缓存**（同 Agent 配置下永不改变） | `SystemMessage` → `messages[0]` |
| **Layer 1: 配置记忆层** | Agent 级别的静态配置：Few-Shot 示例、长效业务规则摘要。与 Layer 0 合并为同一 `SystemMessage` 即可。 | ✅ **100% 缓存**（同 Agent 配置不变则不变） | `SystemMessage` → `messages[0]`（与 Layer 0 拼接）或单独 `AIMessage` → `messages[1]` |
| **Layer 2: 历史对话层** | 本 session 中用户与助手的多轮对话记录。 | ⚡ **滚动命中**（命中至最新变更的前一轮） | `HumanMessage / AIMessage` → `messages[1:-1]` |
| **Layer 3: 当轮动态层** | 本轮特有内容：RAG 召回的知识片段、当轮工具执行结果、用户最新一句提问。**所有动态变量必须放在这里。** | ❌ **不缓存**（每轮必然不同） | `HumanMessage` → `messages[-1]` |

正确组装顺序：`[Layer0_System] + [Layer1_Memory?] + [Layer2_History...] + [Layer3_Dynamic_Human]`

### 2.2 各核心节点重构方案

#### (1) Intent Node（`intent.py`）

**当前问题代码（关键片段）：**

```python
# ❌ 错误：动态内容污染 SystemMessage，且完全丢弃对话历史
sys_prompt = f”””你是一个意图识别与改写专家。
用户原始问题：{last_user_msg}          # ← 动态
相关业务知识：{knowledge_text}          # ← 动态（每次 RAG 结果不同）
## 当前 Agent 可用的动态工具名
{tools_line}                           # ← 半动态（工具配置变更时才变）
...返回 JSON Schema...
“””
response = await llm.ainvoke([SystemMessage(content=sys_prompt)])  # ← 只有一条消息
```

**重构后：**

```python
# ✅ Layer 0: 纯静态 SystemMessage，无任何变量插值
INTENT_STATIC_SYSTEM = “””你是一个意图识别与改写专家。

## 任务说明
请判断用户意图是否清晰，结合业务知识改写问题，并选择正确的路由模式。
如果问题缺乏关键维度（如时间、明确指标），请判定为”不清晰”。

## query_mode 判定规则
- “online_query”：单次向已绑定工具取数即可（单表查询、列表、明细），
  无需多步规划、代码沙箱、多表关联或复杂报表叙事。
- “analysis”：需要任务拆解、多步执行、统计对比、关联、趋势、归因等，
  或不确定用哪个工具。

## 返回格式（严格 JSON，无多余字段）
{
  “rewritten_intent”: “...”,
  “clarify_needed”: true/false,
  “query_mode”: “online_query” 或 “analysis”,
  “target_tool”: “工具名或空字符串”,
  “tool_params”: {}
}
“””

# 从 prompts_overwrite 覆盖时，要求覆盖内容同样不含动态插值
static_sys = prompts_overwrite.get(“intent_system_prompt”, INTENT_STATIC_SYSTEM)

# ✅ Layer 3: 所有动态内容统一放入本轮 HumanMessage
dynamic_human = HumanMessage(content=(
    f”【本轮可用工具（用于 online_query 选型）】：{tools_line}\n\n”
    f”【检索到的相关业务知识】：\n{knowledge_text}\n\n”
    f”【用户当前提问】：{last_user_msg}\n\n”
    f”请基于以上背景，输出路由 JSON。”
))

response = await llm.ainvoke(
    [SystemMessage(content=static_sys)]   # Layer 0: 永远置顶，100% 命中缓存
    + messages[-4:-1]                      # Layer 2: 最近 2 轮历史（帮助消解代词）
    + [dynamic_human]                      # Layer 3: 本轮动态内容
)
```

> **注意**：`tools_line` 虽然来自 Agent 配置（变更频率低），但它取自 `dynamic_tools` 运行时注入，无法在代码编译期确定，因此放入 Layer 3 而非 Layer 0。若要进一步优化，可将工具名写入 Agent 的 `prompts_overwrite[“intent_system_prompt”]` 静态配置中。

#### (2) DAG Node（`dag.py`）

**当前问题代码（关键片段）：**

```python
# ❌ 错误：intent 字符串和工具提示插值进 SystemMessage
sys_prompt = f”””你是一个任务规划专家。
需要分析的目标：{intent}               # ← 动态
...（静态规则 + 示例）...
{custom_tools_hint}                    # ← 半动态（拼接进系统提示末尾）
“””
response = await llm.ainvoke([SystemMessage(content=sys_prompt)])  # ← 只有一条消息
```

**重构后：**

```python
# ✅ Layer 0: 所有静态规则、任务类型说明、示例全部保留在 SystemMessage
DAG_STATIC_SYSTEM = “””你是一个任务规划专家。请将分析目标拆解为具体子任务。

## 支持的任务类型
| type         | 使用场景                                                    |
|--------------|-------------------------------------------------------------|
| [动态工具名]  | 向挂载的外部服务查询原始数据（详见本轮 HumanMessage 的工具列表） |
| code_exec    | 对已查询数据进行计算/统计/关联，须有 deps，params.code 含代码   |
| search_knowledge | 检索知识库                                              |
| render_report    | 生成报告                                                |

## 判断规则（重要）
- 需要”查询/获取数据” → 必须从【可用动态工具】中选 type，deps 可为空
- 需要”统计/汇总/计算/关联”且有前置任务 → 必须用 code_exec
- deps 为空的任务禁止使用 code_exec
- *_query 类型任务的 params 里必须包含 query 或 question 字段

## code_exec 的 Python 代码约定
- input_files: dict，key=前置任务id，value=JSONL 文件绝对路径
- JSONL: 第一行为 meta（含 columns、total），后续每行一条记录
- 可直接使用 pandas（as pd）和 json，无需 import
- 最终结果赋值给 _result（list[dict] 或 dict），并用 print() 输出摘要

## 返回格式（严格 JSON 数组）
每个元素字段：id, type, description, status(“pending”), deps([]), params({})
“””

static_sys = prompts_overwrite.get(“dag_system_prompt”, DAG_STATIC_SYSTEM)

# ✅ Layer 3: 当轮动态内容（意图 + 可用工具）放入 HumanMessage
dynamic_human = HumanMessage(content=(
    f”【可用动态工具列表】：{tools_line}\n\n”
    f”【需要分析的目标】：{intent}\n\n”
    f”请输出 JSON 任务数组。”
))

response = await llm.ainvoke(
    [SystemMessage(content=static_sys)]   # Layer 0
    + [dynamic_human]                      # Layer 3（DAG 节点无需对话历史）
)
```

### 2.3 prompts_overwrite 兼容性说明

当前代码中 `prompts_overwrite` 的 key 是 `intent_prompt` / `dag_prompt` / `insight_prompt`，对应的 value 是**含动态插值的完整 f-string 模板**。重构后，key 命名改为 `*_system_prompt`，value 变为**纯静态字符串**（不含 `{变量}` 插值）。

迁移规则：

| 旧 key（含动态插值） | 新 key（纯静态） | 动态内容去向 |
| :--- | :--- | :--- |
| `intent_prompt` | `intent_system_prompt` | 移至代码构造的 Layer 3 `HumanMessage` |
| `dag_prompt` | `dag_system_prompt` | 移至代码构造的 Layer 3 `HumanMessage` |
| `insight_prompt` | `insight_system_prompt` | `data_summary` 移至 Layer 3；`task_prompt` 追加仍由 `_append_task_prompt_to_system` 处理 |

> **向后兼容处理**：在过渡期，代码可同时兼容旧 key，读取旧 key 时打印 deprecation warning，提示 Agent 配置方迁移。
