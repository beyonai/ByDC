# 文件读写架构优化方案

## 一、Context（背景与动机）

### 1.1 问题描述

当前 `by-datacloud` SDK 中存在**两套互不相通的文件存储抽象**：

| 抽象层 | 出处 | 用途 |
|---|---|---|
| `file_manager` | gateway 注入：`agent_runtime_state.session_manager.file_manager` | analysis agent 沙箱工作区文件读写 |
| `ResultFileStorage` | `datacloud_data_sdk.file_storage.base` | SDK 结果文件（CSV 导出等）持久化 |

`packages/datacloud-analysis/src/datacloud_analysis/tools/file_io.py` 的 `read_file` / `write_file` 走 `file_manager` 路径；而 `packages/datacloud-data/src/datacloud_data_sdk/csv_storage/manager.py` 的导出走 `ResultFileStorage` 路径。两者各自闭环，**跨抽象的写后读不可达**——SDK 经 `ResultFileStorage.write_text` 写出的文件，agent 端用 `file_io.read_file` 无法读回。

### 1.2 决策

- **彻底统一到 `ResultFileStorage`**。`file_manager` 抽象不再使用，`file_io` 工具改走 `ResultFileStorage`。
- 沙箱工作区目前虽是本地路径，但需要通过 `ByclawResultFileStorage` 实现，从而支持分布式部署。
- `session_manager.file_manager` 仅承担 read/write 文件，可整体移除（属下一阶段，本方案仅覆盖 SDK 侧）。

### 1.3 预期收益

1. **单一抽象**：SDK 内只保留 `ResultFileStorage` 一个文件 I/O 抽象。
2. **写读对称**：`file_io.write_file` 与 `file_io.read_file` 共用同一后端，跨宇宙读不到的问题消失。
3. **后端可插拔**：落地侧实现 `ResultFileStorage` ABC 即可接入 byclaw-API / S3 / MinIO / OSS / 本地等任意后端。
4. **依赖倒置保持**：`by-datacloud` 不引入对 `byclaw_data` 的 import，沿用现有 DI 模式。

## 三、实施方案

### Step 1：扩展 `RequestContext` / `InvocationContext`

**文件：** `packages/datacloud-data/src/datacloud_data_sdk/context.py`

- `RequestContext` 增加字段 `result_file_storage: Any = field(default=None, repr=False)`，类型注解为 `Any` 以避免 SDK 内部循环依赖（与 `gateway_context` 同模式）
- `InvocationContext.__init__` 接受同名 kwarg 并写入 `RequestContext`
- 在 docstring 中说明该字段的注入责任与生命周期

**理由：** 沿用现有 `gateway_context` / `workspace_dir` 通过 `RequestContext` 透传给工具的成熟模式，避免引入新机制。

### Step 2：在 `tool_wrapper.dispatch_tool` 处注入 `result_file_storage`

**文件：** `packages/datacloud-analysis/src/datacloud_analysis/orchestration/execution/tool_wrapper.py:716-721`

- `dispatch_tool` 的 `loader` 参数已经存在，且 loader 已持有 `result_file_storage`（由 `loader.configure(...)` 设置）
- 在 `InvocationContext(...)` 构造时新增 `result_file_storage=getattr(loader, "result_file_storage", None)`
- 若 `loader` 为 `None`（非典型路径），传 `None` 即可（下游 `file_io` 会降级到 `LocalResultFileStorage`）

**前置依赖：** 若 `OntologyLoader.configure(result_file_storage=...)` 仅作为私有属性存储，需在 loader 上加只读属性 `result_file_storage` 暴露（最小新增）。

### Step 3：重构 `file_io.py`

**文件：** `packages/datacloud-analysis/src/datacloud_analysis/tools/file_io.py`

1. **删除 `_get_file_manager()` 辅助函数**（不再需要）。
2. **`read_file` / `write_file` 函数体改造：**
   - 移除 `_context: Any = None` 参数（不再依赖 gateway runtime context 注入）
   - 通过 `get_current_context().result_file_storage` 获取存储后端
   - 后端为 `None` 时降级：根据 `workspace_dir` 构造 `LocalResultFileStorage(workspace_dir)`
   - 路径仍通过 `_resolve_safe_path` 做越权防护（沿用现有逻辑）
   - 调用 `storage.read_text(path)` / `storage.write_text(path, content)`
3. **签名变化：**
   - `encoding` 参数将被废弃（`ResultFileStorage` ABC 未承载 encoding 概念，统一 utf-8）
   - 保留参数名以维持 langchain 工具 schema 不变，文档注明「当前仅支持 utf-8，参数预留」
4. **错误处理：**
   - `read_text` 返回 `None` → 视为「文件不存在」
   - 后端抛 `OSError` / `httpx.HTTPError` 等 → 统一捕获为字符串错误返回
5. **保留 `_resolve_safe_path` 与 `_resolve_fallback_workspace_dir`** —— 仍用于 `LocalResultFileStorage` 降级路径与 workspace 安全防护。

### Step 4：测试更新

**文件：** `packages/datacloud-analysis/tests/dca/unit/test_new_tools.py`

- **保留**：`TestFileIOTools`（基于 `InvocationContext(workspace_dir=...)` 的端到端用例，本次走 `LocalResultFileStorage` 降级路径，行为应保持不变）
- **删除**：`TestFileIOToolsWithGateway` 整个类（验证 `file_manager` 委托的 4 个用例已不再适用）
- **新增**：一组测试验证「`InvocationContext` 注入 `ResultFileStorage` 时优先走该后端」：
  - 使用 `MagicMock(spec=ResultFileStorage)` 注入 `InvocationContext(result_file_storage=mock)`
  - 断言 `read_file` / `write_file` 调用 `mock.read_text` / `mock.write_text`，而不是访问磁盘

## 四、不在本次范围内（Phase 2 待跟进）

1. **byclaw-data 落地侧清理**：移除 `session_manager.file_manager` 及其 read/write 实现。该项目独立 venv，需单独提交。
2. **OntologyAgent 透传 `result_file_storage` 到 LangGraph configurable**：当前依赖 `loader.result_file_storage` 取出，若未来 dispatch_tool 不再持有 loader（例如改造为非 ontology 场景），需通过 langgraph `config["configurable"]` 透传。本次保持最小改动。
3. **暴露 `append_file` 工具**：`ResultFileStorage.append_text` 能力当前未通过 langchain @tool 暴露给 agent。如需支持「agent 追加文件」，可在后续单独添加 `append_file` 工具。
4. **encoding 支持**：若需非 utf-8 写入/读取，需扩展 `ResultFileStorage` ABC（跨项目契约变更），由用户单独决策。
