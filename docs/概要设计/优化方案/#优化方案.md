# resource_list 工具白名单 & 虚拟工具前缀可配置化 — 优化方案

## 1. 背景与需求

### 1.1 触发场景
前端发起 `AskAgentCommand` 时，通过 `extra_payload.resource_list` 传递本轮对话所引用的资源清单。当前 `byclaw_data/worker.py` 中：

```python
resource_list = extra_payload.get("resource_list") or []
```

仅消费 `resourceType == "KG_DOC_FILE"` 的附件资源（lines 955–975），其它类型直接被丢弃。

同时，虚拟工具（如 `query_by_project`、`compute_by_project`）的前缀在 `virtual_action_injector.py` 中以字面量硬编码，无法在不同部署环境下定制。

### 1.2 需求一：resource_list 工具白名单
当 `resource_list` 中出现 `resourceType` 为 `VIEW` 或 `OBJECT` 的资源时，业务期望本轮请求 Agent **仅可使用所选 `resouceCode` 名下的全部工具**，且只可使用这些工具。该"白名单"对每个 `resouceCode` 而言包含两类：

| 类别 | 名称示例（OBJECT `by_project`） | 名称示例（VIEW `by_sale_view`） | 来源 |
|------|-------------------------------|-------------------------------|------|
| 虚拟工具 | `query_by_project`、`compute_by_project` | `query_by_sale_view`、`compute_by_sale_view` | `inject_virtual_actions` 注入（DB OBJECT/VIEW 必有 `query_*`，含度量字段时再加 `compute_*`） |
| Action 工具 | OWL 中 `by_project.actions` 暴露的 `action_code`（如 `A1`、`A2`） | `by_sale_view.actions` 暴露的 `action_code`（如 `V1`、`V2`） | `ActionToolGenerator` / `_build_view_tools` |

> 工具名取自 `action.action_code`，不是固定的 `query_/compute_` 前缀。`compute_*` 仅在该资源具备度量字段时才会被注入。`KB OBJECT` 走 `search_{code}`，不在本次需求范围。

### 1.3 需求二：虚拟工具前缀可配置化
将虚拟工具命名前缀（当前硬编码 `"query_"`、`"compute_"`）改为可通过环境变量覆盖的配置项。**默认值保持 `query_` 与 `compute_`**，确保零配置时行为完全向后兼容。

> `KB OBJECT` 的 `search_` 前缀及其 legacy alias 不在本次外化范围；如未来有同样诉求，可按相同模式增量补齐。

### 1.4 输入样例

```json
[
  { "id": "DIG_EMPLOYEE_10000091", "resourceType": "DIG_EMPLOYEE", "resouceCode": "DIG_EMPLOYEE",  "resourceId": "10000091", "resourceName": "销售管理数字员工" },
  { "id": "OBJECT_10000010",      "resourceType": "OBJECT",       "resouceCode": "by_project",    "resourceId": "10000010", "resourceName": "项目信息表" },
  { "id": "VIEW_10000045",        "resourceType": "VIEW",         "resouceCode": "by_sale_view",  "resourceId": "10000045", "resourceName": "销售管理视图" }
]
```

> 原始字段名 `resouceCode` 拼写少一个 `r`，方案需同时兼容未来修复的 `resourceCode`。

## 2. 现有机制回顾

### 2.1 动态路径
`worker.py` 已存在"动态 Agent 路径"（lines 888–898）：当 `extra_payload` 携带非空 `call_object_ids` / `call_view_ids` 时，跳过 `AgentConfig` 查找，直接以这些资源码组装 `mounted_objects`，并通过 `OntologyAgent.ask(view_codes=..., object_codes=...)` 驱动图执行。

### 2.2 工具生成机制
`OntologyToolLoader.load()`（`packages/datacloud-analysis/src/datacloud_analysis/tools/ontology_tool_loader.py`）严格按 `mounted_objects` 装配工具：

| 输入资源 | 加载分支 | 产出工具 |
|----------|----------|----------|
| DB OBJECT `code` | `_build_action_tools` → `inject_virtual_actions` + `ActionToolGenerator` | `query_{code}`、`compute_{code}`（若有度量）、所有 `action.action_code` |
| Non-DB OBJECT `code` | `_build_object_schema_tools` | 仅 OWL `action_code` 工具 |
| VIEW `code` | `_build_view_tools` → `inject_virtual_actions` | `query_{code}`、`compute_{code}`（若有度量）、`view.actions` 中的 `action_code` |
| 列表外资源 | 不加载 | 无 |

**核心结论**：约束 `mounted_objects` ≡ 约束工具白名单，且产出的是**该资源码下的全部工具**（虚拟 + 真实 Action）。

### 2.3 OntologyAgent 接口
`OntologyAgent.ask()` / `resume()` 仅在加载期通过 `view_codes` / `object_codes` 决定可见资源；调用阶段不再支持运行时工具过滤。白名单必须在请求入口（worker.py）完成解析。

### 2.4 配置体系
项目使用 `pydantic-settings`，统一前缀 `DATACLOUD_*`：
- `packages/datacloud-data/src/datacloud_data_service/config.py` 中 `Settings(BaseSettings)` + `Field(default=..., validation_alias="DATACLOUD_*")` 是既有标准范式
- `get_settings()` 通过 `@lru_cache` 暴露单例
- 已有字段：`DATACLOUD_LLM_API_BASE`、`DATACLOUD_LLM_MODEL`、`DATACLOUD_DB_*`、`DATACLOUD_GATEWAY_WORKER_ID` 等

## 4. 方案设计 · 需求一：resource_list 工具白名单

### 4.1 总体思路
"该 `resouceCode` 下的所有工具" 等价于 "把该 code 列入 `mounted_objects`"——`OntologyToolLoader` 已自动覆盖虚拟工具与 Action 工具两类。因此方案只需把 `resource_list` 中的 `VIEW` / `OBJECT` 资源码合并到现有动态路径的 `view_codes` / `object_codes`，即天然满足"白名单 = 所选资源全部工具"的诉求。

### 4.2 触发关系（先讲清楚因果）

`_is_dynamic_agent` **不是白名单的前置条件，而是结果**。判定式：

```python
_is_dynamic_agent = bool(_dyn_object_ids or _dyn_view_ids)
```

而 `_dyn_object_ids` / `_dyn_view_ids` 由两个来源**合并**而成：

1. `extra_payload.call_object_ids` / `call_view_ids`（既有，内部委派用）
2. `resource_list` 中 `resourceType ∈ {OBJECT, VIEW}` 的 `resouceCode`（**本次新增**）

因此关系是单向的：

| 场景 | `_is_dynamic_agent` | 是否走白名单 |
|------|---------------------|------------|
| `resource_list` 含 OBJECT 或 VIEW | **必然 True**（即使没传 `call_*_ids`） | ✅ 走动态路径，工具集 = 这些 code 的全部工具 |
| `resource_list` 仅含 `DIG_EMPLOYEE` / `KG_DOC_FILE` / 空，且无 `call_*_ids` | False | ❌ 走静态路径 `AgentConfig`，无白名单约束 |
| `resource_list` 含 OBJECT/VIEW，同时也有 `call_*_ids` | True | ✅ 两源合并去重，统一作为白名单 |

> 换言之：**白名单的开关就是 "`resource_list` 里有没有 OBJECT/VIEW"**。一旦有，强制走动态路径，旁路静态 `AgentConfig`；没有则一切照旧。

### 4.3 数据流

```
前端 resource_list
        │
        ▼
worker.py 入口解析
   ├─ KG_DOC_FILE  → attached_file_path     (已有)
   ├─ DIG_EMPLOYEE → 不参与工具生成          (忽略)
   ├─ OBJECT       → _res_object_codes      (新增)
   └─ VIEW         → _res_view_codes        (新增)
        │
        ▼
与 call_object_ids / call_view_ids 合并去重（call_*_ids 优先，resource_list 派生码追加）
        │
        ▼
_is_dynamic_agent = (合并后任一列表非空)
        │
        ├── False ──► 静态路径（沿用 AgentConfig）
        │
        └── True  ──► OntologyAgent.ask(view_codes=..., object_codes=...)
                            │
                            ▼
                   OntologyToolLoader 按 mounted_objects 加载，产出每个 code 的：
                      • 虚拟工具：{QUERY_PREFIX}{code} (+ {COMPUTE_PREFIX}{code} 若有度量)
                      • Action 工具：action.action_code
                            │
                            ▼
                   LLM 仅可见上述白名单工具
```

### 4.4 改动点（worker.py）

#### 4.4.1 模块级常量
```python
_RESOURCE_LIST_TOOL_TYPES: frozenset[str] = frozenset({"OBJECT", "VIEW"})
```

#### 4.4.2 辅助函数
```python
def _extract_tool_resource_codes(
    resource_list: list[Any],
) -> tuple[list[str], list[str]]:
    """从 resource_list 中提取 OBJECT / VIEW 的资源码，返回 (object_codes, view_codes)。

    资源码优先取 ``resouceCode``（前端历史拼写），回退 ``resourceCode``。
    非法/空值过滤；保留首次出现顺序，去重。

    每个返回的 code 在下游 ``OntologyToolLoader`` 中会被展开为该资源下的
    全部工具：虚拟 ``{QUERY_PREFIX}{code}`` / ``{COMPUTE_PREFIX}{code}``
    及 OWL/视图定义的 Action 工具。
    """
    object_codes: list[str] = []
    view_codes: list[str] = []
    seen_obj: set[str] = set()
    seen_view: set[str] = set()

    for item in resource_list:
        if not isinstance(item, dict):
            continue
        rtype = str(item.get("resourceType") or "").strip().upper()
        if rtype not in _RESOURCE_LIST_TOOL_TYPES:
            continue
        code_raw = item.get("resouceCode") or item.get("resourceCode") or ""
        code = str(code_raw).strip()
        if not code:
            continue
        if rtype == "OBJECT" and code not in seen_obj:
            object_codes.append(code)
            seen_obj.add(code)
        elif rtype == "VIEW" and code not in seen_view:
            view_codes.append(code)
            seen_view.add(code)

    return object_codes, view_codes
```

### 4.5 优先级与冲突处理

| 来源 | 优先级 | 说明 |
|------|--------|------|
| `extra_payload.call_object_ids` / `call_view_ids` | 高 | 内部调用、Agent 委派的明确指定 |
| `extra_payload.resource_list`（OBJECT/VIEW） | 中 | 前端用户当前选择 |
| `AgentConfig.extra.mounted_objects`（静态路径） | 仅当无任一动态资源时使用 | 兜底 |

合并策略：**追加去重**，首次出现的码胜出。

### 4.6 边界与异常

| 场景 | 行为 |
|------|------|
| `resource_list` 仅含 `DIG_EMPLOYEE` | 不激活动态路径 |
| `resource_list` 含 `KG_DOC_FILE` + `OBJECT` | 附件解析 + 工具白名单同时生效 |
| 同一 code 同时出现在 `call_object_ids` 与 `resource_list.OBJECT` | 去重 |
| 同一 code 既出现在 `OBJECT` 又出现在 `VIEW` | 各自独立去重，按 OWL 类型决定生效分支 |
| `resouceCode` 为空 / 非 str | 跳过 |
| `resource_list` 不是 list | `or []` 兜底 |
| 资源不含度量字段 | `compute_{code}` 不被注入，正常裁剪 |



### 4.7 缓存与一致性

`_build_dynamic_cache_key()` 已用 `mounted_objects` sha1 作为缓存指纹（worker.py:216–221），合并后指纹自动随资源组合变化，不会命中错误的图缓存。无需额外改动。

### 4.8 与 ResumeCommand / paradigm resume 的兼容
- `dyn_thread_id` 不参与 `resource_list`，断点续跑路由不受影响。
- `OntologyAgent.resume()` 接收当前轮的 `view_codes` / `object_codes`，确保子图按当前白名单加载工具。

## 5. 方案设计 · 需求二：虚拟工具前缀可配置化

### 5.1 设计原则
- **零配置兼容**：默认值 `query_`、`compute_`，未设环境变量时行为完全等同改造前。
- **前缀完整字符串**：环境变量值即最终拼接前缀（含分隔符），允许使用者改成 `q_`、`qry_`、`Q-` 等任意形式，也允许设为空字符串得到裸 `{code}` 工具名。代码侧用 `f"{prefix}{code}"` 拼接，不再二次追加下划线。
- **三处必须同步**：前缀外化必须在以下 **三个层面** 同时生效，否则会产生"工具名变了但提示词或检测仍按旧前缀"的不一致问题：
  1. **工具生成** —— 决定实际工具名
  2. **提示词** —— 决定 LLM 调用时使用的工具名
  3. **内部检测** —— 决定 Loader / Plugin 中识别"动态查询/计算工具"的判定逻辑
- **范围聚焦**：本次仅外化 DB OBJECT / VIEW 的 `query_` / `compute_`；`KB OBJECT` 的 `search_` 与 legacy alias 保持现状。

### 5.2 配置项

新增到 `packages/datacloud-data/src/datacloud_data_service/config.py` 的 `Settings`：

```python
virtual_action_query_prefix: str = Field(

    validation_alias="DATACLOUD_VIRTUAL_ACTION_QUERY_PREFIX",
    description="虚拟 query 工具的命名前缀，与资源码直接拼接，例如 query_by_project。",
)
virtual_action_compute_prefix: str = Field(
    default="compute_",
    validation_alias="DATACLOUD_VIRTUAL_ACTION_COMPUTE_PREFIX",
    description="虚拟 compute 工具的命名前缀，与资源码直接拼接，例如 compute_by_project。",
)
```

> 默认值带下划线，与历史行为完全一致；前缀本身即"完整字符串"，便于运维侧灵活替换。

**实现方式（推荐）：**

在 `prompts.py` 顶部新增辅助函数，集中获取前缀，避免分散调用：

```python
from datacloud_data_service.config import get_settings


def _virtual_prefixes() -> tuple[str, str]:
    """返回 (query_prefix, compute_prefix)，与工具生成层共享同一配置源。"""
    s = get_settings()
    return s.virtual_action_query_prefix, s.virtual_action_compute_prefix
```

各 `_get_query_tool_hint_*`、`_build_exec_*` 函数在开头解构：

```python
def _get_query_tool_hint_zh() -> str:
    q, c = _virtual_prefixes()
    return (
        "## 查询工具命名规则\n"
        f"- 查询工具名称格式为 {q}{{对象编码}}（如 {q}ads_enterprise_analysis），"
        f"聚合计算工具格式为 {c}{{对象编码}}（如 {c}ads_enterprise_analysis）。\n"

```

##### `query_clarification_plugin.py`
- `_QUERY_PREFIXES` / `_DATA_TOOL_PREFIXES`（lines 70-71）由模块级常量 `("query_", "compute_")` 改为函数：
  ```python
  def _data_tool_prefixes() -> tuple[str, ...]:
      s = get_settings()
      return (s.virtual_action_query_prefix, s.virtual_action_compute_prefix)
  ```
- line 160 `for prefix in ("query_", "compute_"):` 改为 `for prefix in _data_tool_prefixes():`

### 5.4 与既有缓存/外部消费方的兼容

| 关注点 | 处理 |
|--------|------|
| 工具名作为 LLM Prompt 的一部分被缓存 | 前缀变化会触发提示词文本变化，进而影响 LLM Prompt 缓存命中。属于配置变更的预期开销。 |
| 历史会话存储中已落库的工具名 | 前缀变更后，历史展示可能与新会话形成不一致字符串；不会影响功能，仅展示文案差异。 |
| Action OWL 中 `action_code` 已显式占用了 `query_*` / `compute_*` 命名空间 | OWL 显式 `action_code` 不受前缀影响；若新前缀与已有 `action_code` 冲突，注入侧已有去重保护，按既有逻辑跳过。 |
| `dynamic_cache_key` 与 graph cache | 前缀变化会改变工具名集合的 hash 输入；需在切换前缀后重启 worker 或主动失效缓存，避免使用旧前缀生成的图。 |
| 多包共享配置 | 项目已统一以 `get_settings()` + `lru_cache` 暴露单例，本次新增字段沿用同一通道。 |

### 5.5 校验规则
`Settings` 字段不强制非空，但建议在 `model_post_init` 或专用 validator 中：
- 允许空字符串（语义：去前缀）
- 拒绝包含空白字符的前缀（避免工具名含空格触发 LLM 解析异常）

```python
from pydantic import field_validator

@field_validator("virtual_action_query_prefix", "virtual_action_compute_prefix")
@classmethod
def _validate_prefix(cls, v: str) -> str:
    if any(ch.isspace() for ch in v):
        raise ValueError("虚拟工具前缀不允许包含空白字符")
    return v
```

## 7. 验收用例

### 7.1 需求一：resource_list 工具白名单

#### UC-W1：仅 OBJECT 激活白名单
- **前置**：`call_object_ids` / `call_view_ids` 均未传；`AgentConfig` 中 `by_agent_id` 对应静态配置存在。
- **操作**：发起 `AskAgentCommand`，`extra_payload.resource_list = [{"resourceType": "OBJECT", "resouceCode": "by_project", ...}]`。
- **预期**：
  - `_is_dynamic_agent == True`
  - 日志输出 `resource_list tool whitelist activated: ... object_codes=['by_project'] view_codes=[]`
  - `OntologyAgent.ask` 接收 `object_codes=['by_project']`、`view_codes=[]`
  - LLM 可见工具集 = `by_project` 全部虚拟工具（`query_by_project`、`compute_by_project` 若含度量）+ `by_project.actions[*].action_code`，不含其它资源工具

#### UC-W2：仅 VIEW 激活白名单
- **前置**：同 UC-W1。
- **操作**：`resource_list = [{"resourceType": "VIEW", "resouceCode": "by_sale_view", ...}]`。
- **预期**：动态路径激活；工具集 = `by_sale_view` 全部虚拟 + Action 工具。

#### UC-W3：OBJECT + VIEW 共存
- **操作**：`resource_list` 同时包含 1 个 OBJECT(`by_project`) 与 1 个 VIEW(`by_sale_view`)。
- **预期**：两资源各自的全部工具均出现，互不污染；其它资源工具不出现。

#### UC-W4：仅 DIG_EMPLOYEE 不激活白名单
- **操作**：`resource_list` 仅含 `{"resourceType": "DIG_EMPLOYEE", ...}`。
- **预期**：`_is_dynamic_agent == False`；走静态 `AgentConfig` 路径；行为与改造前一致；不打印白名单激活日志。

#### UC-W5：OBJECT + KG_DOC_FILE 共存
- **操作**：`resource_list = [OBJECT(by_project), KG_DOC_FILE(/path/to/file)]`。
- **预期**：

  - `attached_file_path = "/path/to/file"`，附件提示文本被追加到用户消息

#### UC-W6：`call_*_ids` 与 `resource_list` 同 code 去重

- **预期**：合并后 `_dyn_object_ids == ["by_project"]`（仅一份）；执行结果与仅传 `call_object_ids` 一致。

#### UC-W7：双字段兼容（`resouceCode` 优先于 `resourceCode`）
- **操作**：`resource_list[0]` 同时含 `"resouceCode": "good"` 与 `"resourceCode": "bad"`。
- **预期**：白名单 code = `"good"`。

#### UC-W8：大小写归一
- **操作**：`resourceType` 取值 `"object"` / `"View"` / `"OBJECT"` 任意混合。
- **预期**：均能正确识别为 OBJECT/VIEW；不区分大小写。

#### UC-W9：重复项去重
- **操作**：`resource_list` 含 2 个完全相同的 OBJECT 项。
- **预期**：白名单中该 code 仅出现一次。

#### UC-W10：缺失资源码项被跳过

#### UC-W11：同 code 同时为 OBJECT 与 VIEW
