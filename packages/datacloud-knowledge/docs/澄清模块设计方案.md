## 1. 流程总览

```
StructuredQuery / StructuredCompute（含 complex_conditions）
    │
    ▼
Step 1 ─ 提取术语
    ├─ 主结构：遍历 select / filters / dimensions / metrics / order_by
    └─ complex_conditions：逐条 expand_query → 提取术语
    │
    ▼
Step 2 ─ 所有术语统一 recall 召回
    │
    ▼
Step 3 ─ 一次 LLM 确认（流式，串行）
    输入：主结构术语 + complex_conditions 术语 + 全部召回候选
    输出 ──┬── 主结构：确认后的 StructuredQuery / StructuredCompute
           └── complex_conditions：每条 NL 的术语映射（含 start / end）
    │
    ▼
Step 4 ─ 后处理 → 构建统一 paradigmList
    ├─ 主结构未确定术语 → 对应 paradigmId
    └─ complex_conditions：
       ├─ 已确定术语 → "原词（确认词）"，不参与笛卡尔积
       └─ 未确定术语 → 笛卡尔积展开（按候选排名截断，上限 20）
          → paradigmId="3"，keyword = 原始 NL，recall = [组合列表]
    │
    ▼
Step 5 ─ 一次性给前端
    │
    ▼
用户选择
    │
    ▼
Step 6 ─ format_clarification_query / format_clarification_compute
    ├─ 主结构：按用户选择替换术语
    └─ complex_conditions：用户选的组合句子写回（NL + 括号注释）
    输出：确定的 StructuredQuery / StructuredCompute
```

## 3. 核心模型

### 3.1 ExtractedTerm — 术语提取

```python
@dataclass(frozen=True, slots=True)
class ExtractedTerm:
    """从结构中提取的单个中文术语。"""

    raw_text: str                                   # "营收"
    ktype: str                                      # select | groupBy | whereKey | whereValue | orderBy
    path: str                                       # "select.0" / "filters.1.field" / "metrics.2.expr"
    source: Literal["main", "complex_condition"]    # 来源
    condition_index: int                            # complex_conditions 索引，主结构为 -1
    search_enabled: bool                            # False → 跳过召回
```

#### ktype 映射

| 位置 | ktype | 备注 |
|------|-------|------|
| `select[]` | `select` | |
| `dimensions[]` / `DimensionItem.field` | `groupBy` | 字符串简写 + 对象形式 |
| `filters[].field` / `metrics[].filters[].field` | `whereKey` | |
| `filters[].value` / `metrics[].filters[].value` | `whereValue` | 仅中文，跳过数字 / 日期 |
| `order_by[].field` | `orderBy` | 跳过 `metrics[].as` 别名 |
| `metrics[].field` | `select` | |
| `metrics[].expr` 中的中文 token | `select` | 按 `+-*/()` 分割 |
| `having[].field` | — | 引用 `metrics[].as`，跳过 |
| complex_conditions expand 后 | 按 expand 结果分配 | |

#### 跳过召回的情况

- 纯数字 / 日期（`"2026-04-01"`, `1000000`）
- 非中文字符串（`"total_revenue"`, `"eq"`）
- `having[].field`（引用 `metrics[].as`）
- `order_by[].field` 匹配 `metrics[].as` 时

### 3.2 LLM 确认输出

#### complex_conditions 术语映射

```python
class ConditionTermMapping(BaseModel):
    """complex_condition 中单个术语的确认结果。"""

    original_term: str                  # "亩产效益"
    start: int                          # 0-based 起始位置
    end: int                            # exclusive 结束位置
    confirmed: str | None = None        # 确定 → 填值；不确定 → None
    candidates: list[str] = []          # 按相关度排序


class ConfirmedCondition(BaseModel):
    """单条 complex_condition 的结构化结果。"""

    original_sentence: str              # "亩产效益后30%的地块"
    term_mappings: list[ConditionTermMapping]
```

#### 主结构澄清项

```python
class ClarifyItem(BaseModel):
    """主结构中需要用户澄清的项。"""

    keyword: str            # 原始中文术语
    candidates: list[str]   # 候选列表
    reason: str = ""        # 澄清原因
    source: str = ""        # select / where / group_by / order_by
    path: str = ""          # JSON pointer，供 format 精确替换
```

#### 统一输出

```python
class ConfirmedStructuredQuery(BaseModel):
    """LLM 确认后的 query 结果。"""

    select: list[str]
    filters: list[dict[str, Any]]
    order_by: list[dict[str, Any]]
    limit: int | None = None
    offset: int | None = None
    filter_relation: str = "AND"

    confirmed_conditions: list[ConfirmedCondition] = []
    clarify_items: list[ClarifyItem] = []
    needs_clarification: bool = False


class ConfirmedStructuredCompute(BaseModel):
    """LLM 确认后的 compute 结果。"""

    dimensions: list[str | dict[str, Any]]
    metrics: list[dict[str, Any]]
    filters: list[dict[str, Any]]
    having: list[dict[str, Any]]
    order_by: list[dict[str, Any]]
    limit: int | None = None
    filter_relation: str = "AND"

    confirmed_conditions: list[ConfirmedCondition] = []
    clarify_items: list[ClarifyItem] = []
    needs_clarification: bool = False
```

## 4. 笛卡尔积规则

### 规则

1. 已确定术语（`confirmed is not None`）→ 替换为 `"原词（确认词）"`，不参与组合
2. 未确定术语（`confirmed is None`）→ `candidates` 参与笛卡尔积
3. 拼接格式：按 `start / end` 定位，替换为 `"原词（候选词）"`
4. 截断：按候选排名截断，总组合上限 **20**
5. 全部确定 → 返回单个句子（仍加括号注释）

### 截断策略

从候选最多的术语开始，逐步裁剪末尾候选，直到积 ≤ 20。

```python
def truncate_candidates(
    unconfirmed: list[ConditionTermMapping],
    max_combinations: int = 20,
) -> list[list[str]]:
    """按候选排名截断，确保笛卡尔积 ≤ max_combinations。"""
```

### 示例

```
输入：
  "亩产效益后30%的地块"
  亩产效益@[0:4]  confirmed=None  candidates=[物理网格亩产效益, 管理网格亩产效益]
  地块@[9:11]     confirmed=None  candidates=[物理网格, 管理网格]

输出（4 组合）：
  亩产效益（物理网格亩产效益）后30%的地块（物理网格）
  亩产效益（物理网格亩产效益）后30%的地块（管理网格）
  亩产效益（管理网格亩产效益）后30%的地块（物理网格）
  亩产效益（管理网格亩产效益）后30%的地块（管理网格）

若地块已确定 confirmed=物理网格：
  亩产效益（物理网格亩产效益）后30%的地块（物理网格）
  亩产效益（管理网格亩产效益）后30%的地块（物理网格）
```

## 5. paradigmList 构建

### 映射

| paradigmId | paradigmName | 来源 |
|------------|-------------|------|
| `"1"` | 查询值 | `select[]` / `metrics[].field` / `metrics[].expr` |
| `"2"` | 分组条件 | `dimensions[]` |
| `"3"` | 过滤条件 | `filters` / `metrics[].filters` + **complex_conditions 笛卡尔积** |
| `"4"` | 排序目标 | `order_by[].field` |
| `"5"` | 统计函数 | 空 |

### 主结构格式

已确定（IFieldItem）：

```json
{"keyword": "营收", "recall": ["企业总营收（万元）"], "choiceKeyword": "企业总营收（万元）"}
```

未确定（IFieldItem）：

```json
{"keyword": "效益", "recall": ["企业经济效益等级", "物理网格亩产效益（万元/亩）"]}
```

过滤条件（IConditionItem）：

```json
{
  "field": "行业", "fieldRecall": ["行业类型"],
  "comparison": "in",
  "value": "制造业", "valueRecall": ["制造业", "农业"],
  "choiceField": "行业类型"
}
```

### complex_conditions 格式

放在 paradigmId="3"，使用 IFieldItem：

```json
{
  "keyword": "亩产效益后30%的地块",
  "recall": [
    "亩产效益（物理网格亩产效益）后30%的地块（物理网格）",
    "亩产效益（物理网格亩产效益）后30%的地块（管理网格）",
    "亩产效益（管理网格亩产效益）后30%的地块（物理网格）",
    "亩产效益（管理网格亩产效益）后30%的地块（管理网格）"
  ]
}
```

## 6. LLM 确认 prompt

### 设计原则

- 参考现有 `CONFIRM_SYSTEM_PROMPT` 风格
- complex_conditions 无需主 query 上下文
- 输出通过 tool call schema 约束

### system prompt 骨架

```
你是数据查询确认助手。根据知识库召回结果，将结构化查询中的中文术语映射到真实 schema 字段。

## 输入
- 用户原始查询
- 结构化查询参数（StructuredQuery 或 StructuredCompute）
- 每个中文术语的知识库召回候选
- 维度值线索

## 主结构确认规则
（参考现有 CONFIRM_SYSTEM_PROMPT 的 SELECT 语义化结构、WHERE vs filters 判断等）
- 所有字段名和维度值只能来自召回候选或维度值线索，严禁编造
- 确定的术语直接替换为真实字段名
- 无法确定的放入 clarify_items，标注 source 和 path

## complex_conditions 确认规则
- 对每条 NL 中的中文术语做确认
- 输出 original_term / start / end / confirmed / candidates
- 只有一个高置信候选 → confirmed 填值
- 多候选无法区分 → confirmed = null，candidates 按相关度排序

## 关键规则
- needs_clarification = true ⟺ clarify_items 非空 或任何 confirmed 为 null
```

### recall_context 格式

复用 `_format_recall_context()`，增加 complex_conditions 区域：

```
== 查询值 ==
  营收 (select): ['企业总营收（万元）', ...]

== 分组条件 ==
  行业 (groupBy): ['行业类型', ...]

== 过滤条件 ==
  效能等级 (whereKey): ['企业经济效益等级', ...]

== complex_conditions 术语 ==
  [0] "亩产效益后30%的地块":
    亩产效益 (select): ['物理网格亩产效益（万元/亩）', '管理网格亩产效益', ...]
    地块 (select): ['物理网格', '管理网格', ...]

== 维度值线索 ==
  ...
```

## 7. format_clarification

### 输入

- `structured_query` / `structured_compute`：原始结构
- `form`：前端回传 `{"paradigmList": [...]}`

### 处理

1. 从 `paradigmList` 提取 `choiceKeyword` / `choiceField` / `choiceValue`
2. 从 `knowledge` 提取 path 映射
3. 按 path 精确替换主结构术语
4. complex_conditions：用户选的组合句子写回对应索引

### 输出

```json
{
  "query": "找出亩产效益后30%的地块...",
  "select": ["企业名称", "企业经济效益等级"],
  "filters": [{"field": "企业经济效益等级", "op": "in", "value": ["中效益", "低效益"]}],
  "complex_conditions": ["亩产效益（物理网格亩产效益）后30%的地块（物理网格）"]
}
```

## 10. 风险

| 风险 | 应对 |
|------|------|
| 重复术语（同词不同位置） | 用 path 区分，不按 keyword 做 key |
| expr 子串误伤（`"营收 - 成本"` 替换 `"营收"`） | 提取时记录 token 边界 |
| order_by 引用 metrics.as 别名 | 提取时检测并跳过 |
| LLM 输出 schema 过大 | 验证稳定性，必要时拆分 |
| 笛卡尔积爆炸 | 截断上限 20，从候选最多的术语裁剪 |
