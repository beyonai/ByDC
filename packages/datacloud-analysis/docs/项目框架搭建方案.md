- **底层设施**：VFS/文件、PG 表由 session 与 datacloud-memory 使用；本仓库只依赖配置（如 PG URI）和外部服务。

### 2.2 与现有代码的关系

- 现有 `src/datacloud_analysis/`（下划线）为网关/队列/租户等逻辑，若与「超级分析智能体」同仓部署，可保留；新框架统一放在 **`src/datacloud_analysis/`**（中划线）下，通过 `agent.py` 的 `create_agent` 暴露给 LangGraph / langgraph dev。
- `agent.py` 逐步改为：从 `config` 读多模型与 PG，从 `session.checkpointer` 取 PG Checkpointer，从 `orchestration` 组图（意图 → DAG → Loop → Insight），并挂载 `tools/` 与 `memory/tools`。

## 三、工作空间路径的环境变量设计

### 3.1 设计原则

工作空间（见设计 4.2）的**根目录必须通过环境变量指定**，不在代码中硬编码任何默认路径。原因如下：

- 工作空间本质是用户数据存储区，路径依赖部署环境（本地 Docker、远程集群、云存储挂载点），不应内嵌在业务代码里。
- 多租户部署时，不同用户/应用的隔离路径需在启动时注入，而非由代码决定。
- 测试时可注入临时目录（如 `tmp_path` fixture），不影响正式数据。

### 3.2 工作空间环境变量

| 变量名 | 必填 | 说明 | 示例 |
|--------|------|------|------|
| `DATACLOUD_WORKSPACE_PUBLIC_ROOT` | 是 | 企业公共域根目录（对应设计 `public/datacloud/`） | `/data/public/datacloud` |
| `DATACLOUD_WORKSPACE_PRIVATE_ROOT` | 是 | 用户私有域根目录前缀（对应设计 `{user_id}_private/datacloud/`，`user_id` 由**运行时**拼接，不写在 env 里） | `/data/users` |
| `DATACLOUD_WORKSPACE_TASKS_ROOT` | 否 | 任务沙箱**存放根目录**。若不配则从 `PRIVATE_ROOT` 派生 | `/data/tasks` |

### 3.3 关于 DATACLOUD_WORKSPACE_TASKS_ROOT 的定位说明

**这个环境变量是「静态根前缀」，不是某次具体任务的路径。**

完整的任务工作空间路径由三层拼接而成：

```
{TASKS_ROOT}  ← env var，部署时固定配置（挂载点/磁盘目录）
  └── {user_id}/          ← 运行时注入，由 gateway 领单时确定
        └── task_{task_id}/    ← 运行时生成，每次 Agent 执行唯一
              ├── inputs/
              ├── temp/
              └── outputs/
```

**为什么 TASKS_ROOT 需要是可配置的 env var？**

- 任务执行产生大量临时文件（`/temp`），可能需要挂载到 NVMe/高速盘，与用户私有数据（`PRIVATE_ROOT`）分开存储；
- 在 K8s 场景中，任务沙箱目录可能挂载独立 PVC，与用户 home 目录不同；
- 若不需要单独配置，则留空，代码自动退回到 `{PRIVATE_ROOT}/{user_id}/workspaces/tasks`。

### 3.4 workspace/paths.py 的职责

`workspace/paths.py` 是唯一读取工作空间环境变量的位置，提供如下接口：

```python
# workspace/paths.py

from pathlib import Path
from dataclasses import dataclass
from datacloud_analysis.config.env import WorkspaceSettings

@dataclass(frozen=True)
class TaskPaths:
    inputs: Path          # 可写：任务入参（网关前置下发的材料）
    temp: Path            # 可写：运算过程的临时文件
    outputs: Path         # 可写：最终交付物
    skills_public: Path   # 只读：企业公共技能库挂载点
    skills_private: Path  # 只读：用户私有技能库挂载点

def build_task_paths(user_id: str, task_id: str) -> TaskPaths:
    """
    唯一的路径构造函数。
    user_id / task_id 是运行时参数（来自 gateway 领单），
    根目录来自 env，两者在此函数内合并，外部模块只持有 TaskPaths 对象。
    """
    ws = WorkspaceSettings()

    # tasks_root：优先用独立配置，否则从 private_root 派生
    tasks_root = ws.tasks_root or (ws.private_root / user_id / "workspaces" / "tasks")
    task_base  = tasks_root / user_id / f"task_{task_id}"

    return TaskPaths(
        inputs         = task_base / "inputs",
        temp           = task_base / "temp",
        outputs        = task_base / "outputs",
        skills_public  = ws.public_root / "skills",                             # 只读
        skills_private = ws.private_root / user_id / "workspaces" / "skills",  # 只读
    )
```

其他模块（`workspace/mount.py`、`orchestration/sandbox_executor.py`）只接收 `TaskPaths` 对象，**不直接读取任何路径环境变量**。

## 四、技能（Skills）存储与加载规划

设计 3.1 中提到两处技能：`build_skill`（自举技能，沉淀分析规律）和「动态扩展技能」；设计 4.2 中明确了技能的存储位置（`public/datacloud/skills/`、用户私有 `skills/`）和沙箱内的只读挂载视图（`./skills/`）。本节规划技能的完整存储与加载方案。

### 4.1 技能的分类

| 来源 | 类型 | 存放位置 | 读写性 |
|------|------|----------|--------|
| **系统内置** | SDK 随包附带的官方算子（如通用数据分析、可视化模板） | `src/datacloud_analysis/skills/builtin/` | 只读（随 SDK 发版） |
| **企业上传** | 管理员上传的企业级公共算子 | `{PUBLIC_ROOT}/skills/` | 只读（由管理平台写入） |
| **用户私有** | 用户自己习惯用的私人快捷工具 | `{PRIVATE_ROOT}/{user_id}/workspaces/skills/` | 只读（由用户管理界面写入） |
| **Agent 自举** | `build_skill` 工具沉淀分析规律生成的插件 | `{PRIVATE_ROOT}/{user_id}/workspaces/skills/` | Agent 运行时写入 |

> 系统内置技能随 SDK 源码发版，不依赖部署环境；企业/用户技能通过文件系统或管理 API 写入对应目录；Agent 自举技能写入用户私有目录，下次同一用户的任务即可复用。

### 4.2 技能文件结构约定

每个技能是一个 Python 文件，遵守以下约定，便于 `SkillLoader` 统一扫描：

```python
# 示例：skills/builtin/group_agg.py

SKILL_META = {
    "name": "group_agg",
    "description": "按指定维度分组聚合，支持 sum/mean/count/ratio",
    "version": "1.0.0",
    "author": "system",
}

def run(df, group_by: list[str], agg_col: str, method: str = "sum"):
    """技能主入口，Agent 的 sbx_run_code 工具调用此函数。"""
    return df.groupby(group_by)[agg_col].agg(method).reset_index()
```

约定：
- 必须包含模块级 `SKILL_META` 字典（name、description 必填）；
- 必须包含 `run(...)` 函数作为统一入口；
- `build_skill` 写入新技能时，同样遵守此格式，由 Agent 的编码模型生成。

### 4.3 SkillLoader：技能扫描与注册

在 `workspace/` 下新增 `skills_loader.py`，负责扫描上述三类目录并将技能注册为可供 Agent 调用的工具：

```python
# workspace/skills_loader.py

import importlib.util
from pathlib import Path
from typing import Callable

class SkillLoader:
    """
    扫描并加载三类技能目录，返回已注册的技能字典。
    优先级：用户私有 > 企业公共 > 系统内置（同名技能后者被前者覆盖）。
    """

    def __init__(self, task_paths: TaskPaths):
        self._dirs = [
            # 系统内置（随包）：优先级最低，作为兜底
            Path(__file__).parent.parent / "skills" / "builtin",
            # 企业公共（只读挂载）：覆盖内置
            task_paths.skills_public,
            # 用户私有（只读挂载）：优先级最高
            task_paths.skills_private,
        ]
        self._registry: dict[str, dict] = {}

    def load_all(self) -> dict[str, dict]:
        """扫描所有目录，返回 {skill_name: {"meta": ..., "run": Callable}}。"""
        for directory in self._dirs:
            if not directory.exists():
                continue
            for skill_file in sorted(directory.glob("*.py")):
                self._load_file(skill_file)
        return self._registry

    def _load_file(self, path: Path) -> None:
        spec = importlib.util.spec_from_file_location(path.stem, path)
        module = importlib.util.module_from_spec(spec)
        spec.loader.exec_module(module)
        meta = getattr(module, "SKILL_META", None)
        run_fn: Callable | None = getattr(module, "run", None)
        if meta and run_fn:
            self._registry[meta["name"]] = {"meta": meta, "run": run_fn}
```

### 4.4 技能在沙箱中的挂载与使用

**挂载（`workspace/mount.py`）：**

- 启动任务沙箱时，`mount.py` 将 `skills_public` 与 `skills_private` 目录以**只读**方式挂载到沙箱的 `./skills/`（与设计 4.2 运行视图一致）；
- 系统内置技能因随 SDK 包发布，不需挂载，直接由 `SkillLoader` 从包内路径加载。

**Agent 调用（`tools/sandbox.py`）：**

- `sbx_run_code` 在执行代码时，可从 `./skills/` 路径 `import` 或调用 `run()` 入口；
- `SkillLoader` 扫描结果也作为工具描述注入到系统 Prompt，让 Agent 知道当前可用的技能清单。

**Agent 自举（`tools/skill.py` → `build_skill`）：**

- `build_skill` 工具在 Agent 沉淀分析规律后，由编码模型生成符合约定格式的 `.py` 文件，写入 `skills_private`（即 `TaskPaths.skills_private` 指向的目录）；
- 写入完成后调用 `SkillLoader.load_all()` 重新扫描，新技能在当前任务或后续任务中即可复用。

## 五、SDK 统一初始化设计（`bootstrap.py`）

### 4.1 问题背景

本项目是一个 **SDK**，由上层应用（如任务调度引擎的 Worker 进程）导入使用。SDK 在第一次被调用前需要完成一次性的环境准备：

- 加载并校验所有环境变量（PG 连接、模型配置、工作空间路径）；
- 创建 PG 异步连接池；
- 调用 `checkpointer.setup()` 在 PG 中自动建表（若表已存在则跳过）；
- 初始化 datacloud-memory 的 Store（同样幂等）。

这些操作如果每次 Agent 执行时都触发，会产生不必要的延迟，甚至并发竞争。

### 4.2 初始化入口：`bootstrap.py`

在 `src/datacloud_analysis/bootstrap.py` 中暴露 `setup()` 函数，作为**唯一的 SDK 初始化入口**：

```python
# bootstrap.py

import asyncio
import logging
from typing import Any

_initialized: bool = False
_init_lock: asyncio.Lock | None = None   # 在第一次 setup() 时创建，避免模块导入时就要求事件循环
_pg_pool: Any = None                     # 异步连接池单例

logger = logging.getLogger(__name__)


async def setup() -> None:
    """
    SDK 统一初始化入口。调用方（进程级）应在启动时 await 一次。
    内部通过 asyncio.Lock 保证并发安全，通过 _initialized 标志防止重复执行。
    """
    global _initialized, _init_lock, _pg_pool

    # 懒初始化 Lock（确保在当前事件循环内创建）
    if _init_lock is None:
        _init_lock = asyncio.Lock()

    async with _init_lock:
        if _initialized:
            logger.debug("datacloud-analysis already initialized, skipping.")
            return

        logger.info("datacloud-analysis: starting initialization...")

        # 1. 加载并校验所有环境变量（早失败原则）
        from datacloud_analysis.config.env import Settings
        settings = Settings()           # Pydantic 校验，缺少必填 env 直接抛 ValidationError

        # 2. 创建 PG 异步连接池（进程级单例）
        from psycopg_pool import AsyncConnectionPool
        _pg_pool = AsyncConnectionPool(settings.pg_checkpoint_uri, open=False)
        await _pg_pool.open()

        # 3. PG 建表（LangGraph 内部用 CREATE TABLE IF NOT EXISTS，天然幂等）
        from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
        async with AsyncPostgresSaver.from_conn_string(settings.pg_checkpoint_uri) as checkpointer:
            await checkpointer.setup()
        logger.info("datacloud-analysis: checkpoint tables ready.")

        # 4. 初始化 datacloud-memory Store（同样幂等）
        from datacloud_memory.store import init_store
        await init_store(settings.pg_checkpoint_uri)

        _initialized = True
        logger.info("datacloud-analysis: initialization complete.")


def get_pg_pool() -> Any:
    """返回已初始化的连接池，setup() 未调用时抛出明确错误。"""
    if not _initialized or _pg_pool is None:
        raise RuntimeError(
            "datacloud-analysis is not initialized. "
            "Call `await bootstrap.setup()` at process startup before using the SDK."
        )
    return _pg_pool
```

### 4.3 如何防止重复初始化

防重机制分两层：

| 层次 | 机制 | 作用 |
|------|------|------|
| **逻辑层** | `_initialized` 布尔标志 | 正常流程下，第二次调用直接 return，零开销。 |
| **并发层** | `asyncio.Lock`（`_init_lock`） | 进程启动时若有多个协程同时 `await setup()`（如 Worker 并发领单），只有第一个实际执行，其余等待后因 `_initialized=True` 直接返回。 |
| **DB 层** | `CREATE TABLE IF NOT EXISTS`（由 LangGraph 保证） | 即使 setup() 被意外多次调用到 DB 层，也不会破坏表结构。 |

### 4.4 调用方式（上层应用）

```python
# Worker 进程启动时，main() 中调用一次
import asyncio
from datacloud_analysis import bootstrap

async def main():
    await bootstrap.setup()          # 仅此一次
    # ... 启动任务消费循环、注册 Agent ...

asyncio.run(main())
```

集成测试中可利用 pytest-asyncio 的 session 级 fixture 完成一次初始化：

```python
# tests/conftest.py
import pytest
from datacloud_analysis import bootstrap

@pytest.fixture(scope="session")

```

### 4.5 与目录结构的关系

`bootstrap.py` 处于 `src/datacloud_analysis/` 的**顶层**，对外是 SDK 的「电源按钮」；内部它会调用 `config`、`session`、`datacloud-memory` 各自的初始化，但各子模块本身不需要了解彼此的存在。依赖方向如下：

```
调用方 (Worker/App)
    └──▶ bootstrap.setup()
            ├──▶ config/env.py        (读取并校验所有环境变量)
            ├──▶ session/checkpointer (建 PG checkpoint 表)
            └──▶ datacloud-memory     (建 memory Store 表)
```

## 七、会话管理存储 PG 与依赖环境变量（对应 4.1.3.2）

### 4.1 实施要点回顾

- 使用 `langgraph-checkpoint-postgres` 的 **AsyncPostgresSaver**。
- 在 Agent 网关（或图编译）初始化时调用 `checkpointer.setup()`，由库在 PG 中自动建表（如 `checkpoints`, `checkpoint_writes`, `checkpoint_blobs`）。
- 在 `checkpoints.metadata` 中写入业务溯源：`message_id`, `session_id`, `attachment_ids` 等（见设计 4.1.3.1）。

### 4.2 环境变量设计

建议**仅用环境变量**驱动 PG 连接，不把密码等写进代码或默认值；敏感信息走部署环境或密钥管理。

| 变量名 | 必填 | 说明 | 示例 |
|--------|------|------|------|
| `DATACLOUD_PG_CHECKPOINT_URI` | 是 | 会话/Checkpoint 用 PostgreSQL 连接串（含库名） | `postgresql://user:pass@host:5432/datacloud_db?sslmode=disable` |
| `DATACLOUD_PG_CHECKPOINT_SCHEMA` | 否 | 表所在 schema，空则用默认 `public` | `agent` |

存储方式建议：

- **本地/开发**：`.env` 文件（不提交），由 `python-dotenv` 或 12-factor 在进程启动时加载。
- **生产**：由 K8s Secret / 云厂商密钥 / 配置中心注入同名环境变量，应用只读环境变量。

这样与 4.1.3.2 的「在 Agent 网关初始化时挂载 checkpointer」一致：在 `session/checkpointer.py` 中根据 `DATACLOUD_PG_CHECKPOINT_URI` 创建 `AsyncPostgresSaver` 并执行 `await checkpointer.setup()`。

## 八、多模型与 Embedding 的环境变量设计

项目依赖**四种大模型**与**一种 Embedding 小模型**，建议用统一前缀区分用途，便于按角色配置不同端点或模型名。

### 5.1 命名与用途约定

| 角色 | 用途建议 | 环境变量前缀 |
|------|----------|--------------|
| 快速问答 | 简单问答、路由、意图分类 | `DATACLOUD_LLM_QUICK_` |
| 编码 | 代码生成、脚本编写 | `DATACLOUD_LLM_CODING_` |
| 推理 | 复杂推理、规划、总结 | `DATACLOUD_LLM_REASONING_` |
| 多模态 | 图文/表格等多模态输入 | `DATACLOUD_LLM_MULTIMODAL_` |
| Embedding | 记忆/知识检索向量化 | `DATACLOUD_EMBEDDING_` |

### 5.2 建议的环境变量列表

每种 LLM 使用同一套命名模式（以快速问答为例）：

| 变量名 | 必填 | 说明 | 示例 |
|--------|------|------|------|
| `DATACLOUD_LLM_QUICK_API_BASE` | 是 | API Base URL | `https://lab.xxx.com/gpt-proxy/v1` |
| `DATACLOUD_LLM_QUICK_API_KEY` | 是 | API Key | `sk-xxx` |
| `DATACLOUD_LLM_QUICK_MODEL` | 是 | 模型名 | `Qwen/Qwen3-8B` |
| `DATACLOUD_LLM_QUICK_TEMPERATURE` | 否 | 温度，默认 0.7 | `0.7` |

同理可得：

- **编码**：`DATACLOUD_LLM_CODING_API_BASE`, `_API_KEY`, `_MODEL`, `_TEMPERATURE`
- **推理**：`DATACLOUD_LLM_REASONING_*`
- **多模态**：`DATACLOUD_LLM_MULTIMODAL_*`

Embedding 小模型（一般只需 base、key、model）：

| 变量名 | 必填 | 说明 | 示例 |
|--------|------|------|------|
| `DATACLOUD_EMBEDDING_API_BASE` | 是 | Embedding API Base URL | `https://lab.xxx.com/embeddings/v1` |
| `DATACLOUD_EMBEDDING_API_KEY` | 是 | API Key | `sk-xxx` |
| `DATACLOUD_EMBEDDING_MODEL` | 是 | 模型名 | `text-embedding-3-small` |

### 5.3 存储与加载方式

- **存储**：仅通过**环境变量**存储；不在代码中写默认 key/base。本地用 `.env`，生产用密钥/配置中心注入。
- **加载**：在 `config/env.py`（或 `config/models.py`）中集中读取并校验，暴露为 Pydantic 配置或 dataclass，供 `agent.py` 与各节点按角色选用对应 LLM/Embedding。若某角色未配置则可回退为「与推理模型同源」等策略，并在文档中说明。

## 九、记忆模块 datacloud-memory 的评价与服务规划

### 6.1 单独抽取 datacloud-memory 是否合理

**结论：合理，且与设计一致。**

- 设计 4.3 将**长期记忆的构建与存储**与**智能体主流程**解耦：主流程只「查询记忆」和「触发异步提纯事件」，**提纯与落库**由 Memory Worker 离线完成。
- 把「记忆的构建 + 记忆的查询」集中到 **datacloud-memory**，实现：
  - **构建**：从 Checkpoint 拉取对话 → 归纳 → 写入 Store（PG）；可由本模块内的 Worker 或由调度引擎触发。
  - **查询**：按 namespace（global_rules / experiences）、按关键词/向量检索，供智能体通过 MCP 工具调用。
- 这样智能体侧只依赖「记忆查询 API 或 SDK」，不直接依赖 PG Store 表结构，符合单一职责与可替换实现。

### 6.2 datacloud-memory 内部服务规划

建议在 **datacloud-memory** 内规划以下服务/子包（与设计 4.3.2 双轨挂载一致）：

| 服务/模块 | 职责 | 说明 |
|-----------|------|------|
| **store** | 长期记忆存储抽象 | 对接 PG（LangGraph PostgresStore 或自建表），提供 namespace（如 `global_rules`, `experiences`）的 put/get/search；可选 PgVector 做向量检索。 |
| **build** | 记忆构建（提纯） | 输入：thread_id（或 checkpoint 快照）。从 Checkpoint 拉取原始对话，调用低成本 LLM 做归纳与分类，写入 store。供 Memory Worker 或任务引擎回调调用。 |
| **query** | 记忆查询 | 1）**轻量全局画像**：按 user_id 拉取 `global_rules`，返回可写入 MEMORY.md 的摘要。2）**三层检索**：search（返回 id+title 列表）→ read（按 id 取全文）。供 datacloud-analysis 的 `memory/tools.py` 调用。 |
| **embedding** | 向量化（可选） | 若 experiences 用向量检索，本模块内封装 Embedding 调用（使用第五节约定的 `DATACLOUD_EMBEDDING_*` 或 memory 专属 env），避免在 agent 中重复配置。 |
| **api**（可选） | HTTP/gRPC 接口 | 若 agent 与 memory 跨进程部署，可在此提供「构建触发」「查询」接口，供调度引擎与 agent 调用。 |

当前 **datacloud-memory** 仅有 `__init__.py` 与 redis/msgpack 依赖；设计 4.3 的持久化是 **PG Store**，建议：

- 在 datacloud-memory 中增加对 **PostgreSQL + 可选 PgVector** 的依赖，与设计中的 Store 表结构一致。
- 若短期仍需 Redis 做缓存，可保留为「查询缓存」层，持久层仍以 PG 为主。

## 十、多语言方言（i18n）初始化

### 10.1 问题背景

超级分析智能体面向多语言用户（初期至少支持**中文 `zh_CN`** 与**英文 `en_US`**），以下内容均需要随语言环境变化：

| 影响面 | 说明 |
|--------|------|
| **系统提示（System Prompt）** | Agent 的角色定义、工具使用规则描述需与用户语言一致 |
| **工具/技能描述** | `@tool` 的 `description` 字段影响模型是否调用该工具，需使用对应语言以保证准确率 |
| **错误与日志消息** | 面向用户侧的提示信息（非开发日志）需本地化 |
| **输出格式偏好** | 日期、数字格式随 locale 调整（如 `2026-03-12` vs `03/12/2026`） |

### 10.2 环境变量设计

| 变量名 | 必填 | 说明 | 示例 |
|--------|------|------|------|
| `DATACLOUD_AGENT_LOCALE` | 否 | 默认语言环境，格式 `语言_地区`，默认 `zh_CN` | `zh_CN` / `en_US` |

> **运行时覆盖**：`DATACLOUD_AGENT_LOCALE` 是**进程级默认值**，在多租户场景中，gateway 可从用户 Profile 读取用户级 locale，在构造 Agent 时以参数形式覆盖（优先级：请求参数 > env var）。

### 10.3 目录结构补充

在 `src/datacloud_analysis/`（可导入包）下新增 `i18n/` 子包：

```
src/datacloud_analysis/
└── i18n/
    ├── __init__.py          # 导出 get_system_prompt, get_supported_locales
    └── prompts.py           # 各语言系统提示字典及查询函数
```

### 10.5 i18n/prompts.py

```python
# datacloud_analysis/i18n/prompts.py

_SYSTEM_PROMPTS: dict[str, str] = {
    "zh_CN": (
        "你是 DataCloud 数据云分析助手，帮助用户进行数据分析与业务洞察。\n\n"
        "## 工具使用规则\n"
        "- 当用户询问业务数据（商机、客户、订单、合同等 CRM 记录）时，"
        "**直接调用 `data_query` 工具**，不要委托给子 agent。\n"
        "- `data_query` 是处理自然语言业务数据查询的首选工具，务必优先调用。\n"
        "- 请用中文回复，回答简洁准确。"
    ),
    "en_US": (
        "You are a DataCloud data analysis assistant, "
        "helping users with data analysis and business insights.\n\n"
        "## Tool usage rules\n"
        "- When the user asks about business data (opportunities, customers, "
        "orders, deals, or any CRM records), call the `data_query` tool "
        "**DIRECTLY** — do NOT delegate to a subagent.\n"
        "- `data_query` is always the first choice for natural-language "
        "business data questions.\n"
        "- Please respond in English, concisely and accurately."
    ),
}

_FALLBACK_LOCALE = "zh_CN"


def get_system_prompt(locale: str | None = None) -> str:
    """Return the system prompt for the given locale.

    Falls back to zh_CN if the locale is unsupported.
    If locale is None, reads DATACLOUD_AGENT_LOCALE env var.
    """
    if locale is None:
        from datacloud_analysis.config.env import LocaleSettings
        locale = LocaleSettings().locale
    return _SYSTEM_PROMPTS.get(locale, _SYSTEM_PROMPTS[_FALLBACK_LOCALE])


def get_supported_locales() -> list[str]:
    """Return the list of supported locale codes."""
    return list(_SYSTEM_PROMPTS.keys())
```

### 10.7 bootstrap.py 中的初始化

在 `bootstrap.setup()` 中增加 locale 校验步骤：

### 10.8 扩展新语言的步骤

未来如需支持新语言，仅需：

1. 在 `datacloud_analysis/i18n/prompts.py` 的 `_SYSTEM_PROMPTS` 字典中**新增一个 locale 键值对**；
2. 在 `.env`（或配置中心）将 `DATACLOUD_AGENT_LOCALE` 更新为新 locale 代码；
3. 无需修改 `agent.py`、`bootstrap.py` 或其他业务模块。

## 十一、实施顺序建议（不改变设计）

1. **配置与环境变量**：实现 `config/env.py`（Pydantic Settings），统一 PG、四类 LLM、Embedding、工作空间路径、**Locale** 等全部 env 的读取与校验；缺少必填项则启动时立即抛出 `ValidationError`（早失败）。
2. **工作空间路径**：实现 `workspace/paths.py`，唯一读取 `DATACLOUD_WORKSPACE_*` env 并构造 `TaskPaths`；补充 `workspace/mount.py` 做沙箱挂载逻辑。
3. **SDK 初始化**：实现 `bootstrap.py`，在其中完成连接池创建、PG Checkpoint 建表、datacloud-memory 初始化，并用 `_initialized` + `asyncio.Lock` 防重。
4. **会话层**：实现 `session/checkpointer.py`，提供获取已初始化 checkpointer 的接口（依赖 bootstrap 已运行）；实现 `session/metadata.py` 处理业务 ID 映射。
5. **gateway**：实现与任务调度引擎的对接（领单、回调），不实现引擎本身。
6. **orchestration**：按 3.1 顺序实现意图解析、DAG、Loop、沙箱执行、总结节点，并接入 session 与 workspace。
7. **tools**：实现知识/数据/沙箱/技能/报告等工具，并接入 memory 的查询工具（调 datacloud-memory）。
8. **memory 层**：在 agent 内实现 `memory/loader.py` 与 `memory/tools.py`，调用 datacloud-memory 的 query 能力；在 datacloud-memory 内实现 store / build / query（及可选 api）。
9. **测试**：在 `tests/` 下按 unit / integration 补全用例；集成测试通过 session 级 `initialized_sdk` fixture 调用一次 `bootstrap.setup()`，会话相关集成测试可依赖 testcontainers 或 CI 中的 PG。

## 十二、小结

| 主题 | 结论 |
|------|------|
| 框架根目录 | `src/datacloud_analysis/`，按 3.1 拆分为 gateway、orchestration、session、workspace、memory、tools、config，顶层增加 `bootstrap.py`。 |
| 任务调度引擎 | 第三方，本模块仅通过 gateway 调用。 |
| 工作空间路径 | 通过 `DATACLOUD_WORKSPACE_*` 环境变量指定**静态根前缀**；`TASKS_ROOT` 是所有任务目录的存放根（不是某次任务的路径），实际任务路径 = `{TASKS_ROOT}/{user_id}/task_{task_id}/`，在运行时拼接。`workspace/paths.py` 是唯一路径构造处，外部只持有 `TaskPaths`。 |
| 技能（Skills）存储 | 内置技能随 SDK 包发版（`skills/builtin/`）；企业公共技能存于 `PUBLIC_ROOT/skills/`；用户私有及 `build_skill` 自举技能存于 `PRIVATE_ROOT/{user_id}/workspaces/skills/`。 |
| 技能加载 | `workspace/skills_loader.py` 的 `SkillLoader` 按「内置 → 公共 → 私有」顺序扫描 `.py` 文件，同名技能后者覆盖前者；注册为 Agent 可调用工具，`build_skill` 写入后可即时重扫。 |
| SDK 初始化 | `bootstrap.py` 的 `setup()` 是唯一初始化入口；进程启动时 `await` 一次；通过 `_initialized` 标志 + `asyncio.Lock` 双层防重；DB 层由 LangGraph 的 `CREATE TABLE IF NOT EXISTS` 保证幂等。 |
| 测试目录 | `tests/` 合理，建议 unit / integration（及可选 e2e）分层；集成测试通过 session 级 fixture 调用一次 `bootstrap.setup()`。 |
| 会话 PG | 使用 `DATACLOUD_PG_CHECKPOINT_URI`（及可选 schema），环境变量存储，由 `session/checkpointer.py` 封装；`bootstrap.py` 统一建表。 |
| 多模型与 Embedding | 四类 LLM + 一类 Embedding 用统一前缀环境变量存储，在 config 中集中加载。 |
| datacloud-memory | 单独模块负责记忆构建与查询合理；建议在模块内规划 store、build、query、embedding（可选）、api（可选）服务；`bootstrap.py` 调用其 `init_store` 完成表初始化。 |
| 多语言 i18n | `DATACLOUD_AGENT_LOCALE`（默认 `zh_CN`）控制进程级 locale；`datacloud_analysis/i18n/prompts.py` 集中维护各语言系统提示；`agent.py` 通过 `locale` 参数支持请求级覆盖；新增语言只需在 `_SYSTEM_PROMPTS` 字典中添加键值对。 |

以上方案可直接作为「项目框架搭建」的实施依据；若需拆成按任务拆分的实施计划（含具体文件与测试步骤），可再基于本文档生成 `docs/plans/YYYY-MM-DD-datacloud-analysis-framework.md` 类执行计划。
