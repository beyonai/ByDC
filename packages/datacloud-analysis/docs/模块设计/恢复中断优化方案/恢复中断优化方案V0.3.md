## 1. 方案背景

### 1.1 问题根因回顾

V0.2 已确认：LangGraph `interrupt()` 导致包含它的**整个节点**重新执行。当前 `execution` 节点封装了完整的 `run_react_loop`（含 LLM 调用、工具分发、澄清交互），`interrupt()` 后导致：

1. LLM 被重新调用（10–30 s）
2. `_analyze_clarification()` 被重新调用（10–15 s）

### 1.2 V0.2 的解法

在 `execution` 节点外增加 `analyze_clarify / user_clarify / resume_exec` 三个专用节点，在图级别将昂贵操作与 `interrupt()` 隔离。但 `resume_exec` 需要手动序列化/反序列化 `messages_snapshot`（TC-7 风险）。

### 1.3 本方案（V0.3）的思路

> **如果每个工具调用本身是一个独立 LangGraph 节点，`interrupt()` 只重跑该工具节点，LLM 调用节点完全不受影响。**

这是 LangGraph 官方推荐的 Agent 构建模式（`create_react_agent` / `ToolNode`），天然适配框架的 checkpoint/resume 设计。

## 2. 核心思路：工具即节点

### 2.1 当前架构

```
execution 节点（单一 LangGraph 节点）
  └── run_react_loop（Python for 循环，约 600 行）
        ├── round 0: LLM 调用 → tool_calls → dispatch_tool → before_call_back → interrupt()
        ├── round 1: LLM 调用 → tool_calls → dispatch_tool
        └── round N: finish_react
```

所有操作在同一 LangGraph 节点内，`interrupt()` 导致整体重跑。

### 2.2 工具即节点架构

```
[llm_call_node]
    │
    ├─→ [tool_query_A_node] ─────────────────→ [llm_call_node]（ReAct 循环）
    ├─→ [tool_compute_B_node] ───────────────→ [llm_call_node]
    ├─→ [finish_react_node] ─────────────────→ [respond]
    │
    └─→（经工具节点触发澄清）
           [analyze_clarify_node]
                   │
                   ├─→ [user_clarify_node] → 回到原工具节点（interrupt 仅在此）
                   └─→（自动消歧）→ 回到原工具节点
```

ReAct 循环通过**图边**编码，不再是 Python `for` 循环：
- `llm_call_node` 根据 AIMessage.tool_calls 路由到对应工具节点
- 工具节点执行完毕写 ToolMessage 到 state → 返回 `llm_call_node`（下一轮）
- LLM 调用 `finish_react` → `finish_react_node` → 结束

## 3. 图拓扑设计

### 3.1 动态构图（build_analysis_graph 时）

```python
def build_analysis_graph(tools: dict[str, Any] | None = None, ...) -> StateGraph:
    builder = StateGraph(AgentState)

    # ── 固定节点 ──────────────────────────────────────────────────────────────
    builder.add_node("intend",          intend_node)
    builder.add_node("llm_call",        llm_call_node)
    builder.add_node("finish_react",    finish_react_node)
    builder.add_node("analyze_clarify", analyze_clarify_node)
    builder.add_node("user_clarify",    user_clarify_node)
    builder.add_node("respond",         respond_node)

    # ── 动态节点：每个工具一个节点 ─────────────────────────────────────────────
    tool_names: list[str] = []
    for tool_name, tool_fn in (tools or {}).items():
        builder.add_node(tool_name, make_tool_node(tool_name, tool_fn))
        tool_names.append(tool_name)

    # ── 路由：llm_call → 工具节点 / finish_react ──────────────────────────────
    builder.add_conditional_edges(
        "llm_call",
        _route_after_llm_call,   # 读 AIMessage.tool_calls[0].name
        {name: name for name in tool_names} | {"finish_react": "finish_react"},
    )

    # ── 工具执行完毕：回 llm_call 或进澄清流程 ────────────────────────────────
    for tool_name in tool_names:
        builder.add_conditional_edges(
            tool_name,
            _route_after_tool,   # clarify_needed → analyze_clarify，否则 → llm_call
            {"llm_call": "llm_call", "analyze_clarify": "analyze_clarify"},
        )

    # ── 澄清子流程（与 V0.2 相同设计）────────────────────────────────────────
    builder.add_conditional_edges(
        "analyze_clarify",
        _route_after_analyze,    # paradigm_list 非空 → user_clarify，否则 → 原工具节点
        {"user_clarify": "user_clarify"} | {name: name for name in tool_names},
    )
    builder.add_conditional_edges(
        "user_clarify",
        lambda s: str((s.get("pending_clarification_context") or {}).get("tool_name", "")),
        {name: name for name in tool_names},
    )

    builder.add_edge(START,        "intend")
    builder.add_conditional_edges("intend", _route_after_intend, ...)
    builder.add_edge("llm_call",   "llm_call")   # ← 由 conditional_edges 覆盖，此行仅示意
    builder.add_edge("finish_react", "respond")
    builder.add_edge("respond",    END)
    return builder
```

### 3.2 路由函数

```python
def _route_after_llm_call(state: AgentState) -> str:
    """读最后一条 AIMessage，路由到对应工具节点或 finish_react。"""
    for msg in reversed(state.get("messages") or []):
        if isinstance(msg, AIMessage):
            calls = getattr(msg, "tool_calls", None) or []
            if calls:
                return str(calls[0].get("name", "finish_react"))
            return "finish_react"
    return "finish_react"


def _route_after_tool(state: AgentState) -> str:
    status = str(state.get("execution_status") or "")
    return "analyze_clarify" if status == "clarify_needed" else "llm_call"
```

## 4. 关键技术点分析

### 4.1 澄清流程整合

澄清节点设计与 V0.2 完全相同（`analyze_clarify` / `user_clarify`），无需重新设计。
核心差异：

| 维度 | V0.2 | V0.3 |
|------|------|------|
| 澄清后恢复入口 | `resume_exec`（含 messages_snapshot 反序列化）| 直接路由回原工具节点（无需 snapshot）|
| messages 历史 | 手动序列化存 state，`resume_exec` 还原 | LangGraph `MessagesState` 自动持久化 |
| 工具节点识别 | `pending_clarification_context.tool_name` 传给 `resume_exec` | `user_clarify` 路由函数直接读取，路由到对应节点 |

**关键简化：** 不再需要 `resume_exec` 节点。`user_clarify` 完成后直接路由回触发澄清的工具节点重执行，LangGraph state 中的 `messages` 已包含完整历史，工具节点读取 `clarification_formatted_params` 命中后执行工具即可。

### 4.2 messages 状态管理

AgentState 继承 `MessagesState`，其 `messages` channel 使用 `add_messages` reducer 自动累积：

```python
# llm_call_node 追加 AIMessage
return {"messages": [ai_msg]}

# tool_node 追加 ToolMessage
return {"messages": [ToolMessage(content=result, tool_call_id=tc_id)]}
```

LangGraph checkpoint 自动持久化完整消息历史，无需手动 snapshot。**消除 V0.2 TC-7 的序列化风险。**

### 4.3 工具节点工厂（make_tool_node）

```python
def make_tool_node(tool_name: str, tool_fn: BaseTool) -> Callable:
    async def _node(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
        # 1. 从最后一条 AIMessage 取对应 tool_call
        tc = _extract_tool_call(state, tool_name)
        if tc is None:
            # resume 路径：从 pending_clarification_context 取 structured_input
            tc = _extract_from_pending_context(state, tool_name)

        # 2. 检查 clarification_override（澄清完成后注入）
        override = state.get("clarification_formatted_params")
        if override and override.get("tool_name") == tool_name:
            raw_params = dict(override["params"])
        else:
            raw_params = dict(tc.get("args") or {})

        # 3. 构建 HookContext + 运行 before hook
        ctx = _build_hook_context(state, tool_name, raw_params, config)
        try:
            ctx, before_decision = await hook_manager.run_before(ctx)
        except ClarificationNeededError as exc:
            return {
                "execution_status": "clarify_needed",
                "pending_clarification_context": {
                    **exc.context,
                    "tool_name": tool_name,
                    # 无需 react_messages_snapshot：state["messages"] 已持久化
                },
            }

        # 4. 处理 before_decision（patch / redirect / short_circuit）
        if before_decision:
            result = await _handle_before_decision(before_decision, tools_map, state, ctx)
        else:
            result = await tool_fn.ainvoke(ctx["tool_params"])

        # 5. 写 ToolMessage，清理澄清状态
        tool_msg = ToolMessage(
            content=_compress_tool_result(result, tool_name),
            tool_call_id=str(tc.get("id") or ""),
        )
        return {
            "messages": [tool_msg],
            "execution_status": None,
            "clarification_formatted_params": None,   # 消费后清理
            "pending_clarification_context": None,
            "clarification_analyze_result": None,
        }
    return _node
```

### 4.4 round_idx / max_rounds 控制

当前 `for round_idx in range(max_rounds)` 在 Python 层控制。V0.3 改为 state 字段：

```python
# llm_call_node 每次自增并检查
current_round = int(state.get("react_round_idx") or 0)
if current_round >= max_rounds:
    return {"execution_status": "max_rounds_exceeded"}
return {"messages": [ai_msg], "react_round_idx": current_round + 1}
```

### 4.5 Streaming 支持

`_stream_llm_call` / `_invoke_llm_with_fallback` 从 `run_react_loop` 迁入 `llm_call_node`，逻辑不变，可直接复用现有实现。

### 4.6 并行工具调用（多 tool_calls）

LLM 偶尔在一轮中产生多个 `tool_calls`。V0.3 可用 LangGraph `Send` API 并发扇出：

```python
from langgraph.types import Send

def _route_after_llm_call(state: AgentState) -> str | list[Send]:
    calls = _get_last_tool_calls(state)
    if len(calls) == 1:
        return calls[0]["name"]           # 单工具：直接路由
    return [Send(tc["name"], {}) for tc in calls]   # 多工具：并发
```

> **注意：** 并发扇出时若多个工具均需澄清，须串行化（LangGraph 不支持并发 `interrupt()`）。建议阶段 3 再启用，阶段 1/2 保持串行。

| 维度 | V0.2（节点拆分）| V0.3（工具即节点）|
|------|----------------|-----------------|
| 根治 LLM 重跑 | ✅ | ✅ |

| 回归风险 | 低 | 高（全面重构）|
| 短期可落地 | ✅ | ⚠️ 需专项排期 |

## 6. 可行性结论

**技术上完全可行。** LangGraph 1.0.10 原生支持动态 `add_node`、`MessagesState` 自动累积、`Send` 扇出，架构设计无阻点。

**相对 V0.2 的净增益：**
1. 消除 `messages_snapshot` 手动序列化风险（TC-7）
2. 无需 `resume_exec` 节点，图结构更简洁
3. 与 LangGraph 官方模式对齐，长期可维护性更好

**代价：** `run_react_loop`（约 600 行，含 streaming、fallback、trace、window 等）需全部迁移，是 V0.2 工作量的 3–5 倍，且回归风险高。

## 7. 分阶段实施路径

建议分三阶段，每阶段独立可交付、独立验收。

### 阶段 1：LLM 节点拆分（约 2 天）

将 `run_react_loop` 拆分为 `llm_call_node` + `tool_dispatcher_node`（内部仍用 `dispatch_tool`，暂不做 per-tool 节点）：

```
[llm_call_node] → [tool_dispatcher_node] → [llm_call_node]（循环）
                → [finish_react_node]
```

**收益：** LLM 与工具执行隔离，`interrupt()` 只重跑 `tool_dispatcher_node`，消除 LLM 重跑。

**注意：** `tool_dispatcher_node` 内部仍使用现有 `dispatch_tool` 逻辑，不改动 hook 系统和澄清插件。

### 阶段 2：澄清节点集成（约 1 天）

`tool_dispatcher_node` 内部捕获 `ClarificationNeededError`，接入 V0.2 的 `analyze_clarify / user_clarify` 节点。澄清完成后路由回 `tool_dispatcher_node` 继续执行（无需 messages_snapshot）。

**收益：** 完全解决澄清重跑问题，验收标准与 V0.2 TC-1 ～ TC-10 相同。

### 阶段 3：Per-Tool 节点（约 1–2 天，可选）

将 `tool_dispatcher_node` 拆分为 per-tool 节点，启用 `Send` API 并发。

**收益：** LangGraph 完全原生对齐，支持并行工具调用和 per-tool 超时控制。

## 8. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| `run_react_loop` 迁移引入 streaming 回归 | 高 | 高 | 阶段 1 保留全部 `_stream_llm_call` / `_emit_*` 逻辑原样移入，不做简化 |
| LangGraph `add_messages` reducer 与现有 messages 写法冲突 | 中 | 中 | 统一通过 `return {"messages": [msg]}` 追加，不直接赋值 `state["messages"]` |
| 动态 add_node 导致 graph hash 不稳定（LangGraph 版本差异）| 低 | 低 | 在 CI 中固定 LangGraph 版本并做 graph schema 快照测试 |
| `ClarificationNeededError` 被 tool_node 外层 try/except 吞掉 | 低 | 高 | `make_tool_node` 内只捕获 `ClarificationNeededError`，其他异常 re-raise |
| 阶段 3 并发扇出时多工具同时触发澄清 | 低 | 中 | 阶段 3 的路由函数检测 `clarify_needed` 时降级为串行处理 |

### 10.3 阶段 1：LLM 节点拆分

#### `llm_call_node.py`（新建）

```python
"""llm_call_node：仅负责 LLM 调用，AIMessage 写入 react_messages_log。"""

from __future__ import annotations

import logging
import os
from typing import Any

from langchain_core.messages import HumanMessage
from langchain_core.runnables import RunnableConfig

from datacloud_analysis.orchestration.execution.react_loop import (
    _build_llm,
    _build_system_message,
    _conversation_messages_for_llm,
    _deserialize_messages,
    _invoke_llm_with_fallback,
    _serialize_messages,
    finish_react,
)
from datacloud_analysis.orchestration.state import AgentState

logger = logging.getLogger(__name__)

_DEFAULT_MAX_ROUNDS = int(os.getenv("DATACLOUD_REACT_MAX_ROUNDS", "10"))


async def make_llm_call_node(
    *,
    tools_list: list,
    system_prompt: str,
    stable_system_prompt: str | None = None,
    dynamic_prompt: str | None = None,
    max_rounds: int = _DEFAULT_MAX_ROUNDS,
    gateway_context: Any = None,
):
    """返回 llm_call_node 闭包（在 build_analysis_graph 中调用）。"""

    async def _llm_call(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
        """ReAct LLM 节点：每轮调用 LLM，将 AIMessage 追加到 react_messages_log。

        首轮（react_round_idx 为 None）初始化消息历史；后续轮次从 checkpoint 恢复。
        通过正常 return 持久化 → interrupt() 只重跑 tool_dispatcher，本节点不受影响。
        """
        current_round = int(state.get("react_round_idx") or 0)
        if current_round >= max_rounds:
            logger.warning("[llm_call] max_rounds=%d exceeded", max_rounds)
            return {"execution_status": "max_rounds_exceeded", "react_round_idx": current_round}

        tools_map = {t.name: t for t in tools_list}
        tools_map["finish_react"] = finish_react
        llm = _build_llm(state)
        llm_with_tools = llm.bind_tools(list(tools_map.values()))

        # 恢复或初始化 ReAct 内部消息历史
        log = state.get("react_messages_log")
        if log:
            messages = _deserialize_messages(log)
        else:
            messages = [_build_system_message(system_prompt, stable_system_prompt, dynamic_prompt)]
            conv = _conversation_messages_for_llm(state)
            if conv:
                messages.extend(conv)
            elif query := str(state.get("user_query") or state.get("enriched_query") or ""):
                messages.append(HumanMessage(content=query))

        # LLM 调用（流式 + fallback 逻辑复用 run_react_loop）
        ai_msg, _did_stream = await _invoke_llm_with_fallback(
            llm_with_tools, messages, state, gateway_context=gateway_context, config=config
        )
        messages.append(ai_msg)
        logger.info(
            "[llm_call] round=%d tool_calls=%d streamed=%s",
            current_round,
            len(getattr(ai_msg, "tool_calls", None) or []),
            _did_stream,
        )

        return {
            "react_messages_log": _serialize_messages(messages),
            "react_round_idx": current_round + 1,
            "execution_status": None,
        }

    return _llm_call
```

**关键保证：** `react_messages_log` 经 `return` 正常提交到 checkpoint。`tool_dispatcher_node` 触发 `ClarificationNeededError`（最终导致 `interrupt()`）后，`llm_call_node` **不会重跑**，因为它已在上一步完成并 checkpoint。

"""tool_dispatcher_node：读取最新 AIMessage，分发工具调用，捕获 ClarificationNeededError。"""

from __future__ import annotations

import logging
from typing import Any

from langchain_core.messages import AIMessage, ToolMessage
from langchain_core.runnables import RunnableConfig

from datacloud_analysis.orchestration.execution.react_loop import (
    _deserialize_messages,
    _serialize_messages,
    dispatch_tool,
    finish_react,
)
from datacloud_analysis.orchestration.state import AgentState
from datacloud_analysis.tool_hook_plugins.builtin.query_clarification_plugin import (
    ClarificationNeededError,
)

logger = logging.getLogger(__name__)


def make_tool_dispatcher_node(
    *,
    tools_list: list,
    gateway_context: Any = None,
    loader: Any = None,
    redirect_tools_map: dict[str, Any] | None = None,
):
    """返回 tool_dispatcher_node 闭包。"""

    async def _tool_dispatcher(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
        """工具分发节点：从 react_messages_log 读最新 AIMessage，依次执行工具调用。

        正常完成 → 追加 ToolMessage，路由回 llm_call。
        ClarificationNeededError → 保存上下文，路由到 analyze_clarify。
        finish_react tool_call → 路由到 finish_react_node。
        """
        log = state.get("react_messages_log") or []
        messages = _deserialize_messages(log)

        # 取最后一条 AIMessage
        ai_msg: AIMessage | None = None
        for msg in reversed(messages):
            if isinstance(msg, AIMessage):
                ai_msg = msg
                break
        if ai_msg is None:
            logger.error("[tool_dispatcher] no AIMessage found in react_messages_log")
            return {"execution_status": "error"}

        tools_map: dict[str, Any] = {t.name: t for t in tools_list}
        tools_map["finish_react"] = finish_react
        if redirect_tools_map:
            tools_map.update(redirect_tools_map)

        calls = list(getattr(ai_msg, "tool_calls", None) or [])
        if not calls or calls[0].get("name") == "finish_react":
            return {"execution_status": "finish_react"}

        # resume 路径：澄清后注入的格式化参数
        clarification_override = state.get("clarification_formatted_params")

        for tc in calls:
            tool_name = str(tc.get("name") or "")
            try:
                _tool_id, result = await dispatch_tool(
                    tc,
                    tools_map,
                    state,
                    gateway_context=gateway_context,
                    loader=loader,
                    clarification_override=clarification_override,
                )
            except ClarificationNeededError as exc:
                logger.info(
                    "[tool_dispatcher] ClarificationNeededError tool=%s round=%s",
                    tool_name,
                    state.get("react_round_idx"),
                )
                return {
                    "execution_status": "clarify_needed",
                    "pending_clarification_context": {
                        **exc.context,
                        "tool_name": tool_name,
                        # react_messages_snapshot 含中断前完整历史，供调试；
                        # resume 路径无需反序列化，tool_dispatcher 从 state 中读已 checkpoint 的 log
                        "react_round_idx": int(state.get("react_round_idx") or 1) - 1,
                    },
                }

            tool_msg = ToolMessage(
                content=str(result),
                tool_call_id=str(tc.get("id") or ""),
            )
            messages.append(tool_msg)
            logger.info("[tool_dispatcher] tool=%s done", tool_name)

        return {
            "react_messages_log": _serialize_messages(messages),
            "clarification_formatted_params": None,  # 消费后清理
            "execution_status": None,
        }

    return _tool_dispatcher
```

> **注意：** V0.3 的 `tool_dispatcher_node` 捕获 `ClarificationNeededError` 后 **不需要在 `pending_clarification_context` 中保存 `react_messages_snapshot`**。`react_messages_log` 已由 `llm_call_node` 通过正常 `return` 持久化到 checkpoint，resume 时 `tool_dispatcher_node` 直接从 `state.react_messages_log` 读取即可，无需序列化/反序列化往返（消除 V0.2 TC-7 风险）。

"""make_tool_node：per-tool 节点工厂，每个工具对应独立 LangGraph 节点。

每个工具节点负责：
  1. 从最后一条 AIMessage 提取对应 tool_call（或从 pending_clarification_context 提取 resume 参数）
  2. 检查 clarification_formatted_params（澄清后注入）
  3. 运行 before hook；捕获 ClarificationNeededError
  4. 执行工具，写 ToolMessage 到 state["messages"]（MessagesState 自动累积）
  5. 清理澄清相关 state 字段
"""

from __future__ import annotations

import logging
from typing import Any

from langchain_core.messages import AIMessage, ToolMessage
from langchain_core.runnables import RunnableConfig

from datacloud_analysis.orchestration.state import AgentState
from datacloud_analysis.tool_hook_plugins.builtin.query_clarification_plugin import (
    ClarificationNeededError,

logger = logging.getLogger(__name__)


def make_tool_node(
    tool_name: str,
    tool_fn: Any,
    *,
    gateway_context: Any = None,
    loader: Any = None,
    hook_manager: Any = None,
) -> Any:
    """返回指定工具的 LangGraph 节点函数闭包。"""

    async def _node(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
        # 1. 提取 tool_call 参数（首次调用 or resume 路径）
        tc = _extract_tool_call(state, tool_name)
        if tc is None:
            tc = _extract_from_pending_context(state, tool_name)

        # 2. 检查澄清结果覆盖（澄清完成后注入）
        override = state.get("clarification_formatted_params")
        if override and override.get("tool_name") == tool_name:
            raw_params = dict(override["params"])
        else:
            raw_params = dict(tc.get("args") or {}) if tc else {}

        # 3. 构建 HookContext + 运行 before hook
        ctx = _build_hook_context(state, tool_name, raw_params, gateway_context, loader, config)
        try:
            ctx, before_decision = await hook_manager.run_before(ctx)
        except ClarificationNeededError as exc:
            # 触发澄清：返回 clarify_needed，路由到 analyze_clarify
            # V0.3 阶段 3 无需 messages_snapshot：state["messages"] 已由 MessagesState 自动持久化
            return {
                "execution_status": "clarify_needed",
                "pending_clarification_context": {
                    **exc.context,
                    "tool_name": tool_name,
                    "react_round_idx": int(state.get("react_round_idx") or 0),
                },
            }

        # 4. 处理 before_decision（patch / redirect / short_circuit）
        if before_decision:
            result = await _handle_before_decision(before_decision, state, ctx, gateway_context, loader)
        else:
            result = await tool_fn.ainvoke(ctx["tool_params"])

        # 5. 写 ToolMessage（MessagesState add_messages reducer 自动累积，无需手动 snapshot）
        tool_call_id = str(tc.get("id") or "") if tc else ""
        tool_msg = ToolMessage(
            content=_compress_tool_result(result, tool_name),
            tool_call_id=tool_call_id,
        )
        logger.info("[tool_node:%s] done tool_call_id=%s", tool_name, tool_call_id)

        return {
            "messages": [tool_msg],                    # add_messages reducer 追加
            "execution_status": None,
            "clarification_formatted_params": None,    # 消费后清理
            "pending_clarification_context": None,
            "clarification_analyze_result": None,
        }

    return _node


# ── 辅助函数 ───────────────────────────────────────────────────────────────────

def _extract_tool_call(state: AgentState, tool_name: str) -> dict[str, Any] | None:
    """从 state["messages"] 最后一条 AIMessage 中提取指定工具的 tool_call。"""
    for msg in reversed(state.get("messages") or []):
        if isinstance(msg, AIMessage):
            for tc in (getattr(msg, "tool_calls", None) or []):
                if tc.get("name") == tool_name:
                    return dict(tc)
            return None
    return None


def _extract_from_pending_context(state: AgentState, tool_name: str) -> dict[str, Any] | None:
    """resume 路径：从 pending_clarification_context 重建 tool_call 结构。"""
    ctx = state.get("pending_clarification_context") or {}
    if ctx.get("tool_name") == tool_name:
        return {"name": tool_name, "args": ctx.get("structured_input") or {}, "id": ""}
    return None


def _compress_tool_result(result: Any, tool_name: str) -> str:
    """将工具返回值压缩为字符串写入 ToolMessage（超长截断）。"""
    MAX_LEN = 2000
    text = str(result) if not isinstance(result, str) else result
    if len(text) > MAX_LEN:
        logger.debug("[tool_node:%s] result truncated %d→%d", tool_name, len(text), MAX_LEN)
        return text[:MAX_LEN] + "…[truncated]"
    return text
```

> **关键简化（相对 V0.2）：** per-tool 节点直接向 `state["messages"]` 写入 `ToolMessage`，由 LangGraph `MessagesState` 的 `add_messages` reducer 自动累积并持久化，无需 `react_messages_log` 字段，无需手动序列化。

def _route_after_llm_call_v3(state: AgentState) -> str | list[Send]:
