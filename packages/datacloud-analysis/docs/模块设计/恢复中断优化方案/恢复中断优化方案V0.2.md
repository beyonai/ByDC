## 1. V0.1 方案 A 验证结论

### 1.1 实验设计

V1.0 将方案 A（State 缓存）评为"高可靠性、天然持久化"。本次通过三组控制实验验证该结论。

**实验环境：** LangGraph 1.0.10，PostgreSQL checkpoint saver

**实验 1：interrupt() 前直接 dict 赋值**

```python
async def hook_node(state: AgentState):
    state["_clarification_cache"] = {"key": "value"}  # 直接写入
    interrupt({"ask": "user"})                          # 随即 interrupt
```

结论：**写入不持久**。resume 时 `state["_clarification_cache"]` 为 `None`。

**实验 2：interrupt() 前 ChannelWrite.do_write()**

```python
from langgraph.pregel.write import ChannelWrite
ChannelWrite.do_write(config, [ChannelWriteEntry("_clarification_cache", cache)])
interrupt({"ask": "user"})
```

结论：**写入不持久**，与实验 1 相同。

**实验 3：resume 后节点正常返回**

```python
async def hook_node(state: AgentState):
    resume_val = interrupt({"ask": "user"})
    return Command(update={"_clarification_cache": cache})  # resume 后正常返回
```

结论：**写入持久化** ✅。resume 后下次读取到完整缓存。

### 1.2 根因分析

LangGraph checkpoint 的工作原理：

```
节点执行开始
  │
  ├── 执行期间所有 channel 写操作 → 累积到 CONFIG_KEY_SEND deque
  │
  ├── [正常完成] → deque 内容提交到 checkpoint_pending_writes → 入库
  │
  └── [interrupt() 抛出 GraphInterrupt] → deque 被丢弃 → 写入全部丢失
```

**checkpoint 保存的是执行前快照，不是执行中快照。**

任何在 `interrupt()` 之前写入的内容（无论直接 dict 赋值还是 `ChannelWrite.do_write`）均属于"执行中写入"，在 `GraphInterrupt` 抛出时一并丢弃。

### 1.3 方案 A 判定

| 验证项 | V1.0 预期 | 实测结果 |
|--------|----------|---------|
| interrupt() 前 state 写入是否持久化 | ✅ 天然持久化 | ❌ 不持久 |
| 单元测试（TC-1/2/3）是否通过 | ✅ | ✅（测试不经过 checkpoint 边界）|
| 生产环境 resume 时 CACHE HIT | ✅ | ❌ 始终 CACHE MISS |

**方案 A 在生产环境中无效。** 单元测试通过是因为测试直接传入 dict 引用，不经过 LangGraph checkpoint 序列化/反序列化边界。

## 2. 问题范围扩展：react_loop 同根问题

同一根因导致 `react_loop` 存在相同问题。

### 2.1 现状

`react_loop` 在调用 `interrupt()` 前将消息历史写入 state：

```python
state["react_messages"] = [...]     # 期望 resume 后恢复
state["react_round_idx"] = round_idx
interrupt({"clarification": payload})
```

由于同样的 deque 丢弃机制，`react_messages` 永远不会被持久化。

### 2.2 后果

```
[第一次执行] LLM 调用 → tool_call → before_call_back → interrupt()
[Resume]     LLM 被重新调用（消息历史丢失）→ 可能生成不同 tool_call
             → before_call_back 再次走 CACHE MISS → _analyze_clarification() 再次执行
```

每次澄清交互额外浪费：
- 1 次 LLM 完整调用（10–30 s）
- 1 次 `_analyze_clarification()` SDK 调用（10–15 s）

## 3. 节点拆分方案（方案 B）评估

### 3.1 核心思路

**根因：** `interrupt()` 调用点与昂贵操作（LLM 调用、`_analyze_clarification`）位于同一节点内，节点重执行导致所有操作重复。

**解法：** 将 `interrupt()` 隔离到专用的轻量节点（`user_clarify`）。昂贵操作放到 `interrupt()` 之前的独立节点，通过正常 return 将结果提交到 checkpoint。

```
[execution] 检测到澄清需要 → 返回 clarify_needed 状态
  │
  ▼
[analyze_clarify]  调用 _analyze_clarification()，正常 return → 结果持久化到 checkpoint
  │
  ▼
[user_clarify]     读 state → interrupt() → _format_clarification() → 结果持久化
  │                  ↑ resume 后重跑仅此节点，无 LLM 调用，< 1 s
  ▼
[resume_exec]      恢复 react_loop messages_snapshot，注入格式化参数，继续执行
```

### 3.2 节点重执行代价对比

| 方案 | interrupt() 所在节点 | Resume 重跑内容 | 重跑代价 |
|------|--------------------|--------------|---------| 
| 现状 | execution（含 react_loop） | LLM 调用 + _analyze_clarification | 20–45 s |
| 方案 A（已排除） | execution（含 react_loop） | 同上（缓存实际不命中）| 20–45 s |
| **方案 B（本方案）** | **user_clarify（专用节点）** | **读 state + format_clarification** | **< 1 s** |

### 3.3 图拓扑变更

#### 现有图结构

```
START → intend → execution → respond → END
              └→ END (command_done)
```

#### 新图结构

```
START → intend → execution ──────────────────────────────────────────→ respond → END
              └→ END        └→ analyze_clarify ─── 需要用户确认 ──→ user_clarify
    (command_done)                    │                                      │
                                      │ 自动消歧（paradigm_list 为空）       │
                                      └──────────────────────────────→ resume_exec ──→ respond → END
```

路由规则：
- `execution` 返回 `execution_status == "clarify_needed"` → 转 `analyze_clarify`
- `execution` 返回其他 → 转 `respond`
- `analyze_clarify` 返回 `paradigm_list` **非空** → 转 `user_clarify`（需要用户确认）
- `analyze_clarify` 返回 `paradigm_list` **为空** → 转 `resume_exec`（大模型自动消歧，跳过用户交互）
- `user_clarify` → `resume_exec` → `respond`（固定顺序）

> **代码依据（query_clarification_plugin.py line 606）：**
> `_analyze_clarification()` 返回 `paradigm_list` 为空时，原代码直接应用 `resolved` 结果并跳过 `interrupt()`，无需用户介入。节点拆分后必须保留此分支语义。

### 4.2 query_clarification_plugin.py 修改

**核心改变：** `before_call_back` 不再直接调用 `interrupt()`，改为抛出 `ClarificationNeededError`。

```python
class ClarificationNeededError(Exception):
    """澄清插件检测到需要用户确认时抛出，携带工具调用上下文。"""

    def __init__(self, context: dict[str, Any]) -> None:
        super().__init__("clarification required")
        self.context = context
        # context 包含：tool_name, query, scope_code, structured_input,
        #               is_compute, resolved, is_complex
```

`before_call_back` 在检测到需要澄清时：

```python
# 不再调用 interrupt()，改为抛出信号异常
raise ClarificationNeededError({
    "tool_name": tool_name,
    "query": query,
    "scope_code": scope_code,
    "structured_input": structured_input,
    "is_compute": is_compute,
    "resolved": resolved,
    "is_complex": is_complex,
})
```

新增"已有澄清结果"的命中路径（在入口处最先检查）：

```python
# resume_exec 注入的格式化参数，直接应用，跳过全部分析逻辑
_clarify_result = (ctx.get("metadata") or {}).get("clarification_formatted_params")
if _clarify_result and _clarify_result.get("tool_name") == tool_name:
    _clarified_params = dict(_clarify_result["params"])
    ctx["tool_params"] = _clarified_params
    # ⚠️ 必须保留 is_complex 路由语义（对应原 line 467-468 / 611-613）
    if _clarify_result.get("is_complex"):
        return _build_redirect_decision(tool_name, _clarify_result.get("query", ""), _clarified_params)
    return HookDecision(action="patch", updated_params=_clarified_params)
```

### 4.4 新节点实现

#### analyze_clarify_node

```python
# orchestration/clarification/analyze_clarify_node.py

async def analyze_clarify_node(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
    """调用 _analyze_clarification()，通过正常 return 将结果持久化到 checkpoint。

    该节点不调用 interrupt()，LangGraph 将 return 值正常提交。
    analyze 结果即使 user_clarify 节点重跑也不会重复执行。

    当 paradigm_list 为空时（大模型自动消歧成功），同步生成 pre_filled_params，
    后续 _route_after_analyze 直接路由到 resume_exec，跳过 user_clarify。
    """
    ctx = state["pending_clarification_context"]
    assert ctx is not None

    paradigm_list, clarify_knowledge = _analyze_clarification(
        ctx["query"],
        ctx["scope_code"],
        ctx["structured_input"],
        is_compute=ctx["is_compute"],
    )

    pre_filled_params: dict[str, Any] | None = None
    if not paradigm_list:
        # 自动消歧：直接应用已解析术语，无需用户介入
        # 对应原代码 line 606-613 的逻辑
        pre_filled_params = _apply_resolved_to_params(
            ctx["structured_input"], ctx["resolved"]
        )

    return {
        "clarification_analyze_result": {
            "paradigm_list": paradigm_list,
            "clarify_knowledge": clarify_knowledge,
            "pre_filled_params": pre_filled_params,
        }
    }
```

#### user_clarify_node

```python
# orchestration/clarification/user_clarify_node.py

async def user_clarify_node(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
    """构造澄清 payload，interrupt() 等待用户，格式化结果。
    
    Resume 后重跑此节点时仅执行：
      读 state → interrupt() 直接返回 resume_value → _format_clarification()
    无 LLM 调用，无 _analyze_clarification()，重跑代价 < 1 s。
    """
    analyze_result = state["clarification_analyze_result"]
    ctx_data = state["pending_clarification_context"]
    assert analyze_result is not None and ctx_data is not None

    payload = _build_interrupt_payload(
        analyze_result["paradigm_list"],
        analyze_result["clarify_knowledge"],
        ctx_data,
    )
    resume_value = interrupt(payload)  # 全图唯一 interrupt() 调用点

    formatted_params = _format_clarification(
        ctx_data["tool_name"],
        ctx_data["structured_input"],
        resume_value,
        ctx_data["is_compute"],
    )
    return {
        "clarification_formatted_params": {
            "tool_name": ctx_data["tool_name"],
            "params": formatted_params,
            "is_complex": ctx_data["is_complex"],   # 传递路由标记供 hook 早返回路径使用
            "query": ctx_data["query"],
        },
        "pending_clarification_context": None,   # 清理
        "clarification_analyze_result": None,    # 清理
    }
```

**重跑分析：** resume 后该节点从头执行：读 state（< 1 ms）→ `interrupt()` 直接返回（无 I/O）→ `_format_clarification()`（< 100 ms）。整体 < 1 s。

#### resume_exec_node

```python
# orchestration/clarification/resume_exec_node.py

async def resume_exec_node(state: AgentState, config: RunnableConfig) -> dict[str, Any]:
    """从 messages_snapshot 恢复 react_loop，注入澄清后的工具参数继续执行。

    来源有两条路径：
    - 用户确认路径（经 user_clarify）：clarification_formatted_params 由 user_clarify 写入
    - 自动消歧路径（跳过 user_clarify）：pre_filled_params 由 analyze_clarify 写入
    两种情况均通过 clarification_override 注入 react_loop，before_call_back 命中后跳过分析。
    """
    ctx_data = state["pending_clarification_context"]
    assert ctx_data is not None

    # 优先取用户确认后的格式化参数，其次取自动消歧参数
    formatted = state.get("clarification_formatted_params")
    if formatted is None:
        analyze_result = state.get("clarification_analyze_result") or {}
        pre_filled = analyze_result.get("pre_filled_params")
        assert pre_filled is not None, "resume_exec: 既无 clarification_formatted_params 也无 pre_filled_params"
        formatted = {"tool_name": ctx_data["tool_name"], "params": pre_filled}

    messages = _deserialize_messages(ctx_data["react_messages_snapshot"])

    result = await run_react_loop(
        state,
        config,
        resume_messages=messages,
        resume_round_idx=ctx_data["react_round_idx"],
        clarification_override=formatted,
    )
    return {
        **result,
        "pending_clarification_context": None,    # 清理
        "clarification_analyze_result": None,     # 清理
        "clarification_formatted_params": None,   # 清理
        "execution_status": None,
    }
```

| 风险 | 概率 | 影响 | 缓解措施 |

## 9. 验收标准

### 9.1 验收用例总表

从原代码路径（`before_call_back` 全分支）推导出 10 条验收用例，覆盖正常路径、两条澄清路径、复杂查询路由、多工具序列、多轮中断、状态清理、异常容错。

| TC | 场景 | 触发条件 | 期望路径 | 期望结果 |
|----|------|---------|---------|---------|
| TC-1 | 无歧义术语（正常放行）| catalog 有值，所有术语 1:1 命中 | `execution → respond` | `_analyze_clarification` 不调用；无澄清节点经过 |
| TC-2 | 需要用户确认，简单查询 | unresolved 非空，paradigm_list 非空，is_complex=False | `execution → analyze_clarify → user_clarify → resume_exec → respond` | `_analyze_clarification` 仅调用 1 次；resume 后决策为 patch |
| TC-3 | 需要用户确认，复杂查询 | unresolved 非空，paradigm_list 非空，is_complex=True | 同 TC-2 路径 | resume 后决策为 **redirect**（非 patch）；`is_complex` 须从 `clarification_formatted_params` 正确传递 |
| TC-4 | 自动消歧，简单查询 | unresolved 非空，paradigm_list 为空，is_complex=False | `execution → analyze_clarify → resume_exec → respond`（跳过 user_clarify）| 无 `interrupt()` 调用；`pre_filled_params` 正确注入；决策为 patch |
| TC-5 | 自动消歧，复杂查询 | unresolved 非空，paradigm_list 为空，is_complex=True | 同 TC-4 路径 | 决策为 **redirect** |
| TC-6 | 多工具序列 | 第 1 个工具需澄清，澄清后继续，第 2 个工具无歧义 | 澄清子流程完成后 `resume_exec` 继续 react_loop，第 2 个工具由 hook 正常放行 | 两个工具均成功执行；第 2 个工具不触发 `ClarificationNeededError` |
| TC-7 | 多轮后中断（round > 0）| LLM 在第 2 轮调用工具时触发澄清 | `messages_snapshot` 含前 2 轮完整历史 | `resume_exec` 反序列化后消息数量与中断前一致；react_loop 从正确 round_idx 继续 |
| TC-8 | state 字段清理 | 澄清完成，`resume_exec` 执行完毕 | `resume_exec` 正常 return | `pending_clarification_context` / `clarification_analyze_result` / `clarification_formatted_params` 全部为 None |
| TC-9 | `_analyze_clarification` 抛异常 | analyze_clarify 节点中 SDK 调用失败 | 异常从 `analyze_clarify` 节点向上传播 | 不进入 `user_clarify`；LangGraph 将节点标为 error；上层可捕获并处理 |
| TC-10 | 用户取消澄清（空 resume_value）| user_clarify resume 时 resume_value 为 None 或 `{paradigmList: []}` | `_format_clarification` 被调用，接收空 paradigmList | 不抛异常；返回最低置信度参数或原始 structured_input；日志记录 resume_value 为空 |

### 9.2 关键路径日志模式

#### TC-2：需要用户确认（paradigm_list 非空，is_complex=False）

```
[react_loop] tool_call: query_ads_enterprise_analysis
[query_clarification] clarification needed — raising ClarificationNeededError
[react_loop] ClarificationNeededError caught — snapshot N messages, round_idx=0

[analyze_clarify] done: paradigm_list_count=2 → routing to user_clarify

[user_clarify] interrupt() called — waiting for user

# ---- 用户提交澄清 ----

[user_clarify] resume: interrupt() returned resume_value
[user_clarify] _format_clarification done, is_complex=False

[resume_exec] source=user_clarify, restoring N messages, round_idx=0
[resume_exec] clarification_override injected, is_complex=False → patch decision
```

#### TC-3：复杂查询（is_complex=True）— resume 后走 redirect

```
# 同 TC-2，仅最后一行不同：

```


### 9.3 核心指标

| 指标 | 要求 |
|------|------|
| `_analyze_clarification` 调用次数 | 整个澄清流程中 **= 1** |
| react_loop LLM 调用次数 | 整个澄清流程中 **= 1**（在 execution 节点）|
| resume 到工具执行恢复耗时 | **< 2 s**（不含用户等待）|
| 自动消歧路径 interrupt() 调用次数 | **= 0** |
| resume_exec 完成后三个 state 字段 | **全部为 None** |
| is_complex=True 时 resume 决策类型 | **redirect**（不得为 patch）|
