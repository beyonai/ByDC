**版本：** V1.0  
**日期：** 2026-04-20  
**作者：** 笙歌FL  
**状态：** 待评审

---

## 1. 问题背景

### 1.1 当前行为

`query_clarification_plugin.py` 的 `before_call_back` 回调在触发 `interrupt()` 后，LangGraph 会将当前节点（`before_call_back`）**整体重新执行**。这意味着：

```
[第一次执行] PRE-INTERRUPT#1 → _analyze_clarification() → PRE-INTERRUPT#2 → interrupt(payload) → [暂停]
[恢复后]     PRE-INTERRUPT#1 → _analyze_clarification() → PRE-INTERRUPT#2 → interrupt() 返回 resume_value → [继续]
```

每次澄清交互，`_analyze_clarification()` 被**调用两次**。

### 1.2 已有日志证据

```
# 第一次执行（中断前）
[query_clarification] PRE-INTERRUPT#1 calling analyze_clarification ...
[query_clarification] PRE-INTERRUPT#2 about to call interrupt() ...

# 恢复执行（这两行不应出现）
[query_clarification] PRE-INTERRUPT#1 calling analyze_clarification ...  ← 冗余！
[query_clarification] PRE-INTERRUPT#2 about to call interrupt() ...       ← 冗余！

# 恢复确认点
[query_clarification] RESUME resume_value type=dict value=...
```

### 1.3 代价分析

| 调用 | 内容 | 耗时估算 |
|------|------|---------|
| `_analyze_clarification()` 第 1 次 | LLM 提取 + Embedding 检索 | 10–15 s |
| `_analyze_clarification()` 第 2 次（恢复后冗余） | 同上，结果被丢弃 | 10–15 s |
| **冗余浪费** | | **≈ 10–15 s / 次澄清** |

除耗时外，还额外消耗 API Token 和 Embedding 检索资源。

---

## 2. LangGraph 中断机制原理

LangGraph 的 `interrupt()` 本质是**检查点控制流**：

1. 节点首次执行到 `interrupt(payload)` → 抛出内部异常，图暂停，`payload` 写入 checkpoint
2. 用户提交 resume 后，LangGraph 从上一个 checkpoint 重新调度该节点
3. 节点**从函数开头重新执行**，直到再次调用 `interrupt()` 时，`interrupt()` 直接返回 `resume_value`，不再暂停

这是 LangGraph 的设计约束，**无法在不修改框架的情况下绕过节点重执行本身**，但可以通过以下方案跳过昂贵的副作用操作。

---

## 3. 优化方案

### 方案 A：LangGraph State 缓存（推荐）

#### 核心思路

在调用 `interrupt()` 之前，将 `_analyze_clarification()` 的结果（`paradigm_list` + `clarify_knowledge`）写入 LangGraph 图状态（`state`）。恢复后，节点重新执行时首先检查 state 中是否已有缓存结果，若有则**跳过 SDK 调用**，直接使用缓存值。

#### 实现要点

**Step 1：扩展图状态结构**

在图状态（`GraphState` 或等价 TypedDict）中新增字段：

```python
class ClarificationCache(TypedDict):
    thread_tool_key: str          # "{thread_id}:{tool_name}:{query_hash}"
    paradigm_list: list[dict[str, Any]]
    clarify_knowledge: str

class GraphState(TypedDict, total=False):
    # ... 现有字段 ...
    _clarification_cache: ClarificationCache  # 新增
```

**Step 2：修改 `before_call_back` 回调**

```python
cache_key = _make_cache_key(thread_id, tool_name, query)
cached = ctx.get("state", {}).get("_clarification_cache")

if cached and cached["thread_tool_key"] == cache_key:
    logger.info("[query_clarification] CACHE HIT: skipping _analyze_clarification()")
    paradigm_list = cached["paradigm_list"]
    clarify_knowledge = cached["clarify_knowledge"]
else:
    logger.info("[query_clarification] CACHE MISS: calling _analyze_clarification()")
    paradigm_list, clarify_knowledge = _analyze_clarification(...)
    # 写入 state 缓存（恢复后可命中）
    if ctx.get("state") is not None:
        ctx["state"]["_clarification_cache"] = {
            "thread_tool_key": cache_key,
            "paradigm_list": paradigm_list,
            "clarify_knowledge": clarify_knowledge,
        }
```

**Step 3：缓存失效**

恢复流程结束（`_format_clarification` 完成）后，清除缓存，防止同 thread 下次查询误命中：

```python
if ctx.get("state") is not None:
    ctx["state"].pop("_clarification_cache", None)
```

#### 优缺点

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 中等（需扩展 GraphState + 传递 state 到回调） |
| 可靠性 | 高（state 与 LangGraph checkpoint 同步，天然持久化） |
| 分布式兼容 | 高（checkpoint 存储在数据库，多实例可见） |
| 侵入性 | 需修改 GraphState 和回调签名，影响范围可控 |
| 回滚风险 | 低（新增字段，不破坏现有流程） |

---

### 方案 B：节点拆分（Node Splitting）

#### 核心思路

将当前单一节点拆分为两个节点：

```
[analyze_node]  →  _analyze_clarification() + interrupt(payload)
[format_node]   →  _format_clarification(resume_value) + patch params
```

LangGraph 重新调度时只重新执行 `format_node`，`analyze_node` 不再重跑。

#### 优缺点

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 高（需重构 graph 拓扑、调整节点调度逻辑） |
| 可靠性 | 高（从框架层面彻底解决） |
| 侵入性 | 大（改变图结构，影响所有使用该图的代码路径） |
| 适用场景 | 适合将来大规模重构；当前阶段改造代价过高 |

---

### 方案 C：进程内 LRU 缓存

#### 核心思路

用 `functools.lru_cache` 或自定义 TTL 缓存，以 `(query_hash, scope_code)` 为 key，缓存 `_analyze_clarification()` 结果。

#### 优缺点

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 低 |
| 可靠性 | 中（进程重启后缓存丢失；多进程部署时各自独立） |
| 分布式兼容 | 差（多实例间无共享） |
| 缓存失效控制 | 难（相同 query 在不同会话语境下结果可能不同） |
| 适用场景 | 仅适合单进程开发调试，不推荐生产 |

---

## 4. 方案对比

| 维度 | 方案 A（State 缓存） | 方案 B（节点拆分） | 方案 C（LRU 缓存） |
|------|---------------------|-------------------|-------------------|
| 实现成本 | 中 | 高 | 低 |
| 可靠性 | 高 | 高 | 中 |
| 分布式安全 | ✅ | ✅ | ❌ |
| 框架侵入性 | 低 | 高 | 低 |
| 短期可落地 | ✅ | ❌ | ⚠️ |
| **推荐** | **✅ 首选** | 长期演进 | 不推荐生产 |

## 5. 推荐方案（方案 A）详细设计

### 5.1 缓存 Key 生成

```python
import hashlib

def _make_cache_key(thread_id: str, tool_name: str, query: str) -> str:
    """生成唯一缓存键，绑定到 thread + tool + query 三元组。"""
    digest = hashlib.md5(query.encode()).hexdigest()[:12]  # noqa: S324
    return f"{thread_id}:{tool_name}:{digest}"
```

> MD5 仅用于字符串去重，无安全需求，使用 `# noqa: S324` 消除 ruff S324 警告。

### 5.2 GraphState 扩展

在 `GraphState`（或等价状态 TypedDict）中新增 `_clarification_cache` 字段：

```python
from typing import Any, TypedDict


class ClarificationCache(TypedDict):
    thread_tool_key: str
    paradigm_list: list[dict[str, Any]]
    clarify_knowledge: str


class GraphState(TypedDict, total=False):
    # ... 现有字段（保持不变）...
    _clarification_cache: ClarificationCache  # 新增：澄清 SDK 结果缓存
```

### 5.3 回调修改逻辑（before_call_back）

当前代码（`query_clarification_plugin.py` 约 461–472 行）：

```python
# 哨兵 1
logger.info("[query_clarification] PRE-INTERRUPT#1 calling analyze_clarification ...")
paradigm_list, clarify_knowledge = _analyze_clarification(...)
```

改为：

```python
cache_key = _make_cache_key(
    ctx.get("thread_id", ""), tool_name, query
)
cached: ClarificationCache | None = (ctx.get("state") or {}).get("_clarification_cache")

if cached and cached["thread_tool_key"] == cache_key:
    logger.info("[query_clarification] CACHE HIT — skip _analyze_clarification()")
    paradigm_list = cached["paradigm_list"]
    clarify_knowledge = cached["clarify_knowledge"]
else:
    logger.info("[query_clarification] CACHE MISS — calling _analyze_clarification()")
    paradigm_list, clarify_knowledge = _analyze_clarification(
        query, scope_code, structured_input, is_compute=is_compute
    )
    _state = ctx.get("state")
    if _state is not None:
        _state["_clarification_cache"] = ClarificationCache(
            thread_tool_key=cache_key,
            paradigm_list=paradigm_list,
            clarify_knowledge=clarify_knowledge,
        )
```

在 `_format_clarification` 调用结束后，追加缓存清除：

```python
# 恢复流程完成，清除一次性缓存
_state = ctx.get("state")
if _state is not None:
    _state.pop("_clarification_cache", None)
```

### 5.4 ctx 中 state 的传入

需在 `execution/node.py`（或等价调用处）将 LangGraph `state` 注入 `ctx`：

```python
ctx["state"] = state          # 传入当前图状态引用（可变引用，写入即生效）
ctx["thread_id"] = thread_id  # 传入 thread_id 用于 cache key
plugin.before_call_back(tool_name, tool_params, ctx)
```

> 如果 `ctx` 已经持有 `state` 和 `thread_id`，此步骤可跳过。

## 7. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| `ctx` 未持有 `state` / `thread_id`，需改调用处 | 中 | 低（改动小） | 步骤 1 提前确认 |
| 缓存 key 碰撞（不同语义 query 同 hash） | 极低 | 低（降级为重跑 SDK） | MD5 12 位已足够区分 |
| 缓存未清除导致下次查询误命中 | 低 | 中（返回旧 paradigm_list） | 恢复流程末尾强制清除 |
| 多线程并发同 thread_id 写入缓存 | 低 | 低（LangGraph 单 thread 串行） | 无需额外锁 |

## 8. 验收标准

实施完成后，执行完整澄清流程，日志应符合以下模式：

```
# 首次执行（中断前）—— CACHE MISS，正常调用 SDK
[query_clarification] CACHE MISS — calling _analyze_clarification()
[query_clarification] PRE-INTERRUPT#2 about to call interrupt() ...

# 恢复后 —— CACHE HIT，不再调用 SDK
[query_clarification] CACHE HIT — skip _analyze_clarification()
[query_clarification] RESUME resume_value type=dict value=...
[query_clarification] RESUME parsed: paradigm_list_count=N knowledge_len=N
[query_clarification] format_clarification args: ...
[query_clarification] format_clarification result: tool_params=...
```

**关键指标：** 恢复后日志中不再出现 `CACHE MISS` 或 `calling _analyze_clarification`，视为优化成功，澄清交互耗时减少约 10–15 秒。
