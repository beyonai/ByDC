> 版本：v1.0 | 日期：2026-04-18

## 1. 现状梳理

### 1.1 整体调用链路

> **现状说明**：以下描述改造前的调用链路。当前系统通过 `DATACLOUD_ONTOLOGY_LOAD_MODE` 环境变量在两种模式之间二选一，两种模式能力不互补。新方案统一以 `query_*` / `compute_*` 为入口，通过 `before_callback` 内部路由，不再依赖模式切换（详见第 2 章）。

```
用户问题
  ↓
DataCloudWorker._handle_message()          [byclaw-data/worker.py]
  ↓
create_agent() → execution_node()          [datacloud-analysis/agent.py]
  ↓  (ReAct 循环，LLM 选工具)
query_{code} / compute_{code}              [DATACLOUD_ONTOLOGY_LOAD_MODE=ontology_query]
   或 data_query_{code}                    [DATACLOUD_ONTOLOGY_LOAD_MODE=db_query]
  ↓
QueryExecutor / ComputeExecutor            [datacloud-data-sdk]
   或 PlanAgent → QueryExecutionPlan       [datacloud-data-sdk]
```

### 1.2 工具挂载逻辑（init_agent_conf.py）

**ontology_query 模式**：注入 `query_{code}` / `compute_{code}` 虚拟动作工具。
- 工具通过 `inject_virtual_actions(loader)` → `ActionToolGenerator.generate_tools()` 挂载，每个 OWL 对象/视图各一对。
- 描述由 `virtual_action/generator.py` 的 `build_query_description()` / `build_compute_description()` **动态生成**，内容极为丰富：

  ```
  {对象描述}

  按条件查询{对象名}明细。**field 统一使用字段编码（field_code）**；支持字段过滤、排序、分页；不支持聚合统计。

  **何时使用**：查看具体记录列表时使用；不适用于统计汇总，如需统计请用 compute 动作。

  **可用字段**：
  | 字段编码 | 中文名 | 角色 | 类型 | 可过滤 | 可分组 | 可聚合 | 特殊说明 |
  | --- | --- | --- | --- | --- | --- | --- | --- |
  | field_a   | 字段A | dimension | name       | eq/in            | direct | -          | |
  | amount    | 金额  | measure   | raw_number | gt/gte/lt/lte/eq | -      | sum/avg    | |
  | period    | 账期  | dimension | period     | eq/between       | direct | -          | 必须过滤 |
  ...

  **常见错误**：
  - 使用了字段不支持的 op 操作符
  - field 填了字段中文名而非字段编码（field_code）
  ```

- `compute_{code}` 描述同样包含**维度+度量全部字段的能力矩阵**，并额外说明拍照指标限制、`metrics.agg` 键名规范等。
- 结论：**Agent 选工具阶段已能完整看到字段列表及其操作能力**，ontology_query 模式下字段感知不是问题。

**db_query 模式**：注入 `data_query_{code}` 工具。

- schema 为 `{query: str, contextKnowledge: str}`，接受自然语言直通。
- 工具描述由 `_build_single_db_query_tool()` 生成：`f"数据查询工具: {label}。{desc_suffix}"`，**仅含资源名称和简短说明，不含字段信息**。
- 字段信息依赖 `ObjectViewPayload` 序列化后注入给 `PlanAgent`，Agent 选工具阶段**无法感知字段细节**。

**两种模式的工具描述对比**：

| 维度 | ontology_query（query_* / compute_*） | db_query（data_query_*） |
|------|---------------------------------------|--------------------------|
| 工具描述 | 完整字段能力矩阵（Markdown 表格） | 仅含资源名+简短说明 |
| Agent 选工具时可见信息 | 字段编码、中文名、角色、过滤/分组/聚合操作符 | 对象名称 + 资源描述 |
| 字段映射时机 | 选工具阶段（工具描述即字段规格） | 执行阶段（PlanAgent 内部推理） |
| 选错对象的风险 | 低（字段差异在描述中可见） | 较高（只能靠对象名语义判断） |

### 1.3 query_{code} / compute_{code} 执行路径

```
query_{code}(query, ...)
  ↓
query_clarification_plugin.before_call_back()   [tool_hook_plugin]
  ↓
  ├── 剥除元字段（intent_reason / extraction_confidence / ambiguous_params）
  ├── ambiguous_params 为空 → return None（直接放行）
  └── ambiguous_params 非空 → analyze_query_clarification()
        ↓
        ├── needs_clarification=False → 注入 contextKnowledge，继续查询
        └── needs_clarification=True  → langgraph interrupt，等待用户选择
  ↓
QueryExecutor.execute() / ComputeExecutor.execute()
  → 生成单表 SQL → 执行 → 返回 records
```

**旧方案中断恢复防重触发机制**

旧方案 `interrupt` 后，`react_loop.py` 在 resume 时重放 `react_pending_tool_calls`，重放前**自动剥除三个元字段**：

```python
# react_loop.py — resume 重放时剥除元字段
_tc_args.pop("ambiguous_params", None)
_tc_args.pop("intent_reason", None)
_tc_args.pop("extraction_confidence", None)
```

`before_callback` 收到不含 `ambiguous_params` 的参数后，判断 `ambiguous_params` 为空 → 直接放行，不再触发 `analyze_query_clarification()`，从而避免了重复中断。

**新方案移除了 `ambiguous_params` 依赖**，改由 `before_callback` 主动检测歧义，因此需要新的防重触发机制——在 resume 重放前将用户选择结果写回 `tool_params`，使通道 1 直接命中 1:1 映射，跳过歧义流程（详见 2.2.3）。

---

**现状问题一：单一 analyze_query_clarification 开销高**

`analyze_query_clarification` 是一个重型全流程函数，包含三次 LLM 调用：

| 步骤 | 调用 | 耗时估算 |
|------|------|---------|
| Step 1 | `expand_query`（NL → NatQuery，LLM tool call）| ~1-2s |
| Step 2 | `build_paradigm_resolution_state`（术语召回，向量检索）| ~0.5-1s |
| Step 3 | `llm_confirm`（LLM 审查召回结果）| ~1-2s |

即使是无歧义的简单查询（如"查询本月订单数量"），也会跑完全部三步，**总耗时 2-5s**，用户体验差。

**现状问题二：无复杂度判断，无法路由**

`query_{code}` 和 `compute_{code}` 只能走 OQL 单表路径，遇到以下问题**完全无法处理**：
- 跨对象关联（如"亩产效益后30%地块上的企业"）
- 百分比截断（PERCENT_RANK / 窗口函数）
- 同比环比等时间计算
- 多步依赖（先求阈值，再过滤）

**现状问题三：模式二选一，能力不互补**

`DATACLOUD_ONTOLOGY_LOAD_MODE` 是二值环境变量，同一部署中只能选一种模式：
- `ontology_query`：精度高，但复杂查询无法表达
- `db_query`：全能，但 Schema Linking 依赖 LLM 推理，简单查询成本高

两种能力**无法在同一会话中互补**。

### 1.4 data_query_{code} 执行路径

```
data_query_{code}(query, contextKnowledge)
  ↓
view.query() / object.query()
  ↓
LangGraphPlanGenerator.generate()
  ↓
PlanAgent（LLM + SYSTEM_PROMPT）
  → 生成 QueryExecutionPlan JSON
    ├── steps: [{type: "SQL", sqlTemplate: "WITH ... SELECT ..."}]
    └── aggregation: {strategy: "DIRECT" | "SQLITE_MEM"}
  ↓
执行引擎（跑 SQL / 调 API / SQLITE_MEM 内存聚合）
```

**现状优点**：
- SYSTEM_PROMPT 已内置完整的跨对象/窗口函数/百分比截断规则（150行）
- 支持 CTE、PERCENT_RANK、CASE WHEN 等复杂 SQL 生成
- SQLITE_MEM 策略支持跨数据源查询

**现状不足**：
- SQL 执行错误（如列名歧义）仅靠静态 `PlanValidator` 检测，不反馈实际执行报错
- Schema 输入为 ObjectViewPayload JSON，无字段值样本（M-Schema 缺失），字段映射依赖 description 质量
- `data_query_*` 工具描述同样不含字段信息，LLM 选错视图时整个计划作废

### 1.5 analyze_query_clarification 内部原子能力

```
analyze_query_clarification()
  ├── expand_query()                     → NatQuery（结构化，含 select/where/group_by）
  ├── natquery_to_five_stage()           → 五段式 keys（轻量，纯逻辑）
  ├── build_paradigm_resolution_state()  → 术语召回（向量检索）
  └── llm_confirm()                      → ConfirmedQuery（LLM 判断是否歧义）
```

这四个原子方法**可以独立调用**，为渐进式判断提供基础。

### 1.6 工具入参 / 出参规格

> ⚠️ **基线说明**：本节记录**改造前**的入参规格，反映现状代码实现。新方案在此基础上新增 `query`（必填）和 `complex_conditions`（溢出区），并移除三个元字段（`intent_reason` / `extraction_confidence` / `ambiguous_params`），完整新参数协议详见 2.2.1。

#### 1.6.1 query_{code}（明细查询）

**入参**（全部可选，无必填）：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `select` | `array[string]` | — | 返回字段列表（field_code），为空时返回全部非关联字段 |
| `filters` | `array[FilterItem]` | — | 过滤条件列表，多条件默认 AND |
| `filter_relation` | `"AND"` \| `"OR"` | `"AND"` | 过滤条件连接方式；含账期强制约束时禁止 OR |
| `order_by` | `array[{field, direction}]` | — | 排序规则，`direction` 为 `"asc"` / `"desc"` |
| `limit` | `integer` [1, 1000] | `100` | 返回行数上限 |
| `offset` | `integer` ≥ 0 | `0` | 分页偏移量 |

`FilterItem` 结构：
```json
{"field": "field_code（字段编码）", "op": "eq|in|gt|gte|lt|lte|between|is_null|is_not_null", "value": "字面量或数组"}
```

**入参 JSON 示例**：

```json
// 示例：查询本月新增、营收大于100万、属于"制造业"或"农业"的企业，
//       返回企业名称和营收，按营收降序，取前20条
{
  "select": ["enterprise_name", "revenue", "created_date", "industry"],
  "filters": [
    {"field": "created_date", "op": "between", "value": ["2026-04-01", "2026-04-30"]},
    {"field": "revenue",      "op": "gt",      "value": 1000000},
    {"field": "industry",     "op": "in",      "value": ["manufacturing", "agriculture"]}
  ],
  "filter_relation": "AND",
  "order_by": [{"field": "revenue", "direction": "desc"}],
  "limit": 20,
  "offset": 0
}
```

**出参**（SDK 层，service 层再包装 `{code, message, data}`）：

```json
// 正常返回
{
  "result_type": "normal",
  "records": [{"field_code": "value", ...}, ...],
  "total": 123,
  "meta": {
    "objectId": "对象编码",
    "objectName": "对象名称",
    "columns": [{"name": "field_code", "label": "字段名", "type": "string|number|date"}, ...],
    "total": 123
  },
  "trace": {"request_id": "uuid", "question": "...", "object_id": "..."},

  // 数据量超过阈值（默认10条）时额外返回：
  "pagination": {"page": 1, "page_size": 5, "total": 123, "total_pages": 25, "has_next": true},
  "file": {"file_url": "/path/to/export.csv", "file_id": "uuid"},
  "overflow_notice": "数据量较大（共123条），此处仅返回前5条预览..."
}

// 无法回答（CannotAnswerError）
{"result_type": "rejected", "overflow_notice": "错误信息", "trace": {...}}

// 术语歧义（TermAmbiguousError）
{"result_type": "ask_user", "overflow_notice": "错误信息", "trace": {...}}
```

#### 1.6.2 compute_{code}（分组统计）

**入参**（`metrics` 必填，其余可选）：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `metrics` | `array[MetricItem]` | **是** | 统计指标，至少一项 |
| `dimensions` | `array[string \| DimensionItem]` | 否 | 分组维度；字符串简写等价于 `group_op:"self"` |
| `filters` | `array[FilterItem]` | 否 | 行级过滤（WHERE），同 `query_{code}` 的 filters |
| `having` | `array[HavingItem]` | 否 | 聚合后过滤（HAVING），`field` 必须是 `metrics.as` 别名 |
| `order_by` | `array[{field, direction}]` | 否 | `field` 可为 `metrics.as` 别名或维度 field_code |
| `limit` | `integer` [1, 1000] | 否 | 默认 100 |
| `filter_relation` | `"AND"` \| `"OR"` | 否 | 默认 `"AND"` |

---

`MetricItem` 结构（五种形态，`field` 与 `expr` 二选一，`filters` 可选叠加）：

```json
// ① 行数统计（无需 field / expr）
//    → COUNT(*)
{"agg": "count_all", "as": "enterprise_count"}

// ② 普通字段聚合
//    → SUM(revenue)
{"field": "revenue", "agg": "sum", "as": "total_revenue"}

// ③ 条件字段聚合（指标级 filters → CASE WHEN）
//    → SUM(CASE WHEN region = 'north' THEN revenue ELSE 0 END)
{
  "field": "revenue",
  "filters": [{"field": "region", "op": "eq", "value": "north"}],
  "agg": "sum",
  "as": "north_revenue"
}

// ④ 公式聚合（字段本身是表达式）
//    → AVG(revenue - cost)
{"expr": "revenue - cost", "agg": "avg", "as": "avg_profit"}

// ⑤ 条件公式聚合（expr + filters 叠加）
//    → SUM(CASE WHEN enterprise_level = 'leading' THEN (revenue - cost) ELSE 0 END)
{
  "expr": "revenue - cost",
  "filters": [{"field": "enterprise_level", "op": "eq", "value": "leading"}],
  "agg": "sum",
  "as": "leading_profit"
}
```

---

`DimensionItem` 结构（支持字符串简写）：

```json
// 字符串简写：直接分组，等价于 group_op:"self"
//    → SELECT industry  /  GROUP BY industry
"industry"

// 对象：时间粒度分组
//    → SELECT DATE_FORMAT(created_date,'%Y-%m')  /  GROUP BY DATE_FORMAT(...)
{"field": "created_date", "group_op": "year|month|quarter|day"}

// 对象：range 分桶（需带 buckets）
//    → SELECT CASE WHEN revenue<500000 THEN '低' ... END  /  GROUP BY CASE WHEN ...
{"field": "revenue", "group_op": "range", "buckets": [{"from": null, "to": 500000, "label": "低"}, ...]}
```

> `dimensions` 同时决定 SQL 的 `SELECT` 维度列和 `GROUP BY` 子句，两者表达式完全一致，无需单独维护 `GROUP BY`。

---

**入参 JSON 示例**：

```json
// 示例1：普通字段聚合 + 行数统计
//        按行业、地区统计企业数和总营收，过滤营收>0，按总营收降序取前50
//        → SELECT industry, region, COUNT(*), SUM(revenue) FROM t
//             WHERE revenue > 0 GROUP BY industry, region ORDER BY SUM(revenue) DESC LIMIT 50
{
  "dimensions": ["industry", "region"],
  "metrics": [
    {"agg": "count_all", "as": "enterprise_count"},
    {"field": "revenue", "agg": "sum", "as": "total_revenue"}
  ],
  "filters": [
    {"field": "revenue", "op": "gt", "value": 0}
  ],
  "order_by": [{"field": "total_revenue", "direction": "desc"}],
  "limit": 50
}

// 示例2：条件字段聚合（CASE WHEN）
//        按行业统计总营收，同时单独统计龙头企业营收占比
//        → SUM(revenue) + SUM(CASE WHEN level='leading' THEN revenue ELSE 0 END)
{
  "dimensions": ["industry"],
  "metrics": [
    {"agg": "count_all", "as": "enterprise_count"},
    {"field": "revenue", "agg": "sum", "as": "total_revenue"},
    {
      "field": "revenue",
      "filters": [{"field": "enterprise_level", "op": "eq", "value": "leading"}],
      "agg": "sum",
      "as": "leading_revenue"
    }
  ],
  "order_by": [{"field": "total_revenue", "direction": "desc"}],
  "limit": 50
}

// 示例3：公式聚合 + 条件公式聚合（expr）
//        按行业统计平均利润，以及龙头/骨干企业的总利润
//        → AVG(revenue-cost) + SUM(CASE WHEN level IN(...) THEN (revenue-cost) ELSE 0 END)
{
  "dimensions": ["industry", "region"],
  "metrics": [
    {"agg": "count_all", "as": "enterprise_count"},
    {"field": "revenue", "agg": "sum", "as": "total_revenue"},
    {"expr": "revenue - cost", "agg": "avg", "as": "avg_profit"},
    {
      "expr": "revenue - cost",
      "filters": [{"field": "enterprise_level", "op": "in", "value": ["leading", "backbone"]}],
      "agg": "sum",
      "as": "key_enterprise_profit"
    }
  ],
  "filters": [
    {"field": "created_date", "op": "between", "value": ["2026-01-01", "2026-03-31"]}
  ],
  "having": [
    {"field": "total_revenue", "op": "gt", "value": 1000000}
  ],
  "order_by": [{"field": "avg_profit", "direction": "desc"}],
  "limit": 50
}

// 示例4：时间分组 + 营收分桶 + having
//        按月份 + 营收区间统计企业数和各桶平均利润，过滤不足5家的分组
//        → GROUP BY DATE_FORMAT(created_date,'%Y-%m'), CASE WHEN revenue<500000...
{
  "dimensions": [
    {"field": "created_date", "group_op": "month"},
    {
      "field": "revenue",
      "group_op": "range",
      "buckets": [
        {"from": null,    "to": 500000,  "label": "低收入"},
        {"from": 500000,  "to": 2000000, "label": "中收入"},
        {"from": 2000000, "to": null,    "label": "高收入"}
      ]
    }
  ],
  "metrics": [
    {"agg": "count_all", "as": "enterprise_count"},
    {"field": "revenue", "agg": "avg", "as": "avg_revenue"},
    {"expr": "revenue - cost", "agg": "avg", "as": "avg_profit"}
  ],
  "having": [
    {"field": "enterprise_count", "op": "gte", "value": 5}
  ],
  "order_by": [{"field": "avg_profit", "direction": "desc"}],
  "limit": 100
}
```

**出参**：与 `query_{code}` 结构相同，`records` 中字段名为维度的 field_code 和 metrics 的 `as` 别名。

### 1.7 现状核心问题汇总

| # | 问题 | 影响 |
|---|------|------|
| P1 | `data_query_*` 工具描述不含字段信息，Agent 选工具时无法感知字段细节（ontology_query 模式已有完整字段描述，此问题仅存在于 db_query 模式） | db_query 模式下选错对象/视图 |
| P2 | 所有查询都跑全量澄清流程，简单查询浪费 2-5s | 响应慢，用户体验差 |
| P3 | 无复杂度分类，复杂查询在 OQL 模式下静默失败 | 功能缺口 |
| P4 | 二值模式切换，简单精确 vs 复杂全能不能共存 | 部署灵活性差 |
| P5 | SQL 执行错误不回流重试 | data_query 失败率偏高 |

## 2. 概要设计

### 2.1 设计原则

**工具即对象/视图**：系统中的每个 OWL 对象或预定义视图，对应一对工具 `query_{code}` / `compute_{code}`，其中 `{code}` 是该对象或视图在本体中的**唯一编码**。Agent 选工具即选定了操作目标，工具名本身携带了完整的对象定位信息，无需额外的对象选择步骤。

每一层只做一件事，决策权归属清晰：

| 层次 | 决策主体 | 决策内容 |
|------|----------|----------|
| Agent 选工具 | LLM | 确定目标对象/视图（靠工具描述中的字段能力表），`{code}` 为对象/视图唯一编码 |
| Agent 选工具 | LLM | 确定意图：查明细 → `query_{code}`；做统计 → `compute_{code}` |
| Agent 填参 | LLM | 用自然语言表达意图（字段名、过滤值填中文术语，**不做 code 转换**） |
| before_callback Step 1 | **规则引擎** | `complex_conditions` 非空 → 标记 COMPLEX（< 10ms，无 LLM） |
| before_callback Step 2 通道 1 | **规则引擎** | 参数术语与字段 Schema 直接比对，能否 1:1 映射（< 100ms，无 LLM） |
| before_callback Step 2 通道 2 | 规则 + 轻量 LLM | query 文本关键词补充检测，仅通道 1 存在歧义项时触发（~1s） |
| before_callback Step 2 结果 | 规则 + 按需 LLM | CLEAR → 术语替换为 code；AUTO_DISAMBIG → 自动选定；NEED_CONFIRM → 中断追问 |
| 执行层 | Step 1 + Step 2 组合结果 | SIMPLE → OQL 直查；COMPLEX → redirect → data_query（PlanAgent） |

**两条核心哲学：**

1. **把"能不能用 OQL 表达"的判断，从 LLM 的主观认知转移到参数结构本身**——通过 `complex_conditions` 溢出字段，用规则而非 LLM 推理来识别复杂查询。

2. **把术语→code 的转换职责，从 LLM 转移到系统内部**——LLM 只负责表达意图，系统在 `before_callback` 内部做映射和歧义检测，确保转换错误可被捕获，而非静默执行。

> **工具描述字段表继续保留**：`query_*` / `compute_*` 工具描述中的字段能力矩阵（字段编码、中文名、类型、过滤/分组/聚合操作符）供 Agent **选工具阶段**参考，内容不做删减。调整点仅在于**填参策略**——LLM 选对工具后，不再被要求强制将意图转为 field_code，而是以自然语言填写；术语 → code 的映射由 `before_callback` Step 2 内部完成（见 2.2.2）。

---

### 2.2 流程设计

整体调用链以 `query_{code}` / `compute_{code}` 为**统一入口**，通过 `before_callback` 串联复杂度识别与歧义识别两道判断，最终路由到对应执行路径。

```
用户问题
    ↓
Agent 工具选择（LLM）
    ├── 靠工具描述中的字段能力表 → 确定目标对象/视图
    ├── 查明细 → query_{code}(query, select?, filters?, complex_conditions?, ...)
    └── 做统计 → compute_{code}(query, dimensions?, metrics?, filters?, complex_conditions?, ...)
    ↓
before_callback（两步串行，均在工具调用前执行）
    │
    ├─ Step 1：复杂度识别（< 10ms，纯规则）
    │         complex_conditions 非空 → is_complex = True
    │         complex_conditions 为空 → is_complex = False
    │         ⚠️ 仅做标记，不立即路由，等待 Step 2 完成
    │
    └─ Step 2：歧义识别（约 1s，轻量 LLM，简单/复杂查询均执行）
              │（复杂查询同样存在术语歧义，必须先澄清才能交给 PlanAgent）
              │
              ├── CLEAR（参数无歧义）
              │       is_complex=False → OQL 直查（QueryExecutor）→ 结果
              │       is_complex=True  → redirect → data_query_{code} → PlanAgent → 结果
              │
              ├── AUTO_DISAMBIG（可自动消歧）
              │       补全 filters 参数（自然语言值 → field_code）
              │       is_complex=False → OQL 直查 → 结果
              │       is_complex=True  → redirect → data_query_{code} → PlanAgent → 结果
              │
              └── NEED_CONFIRM（需用户确认）
                      ↓ interrupt() —— State 保存 + 节点挂起
                      用户选择 paradigm
                      ↓ resume —— 节点从顶部重新执行，重放工具调用
                      apply_resume → 用户选择写回 tool_params（自然语言值 → code）
                      before_callback 重放：Step 1 is_complex 重新计算，Step 2 通道1直接 CLEAR
                      is_complex=False → OQL 直查 → 结果
                      is_complex=True  → redirect → data_query_{code} → PlanAgent → 结果
```

**6 种路径组合**：

| 复杂度（Step 1） | 歧义状态（Step 2） | 最终执行路径 |
|-----------------|------------------|-------------|
| SIMPLE | CLEAR | OQL 直查 |
| SIMPLE | AUTO_DISAMBIG | 自动补全参数 → OQL 直查 |
| SIMPLE | NEED_CONFIRM | interrupt → 恢复重放 → OQL 直查 |
| COMPLEX | CLEAR | redirect → data_query（PlanAgent）|
| COMPLEX | AUTO_DISAMBIG | 自动补全参数 → redirect → data_query |
| COMPLEX | NEED_CONFIRM | interrupt → 恢复重放 → redirect → data_query |

> ⚠️ **COMPLEX + NEED_CONFIRM 恢复后路由说明**：用户消歧选择写回 `tool_params`（如将 `{field:"营收"}` 更新为 `{field:"revenue"}`）后，`complex_conditions` 仍然保留——因为复杂度条件本身（"亩产效益后30%"）不因字段歧义消除而消失。Step 1 重新计算仍得 `is_complex=True`，恢复后正确路由到 `data_query`，这是**预期行为**，不是 bug。

#### 2.2.1 复杂问题识别机制

##### 问题根因：旧参数协议依赖 LLM 自报

旧协议中有三个"元字段"由 LLM 在填参时一并自报：

```
intent_reason          → LLM 解释选择理由（主观，不可验证）
extraction_confidence  → LLM 自评置信度（不可靠，高置信≠正确）
ambiguous_params       → LLM 自报歧义字段（漏报率高）
```

依赖 LLM 自报的问题在于：LLM 不知道自己"不知道"，复杂查询在单表路径下静默执行并返回错误结果，而非在入口处被拦截。

##### 新参数协议：用参数结构承载复杂度信号

**新增两个字段，移除三个元字段：**

| 变更 | 字段 | 说明 |
|------|------|------|
| ➕ 新增 | `query: str` | 必填，用户原始问题的完整自然语言描述 |
| ➕ 新增 | `complex_conditions: list[str]` | 溢出区，存放 filter.value 无法表达为字面常量的条件 |
| ➖ 移除 | `intent_reason` | 改由 `query` 字段承担意图表达 |
| ➖ 移除 | `extraction_confidence` | 改由 before_callback 规则主动推断 |
| ➖ 移除 | `ambiguous_params` | 改由 before_callback Step 2 主动检测 |

##### 核心判据：filter.value 能否为字面常量

OQL 单表查询的 filter 结构如下：

```json
{"field": "yield_efficiency", "op": "lt", "value": 500}
```

`value` 必须是一个**查询前已知的字面常量**。当过滤条件的值需要先从数据中动态计算时，LLM 无法填写 `value`，这就是"溢出"：

```
已知（字面常量）→ 填入 filters
  示例：效能等于"中效" / 创建时间在本月 / 营收大于100万

不知道（需先从数据计算）→ 填入 complex_conditions（自然语言描述该条件）
  示例：亩产效益后30% / 营收高于行业平均 / 排名前10名 / 同比增长超过20%
```

`select / dimensions / metrics / order_by` 的值（字段编码、排序方向、聚合函数）在查询时均是已知的，**溢出只发生在 filters**。

##### 识别规则：一条规则，确定性 100%

```python
is_complex = bool(tool_params.get("complex_conditions"))
```

`complex_conditions` 非空 → 标记为 **COMPLEX**（需 PlanAgent 处理）；为空 → 标记为 **SIMPLE**（走 OQL 单表）。

LLM 在填参时只需回答一个问题：**"这个条件的 value，我现在知道是多少吗？"** 不需要理解"窗口函数"或"百分位"的概念。

**参数示例（复杂查询）：**

```json
{
  "query": "找出亩产效益后30%的地块，查询这些地块上的中、低效能的企业清单",
  "select": ["enterprise_name", "efficiency_level", "yield_efficiency"],
  "filters": [
    {"field": "efficiency_level", "op": "in", "value": ["中效", "低效"]}
  ],
  "complex_conditions": ["亩产效益后30%的地块"],
  "order_by": [{"field": "yield_efficiency", "direction": "asc"}],
  "limit": 100
}
```

- `efficiency_level` 的过滤值（中效/低效）是字面常量 → 填入标准 `filters`
- 亩产效益后30% 的阈值需先从数据计算 → 填入 `complex_conditions`
- `select / order_by` 无溢出，照常填写

#### 2.2.2 歧义问题识别机制

##### 参数填写策略：鼓励自然语言，禁止强行转码

**旧策略的问题**：工具描述要求 LLM 将用户意图中的中文术语强行转换为 `field_code` 和枚举 code 后再填入参数。这带来两个隐患：

1. **转错**：LLM 按语义猜测 code，`"营收"` 可能错填为 `net_revenue`（净营收），而实际应为 `gmv`
2. **无法检测歧义**：一旦 LLM 已经完成转换，系统看到的是确定的 code，无从知道原始意图是否有歧义

**新策略**：LLM **不做 code 转换**，直接填写它理解的中文业务术语或自然语言词语；系统在 `before_callback` 内部完成术语 → code 的映射，并在映射过程中检测歧义。

```
旧策略（LLM 强制转码）：
  用户："查询高营收的龙头企业"
  LLM 填参：filters = [{field: "revenue", op: "gt", value: 1000000}]
                                ↑ 猜测转换，可能错

新策略（LLM 填自然语言，系统映射）：
  用户："查询高营收的龙头企业"
  LLM 填参：filters = [{field: "营收", op: "gt", value: "高"}]
                              ↑ 填业务术语    ↑ 填自然语言描述
  → before_callback 做映射检查，发现 "营收" 对应 revenue/net_revenue/gmv 三个候选 → 歧义
```

LLM 的职责从"转码"变为"表达意图"：填写它**想要什么**，不需要知道系统内部的 field_code 是什么。

---

##### 歧义的本质：术语与字段 Schema 无法 1:1 映射

歧义来自**参数中的自然语言术语（字段名、过滤值、select 项）与工具字段 Schema 中已知的 field_code / 枚举 code 之间，存在 0:1 或 1:N 的映射关系**：

| 映射关系 | 含义 | 处理 |
|---------|------|------|
| 1:1 精确匹配 | 术语唯一对应已知 field_code 或枚举值 | CLEAR，直接替换为 code |
| 1:N 多候选 | 术语对应 2+ 个候选 | 进入歧义处理 |
| 0:1 无对应 | 术语在字段 Schema 中无任何匹配 | 进入歧义处理 |

---

##### 识别流程：双路并行检测，合并后判断

歧义来源有两条并行通道：**参数直接检测**和 **query 文本检测**，合并结果统一进入歧义处理阶段。

```
Step 2 歧义识别

┌─ 通道 1：参数直接检测（< 100ms，纯规则）──────────────────────────────┐
│  遍历 LLM 填写的参数术语：                                             │
│    filters.field  → 映射到 field_code（1:1 / 1:N / 无对应）           │
│    filters.value  → 映射到枚举 code（1:1 / 1:N / 无对应）             │
│    select 项      → 映射到 field_code                                  │
│    dimensions/metrics 项 → 映射到 field_code                           │
│  优先检测，能快速发现参数中的明显歧义                                   │
└────────────────────────────────────────────────────────────────────────┘
          ↓ 有歧义项                    ↓ 全部 1:1
┌─ 通道 2：query 文本检测（~1s，1次 LLM 调用）──────────────────────────┐
│  expand_query(query) → NatQuery（含 select/where/group_by 等结构）     │
│  natquery_to_five_stage(NatQuery) → 五段式关键词（纯规则，无 LLM）     │
│  关键词 → 映射到 field_code / 枚举值                                   │
│  补充捕获参数中未显式填写、但 query 中隐含的歧义概念                    │
└────────────────────────────────────────────────────────────────────────┘
          ↓
    合并两个通道的映射结果
          ↓
    ┌── 全部 1:1 映射，且无时间歧义 → CLEAR
    │     直接将术语替换为对应 code，继续执行
    │
    ├── 存在 1:N 或 0:1 → 进入阶段 C：paradigm 召回
    │     build_paradigm_resolution_state() 召回候选 paradigm
    │       ├── 多候选，LLM 能从 query 上下文唯一确定
    │       │     → AUTO_DISAMBIG：自动选定 code，不打断用户
    │       └── 多候选，无法自动确定
    │             → NEED_CONFIRM：触发 interrupt 追问
    │
    └── complex_conditions 中的术语有多义 → 同上进入阶段 C
```

> **为什么参数直接检测优先于 query 文本检测？**
> 参数是 LLM 已经明确表达意图的地方，直接检测无需 LLM 调用（< 100ms），可以在无歧义时完全跳过 expand_query 的 1s 开销。只有参数中存在无法确定的项时，才需要用 query 文本做补充分析。

---

##### 触发歧义处理（阶段 C）的规则

| 规则 | 检测来源 | 触发条件 | 示例 |
|------|---------|---------|------|
| A1 | 参数 / query | 一个术语对应 2+ 个候选 field_code | `"营收"` → `revenue / net_revenue / gmv` |
| A2 | 参数 `filters.value` | enum 字段的过滤值是自然语言，无法映射到已知枚举 code | `"高效"` → `enterprise_level` 枚举值不明确 |
| A3 | 参数 / query | 术语在字段 Schema 中无任何匹配 | `"收益能力"` → 不是任何 field_code |
| A4 | 参数 `filters.value` / query | 时间范围使用相对表达 | `"最近"` / `"今年"` / `"上季度"` → 起止日期不确定 |
| A5 | `complex_conditions` | 条件中的术语本身有多种语义 | `"高亩产地块"` → 绝对值阈值还是相对排名？ |

**直接 CLEAR（跳过阶段 C）的条件：**
- 参数中所有术语均 1:1 映射到已知 field_code 或枚举 code
- 时间过滤值均为绝对日期（如 `"2026-04-01"`），无相对时间表达
- `query` 文本检测补充的关键词同样全部 1:1 映射（参数检测 CLEAR 时才触发此步）

> `llm_confirm` 仅在 NEED_CONFIRM 分支调用，CLEAR 和 AUTO_DISAMBIG 路径均不调用，节省 1-2s。

##### 参数填写示例对比

**示例：查询高营收的龙头企业**

```json
// ❌ 旧策略（LLM 强制转码，风险高）
{
  "query": "查询高营收的龙头企业",
  "filters": [
    {"field": "revenue",          "op": "gt",  "value": 1000000},

// ✅ 新策略（LLM 填自然语言，系统检测歧义）
{
```

#### 2.2.3 问题中断及恢复机制

##### 前提：三类工具均为虚拟注入，不能互相直接调用

`query_{code}`、`compute_{code}`、`data_query_{code}` 都是通过 `inject_virtual_actions()` 动态生成后注册到同一个 `tools_map` 中的**虚拟工具**——它们是注册在映射表里的异步可调用对象，而非真正的类方法。

因此：

- `query_{code}` 的执行体**无法直接调用** `data_query_{code}`，两者之间不存在函数调用关系。
- 路由发生在 **`before_callback` 层**：`tool_wrapper.py` 的 `dispatch_tool()` 在执行真正的工具函数前，先运行 `hook_manager.run_before(ctx)`；如果 hook 返回 `redirect` 决策，`dispatch_tool()` 便从 `tools_map` 查找目标工具并直接调用，**原工具函数不再执行**。

```
tools_map = {
    "query_Enterprise":      <coroutine>,   # 虚拟注入
    "compute_Enterprise":    <coroutine>,   # 虚拟注入
    "data_query_Enterprise": <coroutine>,   # 虚拟注入（db_query 模式）
    ...
}

dispatch_tool("query_Enterprise", args)
  ↓
hook_manager.run_before(ctx)           # before_callback 运行
  ├── COMPLEX → HookDecision(action="redirect", tool="data_query_Enterprise", params=...)
  │      ↓
  │   tools_map["data_query_Enterprise"](**params)   # 从 tools_map 查找并直接调用
  │
  └── SIMPLE → 继续执行 tools_map["query_Enterprise"](**args)
```

> **`redirect` 是需要新增的 HookDecision 类型**：当前 `tool_wrapper.py` 只处理 `short_circuit` 和 `fail`，需扩展以支持 `redirect`。`redirect` 后不再触发 `before_callback`，避免递归。

---

##### 中断粒度：工具调用级重放，而非代码行级

LangGraph 的 `interrupt()` 在本项目中采用**方案 B（State 保存 + 节点重放）**，而非协程帧级挂起。

**中断时**（`interrupt()` 抛出 `GraphBubbleUp`）：

```python
# react_loop.py — 捕获 GraphBubbleUp，保存现场到 State
except GraphBubbleUp:
    _pending = list(ai_msg.tool_calls[tc_idx:])      # 从当前工具调用开始，包括后续未执行的
    state["react_messages"]          = _serialize_messages(messages)
    state["react_pending_tool_calls"] = _pending      # 含原始工具名 + 完整参数
    state["react_round_idx"]         = round_idx
    state["react_last_query_data"]   = _last_query_data
    raise                                              # 继续向上抛，LangGraph 挂起节点
```

**恢复时**（用户完成选择，LangGraph 重新触发节点）：

```python
# react_loop.py — 节点从顶部重新执行，检测到 State 中有保存的现场
_state_msgs = state.get("react_messages")
if _state_msgs is not None:
    # 重建上下文，对 pending_tool_calls 逐个重放
    for tc in pending_tool_calls:
        _tc_args = dict(tc.get("args") or {})
        # 剥除旧版元字段（新方案中这些字段已移除，此处仍保留兼容）
        _tc_args.pop("ambiguous_params", None)
        _tc_args.pop("intent_reason", None)
        _tc_args.pop("extraction_confidence", None)
        tool_id, result = await dispatch_tool(tc, tools_map, state, ...)
```

**关键结论**：

1. **恢复粒度是工具调用级**——节点从顶部重新执行，通过检测 State 快速跳过已完成步骤，然后对 `react_pending_tool_calls` 中的工具调用重新分派（`dispatch_tool`）。
2. **`before_callback` 在恢复时会再次运行**——因为重放走的是完整的 `dispatch_tool()` 路径。
3. **`is_complex` 是重新计算的**，不是保存的局部变量——`complex_conditions` 保存在 `tool_params` 里，而 `tool_params` 保存在 `react_pending_tool_calls` 的 args 中，因此恢复后可以重新读到，结果与中断前完全一致。

---

##### 恢复时 before_callback 重复触发问题

由于 `before_callback` 在恢复时会再次运行，**歧义判断（Step 2）存在重复触发中断的风险**：用户刚完成了一次选择，恢复后 `before_callback` 若再次进入歧义流程，会再次 `interrupt()`，陷入死循环。

**解决方案：用户选择结果写回 tool_params**

用户的选择（paradigm 消歧结果）在 resume 时由框架注入，系统在重放前将选择结果**应用到 `tool_params`** 中（更新 filters 的自然语言值为已确认的 code），使恢复后 `before_callback` 的通道 1 参数直接检测命中 1:1 映射，从而**跳过歧义流程，直接 CLEAR**：

```
用户选择 "营收 → revenue"
  ↓
apply_resume_to_tool_params()
  → tool_params["filters"] 中 {field:"营收"} 更新为 {field:"revenue"}
  ↓
重放 dispatch_tool("query_Enterprise", updated_tool_params)
  ↓
before_callback 通道 1：{field:"revenue"} → 1:1 精确匹配 → CLEAR
  ↓
路由到 OQL 直查（is_complex=False）或 data_query（is_complex=True）
```

此机制同时保证了两条路由的正确性：

| 原始工具 | is_complex（重新计算） | 恢复后执行 | HookDecision |
|---------|----------------------|-----------|--------------|
| `query_{code}` | False | 继续原 `query_{code}`（OQL） | `patch`（或直接放行） |
| `compute_{code}` | False | 继续原 `compute_{code}`（OQL） | `patch`（或直接放行） |
| `query_{code}` / `compute_{code}` | True | `data_query_{scope_code}` | `redirect` |

---

```

      ↓
② Agent 选工具：query_Enterprise(query="...", filters=[{field:"营收",...}], complex_conditions=[])
      ↓
③ dispatch_tool → before_callback（Step 1 SIMPLE, Step 2 通道1检测 "营收" 有多候选）

   → LangGraph 挂起节点，等待用户
      ↓
⑤ 用户选择："营收 = revenue（总营收）"
      ↓
⑥ LangGraph resume → 节点重新从顶部执行

      ↓
⑧ tools_map["query_Enterprise"](updated_args) 执行

```

#### 3.1.3 MetricItem 结构扩展

当前 `_msr_item()` 函数（约第 370 行）生成的 metric schema 仅支持 `field + agg + as` 三字段。根据 1.6.2 节的新协议，需要扩展支持 `expr`（公式字段）和 `filters`（条件聚合）。

在 `_msr_item()` 的 `properties` 字典中补充：

```python
"expr": {
    "type": "string",
    "description": (
        "公式表达式（与 field 互斥）。字段本身是计算表达式时使用，"
        "如 'revenue - cost'。使用 expr 时不填 field。"
    ),
},
"filters": {
    "type": "array",
    "items": {"$ref": "#/definitions/FilterItem"},  # 复用已有 filter 定义
    "description": (
        "条件聚合过滤（CASE WHEN）。仅对满足此过滤条件的行参与当前指标聚合。"
        "与 field/expr 正交，可同时使用。"
    ),
},
```

同时放宽 `required`：`field` 和 `expr` 互斥但均可选，实际 required 仅保留 `["agg", "as"]`：

```python
"required": ["agg", "as"],  # field 与 expr 二选一，均可选填（底层按有无判断）
```

> **注意**：`_msr_item()` 当前返回固定字段的 Schema，`expr`/`filters` 作为可选字段不影响现有调用，向后兼容。

#### 3.2.1 当前状态

#### 3.2.2 改造方案

**不删除函数**，改为重命名并修改内容——新函数名 `inject_query_fields`，注入新的两个字段：

```python
def inject_query_fields(t: BaseTool) -> BaseTool:
    """往数据类工具 Schema 注入 query/complex_conditions 字段（由 LLM 在调用时填写，
    执行前自动剥除，消费由 before_callback 完成）。
    """
    try:
        from pydantic import Field, create_model  # noqa: PLC0415

        original_schema = t.args_schema
        if original_schema is None:
            return t

        new_schema = create_model(
            f"{original_schema.__name__}WithQuery",
            __base__=original_schema,
            query=(
                str,
                Field(
                    default="",
                    description=(
                        "【必填】用户原始查询意图的完整自然语言描述。"
                        "用于歧义判断、复杂查询路由及参数缺失时的兜底推断。"
                    ),
                ),
            ),
            complex_conditions=(
                list[str],
                Field(
                    default_factory=list,
                    description=(

                        "非空时系统路由到全能查询路径（data_query）。"
                    ),
                ),
            ),
        )
        t.args_schema = new_schema

        # 包装 coroutine：调用前剥除两个元字段（before_callback 已消费）
        if hasattr(t, "coroutine") and t.coroutine is not None:
            _orig_coro = t.coroutine

            async def _coro_strip_query_fields(**kw: Any) -> Any:
                kw.pop("query", None)
                kw.pop("complex_conditions", None)
                return await _orig_coro(**kw)

            t.coroutine = _coro_strip_query_fields

        if hasattr(t, "func") and t.func is not None:
            _orig_func = t.func

            def _func_strip_query_fields(**kw: Any) -> Any:
                kw.pop("query", None)
                kw.pop("complex_conditions", None)
                return _orig_func(**kw)

            t.func = _func_strip_query_fields

    except Exception as exc:  # pragma: no cover
        logger.warning(
            "[inject_query_fields] failed for tool=%s: %s", getattr(t, "name", "?"), exc
        )
    return t
```

这是改动量最大的模块。原有 `before_call_back` 仅做单通道澄清（依赖 `ambiguous_params`），需完整替换为**双步双通道**机制。

```python
async def before_call_back(ctx: HookContext) -> HookDecision | None:
    """双步判断：Step 1 复杂度路由，Step 2 双通道歧义判断。"""
    tool_name: str = ctx.get("tool_name", "")
    if not _is_query_or_compute_tool(tool_name):
        return None

    tool_params: dict[str, Any] = dict(ctx.get("tool_params") or {})

    # 剥除新元字段（before_callback 消费，不透传给底层执行体）
    query: str = str(tool_params.pop("query", "") or "")
    complex_conditions: list[str] = list(tool_params.pop("complex_conditions", None) or [])

    # 兼容旧版元字段（过渡期保留，新版 LLM 不再填写）
    tool_params.pop("intent_reason", None)
    tool_params.pop("extraction_confidence", None)
    tool_params.pop("ambiguous_params", None)

    ctx["tool_params"] = tool_params

    logger.info(
        "[before_callback] tool=%s query=%.60r complex_conditions=%s",
        tool_name, query, complex_conditions,
    )

    # ── Step 1：复杂度判断 ────────────────────────────────────────────────────
    is_complex: bool = bool(complex_conditions)

    # ── Step 2：双通道歧义判断（简单和复杂查询均执行）────────────────────────
    ambiguity = await _judge_ambiguity(query, tool_name, tool_params, ctx)

    if ambiguity.status == "NEED_CONFIRM":
        # 触发歧义澄清
        clarification_result = await _analyze_clarification(
            query=query,
            tool_name=tool_name,
            tool_params=tool_params,
        )
        resume_value = interrupt({
            "prompt": "查询条件存在歧义，请确认查询维度",
            "reason_code": "PARADIGM_CLARIFICATION",
            "ask_user_payload": {"paradigmList": clarification_result.paradigm_list},
        })
        # 用户选择后写回 tool_params，防止恢复时再次触发歧义
        _apply_resume(ctx, tool_name, resume_value, query, is_complex)
        return {"action": "patch", "patch": {"tool_params": dict(ctx["tool_params"])}}

    if ambiguity.status == "AUTO_DISAMBIG" and ambiguity.refined_params:
        tool_params = {**tool_params, **ambiguity.refined_params}
        ctx["tool_params"] = tool_params

    # ── 路由决策 ─────────────────────────────────────────────────────────────
    if is_complex:
        return _build_redirect_decision(tool_name, query, tool_params)

    if ambiguity.refined_params:
        return {"action": "patch", "patch": {"tool_params": tool_params}}

    return None
```

#### 3.4.3 双通道歧义判断（Step 2）

```python
@dataclass
class AmbiguityResult:
    status: Literal["CLEAR", "AUTO_DISAMBIG", "NEED_CONFIRM"]
    refined_params: dict[str, Any] | None = None
    paradigm_list: list[dict[str, Any]] | None = None


async def _judge_ambiguity(
    query: str,
    tool_name: str,
    tool_params: dict[str, Any],
    ctx: HookContext,
) -> AmbiguityResult:
    """双通道歧义判断：通道 1 参数直接检测（< 100ms），通道 2 query 文本分析（约 1s）。"""

    # ── 通道 1：参数直接检测（纯规则，无 LLM）────────────────────────────────
    ch1_ambiguous = _check_params_ambiguity(tool_params, tool_name)
    if not ch1_ambiguous:
        # 参数无歧义，跳过通道 2（节省 1s）
        return AmbiguityResult(status="CLEAR")

    # ── 通道 2：query 文本分析（expand_query，约 1s LLM 调用）────────────────
    if not query:
        # 无 query 文本则无法进一步判断，触发追问
        return AmbiguityResult(status="NEED_CONFIRM", paradigm_list=ch1_ambiguous)

    try:
        import asyncio  # noqa: PLC0415
        from datacloud_knowledge.intent import expand_query, natquery_to_five_stage  # type: ignore[import]

        nat_query = await asyncio.to_thread(expand_query, query)
        if nat_query is None:
            return AmbiguityResult(status="NEED_CONFIRM", paradigm_list=ch1_ambiguous)

        five_stage = natquery_to_five_stage(nat_query)
        auto_params = _try_resolve_from_five_stage(five_stage, ch1_ambiguous, tool_params)
        if auto_params is not None:
            return AmbiguityResult(status="AUTO_DISAMBIG", refined_params=auto_params)

        return AmbiguityResult(status="NEED_CONFIRM", paradigm_list=ch1_ambiguous)
    except Exception as exc:
        logger.warning("[before_callback] channel-2 expand_query failed: %s", exc)
        return AmbiguityResult(status="NEED_CONFIRM", paradigm_list=ch1_ambiguous)
```

```python
def _check_params_ambiguity(
    tool_params: dict[str, Any],
    tool_name: str,
) -> list[dict[str, Any]] | None:
    """通道 1：检查参数中是否存在歧义项。返回 None 表示无歧义，返回列表表示有歧义项。

    检测规则（见 2.2.2 节）：
    - A1：filters.field 是业务术语（非 field_code），对应多候选字段
    - A2：filters.value 是自然语言枚举描述，非已知 code
    - A3：filters.field 在字段 Schema 中无任何匹配
    - A4：filters.value 含相对时间表达（"最近"/"今年"/"上季度"等）
    - A5：complex_conditions 中术语有多种语义（暂由 Step 1 路由覆盖，此处不重复处理）
    """
    ambiguous: list[dict[str, Any]] = []
    filters: list[dict[str, Any]] = tool_params.get("filters") or []

    _RELATIVE_TIME_PATTERN = re.compile(
        r"最近|今[年月周天]|上[年月季周]|本[年月季周天]|去年|前\d+[天月年]|[近最]?\d+[天月年]内",
    )

    for f in filters:
        field = str(f.get("field") or "")
        value = f.get("value")
        if not field:
            continue

        # A4：时间相对表达
        if isinstance(value, str) and _RELATIVE_TIME_PATTERN.search(value):
            ambiguous.append({"type": "A4", "field": field, "value": value})
            continue

        # A1/A2/A3：字段名和值的映射检查（依赖 schema 查找，此处返回可疑项供后续处理）
        if not _looks_like_field_code(field):
            # field 是中文或业务术语，非标准 field_code 格式
            ambiguous.append({"type": "A1_A3", "field": field, "value": value})

        elif isinstance(value, str) and not _looks_like_enum_code(value):
            # field 是 field_code 但 value 是自然语言描述
            ambiguous.append({"type": "A2", "field": field, "value": value})

    return ambiguous if ambiguous else None


def _looks_like_field_code(s: str) -> bool:
    """判断字符串是否符合 field_code 格式（全小写英文+下划线+数字）。"""
    return bool(re.match(r"^[a-z][a-z0-9_]*$", s))


def _looks_like_enum_code(s: str) -> bool:
    """判断字符串是否符合枚举 code 格式（英文字母/数字/下划线，无中文）。"""
    return bool(re.match(r"^[A-Za-z0-9_\-]+$", s))
```

```python
def _extract_knowledge_from_paradigm(selected: list[dict[str, Any]]) -> str:
    lines = []
    for item in selected:
        name = item.get("name") or item.get("termName") or ""
        field = item.get("fieldName") or item.get("description") or ""
        if name and field:
            lines.append(f"{name} → {field}")
    return "\n".join(lines)


def _build_oql_params_from_paradigm(
    selected: list[dict[str, Any]],
    query: str,
) -> dict[str, Any]:
    """从用户 paradigm 选择重建 OQL 参数（占位实现）。"""
    filters = []
    for item in selected:
        field_name = item.get("fieldName") or ""
        value = item.get("selectedValue") or item.get("value") or ""
        if field_name:
            filters.append({"field": field_name, "op": "eq", "value": value})
    return {"query": query, "filters": filters}
```

```python
for tc in pending_tool_calls:
    _tc_args = dict(tc.get("args") or {})
    # 兼容旧版：仍然剥除旧元字段（过渡期保留）
    _tc_args.pop("ambiguous_params", None)
    _tc_args.pop("intent_reason", None)
    _tc_args.pop("extraction_confidence", None)

    # 新机制：将用户 paradigm 选择写回 tool_params
    # before_callback 的 _apply_resume 已将 ctx["tool_params"] 更新，
    # 此处 _tc_args 即为 State 中保存的原始参数（中断前的 args），
    # 需要将 resume 选择合并进来，确保通道 1 检测 CLEAR
    if _resume_value is not None:
        _selected = _resume_value.get("paradigmList", []) if isinstance(_resume_value, dict) else []
        # 将已确认的 field_code 写回 filters（覆盖中文术语）
        _tc_args = _merge_resume_to_args(_tc_args, _selected)
        _resume_value = None  # 只对第一个 pending tool call 应用（通常只有一个）

    tc = {**tc, "args": _tc_args}
    tool_id, result = await dispatch_tool(tc, tools_map, state, ...)
```

```python
def _merge_resume_to_args(
    args: dict[str, Any],
    selected: list[dict[str, Any]],
) -> dict[str, Any]:
    """将用户 paradigm 选择合并回 tool_params args。

    将 filters 中字段名为中文术语的项，替换为用户选择的 field_code 和确认值。
    """
    if not selected:
        return args

    args = dict(args)
    filters: list[dict[str, Any]] = list(args.get("filters") or [])
    updated: list[dict[str, Any]] = []

    # 构建 term→field_code 映射（来自用户选择）
    term_to_field: dict[str, str] = {}
    term_to_value: dict[str, Any] = {}
    for item in selected:
        term = str(item.get("name") or item.get("termName") or "")
        field_code = str(item.get("fieldName") or item.get("field_code") or "")
        value = item.get("selectedValue") or item.get("value")
        if term and field_code:
            term_to_field[term] = field_code
            if value is not None:
                term_to_value[term] = value

    for f in filters:
        field = str(f.get("field") or "")
        resolved_code = term_to_field.get(field, field)  # 若无映射则保持原值
        resolved_value = term_to_value.get(field, f.get("value"))
        updated.append({**f, "field": resolved_code, "value": resolved_value})

    args["filters"] = updated
    return args
```

> **说明**：`react_resume_value` 是需要在 LangGraph State schema 中新增的字段，用于承载用户 resume 时提交的内容。具体由 LangGraph 框架层将 `Command(resume=...)` 的值写入 State，再由 react_loop 读取。
