# 复杂问题支撑能力评估与方案

> 文档日期：2026-04-15  
> 评估范围：`packages/datacloud-analysis`（orchestration + react_loop + tool_hook_plugins）

---

## 一、两类典型问题场景

### 场景 A：简单单表提问

> 查询管理网格表的数据清单。

特征：
- 目标明确，单工具调用
- 无跨表依赖
- 无操作类动作

### 场景 B：复杂多步问题

> 查询前 10 的网格（网格表），并查看这些网格中亩产时后 10% 的企业（企业表），总结一下这些企业的签订合同额（合同表），最后发起一个拜访申请待办（操作类动作）。

特征：
- 4 个子任务，跨 4 张表 / 服务
- 后序任务依赖前序结果（网格 ID → 企业筛选 → 合同聚合）
- 最终涉及写操作（发起待办）
- 全程最多需要 4 ~ 6 轮工具调用

---

## 二、现有架构现状

### 2.1 图结构（3 节点管道）

```
START
  ↓
intend_node        意图识别 + 命令路由 + 知识增强
  ↓
execution_node     ReAct 多轮循环（max_rounds 默认 10）
  ↓
respond_node       格式化输出（markdown / json）
  ↓
END
```

没有独立的规划（planning）节点；`AgentState` 中虽然预留了 `plan / todos / todo_md` 等字段，但当前均未被写入或消费。

### 2.2 ReAct 循环能力与限制

| 项目 | 现状 |
|------|------|
| 最大轮次 | 10（`DATACLOUD_REACT_MAX_ROUNDS`，可调） |
| 工具执行模式 | 同轮内**顺序串行**（for 循环，无并发） |
| 消息滑动窗口 | 保留最近 6 轮（`_TRIM_KEEP_ROUNDS=6`） |
| ToolMessage 截断 | 2000 字符；大数据用 `_hint` 替代 |
| 停止条件 | L1 `finish_react` / L2 无 tool_call / L3 max_rounds |
| 中断恢复 | ✅ 支持（`GraphBubbleUp` → state 持久化 → resume） |
| 多轮历史携带 | ✅ 保留 SystemMessage + 第一条 HumanMessage + 最近 6 轮 |
| 任务规划拆解 | ❌ 无（字段预留但未启用） |
| 子代理机制 | ❌ `agent_delegate` 仅做外部委托，无内部分解 |
| 跨轮依赖管理 | ❌ 依赖 LLM 自行在 prompt 中推断 |

### 2.3 场景 A 的执行路径

```
Round 1：LLM → data_query_manage_grid(query="管理网格清单")
Round 2：LLM 收到结果 → finish_react(result_type="query_result")
```

**结论：完全支持，2 轮可完成，无瓶颈。**

### 2.4 场景 B 的执行路径（当前架构）

理论上 LLM 可以自主串联，但存在以下问题：

```
Round 1：data_query_manage_grid(query="亩产前10网格") → 返回 grid_id 列表
Round 2：data_query_enterprise(query="这些网格中亩产后10%企业", grid_ids=[...])
           ↑ 问题①：grid_id 需从 Round 1 结果中提取后拼入参数，依赖 LLM 的
             extract-and-pass 能力，大模型可能截断/遗漏
Round 3：data_query_contract(query="这些企业的合同额", enterprise_ids=[...])
Round 4：compute_xxx 或 LLM 自行聚合
Round 5：action_visit_request(enterprise_ids=[...]) → 发起拜访待办
Round 6：finish_react
```

**当前架构下的瓶颈：**

#### 瓶颈 ① — 跨轮参数传递依赖 LLM 记忆

工具结果经 `_compress_tool_result` 压缩（最长 2000 字符）后写入 ToolMessage，LLM 下一轮需要从这段文本中提取 ID 列表并拼入下一个工具的 args。  
当结果集较大（如 10 个网格 × 多字段）时，LLM 可能提取不完整，导致后续工具参数错误。

#### 瓶颈 ② — 无显式任务规划，LLM 可能"走弯路"

无 planning 节点的情况下，LLM 需在第 1 轮就"看透"整个 4 步任务并规划好顺序。  
复杂 prompt 下 LLM 容易只完成前 1-2 步就调 `finish_react`，遗漏后续任务。

#### 瓶颈 ③ — 消息窗口截断风险

6 轮滑动窗口在 6 步以内的任务中仍可保留完整上下文，但若中间有工具重试（错误后重调），有效轮数会被消耗，早期结果可能滑出窗口。

#### 瓶颈 ④ — 操作类动作缺乏前置确认机制

`action_visit_request` 等写操作工具当前直接执行，没有"先告知用户即将操作、再执行"的中间步骤，用户体验和安全性较弱。

## 三、方案设计

### 方案对比

| 方案 | 描述 | 改动范围 | 复杂度 |
|------|------|----------|--------|
| 方案 1（推荐）| 新增 planning_node，LLM 先拆任务再执行 | 中（新增节点、修改路由） | 中 |
| 方案 2 | 提升 prompt 质量 + 扩大上下文窗口 | 小（仅改 prompt 和参数） | 低 |
| 方案 3 | 多 Agent 并行（子任务委托给子 Agent） | 大（需要 agent_delegate 重构） | 高 |

---

### 方案 1（推荐）：intend_node 路由分流 + planning_node

#### 3.1.1 架构原则：路由判断放在 intend_node

`planning_node` 本身不判断"是否需要规划"——如果让它自己判断，所有问题都会经过一次额外的 LLM 调用，简单问题也不例外，造成不必要的延迟。

**正确做法**：在 `intend_node` 里用**纯规则**（零额外 LLM 调用）判断复杂度，写入 `execution_status`，由图路由分流：

```
START
  ↓
intend_node（意图识别 + 知识增强 + 复杂度判断）
  ↓
[条件路由，基于 execution_status]
  ├─ "command_done"   → END
  ├─ "simple_react"   → execution_node    ← 简单问题，零额外调用
  └─ "complex_react"  → planning_node     ← 复杂问题，先规划（+1次LLM）
                            ↓
                        execution_node
  ↓
respond_node
  ↓
END
```

复杂度判断逻辑（`intend_node` 内，纯规则）：

```python
def _is_complex_query(user_query: str, knowledge_payload: dict) -> bool:
    # 信号1：多步连接词
    multi_step_words = ["并且", "然后", "再", "接着", "最后", "同时", "还要", "并"]
    has_multi_step = any(w in user_query for w in multi_step_words)

    # 信号2：跨多个 object/view（knowledge 里命中多张不同的表）
    try:
        paradigm_list = json.loads(
            knowledge_payload.get("knowledge") or "{}"
        ).get("paradigmList", [])
        distinct_objects = {
            item.get("objectCode") for item in paradigm_list if item.get("objectCode")
        }
        has_multi_table = len(distinct_objects) >= 2
    except Exception:
        has_multi_table = False

    # 信号3：包含操作类意图词
    action_words = ["发起", "创建", "提交", "申请", "新建", "生成"]
    has_action = any(w in user_query for w in action_words)

    return has_multi_step or has_multi_table or has_action
```

`intend_node` 最终写入：
- 简单问题 → `execution_status = "simple_react"`（与当前 `"execution"` 语义等同）
- 复杂问题 → `execution_status = "complex_react"`

| 场景 | 额外 LLM 调用 | 路径 |
|------|-------------|------|
| 简单单表查询 | 0 | intend → execution |
| 复杂多步问题 | 1（planning） | intend → planning → execution |

#### 3.1.2 planning_node 职责

输入：`state.user_query`  
输出：写入 `state.todos`（有序任务列表）

```python
# todos 元素结构
{
    "id": "task-1",
    "title": "查询前10网格",
    "tool_hint": "data_query_manage_grid",   # 提示 LLM 优先用哪个工具
    "depends_on": [],                         # 依赖的前序 task id
    "output_key": "top10_grid_ids",          # 结果存入 state 的 key（供后序任务引用）
    "status": "pending"                       # pending / running / done / failed
}
```

planning_node 调用一次 LLM（非流式）：
- 输入：用户问题 + 可用工具列表摘要
- 输出：JSON 格式的 todos 列表
- 判断是否"复杂问题"：工具调用数 ≥ 2 且存在跨结果依赖

简单问题（单工具调用）直接 skip，不增加额外 LLM 调用开销。

#### 3.1.3 execution_node 变更

当 `state.todos` 非空时，在系统 prompt 中注入任务列表：

```
## 当前任务清单（请按顺序完成）
1. [task-1] 查询前10网格 → 工具建议：data_query_manage_grid，结果存为 top10_grid_ids
2. [task-2] 查询后10%企业 → 依赖 top10_grid_ids，工具建议：data_query_enterprise
3. [task-3] 汇总合同额 → 依赖企业列表，工具建议：data_query_contract
4. [task-4] 发起拜访待办 → 操作类，需先向用户确认
```

这解决了瓶颈②（LLM 不再需要自行推断任务顺序）。

#### 3.1.4 跨轮参数传递增强（解决瓶颈①）

在 `react_loop.py` 中，将工具的**原始结构化结果**（而非压缩文本）暂存到 `state.react_last_query_data`（已有字段），并在下一轮工具调用的 `HookContext` 中以 `previous_results` 字段透传：

```python
# tool_wrapper.py HookContext 新增
"previous_results": state.get("react_last_query_data") or {}
```

ReAct loop 的 ToolMessage 中同时写入结构化摘要（而非只有压缩文本）：

```
[data_query_manage_grid 结果]
共 10 条，grid_id 列表: ["G001","G002",...,"G010"]
亩产效益字段: output_per_mu
完整数据已缓存，可在下一步工具调用中通过 previous_grid_ids 引用。
```

这让 LLM 在下一轮的 prompt 中明确看到可用的 ID 列表，减少提取错误。

#### 3.1.5 操作类动作前置确认（解决瓶颈④）

为所有 `action_*` 工具增加 `before_call_back` hook（新建 `action_confirm_plugin.py`）：

```python
PRIORITY = 50   # 优先级最高，先于其他 hook 执行

async def before_call_back(ctx: HookContext) -> HookDecision | None:
    tool_name = ctx.get("tool_name", "")
    if not tool_name.startswith("action_"):
        return None
    
    # 弹出用户确认框
    from langgraph.types import interrupt
    resume_value = interrupt({
        "prompt": f"即将执行操作：{tool_name}，参数如下：\n{json.dumps(ctx['tool_params'], ensure_ascii=False, indent=2)}\n是否确认执行？",
        "reason_code": "ACTION_CONFIRM",
        "ask_user_payload": {"confirm": True}
    })
    
    if not resume_value.get("confirmed", False):
        return {"action": "short_circuit", "result": {"tool_output": "用户取消了该操作。"}}
    
    return None   # 用户确认，继续执行
```

### 方案 2：提升 prompt + 扩大窗口（兜底方案）

不增加 planning_node，通过以下调整缓解复杂问题：

1. **扩大滑动窗口**：`_TRIM_KEEP_ROUNDS` 从 6 → 10，增加上下文保留
2. **改进 execution_prompt**：加入"复杂任务分步提示"模板，引导 LLM 先列出步骤再执行
3. **ToolMessage 结构化摘要**：压缩时优先保留 ID 字段，而非仅记录行数

优点：改动小，风险低  
缺点：仍依赖 LLM 自行规划，复杂场景成功率不稳定

## 四、场景 B 在方案 1 下的完整执行流程

```
用户输入：查询前10网格，查看亩产后10%企业，总结合同额，发起拜访待办

intend_node
  └─ knowledge_enhancer → 识别"亩产"字段映射 → knowledge_snippets

planning_node（新增）
  └─ LLM 拆解 → todos = [
       {id:"t1", title:"查询亩产前10网格", tool_hint:"data_query_manage_grid", output_key:"grid_ids"},
       {id:"t2", title:"查询亩产后10%企业", tool_hint:"data_query_enterprise", depends_on:["t1"], output_key:"enterprise_ids"},
       {id:"t3", title:"汇总签订合同额",   tool_hint:"data_query_contract",   depends_on:["t2"], output_key:"contract_summary"},
       {id:"t4", title:"发起拜访申请",     tool_hint:"action_visit_request",  depends_on:["t3"]}
     ]

execution_node（注入任务清单到 system_prompt）
  ├─ Round 1: data_query_manage_grid(query="亩产前10网格")
  │             → 返回 grid_ids: [G001..G010]，缓存到 state
  ├─ Round 2: data_query_enterprise(query="亩产后10%企业", grid_ids=[G001..G010])
  │             → 返回 enterprise_ids: [E001..E050]
  ├─ Round 3: data_query_contract(query="合同额汇总", enterprise_ids=[E001..E050])
  │             → 返回 contract_summary
  ├─ Round 4: action_visit_request(enterprise_ids=[E001..E050])
  │             → action_confirm_plugin.before_call_back 触发 interrupt
  │             → 用户确认 → resume → 执行操作
  └─ Round 5: finish_react(result_type="query_result", answer="...")

respond_node
  └─ 输出合同额表格 + 拜访待办创建结果
```

总轮次：5 轮，在默认 max_rounds=10 内完整执行。

## 六、关键风险与注意事项

### 风险 1：planning_node 额外 LLM 调用增加延迟

- planning_node 会增加 1 次 LLM 调用（约 1-3 秒）
- **缓解**：简单问题判断逻辑放在 intend_node，命中"简单"则跳过 planning_node
- **判断标准**：用户问题中无"并"/"同时"/"再"/"然后"/"最后"等多步连接词，且 knowledge_snippets 无多表映射时视为简单问题

### 风险 2：todos 规划错误导致执行偏差

- LLM 规划的 todos 可能遗漏或顺序错误
- **缓解**：execution_node 注入任务清单后，LLM 在 ReAct 中仍可自行调整；todos 仅作"软约束提示"而非"硬约束调度"

### 风险 3：跨轮 ID 传递量超出上下文

- 前 10 个网格 ID 数据量小，问题不大
- 若后续需传递大量 ID（如 50 个企业 ID），ToolMessage 2000 字符上限可能截断
- **缓解**：将大 ID 列表存入 workspace 临时文件，ToolMessage 中写文件路径，后序工具从文件读取

### 风险 4：操作类动作在 paradigm interrupt 中断后的死循环

- 与 query_clarification_plugin 类似，action_confirm_plugin 在 interrupt 恢复后需清除"已确认"标志
- **实现要点**：恢复后立即在 ctx 中写入 `{"action_confirmed": True}`，避免同一操作被重复弹出确认

## 七、现状评估结论

| 场景 | 当前支持情况 | 改动后支持情况 |
|------|------------|--------------|
| 简单单表查询（场景 A） | ✅ 完全支持 | ✅ 不变 |
| 多步跨表查询（场景 B，纯查询部分） | ⚠️ 依赖 LLM 自行规划，成功率约 60-70% | ✅ planning_node 后稳定支持 |
| 查询 + 操作类动作混合（场景 B 完整） | ❌ 操作无确认机制，跨轮传参不稳定 | ✅ action_confirm + todos 注入后支持 |
| 超过 10 步的超长任务链 | ❌ max_rounds 限制 | ⚠️ 可调参，但需子代理机制才能真正解决 |
