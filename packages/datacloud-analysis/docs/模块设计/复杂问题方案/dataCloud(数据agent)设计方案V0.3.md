# dataCloud(数据agent)设计方案

## 1.背景

本方案是《复杂问题方案.md》迭代优化版本。



## 2.设计思想

使用前完成本体治理，运行层面，在agent层贪心，在工具渐进

### 2.1**前置依赖**

使用本框架依赖以下的本体构建的治理工作。

1、术语治理：需要梳理字典术语、列表术语以及已知的同义词。

2、本体治理：需要提前把api或数据库表建设成本体对象/本体视图。其中本体的属性根据需要绑定字典术语或列表术语。同时本体属性再进一步细分为维度、度量类型。

### 2.2**分类思想**

本框架对数据读取、任务复杂度、本体属性进行了标准化分类。本框架基于这类分类体系进行运行。

**1、数据读取任务分类**：本方案把数据的读取分为查询和统计两类，分别对应quey_*,compute__*两类工具。

**2、任务复杂度分类**：根据使用标准的query_*/compute_*, 分为标准任务和定制任务。在实现上，定义任务通过complex*参数进行识别，目前仅实现了查询条件出现跨对象的查询条件的类型，后续可扩展更多类型。

**3、本体属性分类**：对象属性分为维度和度量两大类，每类分为若干小类，不同类型的属性在分组、条件过滤、统计计算上有不同的规则约束。

### 2.3**贪心思想**

本方案技术上利用agent的reAct & function call机制落地，尽可能在agent调用工具前识别尽可能多任务识别。

| 任务                   | 决策内容                                                     |
| ---------------------- | ------------------------------------------------------------ |
| 选工具(本体)           | 确定目标对象/视图（靠工具描述中的字段能力表），`{code}` 为对象/视图唯一编码 |
| 选工具(任务分类)       | 确定意图：查明细 → `query_{code}`；做统计 → `compute_{code}` |
| 参数抽取(标准参数)     | 从自然语言中抽取标准的 `query_{code}`、compute_{code} 参数抽取。 |
| 参数抽取(歧义识别)     | 从自然语言中识别出来无法直接推理的参数，让用户在原词中表达。 |
| 参数抽取(复杂任务识别) | 从自然语言中抽取无法表达的任务（complex参数）                |

### 2.4**渐进思想**

1、任务复杂渐进：先通过query_{code}、compute_{code} 引导能够用简单任务，规则转SQL，如标准化搞不定（包含complex.. 参数），则转化为text2SQl。

2、意图澄清渐进：

​	2.1）先通过规则进行歧义意图判断（标准术语+术语同义词）。

​	2.2）能够自动修正尽可能自动修正。

​		2.2.1)能够局部修正尽量局部修正（例如只有select 有歧义则只修正select， where有歧义则只修正where）。

​        	2.2.2)无法局部修正整体修正（重新调整select， from， where）

​	3）无法自动修正列出可能性，让人工确认（列出可能组合可能，让人工确认）。

## 3.建设目标

1、性能目标：

1）首字节响应 及平均响应：首字节1秒，平均每秒20个文字或字母输出。

2）完成时间：简单任务(text2Dsl)：5-15秒完成任务处理，复杂任务(text2Sql)：15-3分钟完成处理。

2、准确率目标：基于XXX评测集准确率达95%。
