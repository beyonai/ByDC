# 澄清中断/恢复机制 —— 实证与完善方案

> 版本：V0.2 | 日期：2026-04-19
> 背景：通过断点实测（断在 `query_clarification_plugin.py` 第 460 行），
> 确认了中断消息链路的真实行为，发现与原设计文档存在两处关键偏差，本文记录实证结论和修复方案。

---

## 1. 实证：中断/恢复消息链路

### 1.1 中断时：从 interrupt() 到前端 SSE

```
① plugin: interrupt({
       "prompt":           "查询条件存在歧义，请确认查询维度",
       "reason_code":      "PARADIGM_CLARIFICATION",
       "ask_user_payload": {"paradigmList": [...], "query": "..."},
       "_clarify_knowledge": "...",
   })
   → 抛出 GraphBubbleUp

② react_loop.py 捕获 GraphBubbleUp
   → 把 react_messages / react_pending_tool_calls / react_round_idx 写入 state
   → 重新抛出 GraphBubbleUp，LangGraph checkpoint 保存 state

③ 图流结束后，worker.py: aget_state(snapshot_config)
   读取 snapshot.interrupts[0].value = interrupt_value

④ worker.py 从 interrupt_value 中提取字段：
       prompt             ← interrupt_value["prompt"]
       interrupt_reason   ← interrupt_value["reason_code"]
       paradigm_list      ← interrupt_value["ask_user_payload"]["paradigmList"]
       clarify_query      ← interrupt_value["ask_user_payload"]["query"]
       clarify_knowledge  ← interrupt_value["_clarify_knowledge"]

⑤ worker.py 从 snapshot 中提取框架元数据：
       checkpoint_id    ← snapshot.config["configurable"]["checkpoint_id"]
       checkpoint_ns    ← snapshot.config["configurable"]["checkpoint_ns"]
       thread_id        ← config["configurable"]["thread_id"]
       todo_active_id   ← snapshot.values["todo_active_id"]
       pending_capability ← snapshot.values["active_tools"][0]

⑥ worker.py 手动组装 metadata，调用 complex_ask_user()
   → emit SSE: contentType=3012
       content  = json.dumps({"paradigmList": paradigm_list})
       metadata = {
           "checkpoint_id":      "...",
           "checkpoint_ns":      "",
           "thread_id":          "10000587:10039577",
           "agent_id":           10000587,
           "conf_hash":          "...",
           "interrupt_reason":   "PARADIGM_CLARIFICATION",
           "paradigmList":       [...],
           "query":              "...",         ← 用户原始查询
           "clarify_knowledge":  "...",         ← KnowledgeMeta JSON
           "todo_active_id":     "",
           "react_step_id":      "",
           "pending_capability": "",
       }
```

**前端实际收到的 SSE（实测）：**

```
event: answerDelta
data: {
  "contentType": "3012",
  "choices": [{
    "delta": {
      "content": "{\"paradigmList\": [...]}"   ← 5个 paradigm 的 JSON 字符串
    }
  }],
  "metadata": "{...checkpoint_id, thread_id, paradigmList, query, clarify_knowledge...}"
}
```

### 1.4 knowledge 字段内容与调用链路

#### knowledge 放的是什么

`result.knowledge` 是 **`KnowledgeMeta` 的 JSON 序列化字符串**，本质是一张"写回路由表"：

```json
{
    "path_mapping": {
        "1":  "select.0",
        "2":  "select.1",
        "f1": "filters.0.field",
        "o1": "order_by.0.field"
    },
    "confirmed_conditions": [...],
    "mode": "query"
}
```

`path_mapping` 的含义：`kid`（paradigm item 的唯一 ID）→ 用户选择结果要写回 `structured_query` 的 JSON 路径。

`format_clarification_query` 收到用户选择后，用 `path_mapping["f1"] = "filters.0.field"` 找到目标位置，
把用户选的 field_code 精确写到 `structured_query["filters"][0]["field"]`。
**没有 path_mapping，format 不知道往哪写，字段替换失败或退化。**

#### 调用链路

```
before_call_back(ctx)                                   [plugin.py:366]
  └─ _analyze_clarification(query, scope_code,          [plugin.py:323]
         structured_input, is_compute)
       └─ _sdk_analyze_query / _sdk_analyze_compute     [plugin.py:332-334]
          = analyze_query_clarification_query/compute() [clarification/api.py]
              │
              ├─ Step 1: extract_terms_query()          → all_terms（术语列表）
              ├─ Step 2: _unified_recall()              → recall_map（向量召回）
              ├─ Step 3: llm_confirm_structured()       → confirmed（LLM 确认，含 clarify_items）
              └─ Step 4: build_paradigm_list(
                             confirmed, all_terms,
                             recall_map)
                   │
                   ├─ serialize_paradigm_payload(paradigm_list)
                   │    → form_payload = '{"paradigmList":[...]}'
                   │                           → ClarificationResult.form  ← 透传给前端
                   │
                   └─ serialize_knowledge_meta(KnowledgeMeta)
                        → knowledge_payload = '{"path_mapping":{kid→path},"mode":"query",...}'
                                                    → ClarificationResult.knowledge  ← 此处生成
          ↓
  _analyze_clarification 返回 (paradigm_list, result.knowledge)
  interrupt(...)  → 经 worker.py 透传至前端 metadata → resume
       ↓
  _format_clarification(query, structured_input, form_str, knowledge)  [plugin.py:347]
    └─ format_clarification_query/compute()             [clarification/format.py]
         ├─ 解析 form_str → paradigm_list（用户选择结果）
         ├─ 解析 knowledge → KnowledgeMeta（含 path_mapping）
         └─ 对每个 paradigm item：
              path = path_mapping[item["kid"]]          → e.g. "filters.0.field"
              _set_by_path(structured_query, path, user_choice)  ← 精确写回
```

## 2. 原有 Bug

### Bug 1：form_str 序列化了整个 resume_value，格式错误

**位置：** `query_clarification_plugin.py` 第 466-470 行（修复前）

**问题代码：**
```python
form_str = (
    json.dumps(resume_value, ensure_ascii=False)
    if isinstance(resume_value, dict)
    else "{}"
)
```

**问题：** SDK 的 `format_clarification_query/compute` 期望 `form` 参数格式为：
```json
{"paradigmList": [...用户已填写的选择...]}
```

实际传入的是整个 `resume_value`（顶层有 `paradigmResult`、`metadata`，无 `paradigmList`），
SDK 在顶层找不到 `paradigmList` key，导致用户选择被忽略，format 结果退化为原始参数。

### Bug 2：knowledge 永远为空字符串

**位置：** `query_clarification_plugin.py` 第 460-464 行（修复前）

**问题代码：**
```python
knowledge = ""
if isinstance(resume_value, dict):
    knowledge = str(resume_value.get("_clarify_knowledge", ""))  # 始终为 ""
```

**根因：** `_clarify_knowledge` 放在 interrupt payload 顶层，worker.py 从未读取它，
前端收不到，resume_value 中自然也没有，`get()` 永远返回默认值 `""`。

`knowledge`（KnowledgeMeta JSON）是 `format_clarification_*` 做精确字段路径映射的必要内部状态。
knowledge 为空时 SDK 仍能运行，但 path_mapping 缺失，字段写回精度下降。

## 3. 解决方案：前端 metadata echo（方案 B）

### 3.1 核心思路

前端会将 SSE metadata 中所有字段原样 echo 回 `resume_value["metadata"]`。
只需让 `worker.py` 把 `clarify_knowledge` 和 `query` 写入 metadata，
前端自动带回，plugin resume 端直接从 `resume_value["metadata"]` 读取。

```
interrupt 前: ask_user_payload = {"paradigmList": [...], "query": query}
              _clarify_knowledge = clarify_knowledge（在 interrupt payload 顶层）
                    ↓
worker.py: clarify_knowledge = interrupt_value["_clarify_knowledge"]
           clarify_query     = ask_user_payload["query"]
           metadata["clarify_knowledge"] = clarify_knowledge
           metadata["query"]             = clarify_query
                    ↓ SSE → 前端 → resume
resume_value["metadata"]["clarify_knowledge"]  ← plugin 读取 knowledge
resume_value["metadata"]["query"]              ← 前端展示用（plugin 无需读）
```

#### 改动 2：query_clarification_plugin.py —— resume 端修复 knowledge 和 form_str 提取

```python
# 改前（两处 bug）
knowledge = ""
if isinstance(resume_value, dict):
    knowledge = str(resume_value.get("_clarify_knowledge", ""))  # 永远 ""
form_str = json.dumps(resume_value, ensure_ascii=False)          # 格式错误

# 改后
knowledge = ""
paradigm_list_from_resume: list[dict[str, Any]] = []
if isinstance(resume_value, dict):
    meta = resume_value.get("metadata") or {}
    knowledge = str(meta.get("clarify_knowledge") or "")
    para_result = resume_value.get("paradigmResult") or []
    if para_result and isinstance(para_result[0], dict):
        paradigm_list_from_resume = list(para_result[0].get("paradigmList") or [])
    if not paradigm_list_from_resume:
        paradigm_list_from_resume = list(meta.get("paradigmList") or [])

form_str = json.dumps({"paradigmList": paradigm_list_from_resume}, ensure_ascii=False)
```

#### 改动 3：byclaw-data/worker.py —— 从 interrupt_value 提取并注入 metadata

```python
# 改前
paradigm_list = []
if isinstance(interrupt_value, dict):
    if isinstance(interrupt_value.get("ask_user_payload"), dict):
        ask_user_payload = interrupt_value.get("ask_user_payload")
```

### 3.3 修复前后数据流对比

```
修复前：
  interrupt({..., "_clarify_knowledge": knowledge, "_clarify_query": query})
       ↓ worker.py 丢弃两个字段
  resume_value["metadata"] 无 clarify_knowledge
  knowledge = ""                               ← 永远空
  form_str  = json.dumps(resume_value)         ← 整个 resume_value，格式错误
  → format_*: 无 knowledge + 找不到 paradigmList → 字段写回失败

修复后：
  interrupt({"ask_user_payload": {"paradigmList":[...], "query": query},
             "_clarify_knowledge": knowledge})
       ↓ worker.py 提取并注入 metadata
  resume_value["metadata"]["clarify_knowledge"] = knowledge  ← 前端 echo 回来
  knowledge = resume_value["metadata"]["clarify_knowledge"]  ← 正确取回
  form_str  = {"paradigmList": resume_value["paradigmResult"][0]["paradigmList"]}
  → format_*: 有 knowledge + 正确 paradigmList → 字段写回准确
```

### 3.4 边界处理

| 场景 | 处理 |
|------|------|
| `_clarify_knowledge` 为空（SDK 未返回） | knowledge = ""，format_* 降级运行 |
| `paradigmResult` 为空或结构异常 | paradigm_list_from_resume = []，form_str = `{"paradigmList":[]}`，format_* 返回原始参数 |
| `_format_clarification` 抛出异常 | 现有代码已有 try/except 兜底，降级到透传原始 query |
