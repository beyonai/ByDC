# 大模型报错兼容方案

> 版本：v1.1 | 日期：2026-04-13

---

## 背景与问题

在生产环境中，大模型调用（LLM API）存在以下不稳定场景：

| 场景 | 典型错误 | 频率 |
|------|---------|------|
| API 限流 | HTTP 429 Too Many Requests | 高峰期常见 |
| 服务端偶发故障 | HTTP 500 / 502 / 503 / 504 | 偶发 |
| 网络超时 | Connection timeout / Read timeout | 弱网环境 |
| 模型服务不可用 | Connection refused / DNS 解析失败 | 偶发 |

当前代码无任何重试或容错机制，一次报错直接导致整个问答失败，用户体验差。

---

## 设计目标

1. **透明**：不改变正常请求路径，失败时自动恢复，用户无感知
2. **可配置**：通过环境变量控制重试策略，不需要改代码
3. **渐进**：三层降级，每层解决不同粒度的问题
4. **可观测**：所有重试和切换行为写入日志，便于排查

---

## 三层容错方案

```
请求
  │
  ├─ Layer 1：请求级重试（指数退避）
  │      失败 → 等待 → 重试（最多 N 次）
  │
  ├─ Layer 2：轮次级 Checkpoint 恢复
  │      一轮 react 完全失败 → 保存断点 → 下次对话从断点继续
  │
  └─ Layer 3：备用模型切换
         主模型不可用 → 切换到 fallback 模型
```

---

## Layer 1：请求级重试（指数退避）

### 策略

- 只重试**可重试错误**（临时性故障），不重试**不可重试错误**（认证失败、参数错误等）
- 指数退避：`wait = min(min_wait × 2^attempt, max_wait)`
- HTTP 429 时，优先读取响应头 `Retry-After`，按提示时间等待

### 可重试 vs 不可重试

| 错误类型 | HTTP 状态 / 异常 | 是否重试 |
|---------|----------------|---------|
| 限流 | 429 | ✅ |
| 服务端错误 | 500 / 502 / 503 / 504 | ✅ |
| 超时 | TimeoutError / ConnectTimeout | ✅ |
| 网络断开 | ConnectionError / OSError | ✅ |
| 认证失败 | 401 | ❌ |
| 请求参数错误 | 400 | ❌ |
| 内容过滤 | 400 (content_filter) | ❌ |

### 环境变量

| 变量名 | 默认值 | 说明 |
|-------|-------|------|
| `DATACLOUD_LLM_MAX_RETRIES` | `3` | 最大重试次数（不含首次） |
| `DATACLOUD_LLM_RETRY_MIN_WAIT` | `1.0` | 首次重试前等待秒数 |
| `DATACLOUD_LLM_RETRY_MAX_WAIT` | `30.0` | 最大等待秒数上限 |
| `DATACLOUD_LLM_RETRY_RATE_LIMIT_WAIT` | `10.0` | 429 时额外附加等待秒数 |

### 伪代码

```python
import asyncio, os, re, logging
from typing import Any, AsyncIterator, Callable, TypeVar

logger = logging.getLogger(__name__)

_MAX_RETRIES        = int(os.environ.get("DATACLOUD_LLM_MAX_RETRIES", "3"))
_RETRY_MIN_WAIT     = float(os.environ.get("DATACLOUD_LLM_RETRY_MIN_WAIT", "1.0"))
_RETRY_MAX_WAIT     = float(os.environ.get("DATACLOUD_LLM_RETRY_MAX_WAIT", "30.0"))
_RATE_LIMIT_WAIT    = float(os.environ.get("DATACLOUD_LLM_RETRY_RATE_LIMIT_WAIT", "10.0"))

_RETRYABLE_HTTP = {429, 500, 502, 503, 504}
_NON_RETRYABLE_HTTP = {400, 401, 403}


def _is_retryable(exc: BaseException) -> bool:
    """判断异常是否值得重试。"""
    status = getattr(exc, "status_code", None) or getattr(exc, "status", None)
    if status is not None:
        return status in _RETRYABLE_HTTP
    # 网络层异常（无 status）一律重试
    return True


def _parse_retry_after(exc: BaseException) -> float:
    """从 429 响应头解析 Retry-After 值（秒）。"""
    headers = getattr(exc, "response", None)
    headers = getattr(headers, "headers", {}) if headers else {}
    ra = headers.get("Retry-After") or headers.get("retry-after")
    if ra and re.match(r"^\d+$", str(ra)):
        return float(ra)
    return 0.0


async def stream_llm_call_with_retry(
    llm_call: Callable[..., Any],
    *args: Any,
    **kwargs: Any,
) -> Any:
    """
    对 llm_call(*args, **kwargs) 进行重试包装。
    llm_call 可以是普通 async 函数或返回 async iterator 的函数。
    """
    last_exc: BaseException | None = None
    for attempt in range(_MAX_RETRIES + 1):
        try:
            return await llm_call(*args, **kwargs)
        except Exception as exc:
            last_exc = exc
            is_rate_limit = (
                getattr(exc, "status_code", None) == 429
                or getattr(exc, "status", None) == 429
            )
            if not _is_retryable(exc) or attempt >= _MAX_RETRIES:
                logger.error(
                    "[LLM] 不可重试错误或达到最大重试次数 attempt=%d/%d: %s",
                    attempt, _MAX_RETRIES, exc,
                )
                raise

            # 计算等待时间
            wait = min(_RETRY_MIN_WAIT * (2 ** attempt), _RETRY_MAX_WAIT)
            if is_rate_limit:
                wait += _parse_retry_after(exc) or _RATE_LIMIT_WAIT

            logger.warning(
                "[LLM] 调用失败，第 %d/%d 次重试，等待 %.1f 秒: %s",
                attempt + 1, _MAX_RETRIES, wait, exc,
            )
            await asyncio.sleep(wait)

    raise last_exc  # type: ignore[misc]
```

### 接入位置

在 `react_loop.py` 中的 LLM 调用处（`llm.ainvoke` / `llm.astream`）包裹上述函数：

```python
# 修改前
response = await llm.ainvoke(messages)

# 修改后
response = await stream_llm_call_with_retry(llm.ainvoke, messages)
```

## Layer 2：轮次级 Checkpoint 恢复

### 问题场景

Layer 1 重试 N 次后仍失败（如模型长时间不可用）。此时一整个 react 轮次失败，用户下次重新提问时需要从头开始。

### 方案

在 `react_loop.py` 中，每轮 react 执行完工具调用后，将**当前状态**持久化到 Redis（或数据库）。若下一步 LLM 推理失败，则：

1. 向前端返回**用户可见的提示**（"模型暂时不可用，请稍后继续"）
2. 在 Redis 写入**断点 key**（包含当前 state、已执行的 tool 结果）
3. 用户下次发起对话时，后端检测断点 key → 直接从断点 state 继续，跳过已完成的工具调用

### 断点数据结构

```python
@dataclass
class LLMFailureCheckpoint:
    session_id: str          # 会话 ID
    turn_id: str             # 轮次 ID
    saved_at: str            # ISO8601 时间戳
    state_snapshot: dict     # 当前 GraphState 快照
    completed_steps: int     # 已完成的 react step 数
    error_type: str          # 触发断点的错误类型
    error_message: str       # 错误信息摘要
```

### Redis Key 规范

```
llm:checkpoint:{session_id}:{turn_id}
TTL: 3600 秒（1小时，超时自动清除）
```

### 伪代码

```python
async def _save_llm_failure_checkpoint(
    redis_client,
    session_id: str,
    turn_id: str,
    state: dict,
    completed_steps: int,
    exc: Exception,
) -> None:
    import json, datetime
    checkpoint = {
        "session_id": session_id,
        "turn_id": turn_id,
        "saved_at": datetime.datetime.utcnow().isoformat(),
        "state_snapshot": state,
        "completed_steps": completed_steps,
        "error_type": type(exc).__name__,
        "error_message": str(exc)[:200],
    }
    key = f"llm:checkpoint:{session_id}:{turn_id}"
    await redis_client.setex(key, 3600, json.dumps(checkpoint, ensure_ascii=False))
    logger.warning("[LLM] 已保存断点 key=%s completed_steps=%d", key, completed_steps)


async def _load_llm_checkpoint(redis_client, session_id: str, turn_id: str) -> dict | None:
    import json
    key = f"llm:checkpoint:{session_id}:{turn_id}"
    data = await redis_client.get(key)
    return json.loads(data) if data else None
```

### 全部模型不可用时的用户引导回复

当 Layer 1 重试耗尽、Layer 3 备用模型也失败后，系统保存 Checkpoint，并向用户返回一条**固定的引导文案**，明确告知重试方式：

```
⚠️ 当前 AI 服务暂时不可用（服务器繁忙或网络异常），您的问题已被记录。

请稍等片刻后，**重新发送同样的问题**，系统将自动从中断处继续处理，无需重新开始。
```

**要点说明**

| 要点 | 设计决策 |
|------|---------|
| 引导动作 | 明确告知"重新发送同样的问题"，不要求用户做其他操作 |
| 不暴露技术细节 | 不提 checkpoint、session_id、Redis 等内部概念 |
| 断点有效期 | Checkpoint TTL 为 1 小时，超时后回复变为"请重新提问" |
| 前端无需改动 | 用户正常重发消息即可，后端自动检测断点并恢复 |

**后端返回结构（SSE 最后一帧）**

```python
# react_loop.py 中，全部模型失败后执行
await _save_llm_failure_checkpoint(redis, session_id, turn_id, state, completed_steps, exc)

# 通过 context.emit_chunk 向前端推送引导文案
_CHECKPOINT_REPLY = (
    "⚠️ 当前 AI 服务暂时不可用（服务器繁忙或网络异常），您的问题已被记录。\n\n"
    "请稍等片刻后，**重新发送同样的问题**，系统将自动从中断处继续处理，无需重新开始。"
)
await context.emit_chunk(_CHECKPOINT_REPLY, event_type="answerDelta", content_type="text")
```

**断点过期后的降级回复**

若用户超过 1 小时才重试，Checkpoint 已过期，此时正常走全新流程，同时返回提示：

```
您之前的问题处理记录已过期，将重新为您查询，请稍候。
```

### 优先级

Checkpoint 恢复是**可选增强**，首期可以只实现 Layer 1 + Layer 3，Layer 2 在业务有明确需求时再实施。

## Layer 3：备用模型切换

### 问题场景

主模型（如 GPT-4o）达到每日配额上限或服务不可用，需要临时切换到备用模型（如 GPT-3.5-turbo 或其他厂商模型）。

### 方案

在 `react_loop.py` 初始化阶段，读取备用模型配置，构建 `fallback_llm`。当 Layer 1 重试全部失败后，捕获异常，用 `fallback_llm` 替换 `llm` 继续执行。

### 环境变量

| 变量名 | 默认值 | 说明 |
|-------|-------|------|
| `DATACLOUD_LLM_FALLBACK_ENABLED` | `false` | 是否启用备用模型 |
| `DATACLOUD_LLM_FALLBACK_MODEL` | `gpt-3.5-turbo` | 备用模型名称 |
| `DATACLOUD_LLM_FALLBACK_BASE_URL` | （同主模型） | 备用模型 API 地址 |
| `DATACLOUD_LLM_FALLBACK_API_KEY` | （同主模型） | 备用模型 API Key |

### 伪代码

```python
def _build_fallback_llm() -> Any | None:
    """构建备用 LLM，未配置时返回 None。"""
    if os.environ.get("DATACLOUD_LLM_FALLBACK_ENABLED", "false").lower() != "true":
        return None
    from langchain_openai import ChatOpenAI  # 或其他 LLM SDK
    return ChatOpenAI(
        model=os.environ.get("DATACLOUD_LLM_FALLBACK_MODEL", "gpt-3.5-turbo"),
        base_url=os.environ.get("DATACLOUD_LLM_FALLBACK_BASE_URL"),
        api_key=os.environ.get("DATACLOUD_LLM_FALLBACK_API_KEY"),
        streaming=True,
    )


# react_loop.py 中的调用逻辑
async def run_react_loop(state, llm, ...):
    fallback_llm = _build_fallback_llm()
    current_llm = llm

    for step in range(max_steps):
        try:
            response = await stream_llm_call_with_retry(current_llm.ainvoke, messages)
        except Exception as exc:
            if fallback_llm is not None and current_llm is not fallback_llm:
                logger.warning(
                    "[LLM] 主模型失败，切换到备用模型: %s",
                    os.environ.get("DATACLOUD_LLM_FALLBACK_MODEL"),
                )
                current_llm = fallback_llm
                response = await stream_llm_call_with_retry(current_llm.ainvoke, messages)
            else:
                raise
```

## 优先级矩阵

| Layer | 实现复杂度 | 效果 | 优先级 |
|-------|----------|------|-------|
| Layer 1：请求级重试 | 低（~50 行） | 覆盖 90% 的偶发性失败 | **P0 立即实施** |
| Layer 3：备用模型切换 | 低（~30 行） | 覆盖主模型不可用场景 | **P1 按需实施** |
| Layer 2：Checkpoint 恢复 | 中（~100 行 + Redis） | 覆盖长时间不可用场景 | P2 有需求再做 |

## 日志规范

所有重试和切换行为统一使用 `WARNING` 级别日志，格式：

```
[LLM] 调用失败，第 1/3 次重试，等待 1.0 秒: <错误描述>
[LLM] 调用失败，第 2/3 次重试，等待 2.0 秒: <错误描述>
[LLM] 不可重试错误或达到最大重试次数 attempt=3/3: <错误描述>
[LLM] 主模型失败，切换到备用模型: gpt-3.5-turbo
[LLM] 已保存断点 key=llm:checkpoint:xxx:yyy completed_steps=2
```

## 注意事项

1. **不要重试非幂等操作**：若 LLM 调用已产生副作用（如写 DB），重试前需确认安全
2. **流式调用（astream）的重试**：流式调用需要重新发起请求，不能从中断位置续传；重试时需重新传完整 messages
3. **日志采样**：高频限流场景下日志量大，可设置 `DEBUG` 模式下才打印每次重试详情
4. **超时设置**：建议同时为 LLM 客户端设置合理的 `timeout`（如 60s），避免 hang 住
