# 本体调用优化设计

## 问题背景

### 问题1：system_prompt 未生效
当前在 `worker.py:587` 中传入了 `prompts_dict`（包含 `system_prompt` 和 `task_prompt`）到 `create_agent` 的 `prompts_overwrite` 参数，但是：
- `agent.py:48` 注释中标记 `prompts_overwrite` 为 legacy，已被忽略
- `agent.py:71` 实际传入 `_create_deep_agent` 的 `system_prompt` 参数为 None
- 导致业务配置的 system_prompt 无法生效

示例配置（未生效）：
```python
prompts_overwrite = {
    'system_prompt': '我是亦庄产业大脑数字员工，我可以提供亦庄产业链分析、管理网格综合分析、企业综合分析表服务。',
    'task_prompt': '请根据以下原则来处理问题。\n# 单一技能优先原则：...'
}
```

### 问题2：tools_dict 直接注入到 agent
当前在 `worker.py:588` 中，`tools_dict` 直接注入到 agent 中：
```python
create_agent(
    prompts_overwrite=prompts_dict,
    tools=tools_dict,  # 直接注入
)
```

在 `agent.py:122` 中，这些工具被添加到 `all_tools`：
```python
all_tools = register_all_tools()
if tools:
    all_tools.extend(tools.values())  # 直接注册
```

**问题**：
1. 工具数量膨胀：每个业务域都会注册大量视图/对象工具，导致 LLM 上下文过大
2. 维护困难：业务视图变更需要重新构建 agent graph
3. 缺乏灵活性：无法根据运行时上下文动态选择可用工具

### 问题3：知识注入不完整
当前 `KnowledgeInjectionMiddleware` 仅注入了简单的术语匹配信息：
```python
# 当前实现（knowledge_injection.py:78-86）
schema_lines = ["<ontology_context>"]
schema_lines.append("相关本体术语：")
for match in term_matches[:5]:
    term_name = match.get("term_name", "")
    term_type = match.get("term_type_code", "")
    match_type = match.get("match_type", "")
    schema_lines.append(f"- {term_name} ({term_type}) [匹配类型: {match_type}]")
```

**缺少关键信息**：
- 对象的完整结构（属性列表、属性类型）
- 对象间的关系
- 对象支持的 action
- 属性的编码和别名（用于参数抽取）

### 问题4：思考过程推送缺失

当前 Deep Agents SDK 架构下，缺少关键思考过程的前端推送：

**4.1 知识注入过程不可见**
- `KnowledgeInjectionMiddleware` 在每次 LLM 调用前注入本体知识
- 注入的对象元数据（属性、关系、动作）对用户不可见
- 用户无法了解 agent 获取了哪些上下文信息

**4.2 工具调用过程不透明**
- 虽然 Deep Agents SDK 内部有工具调用机制
- 但需要确保与老版本一致的推送格式
- 需要推送：工具名称、调用原因（reason）、入参、返参

**老版本推送机制**（参考 `tool_wrapper.py`）：
```python
# 工具调用层级结构
async with gateway_context.sub_step(tool_name):
    # 第一层：工具名称
    if reason:
        await _emit_child_think(gateway_context, reason)  # 第二层：调用原因
    
    await _run_tool()  # 执行工具
    
    # 第三层：返回内容
    await _emit_tool_detail(
        gateway_context,
        "工具返回",
        ctx.get("tool_output"),
    )
```

**推送格式**：
- 使用 `gateway_context.sub_step(title)` 创建层级节点
- 使用 `_emit_child_think(gateway_context, content)` 推送子节点内容
- 使用 `_emit_tool_detail(gateway_context, title, detail)` 推送详细信息
- 事件类型：`event_type="1001"`, `content_type="1002"` (思考过程)

## 设计目标

1. **system_prompt 生效**：确保 worker 传入的业务自定义 system_prompt 能够正确覆盖默认 prompt
2. **工具动态化**：OBJECT/VIEW 类型不再直接注入 agent，而是通过知识注入传递元数据；其他类型（AGENT、FUNCTION 等）保持原有注册方式
3. **知识注入增强**：注入对象的完整元数据（名称、编码、描述、属性、关系、action）
4. **统一调用入口**：所有本体操作统一通过 `query_objects` 和 `execute_action` 两个核心工具
5. **兼容性保证**：非 OBJECT/VIEW 类型的工具（如 AGENT 委托）保持原有功能不变
6. **思考过程可见**：知识注入和工具调用过程实时推送到前端，保持与老版本一致的用户体验

## 实施方案

### 阶段执行顺序

本方案分为4个核心阶段 + 1个未来优化方向，建议按以下顺序实施：

**核心实施阶段**：

1. **阶段1：修复 system_prompt**（基础 - 必须先完成）
   - 确保 worker 传入的 prompts_overwrite 能够生效
   - 为后续所有阶段提供 prompts 处理基础
   - **依赖**：无

2. **阶段2：工具类型分离与知识注入**（核心 - 架构改造）
   - 将 OBJECT/VIEW 类型工具改为通过知识注入传递
   - 实现完整的知识注入功能
   - 保持其他类型工具（AGENT/FUNCTION）的兼容性
   - **依赖**：阶段1

3. **阶段3：统一 query_objects 工具**（可选 - 参数规范化）
   - 统一对象查询和视图查询的参数格式
   - **依赖**：无（可与阶段4并行，或在阶段4之前/之后）
   - **优先级**：低（主要优化命名统一性，不影响核心功能）

4. **阶段4：思考过程推送**（用户体验增强）
   - 实现知识注入和工具调用的流式推送
   - 包含 AGENT 类型工具的异步调用支持
   - **依赖**：阶段2（需要 middleware 架构）

**未来优化方向**（可选）：

5. **基于意图的精细化知识注入**
   - 当挂载对象数量>10个或上下文压力增大时实施
   - 使用 search_knowledge 分析用户意图，仅注入相关对象
   - **前提**：阶段2已满足当前需求，暂无明显瓶颈

```python
def create_agent(
    *,
    model: str | None = None,
    api_key: str | None = None,
    base_url: str | None = None,
    temperature: float = 0.7,
    locale: str | None = None,
    system_prompt: str | None = None,
    prompts_overwrite: dict[str, Any] | None = None,
    tools: dict[str, Any] | None = None,
) -> Any:
    """Create a deep agent for DataCloud using Deep Agents SDK.

    Args:
        model: LLM model name
        api_key: API key for LLM provider
        base_url: Base URL for LLM provider
        temperature: LLM temperature
        locale: Locale for agent (default: zh_CN)
        system_prompt: Custom system prompt
        prompts_overwrite: Prompt overrides from worker (优先级最高)
            - system_prompt: 覆盖默认 system prompt
            - task_prompt: 附加任务指导（将追加到 system_prompt）
        tools: Additional tools to register

    Returns:
        Compiled agent graph
    """
    # Resolve locale
    resolved_locale = locale or os.getenv("DATACLOUD_AGENT_LOCALE", "zh_CN")
    supported = get_supported_locales()
    if resolved_locale not in supported:
        logger.warning(
            "create_agent: locale=%r is not supported (supported: %s). Falling back to zh_CN.",
            resolved_locale,
            supported,
        )
        resolved_locale = "zh_CN"

    # 优先级: prompts_overwrite['system_prompt'] > system_prompt > default
    final_system_prompt = system_prompt
    task_prompt = None
    
    if prompts_overwrite:
        if "system_prompt" in prompts_overwrite:
            final_system_prompt = prompts_overwrite["system_prompt"]
            logger.info("create_agent: using system_prompt from prompts_overwrite")
        if "task_prompt" in prompts_overwrite:
            task_prompt = prompts_overwrite["task_prompt"]
            logger.info("create_agent: using task_prompt from prompts_overwrite")

    return _create_deep_agent(
        model=model,
        api_key=api_key,
        base_url=base_url,
        temperature=temperature,
        locale=resolved_locale,
        system_prompt=final_system_prompt,  # 传递解析后的 system_prompt
        task_prompt=task_prompt,  # 新增参数
        tools=tools,
    )
```

#### 1.2 修改 `_create_deep_agent` 函数

```python
def _create_deep_agent(
    *,
    model: str | None = None,
    api_key: str | None = None,
    base_url: str | None = None,
    temperature: float = 0.7,
    locale: str = "zh_CN",
    system_prompt: str | None = None,
    task_prompt: str | None = None,  # 新增
    tools: dict[str, Any] | None = None,
    mounted_objects: list[str] | None = None,  # 新增：阶段2传入
) -> Any:
    """Create agent using Deep Agents SDK.

    Args:
        model: LLM model name
        api_key: API key
        base_url: Base URL
        temperature: Temperature
        locale: Locale
        system_prompt: System prompt (优先使用此参数)
        task_prompt: Additional task guidance (追加到 system_prompt)
        tools: Additional tools (阶段2后为 other_tools，不包含 OBJECT/VIEW)
        mounted_objects: 挂载的对象/视图列表（阶段2新增）

    Returns:
        Compiled agent
    """
    import pathlib

    from deepagents import create_deep_agent
    from datacloud_analysis.backend import create_datacloud_backend
    from datacloud_analysis.context import DatacloudContext
    from datacloud_analysis.middlewares import (
        DatacloudOutputMiddleware,
        KnowledgeInjectionMiddleware,
        WorkspaceInitMiddleware,
    )
    from datacloud_analysis.subagents import CODE_EXECUTOR_SUBAGENT
    from datacloud_analysis.tools.registry import register_all_tools

    logger.info("create_agent: Using Deep Agents SDK (locale=%s)", locale)

    # Register OQL tools (emit_result injected by DatacloudOutputMiddleware)
    all_tools = register_all_tools()
    if tools:
        # 阶段2后，tools 仅包含 other_tools (AGENT/FUNCTION等类型)
        # OBJECT/VIEW 类型通过 mounted_objects 传递给 KnowledgeInjectionMiddleware
        all_tools.extend(tools.values())

    # Build system prompt: default + custom + task_prompt
    final_system_prompt = system_prompt or _build_default_system_prompt(locale)
    
    if task_prompt:
        final_system_prompt = f"{final_system_prompt}\n\n# 任务处理指导\n{task_prompt}"
        logger.info("create_agent: appended task_prompt to system_prompt")

    # Workspace dir for backend
    workspace_dir = os.getcwd()

    # DatacloudBackend — limits file operations to workspace_dir
    backend = create_datacloud_backend(workspace_dir)

    # Built-in skills directory
    builtin_skills_dir = str(pathlib.Path(__file__).parent / "skills" / "builtin")
    skill_sources = [builtin_skills_dir]

    # Middleware stack
    middlewares = [
        KnowledgeInjectionMiddleware(mounted_objects=mounted_objects),  # 阶段2传递挂载对象
        DatacloudOutputMiddleware(),
        WorkspaceInitMiddleware(
            workspace_dir=workspace_dir,
            agent_name="DataCloud Agent",
        ),
    ]

    # Resolve checkpointer
    checkpointer: Any = None
    try:
        from datacloud_analysis.session.checkpointer import get_checkpointer

        checkpointer = get_checkpointer()
        logger.info("create_agent: using PG checkpointer")
    except RuntimeError:
        logger.warning(
            "create_agent: checkpointer not initialized — running without checkpointing. "
            "Call `await bootstrap.setup()` before create_agent() to enable interrupt/resume."
        )

    # Resolve model
    resolved_model: Any
    if base_url:
        from langchain_openai import ChatOpenAI

        resolved_model = ChatOpenAI(
            model=model or "gpt-4o",
            base_url=base_url,
            api_key=api_key or "sk-placeholder",
            temperature=temperature,
        )
        logger.info(
            "create_agent: using ChatOpenAI with custom base_url=%s model=%s",
            base_url,
            model,
        )
    else:
        resolved_model = model or "claude-sonnet-4-6"

    # Create compiled agent with Deep Agents SDK
    compiled = create_deep_agent(
        model=resolved_model,
        tools=all_tools,
        system_prompt=final_system_prompt,  # 使用构建后的完整 prompt
        middleware=middlewares,
        context_schema=DatacloudContext,
        backend=backend,
        skills=skill_sources,
        subagents=[CODE_EXECUTOR_SUBAGENT],
        checkpointer=checkpointer,
    )

    return compiled
```

### 阶段2：工具类型分离与知识注入

#### 目标
将 worker 中的 `tools_dict` 按类型分离处理：
- **OBJECT/VIEW 类型**：改为通过知识注入方式传递元数据（不再注册为工具函数）
- **其他类型（AGENT/FUNCTION等）**：保持原有方式，继续通过 `tools` 参数注册为工具函数

**核心改变**：
1. OBJECT/VIEW 类型工具 → 提取为 `mounted_objects` 列表，传递给 `KnowledgeInjectionMiddleware`
2. 其他类型工具 → 保留为 `other_tools`，传递给 `create_agent` 的 `tools` 参数
3. 实现完整的知识注入机制，将 OBJECT/VIEW 的元数据注入到 system_message

**兼容性保证**：非 OBJECT/VIEW 类型的工具（如 AGENT 委托、自定义 FUNCTION）不受影响，仍然正常工作。

#### 2.6 完整的知识注入实现

**核心思路**：在每个问题（LLM 调用）时通过 Middleware 注入 Schema，以便未来支持根据用户问题精细化过滤对象。

**为什么必须在 Middleware 中注入？**

✅ **正确方式**（问题时注入 + 状态去重）：
```python
# 在 KnowledgeInjectionMiddleware.awrap_model_call 中注入
async def awrap_model_call(self, request: ModelRequest, handler) -> Any:
    # ✅ 使用 LangGraph state 检查是否已注入（而不是检查 request.system_message）
    state = request.runtime.state
    if state.get("_ontology_injected", False):
        return await handler(request)  # 已注入，跳过
    
    # 1. TODO: 精细化过滤（待罗彦卓实现）
    #    根据用户问题从 mounted_objects 中过滤出相关对象
    user_query = self._extract_user_query(request)
    relevant_objects = self._filter_relevant_objects(user_query, self.mounted_objects)
    
    # 2. 注入相关对象的 Schema
    ontology_schema = await self._inject_mounted_objects(relevant_objects)
    request = request.override(system_message=append_to_system_message(...))
    
    # 3. 标记已注入（写入 state，持久化到 checkpoint）
    state["_ontology_injected"] = True
    
    return await handler(request)
```

**为什么不在 Agent 启动时注入？**

❌ **错误方式**（在 `_create_deep_agent` 中一次性注入）：
- 无法支持未来优化：根据用户问题精细化过滤对象
- 例如：用户问题只涉及"企业"对象，但启动时已注入所有5个对象
- 无法在运行时动态调整注入内容

**Middleware 方案的优势**：
1. **支持精细化过滤**：可以在每个问题时分析用户意图，只注入相关对象
2. **去重机制可靠**：使用 LangGraph state（持久化到 checkpoint），而不是 request.system_message
3. **未来可扩展**：为罗彦卓实现意图分析预留接口

✅ **完整实现**：

```python
# datacloud-analysis/src/datacloud_analysis/middlewares/knowledge_injection.py

"""知识注入中间件 - 在每个问题时注入本体 Schema"""

from __future__ import annotations
from typing import Any, Callable, Awaitable
import logging

from langchain.agents.middleware.types import AgentMiddleware, ModelRequest

logger = logging.getLogger(__name__)


class KnowledgeInjectionMiddleware(AgentMiddleware):
    """
    知识注入中间件
    
    在每次 LLM 调用时（每个问题）注入本体 Schema，支持未来精细化过滤。
    使用 LangGraph state 去重，避免重复注入。
    """
    
    tools: list = []
    
    def __init__(self, mounted_objects: list[str] | None = None):
        """
        Args:
            mounted_objects: 挂载的对象/视图列表（完整列表）
        """
        super().__init__()
        self.mounted_objects = mounted_objects or []
    
    async def awrap_model_call(
        self,
        request: ModelRequest,
        handler: Callable[[ModelRequest], Awaitable[Any]],
    ) -> Any:
        """在 LLM 调用前注入本体 Schema。"""
        # 去重检查：使用 LangGraph state（持久化到 checkpoint）
        state = request.runtime.state
        if state.get("_ontology_injected", False):
            # 已注入，跳过
            return await handler(request)
        
        if not self.mounted_objects:
            # 没有挂载对象，跳过
            return await handler(request)
        
        try:
            # 1. 提取用户问题
            user_query = self._extract_user_query(request)
            
            # 2. 精细化过滤（根据用户问题筛选相关对象）
            #    TODO: 待罗彦卓实现，当前返回所有挂载对象
            relevant_objects = self._filter_relevant_objects(user_query, self.mounted_objects)
            
            # 3. 构建 Schema
            ontology_schema = await self._build_ontology_schema(relevant_objects)
            
            if ontology_schema:
                # 4. 注入到 system_message
                from datacloud_analysis.utils.prompt_utils import append_to_system_message
                
                updated_request = request.override(
                    system_message=append_to_system_message(
                        request.system_message,
                        ontology_schema
                    )
                )
                
                # 5. 标记已注入（写入 state）
                state["_ontology_injected"] = True
```

```python
def format_object_schema(
    ontology_class,  # OntologyClass
    all_relations,  # list[OntologyRelation]
    ontology_loader
) -> str:
    """格式化单个对象的完整 Schema。"""
    lines = []
    
    # 1. 对象基本信息（名称、编码、描述）
    lines.append(f"### 对象类型：{ontology_class.object_name}")
    lines.append(f"**object_type**: `{ontology_class.object_code}`")
    
    # 判断是否为视图
    is_view = "view" in ontology_class.object_code.lower() or "ads_" in ontology_class.object_code
    if is_view:
        lines.append("**类型**: 预定义视图")
    
    if ontology_class.description:
        lines.append(f"**描述**: {ontology_class.description}")
    lines.append("")
    
    # 2. 属性列表（属性名称、属性编码、类型、说明、别名）
    if ontology_class.fields:
        lines.append("#### 属性列表")
        lines.append("| 属性名称 | 属性编码 | 类型 | 说明 | 别名 |")
        lines.append("|---------|---------|------|------|------|")
        
        for field in ontology_class.fields[:10]:  # 最多显示10个属性
            field_name = field.field_name
            field_code = field.field_code
            field_type = field.field_type
            description = field.description or ""
            aliases = ", ".join(field.aliases[:3]) if field.aliases else ""  # 最多3个别名
            
            lines.append(f"| {field_name} | {field_code} | {field_type} | {description} | {aliases} |")
        
        lines.append("")
    
    # 3. 关联关系（仅对象，视图不显示关系）
    if not is_view:
        object_relations = filter_object_relations(
            ontology_class.object_code, 
            all_relations,
            ontology_loader
        )
        if object_relations:
            lines.append("#### 关联关系")
            for rel in object_relations[:5]:  # 最多显示5个关系
                source_name = rel["source_name"]
                target_name = rel["target_name"]
                rel_type = rel["relation_type"]
                desc = rel["description"] or rel["relation_name"]
                
                lines.append(f"- **{source_name} → {target_name}** ({rel_type}): {desc}")
            lines.append("")
    
    # 4. 可用动作（动作名称、动作编码、参数列表）
    if ontology_class.actions:
        lines.append("#### 可用动作")
        for action in ontology_class.actions[:5]:  # 最多显示5个动作
            action_name = action.action_name
            action_code = action.action_code
            description = action.description or ""
            
            lines.append(f"- **{action_name}** (`{action_code}`)")
            if description:
                lines.append(f"  - 描述: {description}")
            
            # 参数列表
            if action.params:
                params_str = []
                for param in action.params[:5]:  # 最多显示5个参数
                    param_code = param.param_code
                    param_type = param.param_type
                    required = "必填" if param.required else "可选"
                    params_str.append(f"`{param_code}` ({param_type}, {required})")
                
                if params_str:
                    lines.append(f"  - 参数: {', '.join(params_str)}")
            
            lines.append("")
    
    return "\n".join(lines)

def filter_object_relations(
    object_code: str,
    all_relations: list,  # list[OntologyRelation]
    ontology_loader
) -> list[dict]:
    """过滤与指定对象相关的关系。"""
    relations = []
    
    for rel in all_relations:
        if rel.source_class == object_code or rel.target_class == object_code:
            # 获取源和目标对象的名称
            try:
                source_obj = ontology_loader.get_ontology_class(rel.source_class)
                source_name = source_obj.object_name
            except:
                source_name = rel.source_class
            
            try:
                target_obj = ontology_loader.get_ontology_class(rel.target_class)
                target_name = target_obj.object_name
            except:
                target_name = rel.target_class
            
            relations.append({
                "source_class": rel.source_class,
                "source_name": source_name,
                "target_class": rel.target_class,
                "target_name": target_name,
                "relation_type": rel.relation_type,
                "relation_name": rel.relation_name,
                "description": rel.description,
            })
    
    return relations
```

#### 4.1 思考过程推送机制说明

**重要说明**：经过对 Deep Agents SDK 和 LangChain 源码的深入分析，发现以下关键信息：

**LangChain / Deep Agents 的思考过程推送机制**：

1. **LangChain Callback 系统**（标准机制）：
   - `on_tool_start(serialized, input_str, run_id, ...)`: 工具开始执行时触发
   - `on_tool_end(output, run_id, ...)`: 工具执行成功后触发
   - `on_tool_error(error, run_id, ...)`: 工具执行出错时触发
   - 这些回调在工具的 `run()` / `arun()` 方法中自动调用（`langchain_core/tools/base.py:942-1003`）

2. **LangChain 自定义事件机制**（推送任意思考过程）：
   - `adispatch_custom_event(name, data, config=config)`: 异步推送自定义事件
   - `dispatch_custom_event(name, data, config=config)`: 同步推送自定义事件
   - 通过 Callback Handler 的 `on_custom_event(name, data, run_id, ...)` 接收
   - **用途**：在任意位置（如中间件、工具内部）推送自定义的思考过程信息

3. **Deep Agents SDK 的设计**：
   - Deep Agents **不提供**额外的工具调用推送机制
   - 完全依赖 LangChain 的标准 Callback 系统
   - 工具调用自动通过 `on_tool_start/end/error` 推送
   - 自定义思考过程需要在中间件或工具中手动调用 `adispatch_custom_event`

**当前代码架构的现状**：

1. **当前代码使用 Deep Agents SDK 架构**：
   - 工具直接通过 `register_all_tools()` 注册到 `create_deep_agent(tools=[...])`
   - 没有自定义的 ReAct 循环或 tool_wrapper 层
   - Hook 插件系统代码存在（`tool_hook_plugins/`）但**未被使用**

2. **老版本代码使用自定义 ReAct 循环**：
   - 通过 `tool_wrapper.py` 包装工具调用
   - 集成 Hook 插件系统（`semantic_param_enhancer`）
   - 在 `react_loop.py` 中手动管理工具调度和推送

**当前架构下的实现方案**：

由于当前代码已经迁移到 Deep Agents SDK，我们需要采用新的实现方式。

**核心需求**：
1. ✅ 知识注入推送（使用 `adispatch_custom_event`）
2. ✅ 工具调用推送（工具名称、入参、出参）- 需要层级结构
3. ❌ 参数增强（暂不实现）

**技术选型**：

经过对比 LangChain Callback 和 Middleware 机制，发现：
- **LangChain Callback**：无法访问 `gateway_context`，无法创建层级结构（`sub_step`、`message_id`）
- **Middleware `awrap_tool_call`**：可以访问 `gateway_context`，可以创建层级推送

**结论**：必须使用 **Middleware 的 `awrap_tool_call`** 实现工具调用推送。

**方案：创建 ToolCallLoggingMiddleware**

功能：
- ✅ 推送工具调用过程（工具名称、入参、出参）
- ✅ 创建层级结构（与老版本一致）
- ❌ 不做参数增强

#### 4.2 创建 ToolCallLoggingMiddleware

```python
# datacloud-analysis/src/datacloud_analysis/middlewares/tool_logging.py

"""工具调用日志中间件 - 推送工具调用过程到前端"""

from __future__ import annotations
from typing import Any, Callable, Awaitable
import logging

from langchain.agents.middleware.types import AgentMiddleware
from langgraph.prebuilt.tool_node import ToolCallRequest
from datacloud_data_sdk.stream_text import coerce_stream_chunk_text

logger = logging.getLogger(__name__)


class ToolCallLoggingMiddleware(AgentMiddleware):
    """
    工具调用日志中间件
    
    推送工具调用过程到前端，包括：
    - 工具名称
    - 入参（tool_call["args"]）
    - 出参（工具返回值）
    
    创建层级结构（与老版本一致）：
    - 层级1：工具名称（使用 sub_step）
    - 层级2：入参详情
    - 层级3：出参详情
    """
    
    tools: list = []
    
    async def awrap_tool_call(
        self,
        request: ToolCallRequest,
        handler: Callable[[ToolCallRequest], Awaitable[Any]],
    ) -> Any:
        """在工具调用前后推送消息到前端。"""
        # 获取 gateway_context
        gateway_context = self._get_gateway_context(request)
        if gateway_context is None:
            # 没有 gateway_context，直接执行工具
            return await handler(request)
        
        # 提取工具信息
        tool_name = request.tool_call.get("name", "unknown_tool")
        tool_args = request.tool_call.get("args", {})
        
        # 检查是否是 AGENT 类型工具
        is_agent_delegate = self._is_agent_delegate_tool(request)
        
        try:
            # 层级1：创建工具调用节点
            async with gateway_context.sub_step(tool_name):
                # 层级2：推送入参
                await self._emit_child_think(
                    gateway_context,
                    f"调用参数：{self._format_args(tool_args)}"
                )
                
                if is_agent_delegate:
                    # AGENT 类型工具：使用 delegate_parent_scope
                    result = await self._handle_agent_delegate(
                        gateway_context,
                        request,
                        handler,
                    )
                else:
                    # 普通工具：直接执行
                    result = await handler(request)
                
                # 层级3：推送出参
                await self._emit_child_think(
                    gateway_context,
                    f"返回结果：{self._format_result(result)}"
                )
                
                return result
        
        except Exception as exc:
            # 推送错误信息
            await self._emit_child_think(
                gateway_context,
                f"执行失败：{str(exc)}"
            )
            raise
    
    def _is_agent_delegate_tool(self, request: ToolCallRequest) -> bool:
        """检查工具是否是 AGENT 类型（agent_delegate）。"""
        try:
            tool = request.tool
            is_delegate_flag = getattr(tool, "_is_agent_delegate", False)
            return isinstance(is_delegate_flag, bool) and is_delegate_flag
        except Exception:
            return False
    
    async def _handle_agent_delegate(
        self,
        gateway_context: Any,
        request: ToolCallRequest,
        handler: Callable[[ToolCallRequest], Awaitable[Any]],
    ) -> Any:
        """处理 AGENT 类型工具调用（子 Agent 异步调用）。
        
        使用 delegate_parent_scope 创建代理作用域，确保子 Agent 的流式输出
        能够正确挂载到父 Agent 的消息层级下。
        """
        from contextlib import nullcontext
        
        # 获取代理作用域工厂
        delegate_parent_scope_factory = getattr(
            gateway_context,
            "delegate_parent_scope",
            None,
        )
        delegate_parent_scope = nullcontext()
        
        if callable(delegate_parent_scope_factory):
            # 创建代理作用域，传入当前 message_id
            current_message_id = str(getattr(gateway_context, "message_id", "") or "")
            delegate_parent_scope = delegate_parent_scope_factory(current_message_id)
        
        # 在代理作用域内执行工具
        with delegate_parent_scope:
            # 调用工具（内部会触发 interrupt，等待子 Agent 完成）
            result = await handler(request)
        
        return result
    
    def _get_gateway_context(self, request: ToolCallRequest) -> Any | None:
        """从 request 中获取 gateway_context。"""
        try:
            return request.runtime.config.get("configurable", {}).get("gateway_context")
        except Exception:
            return None
    
    async def _emit_child_think(
        self,
        gateway_context: Any,
        text: str,
    ) -> None:
        """推送子节点思考内容（与老版本实现一致）。"""
        try:
            from by_framework import StreamChunkEvent
            
            # 生成子节点 message_id
            child_message_id = self._new_message_id(gateway_context)
            parent_message_id = str(getattr(gateway_context, "message_id", "") or "")
            
            # 创建消息块
            chunk = StreamChunkEvent(content=coerce_stream_chunk_text(text))
            
            # 推送参数
            emit_kwargs = {
                "event_type": "1001",  # 思考过程事件类型
                "content_type": "1002",  # 思考内容类型
            }
            if child_message_id:
                emit_kwargs["message_id"] = child_message_id
            if parent_message_id:
                emit_kwargs["parent_message_id"] = parent_message_id
            
            await gateway_context.emit_chunk(chunk, **emit_kwargs)
        
        except Exception as exc:
            logger.debug("_emit_child_think failed: %s", exc)
    
    def _new_message_id(self, gateway_context: Any) -> str:
        """生成新的 message_id（与老版本实现一致）。"""
        generate_message_id = getattr(gateway_context, "generate_message_id", None)
        if callable(generate_message_id):
            try:
                return str(generate_message_id() or "")
            except Exception:
                return ""
        return ""
    
    def _format_args(self, args: dict[str, Any]) -> str:
        """格式化入参为可读字符串。"""
        if not args:
            return "{}"
        
        # 简化展示，避免过长
        import json
        try:
            formatted = json.dumps(args, ensure_ascii=False, indent=2)
            # 限制长度
            if len(formatted) > 500:
                return formatted[:500] + "\n... (已截断)"
            return formatted
        except Exception:
            return str(args)
    
    def _format_result(self, result: Any) -> str:
        """格式化返回结果为可读字符串。"""
        # 如果是字典且包含 status 字段，提取关键信息
        if isinstance(result, dict):
            status = result.get("status")
            if status:
                record_count = 0
                if isinstance(result.get("result"), dict):
                    records = result["result"].get("records", [])
                    if isinstance(records, list):
                        record_count = len(records)
                
                return f"状态: {status}, 记录数: {record_count}"
        
        # 通用格式化
        import json
        try:
            formatted = json.dumps(result, ensure_ascii=False, indent=2)
            if len(formatted) > 500:
                return formatted[:500] + "\n... (已截断)"
            return formatted
        except Exception:
            return str(result)[:500]
```

### 风险1：知识注入上下文过大

  - 最多注入 3 个对象

  - 使用结构化的 Markdown 格式
  - 提供清晰的调用示例
  - 在 system_prompt 中强调如何使用注入的知识

### 风险3：非 OBJECT/VIEW 类型工具兼容性
- **风险等级**：低
- **影响范围**：AGENT 委托、自定义 FUNCTION 等工具类型
- **AGENT 类型工具特殊处理**：
  - resourceBizType == "AGENT" 的工具会被标记 `_is_agent_delegate = True`
  - 在 ToolCallLoggingMiddleware 中检测并特殊处理
  - 使用 `delegate_parent_scope()` 创建代理作用域
  - 通过 `interrupt()` 机制实现异步调用和恢复
  - 子 Agent 的流式输出正确挂载到父消息层级下
- **缓解措施**：
  - 在 Worker 的 `_build_graph` 中明确分离工具类型
  - OBJECT/VIEW 类型提取为 mounted_objects，通过知识注入处理
  - 其他类型（AGENT、FUNCTION 等）保留为 other_tools，直接注册到 agent
  - 添加测试用例验证兼容性（测试4、测试6）
  - 确保现有 AGENT 委托功能不受影响
  - ToolCallLoggingMiddleware 中实现 `_is_agent_delegate_tool()` 检测
  - ToolCallLoggingMiddleware 中实现 `_handle_agent_delegate()` 特殊处理

### 风险4：视图查询性能
- **风险等级**：低
- **缓解措施**：
  - OQLRouter 内部缓存视图元数据
  - 视图查询直接转发到 OQL 引擎，无额外开销
  - 监控查询性能指标

### 风险5：思考过程推送性能影响
- **风险等级**：低
- **影响范围**：知识注入和工具调用的推送可能增加响应延迟
- **缓解措施**：
  - 推送操作异步执行，不阻塞主流程
  - 推送内容精简，避免大量数据传输
  - 推送失败时静默处理，不影响核心功能
  - 使用 `try/except` 包裹推送逻辑，确保异常不中断执行

### 风险6：子 Agent 异步调用的中断/恢复机制
- **风险等级**：中
- **影响范围**：AGENT 类型工具的异步调用和恢复流程
- **关键机制**：
  - **interrupt() 挂起**：调用 AGENT 工具时触发 interrupt，等待子 Agent 完成
  - **ResumeCommand 恢复**：子 Agent 完成后发送 ResumeCommand 恢复父 Agent
  - **重放抑制**：恢复时避免重复推送工具结果（`_consume_delegate_resume_replay_suppression`）
  - **消息层级维护**：通过 `delegate_parent_message_id` 维护父子消息关系
- **潜在风险**：
  - 子 Agent 调用失败导致父 Agent 永久挂起
  - 恢复时消息层级关系错乱
  - 重复推送工具结果导致前端显示混乱
  - LangGraph checkpoint 恢复时重放工具调用
- **缓解措施**：
  - Worker 端设置超时机制，避免永久挂起
  - 在 metadata 中记录 `delegate_parent_message_id` 和 `parent_resume_target`
  - 使用 `_consume_delegate_resume_replay_suppression` 标志避免重放时重复推送
  - delegate_parent_scope 确保子 Agent 消息正确挂载
  - 添加测试用例验证中断/恢复流程（测试6）
  - 日志记录关键步骤（interrupt、resume、delegate_scope）便于排查问题
