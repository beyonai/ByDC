# 知识增强方案

## 1. 背景与问题

知识增强链路负责：识别业务术语 → 补充知识上下文 → 将增强结果交给 planning / execution / end 节点。但结合现有代码与最新联调日志，暴露出以下问题：

1. knowledge_enhance 与后续节点的输入输出契约模糊，难以定位数据在链路中何处丢失。
2. enriched_query 仍停留在“术语替换/轻量改写”，未真正承载“原始问题 + 已确权知识摘要”。
3. knowledge_payload 虽保留完整知识，但下游不会主动读取，导致知识补充无法落地。
4. 有歧义术语时仍可能阻断流程；前端思考过程也缺少“原始问题 → 增强后问题 → 知识摘要”闭环。
5. 无统一的摘要去噪、截断和去重规则，prompt 长度与质量难控。

## 2. 设计目标

1. 仅对**已确权术语**补充知识；歧义术语只做日志提示。
2. enriched_query 升级为“原始问题 + 已确权知识摘要”的主输入文本，供 planning / execution / react_runtime 消费。
3. 完整知识保存在 knowledge_payload 这一结构化快照中，便于回溯和二次加工，但不会在下游被直接解析。
4. knowledge_snippets 继续承担轻量上下文补充，用于工具调用时提供局部解释；与 enriched_query 形成分工。
5. 前端思考过程必须可视化“原始问题、歧义术语、知识摘要、增强后问题”，提高可观测性。
6. 明确各字段契约、优先级与长度限制，保证 prompt 稳定。

## 3. 核心结论

### 3.1 enriched_query 的正式结构

enriched_query 必须输出统一模板，默认使用 Markdown：

`	ext
原始问题：
{user_query}

补充知识：
- {term_name_1}：{definition_summary_1}
- {term_name_2}：{definition_summary_2}
`

- 无可用知识时直接回退 enriched_query = user_query。
- 补充知识只包含 confirmed_terms，按置信度排序，最多 5 条。
- 每条摘要限制在 80~150 字，优先保留结论/使用方法，过滤示例、贴现率等冗余信息。

### 3.2 knowledge_payload 的定位

- 保存完整知识原文（术语定义、字段、指标、来源等），格式化为结构化字典。
- 仅作为链路内的“原始数据仓”，供调试、审计、测试使用。
- knowledge_enhance 负责从中提炼摘要，后续节点不直接解析。

### 3.3 歧义术语处理

- mbiguous_terms 不再触发澄清中断，只作为日志与 SSE 提示。
- 每个歧义项记录 mention / candidates / reason，写入 	hinking_log 与调试面板。
- 若所有术语都歧义，enriched_query 仍返回原始问题，规划继续，只是告警“无确权知识”。

### 3.4 planning 输入优先级

- 新契约：planning_input = enriched_query ?? intent ?? user_query。
- 在状态树中保留字段顺序，并在日志中打印 planning_input_source=enriched_query|intent|user_query，可追踪覆盖情况。

### 3.5 knowledge_snippets vs enriched_query

- knowledge_snippets：面向工具/函数调用的轻量上下文（JSON/列表），可包含示例语句、过滤条件等。
- enriched_query：面向 LLM 的自然语言上下文，只包含最小摘要。
- 两者共用同一份 knowledge_payload 来源，但必须通过去重规则避免重复堆砌：摘要中出现的句子不得原样再写入 snippets。

## 4. 字段职责划分

| 字段 | 作用 | 是否供后续节点直接消费 |
| --- | --- | --- |
| user_query | 用户原始问题 | 是 |
| confirmed_terms | 已确权术语 | 是 |
| mbiguous_terms | 歧义术语列表，含候选 | 否，仅日志/展示 |
| knowledge_payload | 完整知识快照 | 否 |
| enriched_query | 增强后问题文本 | 是 |
| 	erm_hints | 兼容旧规划的结构化提示 | 是 |
| knowledge_snippets | 工具上下文补充 | 是 |
| enriched_query_source | 增强来源（confirmed_terms / fallback） | 否 |
| enriched_query_confidence | 增强置信度（0~1） | 否 |

## 5. 目标调用链路

`	ext
用户问题
  -> knowledge_enhance
     -> 术语提取、候选搜索、歧义消解
     -> 加载 confirmed_terms 完整知识
     -> 生成 knowledge_payload
     -> 生成去噪摘要 + knowledge_snippets
     -> 输出 enriched_query（原始问题 + 摘要）
  -> planning
     -> 使用 enriched_query 规划任务
  -> execution/react_runtime
     -> 使用 enriched_query / snippets / term_hints 执行 function call
  -> end
     -> 汇总结果、展示增强上下文
`

## 6. 当前代码现状

- knowledge_enhance_node 仍按旧逻辑改写问题；enriched_query 只是术语替换。
- planning_node 仍优先 intent；enriched_query 常被覆盖，导致增强失效。
- execution/react_runtime 已消费 enriched_query，但对长文本格式未明确测试。
- knowledge_payload 仅保存，不被读取；knowledge_preview 等旧字段仍存在。

## 7. 方案细节

### 7.1 摘要生成规则

- 输入：knowledge_payload 中的 confirmed_terms 相关结构。
- 步骤：
  1. 先按置信度、时间更新排序，去重相同 	erm_id；
  2. 使用正则或模板裁剪冗余句，如“本条数据来源于……”；
  3. 截断到 80~150 字（中文字符数），必要时补充省略号；
  4. 记录 summary_quality，若 <0.4 则不写入 enriched_query，仅保留在 knowledge_snippets。

### 7.2 enriched_query 的格式规范

- 使用 Markdown 标题 + 列表，保持两段。
- 模板放在代码常量中，统一 原始问题 / 补充知识 关键词。
- 若只有单条知识，仍保留列表前缀，避免 prompt 结构漂移。
- 在状态里写入 enriched_query_source=confirmed_terms|fallback。

### 7.3 歧义术语与可观测性

- 	hinking_log 输出：
  - 
  aw_question
  - mbiguous_terms（含 mention/candidates）
  - confirmed_terms + 摘要预览
  - enriched_query
- SSE / 前端同时展示，便于人工排查。
- 若需回溯，日志中包含 mbiguous_terms_count、confirmed_terms_count。

### 7.4 planning 输入契约

- planning_node 在读取状态时，先尝试 state.enriched_query。
- 若为空，再用 intent；仍为空再退回 user_query。
- 在日志中输出 planning_input_source，同时写入 state.planning_input_source 供调试。

### 7.5 knowledge_snippets 的角色

- 面向工具：允许包括字段映射、SQL 片段、枚举值等结构化文本。
- 生成规则：
  - 可使用比摘要更长的文本，但每条不超过 300 字；
  - 与 enriched_query 摘要重复度 >0.8 时自动去除。

### 7.6 长文本与 prompt 风险

- 预计 enriched_query 长度 ≤ 1200 字；若超过阈值，退化为“原始问题 + 最多 3 条摘要”。
- 在 
eact_runtime 中记录 prompt_char_len，超阈值报警但不报错。

## 8. 详细改动点

1. **orchestration/knowledge_enhance/node.py**
   - 新增 _build_confirmed_knowledge_summaries，实现排序、去重、截断。
   - 新增 _compose_enriched_query，统一模板与置信度。
   - 新增 _build_knowledge_snippets 去重逻辑，避免与摘要重复。
   - 在主流程中记录 	hinking_log（原问题/歧义/摘要/增强问题）。
2. **orchestration/planning/node.py**
   - 引入 planning_input = enriched_query or intent or user_query。
   - 记录 planning_input_source 到日志和状态。
3. **orchestration/execution/react_runtime.py**
   - 记录 prompt_char_len、enriched_query_source，在日志中打印。
   - 增加针对 Markdown 模板的回归单测（确保 function call prompt 兼容）。
4. **orchestration/execution/sandbox_executor.py**
   - 继续透传 enriched_query/knowledge_snippets/term_hints，确保 hooks 能拿到一致的上下文。
5. **orchestration/end/node.py**
   - 展示 enriched_query、confirmed_terms 摘要，用于最终说明。
6. **字段清理**
   - 删除 knowledge_preview；
   - 新增 planning_input_source；
   - 文档化 knowledge_payload、enriched_query_source/confidence。
