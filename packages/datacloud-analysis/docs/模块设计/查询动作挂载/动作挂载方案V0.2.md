# 查询动作挂载方案

## 1. 现状架构

### 1.1 MCP Server 侧工具链

MCP Server 暴露两套工具模式，由 `tool_list_mode` 控制：

#### unified 模式（默认，生产推荐）

```
MCP client: list_tools()
  → ToolRegistry.list_tools(tool_list_mode="unified")
  → 只暴露：unified_query_tool（自然语言查询）+ operation 族工具

MCP client: call_tool("unified_data_query", {question, object_ids})
  → UnifiedQuery.execute(question, object_ids)
       → view.query(question) / obj.query(question)
       → LLM 生成查询计划（SQL 片段、过滤条件）
       → QueryExecutor / ComputeExecutor 执行 SQL
       → 返回结构化查询结果
```

> `query_X`、`compute_X` 等动作工具在 unified 模式下**不对 MCP 客户端可见**。

#### skill 模式（测试 / 精细调用）

```
MCP client: list_tools(tool_list_mode="skill")
  → ToolRegistry 全量列出：
       query_{obj_code}   — 结构化查询（select/filters/order_by/limit）
       compute_{obj_code} — 聚合计算（measures/dimensions/filters）
       search_{obj_code}  — 知识库检索
       query_view_{code}  — VIEW 结构化查询
       自定义 OWL action  — OWL 文件内定义的业务动作

MCP client: call_tool("query_ads_chain_analysis", {...})
  → snapshot.action_routes[action_code]             ← O(1) 路由
  → ActionExecutor.execute(object_code, action_code, args)
       → OntologyLoader.get_object(object_code)
       → obj.invoke_action(action_code, args)
       → Action.execute()
            → QueryExecutor（query族）/ ComputeExecutor（compute族）
            → SQL → 数据库
```

#### query_*/compute_* 工具的 Schema 来源

```
LoaderRuntimeManager._build_loader()
  → inject_virtual_actions(loader)
       → build_query_schema()   ← 生成 query 族 JSON Schema
       → build_compute_schema() ← 生成 compute 族 JSON Schema
       → VirtualAction 注入 loader.cls.actions / view._virtual_actions
       → VirtualActionRegistry 全局注册
```

Schema 中的 `select`、`filters.field`、`order_by.field` 均使用 `field_code`（英文枚举），在注入阶段**一次性生成**，MCP 与 agent 侧**共用同一份 Schema**。

---

### 1.2 worker.py 侧工具链（agent 侧）

以 `byclaw-data/worker.py` 为入口的 agent 侧，工具链分两层：

```
worker._build_graph()
  → create_agent(mounted_objects, loader,
                 tools=tools_dict,
                 skip_action_families=frozenset({"query","compute"}))
```

#### LLM 可见工具（OntologyToolLoader 产出）

```
OntologyToolLoader(mounted_objects, loader,
                   skip_action_families=frozenset()).load()
  ├─ OBJECT 路径：
  │    ActionToolGenerator.generate_tools(obj_code)
  │      → query_{obj_code}, compute_{obj_code}, 自定义 OWL action
  │      → StructuredTool(coroutine=ActionExecutor.execute)
  │           → obj.invoke_action() → Action.execute() → SQL
  └─ VIEW 路径：
       view.actions → query_view_{code}, compute_view_{code}
       → StructuredTool(coroutine=view.invoke_action)
            → Action.execute() → SQL
```

> 当 `skip_action_families=frozenset({"query","compute"})` 时，`query_X`/`compute_X` 被跳过，只保留自定义 OWL action。

#### LLM 不可见工具（redirect_tools，db_query 模式）

```
worker._build_dynamic_tools()
  → data_query_{obj_code}（redirect_tool，LLM 不可见）
       → HTTP 请求 MCP Server: call_tool("unified_data_query", ...)
            → UnifiedQuery.execute() → LLM 计划 → SQL
```

`data_query_*` 工具不进入 agent 的 tool 列表，仅作为内部路由使用（agent 通过 `unified_data_query` 自然语言接口访问，LLM 不感知底层分发细节）。

#### 完整工具层级关系

| 工具名 | LLM 可见 | 执行路径 | 底层 |
|---|---|---|---|
| `query_{code}` | ✅ | ActionExecutor → obj.invoke_action | SQL (QueryExecutor) |
| `compute_{code}` | ✅ | ActionExecutor → obj.invoke_action | SQL (ComputeExecutor) |
| 自定义 OWL action | ✅ | ActionExecutor / view.invoke_action | SQL / 自定义逻辑 |
| `data_query_{code}` | ❌ | redirect → HTTP → MCP unified_data_query | LLM 计划 → SQL |

### 1.3 Schema 个性化：当前不支持

**现状**：`query_X`/`compute_X` 的 JSON Schema（`select`、`filters`、`order_by` 等字段）由 `inject_virtual_actions` 在加载期**一次性生成**，MCP Server 与 agent 侧**共用同一份 Schema 定义**，不存在针对 agent 端的定制机制。

**执行期中文翻译**：`action.py: _translate_view_params()` 在执行时可将中文字段名转换为 `field_code`，但这是容错逻辑，Schema 本身不声明接受中文输入，agent 也无法感知。

**实际影响**：

- 若希望 agent 侧看到的 Schema 字段说明、枚举值、必填项与 MCP 侧有所不同（如对 agent 隐藏部分参数、增加业务语义描述），**当前架构不支持**。
- `skip_action_families` 只能整族跳过，无法做字段级 Schema 裁剪。
- 如需个性化，需在 `OntologyToolLoader` 层对 `StructuredTool` 的 `args_schema` 做后处理覆盖（目前无此机制）。

## 2.目标架构

### 2.1 分工清晰

#### 现有问题：系统 2 与系统 3 耦合

当前 `byclaw-data`（系统 3）在初始化 agent 时需要直接承担 OWL 加载职责：

```python
# 系统3现状（worker.py / init_agent_conf.py 简化）
loader = OntologyLoader()
loader.load_from_owl_directory(ontology_path)
inject_virtual_actions(loader)          # ← 系统3需要知道这个细节

tools = OntologyToolLoader(
    mounted_objects, loader
).load()
```

**问题**：
1. 每个落地项目（byclaw-data、新业务项目）都必须知道 `inject_virtual_actions` 的存在和调用时机。
2. 若 `inject_virtual_actions` 内部逻辑变更（如新增 `search_X` 注入），所有项目均需同步更新。
3. `OntologyLoader` 的构建细节（路径、加载方式）泄漏到应用层。

---

#### 目标：OntologyToolLoader 自闭合

系统 3 只需提供**业务配置**（挂载对象 + OWL 路径），加载与注入细节全部内聚到系统 2。

**新接口设计：**

```python
# 系统3目标（worker.py 简化）
tools = OntologyToolLoader(
    mounted_objects=["enterprise", "manage_grid"],
    ontology_path="/path/to/owl",           # ← 新增参数
    skip_action_families=frozenset(),
).load()
# OntologyToolLoader 内部完成：
#   load_from_owl_directory(ontology_path)
#   inject_virtual_actions(loader)
#   → 产出 StructuredTool 列表
```

**OntologyToolLoader 内部变更：**

```python
class OntologyToolLoader:
    def __init__(
        self,
        mounted_objects: list[str],
        ontology_path: str | Path | None = None,   # 新增
        loader: OntologyLoader | None = None,       # 保留（兼容 MCP 侧传入已构建 loader）
        skip_action_families: frozenset[str] = frozenset(),
    ) -> None:
        if loader is not None:
            self._loader = loader      # 外部传入（MCP 侧、单测场景）
        elif ontology_path is not None:
            self._loader = self._build_loader(ontology_path)  # 自建
        else:
            raise ValueError("必须提供 loader 或 ontology_path 之一")
        ...

    @staticmethod
    def _build_loader(ontology_path: str | Path) -> OntologyLoader:
        loader = OntologyLoader()
        loader.load_from_owl_directory(str(ontology_path))
        inject_virtual_actions(loader)
        return loader
```

---

#### 分工边界（目标状态）

| 职责 | 系统 2（OntologyToolLoader） | 系统 3（byclaw-data / 落地项目） |
|---|---|---|
| OWL 文件加载 | ✅ 内聚 | ❌ 不感知 |
| inject_virtual_actions | ✅ 内聚 | ❌ 不感知 |
| 挂载对象列表 | 透传 | ✅ 业务配置 |
| ontology_path | 接收 | ✅ 业务配置 |
| skip_action_families | 接收 | ✅ 按需传入 |
| StructuredTool 产出 | ✅ 唯一产出方 | 消费方 |

---

#### 兼容性保障

- `loader` 参数保留，MCP Server 的 `LoaderRuntimeManager` 传入已构建的 `loader` 时行为不变。
- `ontology_path` 参数为可选，不传时沿用原有 `loader` 参数逻辑，**不破坏任何现有调用方**。
- `inject_virtual_actions` 在 `_build_loader` 内部调用，若外部 `loader` 已注入则不重复注入（`VirtualActionRegistry` 的幂等性保障）。

#### 迁移步骤

1. `OntologyToolLoader.__init__` 新增 `ontology_path` 参数，实现 `_build_loader` 静态方法。
2. `byclaw-data/worker.py`：移除 `OntologyLoader()` + `load_from_owl_directory()` + `inject_virtual_actions()` 三行，改为传 `ontology_path`。
3. `examples/e_commerce_demo/backend/init_agent_conf.py`：同步迁移（若仍保留该文件）。
4. 单测：验证 `OntologyToolLoader(ontology_path=...)` 与 `OntologyToolLoader(loader=...)` 两路均正常产出工具。

### 2.2 Schema 个性化架构

#### 代码核查结论

核查 `generator.py`、`ontology_tool_loader.py` 及 `query_clarification_plugin.py` 后，对问题的理解需要大幅修正：

**已确认属实：**
- `select` / `order_by.field`：enum 只含英文 field_code
- `build_query_schema`（第 294 行）/ `build_compute_schema`（第 439 行）均硬编码 `strict_field_code=True`
- `OntologyToolLoader` 无 schema 后处理机制

**原描述不准确之处：**
- `filters.field` 不是全量 enum，而是 `anyOf` 结构（每字段单独 item），description 已内嵌中文名
- **中文字段名翻译在执行层其实已经解决**——`query_clarification_plugin.py` 已做了完整处理（见下节）

---

#### 现有 Hook 层已覆盖的能力（被遗漏的关键组件）

`query_clarification_plugin.py` 作为 `before_call_back` hook，在每次 `query_*/compute_*` 工具调用前拦截，执行以下完整流程：

```
LLM call: query_{obj_code}({query, filters, select, ...})
        ↓
before_call_back(ctx)   ← query_clarification_plugin 拦截
  1. _normalize_json_fields()   — JSON 字符串列表字段解码为 list
  2. 弹出元字段：query, complex_conditions
  3. Step1 复杂度判断：complex_conditions 非空 → is_complex=True
  4. _get_field_catalog(tool_name, ctx)
       → loader.get_ontology_class / get_view → {field_name: field_code}
       → 同时收录 field_code 本身 + aliases
  5. _collect_terms_from_params(tool_params)
       → 从 filters.field_name_cn / filters.field / select / dimensions / metrics / order_by 提取术语
       → _is_field_code() 区分：ASCII field_code 直通，中文名收集待解析
  6. _resolve_terms(terms, catalog)
       → CLEAR（全部命中）→ _apply_resolved_to_params() 写回 field_code
       → NEED_CONFIRM（有未命中）→ analyze_query_clarification SDK → 知识图谱歧义消除
           → needs_clarification=True → ClarificationNeededError → 前端弹确认表单
           → needs_clarification=False → 自动 format_clarification 写回，跳过弹表单
  7. is_complex=True → redirect → data_query_{obj_code}（接 UnifiedQuery 走 LLM 计划）
     is_complex=False → patch tool_params → 直接走 QueryExecutor（OQL 直查）
```

**结论**：LLM 填入中文字段名（无论放在 `field` 还是 `field_name_cn`），hook 层**已能完整处理**：解析、歧义确认、最终写回 field_code，整个执行链无需改动。

---

#### 当前架构全貌（含 Hook）

```
Schema 层（strict_field_code=True）
  LLM 被引导填 field_code（英文）← 实际也可填中文名，Schema 未声明

              ↓ LLM tool call

Hook 层（query_clarification_plugin.before_call_back）
  ├─ 中文→field_code 映射（catalog 查询）
  ├─ 歧义判断 → 知识图谱澄清（interrupt / ClarificationNeededError）
  └─ 复杂度判断 → redirect data_query_* / patch → 直接执行

              ↓

执行层
  ├─ QueryExecutor（简单 OQL 直查）
  └─ UnifiedQuery（LLM 计划 → SQL，经 data_query_* redirect）
```

**实际问题不是"无法处理中文"，而是"Schema 没有告诉 LLM 可以填中文"**。LLM 在 strict schema 约束下会尽量填 field_code，有时会猜错，有时因不认识 field_code 而填错。Hook 层是兜底，不是主路径。

---

#### 根因与目标

| 层 | 现状 | 目标 |
|---|---|---|
| Schema（LLM 引导层） | strict 枚举，LLM 被引导填 field_code，不知道中文名可用 | 声明支持中文名输入，引导 LLM 填"企业名称"或"enterprise_name" |
| Hook（翻译/路由层） | 已处理中文→field_code + 歧义澄清 + redirect | 不变，继续作为兜底保障 |
| 执行层 | OQL 直查 / UnifiedQuery | 不变 |

---

#### 方案：Schema 层对齐 Hook 层能力（`agent_friendly=True`）

在 `OntologyToolLoader` 中新增 `agent_friendly: bool = True`。当为 True 时，对 query/compute 工具的 filters schema 做**最小化替换**：用 `strict_field_code=False` 重建 filters 部分，使 Schema 与 hook 层声明对齐。

> `generator.py` 的 `_build_filters_schema(strict_field_code=False)` 路径**已经实现**，描述明确说「field 可填写字段中文名或字段编码，系统自动识别映射」——只需在 OntologyToolLoader 调用该路径即可。

```
OntologyToolLoader.load()
  → _build_action_tools(obj_code)
  → 取得 tool_def["inputSchema"]（来自 inject_virtual_actions，strict）
  → 若 agent_friendly=True 且 action_type in {"query","compute"}：
       obj = loader.get_object(obj_code)
       relaxed_filters = _build_filters_schema(obj.fields, strict_field_code=False)
       patched_schema = {**inputSchema, "properties": {
           **inputSchema["properties"],
           "filters": relaxed_filters,   ← 替换为 relaxed
       }}
  → _json_schema_to_pydantic(patched_schema)
```

效果：
- filters description 变为「支持字段中文名或编码，系统自动识别映射」
- field 属性无 enum 约束，LLM 可直接填"企业名称"（与 hook 层已有能力对齐）
- hook 层不变，仍保障兜底翻译和歧义消除

**对 `select` / `order_by.field`**：仍为 field_code enum（中文名在 `x-dc-field-catalog` 扩展字段），hook 层 `_collect_terms_from_params` 已能处理 select 中的中文名——Schema 层暂不调整，后续可通过 SchemaTransformer 补充（见下）。

---

#### 方案补充：SchemaTransformer 钩子（按需）

如果未来需要：select 枚举附带中文别名描述、description 补充业务语境、字段子集裁剪等，在 `OntologyToolLoader` 追加 `schema_transformer` 钩子作为补充，**不与 agent_friendly 冲突**：

```python
class SchemaTransformer(Protocol):
    def transform(self, action_code: str, action_type: str, schema: dict[str, Any]) -> dict[str, Any]: ...
```

典型扩展：
| Transformer | 用途 |
|---|---|
| `SelectEnrichTransformer` | 为 select enum 的 description 补充中文别名（来自 x-dc-field-catalog） |
| `FieldSubsetTransformer` | 不同业务场景只暴露字段子集 |
| `DescriptionPatchTransformer` | 追加业务语境到参数描述 |

> 当前**只需实现 `agent_friendly=True`**，SchemaTransformer 备用。

---

#### 三层协作关系（目标状态）

```
Schema 层（agent_friendly=True，relaxed filters）
  ↓ 引导 LLM：field 可填中文名或 field_code，select 仍为 field_code enum

Hook 层（query_clarification_plugin，不变）
  ↓ 解析中文→field_code，歧义消除，复杂度判断→redirect

执行层（QueryExecutor / UnifiedQuery，不变）
```

| 职责 | 位置 | 目标状态 |
|---|---|---|
| 告知 LLM 可填中文名 | Schema（agent_friendly） | ✅ 新增 |
| 中文→field_code 翻译 | Hook（query_clarification_plugin） | ✅ 已有，不变 |
| 歧义消除（知识图谱） | Hook（query_clarification_plugin） | ✅ 已有，不变 |
| 复杂查询 → redirect data_query_* | Hook（query_clarification_plugin） | ✅ 已有，不变 |
| 简单查询 OQL 直查 | 执行层（QueryExecutor） | ✅ 已有，不变 |
| MCP strict schema 不受影响 | inject_virtual_actions | ✅ 不动 |

### 3.2 OntologyToolLoader 接口变更

```python
# datacloud_analysis/tools/ontology_tool_loader.py

class OntologyToolLoader:
    def __init__(
        self,
        mounted_objects: list[str] | None = None,
        loader: Any | None = None,                          # 保留：外部传入已构建 loader（MCP / 单测）
        ontology_path: str | Path | None = None,            # 新增：自建 loader 路径（agent 侧）
        skip_action_families: frozenset[str] = frozenset(),
        agent_friendly: bool = True,                        # 新增：放宽 filters 的字段名约束
    ) -> None:
        if loader is not None:
            self._loader = loader
        elif ontology_path is not None:
            self._loader = self._build_loader(Path(ontology_path))
        else:
            raise ValueError("必须提供 loader 或 ontology_path 之一")

        self._mounted_objects: list[str] = list(mounted_objects or [])
        self._skip_action_families = skip_action_families
        self._agent_friendly = agent_friendly
```

**兼容性保障**：
- `loader` 参数保留且优先级最高，现有调用方（MCP Server、单测）零改动
- `ontology_path` 为可选，只有在 `loader=None` 时才触发自建流程
- `agent_friendly` 默认 `True`，对已有调用方（未传该参数）自动启用 relaxed filters；如需强制 strict，显式传 `agent_friendly=False`

### 3.3 `_build_loader` 静态方法

```python
@staticmethod
def _build_loader(ontology_path: Path) -> Any:
    """自建 OntologyLoader：加载 OWL 文件并注入虚拟动作。

    仅在 agent 侧通过 ontology_path 构建时调用；
    外部传入 loader 时不经过此方法。
    """
    from datacloud_data_service.loader.ontology_loader import OntologyLoader
    from datacloud_data_service.tools.virtual_action_injector import inject_virtual_actions

    loader = OntologyLoader()
    loader.load_from_owl_directory(str(ontology_path))
    inject_virtual_actions(loader)
    return loader
```

### 3.4 `_apply_agent_schema_patches` 方法（原 `_relax_filters_schema`，扩展为全量修正）

> 基于《dataCloud设计方案V0.3.md》§5.1.1~5.1.6 的分析，`agent_friendly=True` 需要同时修正五处 schema/description，而不仅仅是替换 filters。

#### 3.4.0 常量文件：`_agent_schema_patches.py`

```python
    "触发场景：\n"
    "1. 相对排名（如'后30%'、'前N名'）；\n"
    "2. 跨对象子查询（如'亩产效益后30%的地块'作为过滤范围）；\n"
    "3. 动态比较值（如'高于行业平均'）。\n"
    "⚠️ 字段名在字段列表中找不到时，不写入此列表——直接在 select/filters/order_by 中填原词。\n"
    "字段名透传与 complex_conditions 是两个独立规则，不能混用。\n"
    "写入内容：只写无法字面化的那个条件片段，不写整句查询。\n"
    "此列表非空时系统自动路由到全能查询路径（data_query），无需手动调用。"
)
```

> **选择新建文件而非放入 `i18n/prompts.py` 的原因**：`prompts.py` 存放的是 ReAct 执行规则（agent 级别系统提示），而此处三个常量是工具参数级别的 JSON Schema description（tool 级别）。层次不同，不宜混放。

#### 3.4.1 需要修正的五处问题（来自 V0.3 分析）

| 问题 | 来源 | 当前内容（有误） | 目标内容 |
|---|---|---|---|
| **P1** `complex_conditions` 触发条件 2 | `build_query_schema` / `build_compute_schema` | "字段名找不到精确对应 → 写入此列表" | 删除该条件；仅收"值无法字面化"的条件；补充"字段名找不到时填原词到标准参数，不进此列表" |
| **P2** `filters.field` 缺少原词透传指令 | `_filter_item_schema`（strict_field_code=False 路径） | "系统自动识别映射" | 补充"若字段名找不到精确对应，直接填用户原词，禁止猜测替换为相近字段名" |
| **P3** `filters` 缺 catch-all 兜底 | `_build_filters_schema` | strict=True 路径无 catch-all | strict=False 路径已有 `_FILTER_CATCHALL_SCHEMA`，使用该路径即可覆盖 |
| **P4** `select` / `order_by.field` 仍声明"必须用属性编码" | `build_query_schema` | description 中"统一填写属性编码" / "只允许使用属性编码" | 改为"支持字段编码或字段中文名；找不到时填用户原词，禁止猜测替换" |
| **P5** 工具顶层 description 仍写"field 统一使用属性编码" | `build_query_description` Markdown | "select / filters.field / order_by.field 统一使用属性编码" / "field 填了不存在的字段名（系统无法映射时会报错）" | 改为"支持字段编码或中文名；找不到时填原词" / 删除"会报错"改为"系统后端负责解析" |

#### 3.4.2 实现

```python
def _apply_agent_schema_patches(
    self, scope_code: str, input_schema: dict[str, Any]
) -> dict[str, Any]:
    """agent_friendly=True 时对 query/compute 工具 schema 做全量修正。

    修正点：
      1. filters → strict_field_code=False（catch-all 兜底 + field 支持中文名 + 原词透传指令）
      2. select → description 补充"支持中文名或编码，找不到时填原词"
      3. order_by.field → description 补充"支持中文名或编码，找不到时填原词"
      4. complex_conditions → description 删除"字段名未命中 → 写入此列表"，补充"字段名找不到时填原词到标准参数"
    loader 查找失败时原样返回，降级为 strict 行为。
    """
    from datacloud_analysis.tools._agent_schema_patches import (  # noqa: PLC2701
        AGENT_COMPLEX_CONDITIONS_DESCRIPTION,
        AGENT_ORDER_BY_FIELD_DESCRIPTION,
        AGENT_SELECT_DESCRIPTION,
    )
    from datacloud_data_sdk.virtual_action.generator import _build_filters_schema  # noqa: PLC2701

    # 获取字段列表（OBJECT / VIEW 两路）
    fields: list[Any] = []
    try:
        obj = self._loader.get_ontology_class(scope_code)
        fields = list(obj.fields)
    except Exception:  # noqa: BLE001
        try:
            view = self._loader.get_view(scope_code)
            fields = list(getattr(view, "fields", []))
        except Exception:  # noqa: BLE001
            return input_schema  # 降级：loader 查找失败，原样返回

    props = dict(input_schema.get("properties") or {})

    # P1 + P2 + P3：替换 filters（strict_field_code=False 路径已包含 catch-all + 原词透传指令）
    props["filters"] = _build_filters_schema(fields, strict_field_code=False)

    # P4-a：select — 保留 enum（LLM 参考），补充 description 说明中文名可用
    if "select" in props:
        select = dict(props["select"])
        select["description"] = AGENT_SELECT_DESCRIPTION
        props["select"] = select

    # P4-b：order_by.field — 保留 enum（LLM 参考），补充 description
    if "order_by" in props:
        order_by = dict(props["order_by"])
        items = dict(order_by.get("items") or {})
        item_props = dict(items.get("properties") or {})
        if "field" in item_props:
            field_prop = dict(item_props["field"])
            field_prop["description"] = AGENT_ORDER_BY_FIELD_DESCRIPTION
            item_props["field"] = field_prop
        items["properties"] = item_props
        order_by["items"] = items
        props["order_by"] = order_by

    # P1：complex_conditions — 删除"字段名未命中 → 写此列表"条件，补充原词透传声明
    if "complex_conditions" in props:
        props["complex_conditions"] = {
            **props["complex_conditions"],
            "description": AGENT_COMPLEX_CONDITIONS_DESCRIPTION,
        }

    return {**input_schema, "properties": props}
```
