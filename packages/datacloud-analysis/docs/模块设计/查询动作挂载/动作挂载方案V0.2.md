# 查询动作挂载方案

## 1. 现状分析

> **说明**：本节在 V0.1 基础上，逐一核实三套系统的代码现状（截至 2026-04-21），
> 标注哪些描述已属实、哪些已发生变化，并补充新增细节。

---

### 1.1 三套系统（架构格局未变，各自均已演进）

#### 系统 1：MCP Server（对外协议层）

**V0.1 描述的调用链：**
```
routes.py lifespan
  → OntologyLoader.load_from_owl_directory()
  → inject_virtual_actions(loader)
  ↓
mcp_sdk_handler.py: list_tools() → ToolRegistry → ActionToolGenerator（遍历 cls.actions）
mcp_sdk_handler.py: call_tool()  → _find_action_scope() → VirtualActionRegistry（O(1)）
```

**当前实际状态（已演进）：**

1. **OntologyLoader 生命周期升级为 `LoaderRuntimeManager`**
   - `routes.py` lifespan 不再直接调 `load_from_owl_directory` + `inject_virtual_actions`
   - 改为启动 `LoaderRuntimeManager`，由其内部 `_build_loader()` 完成加载与注入
   - `inject_virtual_actions(loader)` 仍然调用，位置在 `loader_runtime.py: _build_loader()`
   - 支持热重载（`LoaderRuntimeManager.reload()`），每次重载生成新的 `LoaderSnapshot`

2. **新增 `LoaderSnapshot`（快照级路由索引）**
   ```python
   @dataclass(frozen=True)
   class LoaderSnapshot:
       loader: OntologyLoader
       action_routes: dict[str, ActionRouteRef]  # action_code → (scope_type, scope_code)
   ```
   - `action_routes` 由 `build_action_routes()` 构建，遍历 `cls.actions` + `view._virtual_actions`
   - `call_tool()` 路由查找优先级调整为：
     1. `snapshot.action_routes`（O(1)，快照本地副本，**最优先**）
     2. `VirtualActionRegistry`（全局单例，降级）
     3. 遍历 `cls.actions` / `view._virtual_actions`（最后兜底）

3. **`list_tools()` 增加 `tool_list_mode` 过滤**
   - `unified` 模式（默认）：只暴露 `unified_query_tool` + `operation` 族工具，**query/compute 等动作工具不再列出**
   - 非 unified 模式（如 `skill`）：列出所有工具（包含 query/compute/自定义 action）
   - V0.1 描述的"遍历 cls.actions 全量生成"**仅在非 unified 模式下成立**

4. **`call_tool()` 新增 `unified_data_query` 专用路径**
   - 调用 `UnifiedQuery.execute(question, object_ids)` 处理统一查询
   - 其余工具（含 query_X、compute_X）仍走 `_find_action_scope()` → `ActionExecutor` / `view.invoke_action()`

5. **`ToolRegistry._append_view_tools()` 新增 `exposure_policy` 过滤**
   - VIEW 工具按 `action.exposure_policy`（`direct`/`skill_only`/`hidden`）过滤
   - `unified` 模式跳过 `skill_only` 类工具

```
routes.py lifespan
  → LoaderRuntimeManager.start()
       → _build_loader()
            → load_from_path()
            → inject_virtual_actions(loader)   ← 仍然注入
            → build_action_routes(loader)      ← 新增：构建快照路由索引
       → LoaderSnapshot(loader, action_routes)
  ↓
mcp_sdk_handler.py: list_tools()
  → ToolRegistry(loader).list_tools(tool_list_mode="unified")
     → [unified_query_tool] + operation 族工具（query/compute 不列出）
  ↓
mcp_sdk_handler.py: call_tool()
  → "unified_data_query" → UnifiedQuery.execute()          ← 新增专用路径
  → 其他工具 → snapshot.action_routes（O(1)）             ← 优先
            → VirtualActionRegistry                        ← 降级
            → ActionExecutor / view.invoke_action()
```

#### 系统 2：OntologyToolLoader（agent 侧，`datacloud-analysis` 包）

**V0.1 描述的问题：**
- Bug 1：`meta.get("action_type")` 返回族名被当作唯一 `action_code`，动作工具全部失败
- Bug 2：VIEW 类型不支持（`ObjectNotFoundError`）
- `_build_action_tools` 失败告警，agent 侧无动作工具产出

**当前实际状态（V0.1 中两个 Bug 均已修复，并已按目标架构重写）：**

1. **`action_tool_generator.py: _meta` 已补全 `action_code` 字段**（Bug 1 前提修复）
   ```python
   tool["_meta"] = {
       "object_code": object_code,
       "action_type": action.action_type,   # 族名，如 "query"/"compute"
       "action_code": action.action_code,   # 唯一码，如 "query_ads_chain_analysis"
   }
   ```

2. **Bug 1 已修复**：`_build_action_tools()` 使用 `action_code` 而非 `action_type`
   ```python
   action_code: str = meta.get("action_code") or name
   # fallback 到 name（ActionToolGenerator 保证 name == action.action_code）
   ```

3. **Bug 2 已修复**：`load()` 方法区分 OBJECT / VIEW，`_build_view_tools()` 已实现
   - VIEW 路径：`loader.get_view(view_code).actions` → `view.invoke_action()`
   - VIEW 同样支持 `skip_action_families` 过滤

4. **新增 `skip_action_families` 参数**（支持 `db_query` 模式）
   - 传入 `frozenset({"query", "compute"})` 可跳过虚拟注入动作，只保留 OWL 原生自定义 action

5. **降级兜底逻辑改为条件触发**
   - `_build_query_tool()`（DynamicQueryToolGenerator）保留
   - 仅当 `_build_action_tools` 未产出任何 `query_*` 工具时才触发（inject_virtual_actions 未调用的旧环境兜底）

6. **执行路径完全对齐 MCP Server**
   - OBJECT → `_make_object_action_coroutine` → `ActionExecutor.execute(obj_code, action_code, args)`
   - VIEW   → `_make_view_action_coroutine`   → `view.invoke_action(action_code, args)`

```
OntologyToolLoader(mounted_objects, loader,
                   skip_action_families=frozenset()).load()
  ├─ obj_code 在 loader._scenes → VIEW 路径
  │     _build_view_tools(obj_code)
  │       → view.actions（过滤 skip_action_families + exposure_policy）
  │       → StructuredTool(coroutine=view.invoke_action)
  └─ obj_code 不在 _scenes → OBJECT 路径
        _build_action_tools(action_gen, obj_code)
          → action_gen.generate_tools(obj_code)  [需 inject_virtual_actions 已调用]
          → 过滤 skip_action_families
          → StructuredTool(coroutine=ActionExecutor.execute)
        若无 query_* 产出 → _build_query_tool fallback（DynamicQueryToolGenerator）
```

#### 系统 3：`init_agent_conf`（agent 侧，demo 应用层）

**V0.1 描述的问题（基于 `byclaw-data` 包）：**
- `_build_shared_loader` 未调 `inject_virtual_actions`
- `_build_ontology_query_wrappers` 生成重复 `query_{code}` 工具并覆盖系统 2 产出
- 合并逻辑中系统 3 胜出，系统 2 产出被覆盖

**当前实际状态（`examples/e_commerce_demo/backend/` 版本已大幅重构）：**

> ⚠️ 注意：`byclaw-data` 为独立仓库，本仓库中仅有 `examples/e_commerce_demo/backend/`
> 版本的 `init_agent_conf.py`，以下描述基于该文件。

1. **`_build_shared_loader` 方法不存在**
   - 对应功能已整合到 `_build_dynamic_tools_with_diagnostics` 流程中

2. **`_build_ontology_query_wrappers` 已删除**
   - 不再单独为每个 OBJECT/VIEW 生成 `query_{code}` 专用工具

3. **OBJECT/VIEW 类型跳过专用工具生成，改用通用工具**
   ```python
   # _build_ontology_tools_with_diagnostics 中：
   if resource_biz_type in {"OBJECT", "VIEW"}:
       report["skipped"].append({..., "reason": "object_view_use_generic_query_objects_tool"})
       continue  # OBJECT/VIEW 统一走通用 query_objects 工具
   ```

4. **新增诊断包装层**
   - `_build_ontology_tools_with_diagnostics` / `_build_delegate_tools_with_diagnostics`
   - 每次工具构建均记录诊断报告

5. **工具合并覆盖问题已不复存在**
   - 系统 2 / 系统 3 的 `query_{code}` 覆盖冲突已随 `_build_ontology_query_wrappers` 删除而消失
