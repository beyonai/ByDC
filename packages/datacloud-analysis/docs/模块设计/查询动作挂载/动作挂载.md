# 查询动作挂载方案

## 1. 现状分析

### 1.1 三套并行系统

当前存在三套独立实现，均承担"将本体对象转换为可调用工具"的职责：

#### 系统 1：MCP Server（对外协议层）

```
routes.py lifespan
  → OntologyLoader.load_from_owl_directory()
  → inject_virtual_actions(loader)          # 把 query_X / compute_X 注入进 cls.actions
  ↓
mcp_sdk_handler.py: list_tools()
  → ToolRegistry(loader).list_tools()
     → ActionToolGenerator.generate_tools(obj_code)  # 遍历 cls.actions
     → ToolRegistry._append_view_tools(view.actions) # VIEW 单独处理
  ↓
mcp_sdk_handler.py: call_tool()
  → _find_action_scope()  → VirtualActionRegistry (O(1) 路由)
  → ActionExecutor.execute(scope_code, action_code, args)
  or view.invoke_action(action_code, args)
```

- **工具来源**：`cls.actions`（经 virtual_action_injector 注入后）
- **VIEW 支持**：有，`view.actions` 单独处理
- **执行路由**：`VirtualActionRegistry`

---

#### 系统 2：OntologyToolLoader（agent 侧，`datacloud-analysis` 包）

```
agent.py: create_agent()
  → OntologyToolLoader(mounted_objects, loader).load()
     → _build_query_tool()
        → DynamicQueryToolGenerator.generate(obj_code)           # 独立生成 schema
        → DynamicQueryToolGenerator.generate_ontology_action()   # 现场构造 OntologyAction
        → StructuredTool(coroutine=Action(ont_action, loader).execute)
     → _build_action_tools()
        → ActionToolGenerator.generate_tools(obj_code)           # 遍历 cls.actions
        → loader.get_action(obj_code, meta["action_type"])       # Bug: 用族名当唯一码
        → 失败，告警                                              # 无工具产出
```

- **工具来源**：`DynamicQueryToolGenerator`（绕过 virtual injection），只产出 `query_{code}`
- **VIEW 支持**：无（Bug 2：get_ontology_class 只查 `_classes`，不查 `_scenes`）
- **执行路由**：`Action(ont_action, loader).execute()` 直接调用

---

#### 系统 3：`init_agent_conf._build_ontology_query_wrappers`（agent 侧，`byclaw-data` 包）

```
_handle_single_agent_detail() [ontology_query 模式]
  → _build_shared_loader()
     → OntologyLoader.load_from_owl_directory()
     → ⚠️ 未调用 inject_virtual_actions()
  → _build_ontology_query_wrappers()
     → _build_single_ontology_query_tool(resource_code)
        → _legacy_executor: loader.get_view/object().query(question)   # 自然语言兜底
        → _structured_executor: DynamicQueryToolGenerator
                                  .generate_ontology_action(code)
                                  → Action(ont_action, loader).execute(payload)
        → StructuredTool(name=f"query_{code}", five-stage schema)
```

- **工具来源**：`DynamicQueryToolGenerator`（同样绕过 virtual injection）
- **VIEW 支持**：有（先 try `get_view`，再 fallback `get_object`）
- **执行路由**：`Action(ont_action, loader).execute()`

---

## 2. 目标架构

**统一以系统 1（MCP Server）的模式为标准**，agent 侧工具生成与 MCP 侧保持一致：

- 工具 schema 来源：`inject_virtual_actions` 注入后的 `cls.actions` / `view.actions`，通过 `ActionToolGenerator` + VIEW 单独处理
- 执行路由：`VirtualActionRegistry`（对象），`view.invoke_action()`（视图）
- 消除 `DynamicQueryToolGenerator` 在 agent 侧作为主路径的重复使用
- 消除 `init_agent_conf._build_ontology_query_wrappers` 的重复实现

---

## 3. 两种加载模式的适配

`ONTOLOGY_LOAD_MODE` 控制 agent 侧工具生成策略，两种模式的定位、工具命名、执行路径完全不同，需分别处理。

### 3.1 模式对比

| 维度 | `ontology_query`（默认） | `db_query` |
|---|---|---|
| **适用场景** | LLM 填结构化 JSON → 直接执行 | 自然语言 → LLM 规划 → SQL 执行 |
| **工具名** | `query_{code}`、`compute_{code}` | `data_query_{code}` |
| **schema 来源** | `ActionToolGenerator`（`cls.actions`） | 固定：`{ question: str }` |
| **执行路径** | `ActionExecutor` / `view.invoke_action()` | `obj.query(question)` / `view.query(question)` |
| **mounted_objects** | 有值，传给 `AgentConfig.extra` | 空列表 `[]` |
| **shared_loader** | 非 None，传给 `AgentConfig.extra` | `None` |
| **OntologyToolLoader 介入** | 是（`create_agent()` 时动态生成） | 否（`mounted_objects=[]`，返回 `{}`） |
| **System 1 统一是否适用** | **是，本方案主体** | **否，保持现状不变** |

### 3.3 `db_query` 模式（路线 A：自然语言 → HTTP → MCP）

#### 3.3.1 现状问题

`db_query` 模式当前用 `_build_single_db_query_tool` 直接调 SDK：

```
_build_single_db_query_tool(resource_code)
  → _legacy_executor: obj.query(question) / view.query(question)  ← 直接调 SDK，进程内
     五阶段 schema → five_stage_to_legacy_question → LangGraphPlanGenerator → SQL
```

存在两个问题：
1. **OWL 自定义 action 缺失**：只生成 `data_query_{code}` 一个工具，`compute_{code}` 和 OWL 里的其他 action 完全没挂载
2. **执行逻辑与系统 1 割裂**：查询规划逻辑在 agent 进程内重复，与 MCP Server（`routes.py`）各维护一份

#### 3.3.2 改造目标

`data_query_{code}` **保持自然语言语义**，执行改为 HTTP 调用 `routes.py` 的 `unified_data_query`；同时补齐 compute 和 OWL 自定义 action 工具。

| 工具类型 | 改造前 | 改造后 |
|---|---|---|
| `data_query_{code}` | 五阶段自然语言，进程内 SDK 直调 | 自然语言 `{question}`，HTTP → MCP `unified_data_query` |
| `compute_{code}` | 无 | **无**（compute 是虚拟注入 action，不适用于自然语言模式） |
| OWL 自定义 action | 无 | `ActionExecutor`（系统 1）— 仅 OWL 原生定义的 action |
| VIEW 查询 | 五阶段自然语言，进程内 SDK | `view.invoke_action()`（系统 1） |

#### 3.3.3 `data_query_{code}` 工具：HTTP 调用 `unified_data_query`

`routes.py` 的 `unified_data_query` 接受 `object_ids` 参数，传单个对象时等价于 `data_query_X(question)`：

```
MCP call_tool("unified_data_query", {
    "question": "...",
    "object_ids": ["ads_chain_analysis"]
})
→ UnifiedQuery.execute(question, object_ids=["ads_chain_analysis"])
→ obj.query(question) → LangGraphPlanGenerator → SQL
```

**`_build_db_query_tools` 改造**：将每个 `data_query_{code}` 工具的执行改为 HTTP 调用：

```python
async def _http_data_query(question: str, resource_code: str) -> Any:
    """通过 HTTP 调用 MCP unified_data_query，等价于原 obj.query(question)。"""
    import httpx
    endpoint = os.environ.get("ONTOLOGY_MCP_ENDPOINT", "http://127.0.0.1:8080/api/v1/mcp")
    payload = {
        "jsonrpc": "2.0",
        "method": "tools/call",
        "id": 1,
        "params": {
            "name": "unified_data_query",
            "arguments": {
                "question": question,
                "object_ids": [resource_code],
            },
        },
    }
    async with httpx.AsyncClient() as client:
        resp = await client.post(endpoint, json=payload, timeout=120)
        resp.raise_for_status()
        return resp.json()
```

工具 schema 简化为纯 `{question: str}`（移除五阶段结构，LLM 直接传自然语言）：

```python
StructuredTool.from_function(
    name=f"data_query_{resource_code}",
    description=f"自然语言查询 {resource_name}。{resource_desc}",
    args_schema=DataQuerySchema,   # {question: str}
    coroutine=lambda question: _http_data_query(question, resource_code),
)
```

#### 3.3.4 OWL 自定义 action：复用 `OntologyToolLoader`

`db_query` 模式下，查询工具走 HTTP，`compute_X` 虚拟动作不挂载（与结构化聚合语义不符），但 OWL 原生自定义 action 仍需挂载。

**区分虚拟 action 与 OWL 原生 action 的原则**：
- **虚拟 action**：由 `inject_virtual_actions` 注入，`action_type` 固定为 `"query"` 或 `"compute"`
- **OWL 原生 action**：OWL 文件中 `object` / `view` 定义，`action_type` 为其他自定义值

**`OntologyToolLoader` 增加 `skip_action_families` 参数**，在 `db_query` 模式下跳过 query 和 compute 两个虚拟族，只保留 OWL 原生 action：

```python
class OntologyToolLoader:
    def __init__(
        self,
        mounted_objects: list[str] | None = None,
        loader: Any | None = None,
        skip_action_families: frozenset[str] = frozenset(),
        # db_query 模式传入 frozenset({"query", "compute"})
    ) -> None:
        ...
        self._skip_action_families = skip_action_families

    def _build_action_tools(self, action_gen, obj_code):
        for tool_def in action_gen.generate_tools(obj_code):
            action_family = tool_def.get("_meta", {}).get("action_type", "")
            if action_family in self._skip_action_families:
                continue   # db_query 模式：跳过 query / compute 虚拟动作
            ...
```

#### 3.3.5 `init_agent_conf` 的调整

`db_query` 模式分支改为：
1. `_build_db_query_tools()` 生成 HTTP 版 `data_query_{code}` → 放入 `AgentConfig.tools`
2. 填充 `mounted_objects` 和 `shared_loader` → 供 `OntologyToolLoader` 生成 compute + 自定义 action

```python
if load_mode == "db_query":
    db_loader = self._build_shared_loader(rel_resource_list)  # 含 inject_virtual_actions
    # 1. HTTP 版 data_query 工具
    db_query_tools = self._build_db_query_tools(
        agent_id=agent_id,
        rel_resource_list=rel_resource_list,
        loader=db_loader,          # 仅用于获取描述信息
    )
    dynamic_tools = {**dynamic_tools, **db_query_tools}
    # 2. 保留 mounted_objects，让 OntologyToolLoader 仅生成 OWL 原生自定义 action
    #    （query 和 compute 族虚拟 action 均跳过：query 已由 HTTP 工具替代，compute 不适用）
    mounted_objects = [
        snapshot["resourceCode"]
        for rel in rel_resource_list
        if (snapshot := self._rel_resource_snapshot(rel))["resourceBizType"] in {"OBJECT", "VIEW"}
        and snapshot["resourceCode"]
    ]
    shared_loader = db_loader
    # extra 标记，告知 create_agent 跳过 query 和 compute 族虚拟动作
    # （create_agent 透传给 OntologyToolLoader(skip_action_families={"query","compute"})）
```

#### 4.3.2 修复 Bug 1 执行路径：用 `VirtualActionRegistry` 路由替代 `loader.get_action()`

**修改位置**：`_build_action_tools`，`ont_action = self._loader.get_action(...)` 这一行

当前代码通过 `loader.get_action(obj_code, action_code)` 获取 `OntologyAction` 对象。对于虚拟动作，应直接走执行器，不需要再反查 `OntologyAction`：

```python
# 修改前（内层 try 块）
meta = tool_def.get("_meta", {})
action_code: str = meta.get("action_type") or meta.get("action_code", "")
ont_action = self._loader.get_action(obj_code, action_code)
result[name] = StructuredTool(
    name=name,
    ...
    coroutine=_make_tool_coroutine(ont_action, self._loader),
)

# 修改后：按 _meta 的 scope 信息路由，无需反查 OntologyAction
meta = tool_def.get("_meta", {})
action_code: str = meta.get("action_code") or meta.get("action_type", "")
result[name] = StructuredTool(
    name=name,
    ...
    coroutine=_make_object_action_coroutine(obj_code, action_code, self._loader),
)
```

新增执行闭包（与 MCP `call_tool` 中对象分支对齐）：

```python
def _make_object_action_coroutine(obj_code: str, action_code: str, loader: Any) -> Any:
    async def _execute(**kwargs: Any) -> Any:
        try:
            from datacloud_data_service.tools.action_executor import ActionExecutor
            executor = ActionExecutor(loader)
            return await executor.execute(obj_code, action_code, kwargs)
        except Exception as exc:
            logger.error("OntologyToolLoader: 工具执行失败: %s", exc, exc_info=True)
            return {"error": str(exc)}
    return _execute
```

#### 4.3.3 修复 Bug 2：增加 VIEW 类型支持

**修改位置**：`load()` 方法，增加 VIEW 处理分支（对齐 `ToolRegistry._append_view_tools`）

```python
def load(self) -> dict[str, Any]:
    ...
    for obj_code in self._mounted_objects:
        # 先判断是否为 VIEW（在 _scenes 中）
        if self._is_view(obj_code):
            view_tools = self._build_view_tools(obj_code)
            tools.update(view_tools)
        else:
            # OBJECT 路径（原有逻辑，修复 Bug 1 后）
            query_tool = self._build_query_tool(query_gen, obj_code)
            if query_tool is not None:
                tools[query_tool.name] = query_tool
            action_tools = self._build_action_tools(action_gen, obj_code)
            tools.update(action_tools)
    ...

def _is_view(self, code: str) -> bool:
    scenes = getattr(self._loader, "_scenes", {})
    return code in scenes

def _build_view_tools(self, view_code: str) -> dict[str, Any]:
    """为 VIEW 类型生成工具，对齐 ToolRegistry._append_view_tools 逻辑。"""
    result: dict[str, Any] = {}
    try:
        from langchain_core.tools import StructuredTool

        view = self._loader.get_view(view_code)
        for action in view.actions:
            exposure = getattr(action, "exposure_policy", "direct")
            if exposure == "hidden":
                continue
            if not action.input_schema:
                continue
            result[action.action_code] = StructuredTool(
                name=action.action_code,
                description=action.description or action.action_name,
                args_schema=_json_schema_to_pydantic(
                    action.input_schema,
                    f"_{action.action_code}_Schema",
                ),
                coroutine=_make_view_action_coroutine(view_code, action.action_code, self._loader),
            )
    except Exception as exc:
        logger.warning("OntologyToolLoader: 构建 VIEW %s 工具失败: %s", view_code, exc)
    return result
```

新增 VIEW 执行闭包（与 MCP `call_tool` 中视图分支对齐）：

```python
def _make_view_action_coroutine(view_code: str, action_code: str, loader: Any) -> Any:
    async def _execute(**kwargs: Any) -> Any:
        try:
            view = loader.get_view(view_code)
            return await view.invoke_action(action_code, kwargs)
        except Exception as exc:
            logger.error("OntologyToolLoader: VIEW 工具执行失败: %s", exc, exc_info=True)
            return {"error": str(exc)}
    return _execute
```

#### 4.3.4 `_build_query_tool` 的处置

当 `inject_virtual_actions` 已调用后，`cls.actions` 中已有 `query_{code}` 虚拟动作，`ActionToolGenerator.generate_tools()` 会直接生成，不再需要 `DynamicQueryToolGenerator` 单独走一条路。

`_build_query_tool` 可以**保留为降级兜底**（当 virtual injection 未调用时作为 fallback），但正常路径不再走此处。或直接删除，强制要求调用方必须先注入虚拟动作。

建议：**保留但降级为 debug-only fallback**，在 `load()` 中优先走 `_build_action_tools`，如果 `_build_action_tools` 未产出任何工具再尝试 `_build_query_tool`。

## 5. 修改后数据流（统一架构）

```
[启动阶段]
_build_shared_loader()
  └─ OntologyLoader.load_from_owl_directory()
  └─ inject_virtual_actions(loader)       ← 新增
       ├─ cls.actions ← [query_X, compute_X, ...]  (OBJECT)
       └─ scene["_virtual_actions"] ← [query_view_X, compute_view_X, ...]  (VIEW)

[每次请求 create_agent()]
OntologyToolLoader(mounted_objects, loader).load()
  ├─ obj_code 是 OBJECT → ActionToolGenerator.generate_tools(obj_code)
  │     读 cls.actions → StructuredTool(coroutine=ActionExecutor.execute)
  └─ obj_code 是 VIEW  → loader.get_view(code).actions
        → StructuredTool(coroutine=view.invoke_action)

merged_tools = {**ontology_tools, **(extra_tools or {})}
# extra_tools 不再包含重复的 query_{code}（已移除 _build_ontology_query_wrappers）
```
