# 日志方案设计

**版本：** v1.1  
**日期：** 2026-04-21  
**覆盖范围：** `datacloud-analysis` SDK + `byclaw-data` Worker 入口

---

## 1. 现状概述

| 维度 | byclaw-data | datacloud-analysis |
|------|-------------|-------------------|
| 日志获取 | `by_framework.common.logger`（worker.py）/ `logging.getLogger(__name__)`（其余文件） | `logging.getLogger(__name__)` 统一 |
| 初始化位置 | 框架层（by_framework），自身无配置 | `agent.py` 的 `basicConfig`（仅 console） |
| 文件输出 | ❌ 无 | ❌ 无 |
| 日志轮转 | ❌ 无 | ❌ 无 |
| 压缩归档 | ❌ 无 | ❌ 无 |
| 外部日志库 | ❌ 无 | ❌ 无 |

**痛点：**
1. 两模块日志输出均只写 console，生产环境无持久化记录
2. 无按天分割，长时间运行后日志无限膨胀
3. 旧日志不压缩，磁盘占用高
4. `agent.py` 的 `basicConfig` 写死在 SDK 内部，调用方无法覆盖格式/级别

---

## 2. 设计目标

1. **统一入口**：提供 `setup_logging()` 一键初始化，`worker.py` 启动时调用一次
2. **分级输出**：Console 输出 INFO+，文件输出 INFO+；单独 error 文件仅记录 ERROR+
3. **按天轮转**：每天 00:00 自动切割日志文件
4. **自动压缩**：轮转时对旧文件执行 gzip 压缩，超出保留天数自动删除
5. **零额外依赖**：纯 Python 标准库（`logging`、`gzip`、`shutil`），不引入第三方库
6. **不干扰框架**：只配置 `datacloud_analysis` / `byclaw_data` 两个命名空间，不触碰 `by_framework` 的 root logger
7. **侵入性最小**：现有业务代码中的 `logging.getLogger(__name__)` 一行不改

---

## 3. 日志文件规划

```
logs/                          ← 统一日志目录（由环境变量控制，默认 ./logs）
├── app.log                    ← 当前日志（INFO+，所有 logger）
├── app.log.2026-04-20.gz      ← 昨天已压缩归档
├── app.log.2026-04-19.gz
├── ...（保留 30 份）
├── error.log                  ← 当前错误日志（ERROR+）
├── error.log.2026-04-20.gz
└── ...（保留 90 份）
```

```python

```

```python
"%(asctime)s [%(levelname)-5s] %(process)d %(name)s: %(message)s"
```

## 5. 核心实现

### 5.1 模块位置

```
packages/datacloud-analysis/src/datacloud_analysis/
└── logging_setup.py          ← 新增，唯一日志配置入口
```

### 5.2 完整代码

```python
"""统一日志配置模块。

只配置 datacloud_analysis / byclaw_data 两个命名空间，不干扰
by_framework 或其他第三方库已安装的 root handler。

调用方式（在进程启动时调用一次）：

    from datacloud_analysis.logging_setup import setup_logging
    setup_logging()            # 使用环境变量 / 默认值
    setup_logging(             # 显式指定
        log_dir="/var/log/datacloud",
        level="DEBUG",
        app_backup_count=30,
        error_backup_count=90,
    )

仅供测试使用的重置接口：

    from datacloud_analysis.logging_setup import reset_logging
    reset_logging()
"""

from __future__ import annotations

import gzip
import logging
import logging.handlers
import os
import shutil
from pathlib import Path

_SETUP_DONE: bool = False

_FMT_CONSOLE = "%(asctime)s [%(levelname)-5s] %(name)s: %(message)s"
_FMT_FILE    = "%(asctime)s [%(levelname)-5s] %(process)d %(name)s: %(message)s"
_DATE_FMT    = "%Y-%m-%d %H:%M:%S"

# 日志配置覆盖的命名空间（不触碰 root logger）
_MANAGED_NAMESPACES = ("datacloud_analysis", "byclaw_data")

# 仅对噪音最大的子模块提升级别，保留 langchain_core 整体的 INFO 诊断
_NOISY_LOGGERS = (
    "httpx",
    "httpcore",
    "urllib3",
    "langchain_core.tracers",
    "langchain_core.callbacks",
    "openai._base_client",
)


def _gz_namer(name: str) -> str:
    """轮转文件名追加 .gz 后缀。

    TimedRotatingFileHandler.getFilesToDelete() 用此函数生成预期文件名，
    再去磁盘匹配删除。namer 必须返回磁盘上的真实文件名（含 .gz），
    cleanup 才能正确触发，保证 backupCount 生效。
    """
    return name + ".gz"


def _gzip_rotator(source: str, dest: str) -> None:
    """轮转回调：gzip 压缩旧日志文件。

    dest 已由 _gz_namer 追加了 .gz，此处直接写入即可。
    """
    with open(source, "rb") as f_in, gzip.open(dest, "wb") as f_out:
        shutil.copyfileobj(f_in, f_out)
    os.remove(source)


def _make_timed_handler(
    log_path: Path,
    level: int,
    backup_count: int,
) -> logging.handlers.TimedRotatingFileHandler:
    """创建按天轮转 + gzip 压缩的 FileHandler。"""
    handler = logging.handlers.TimedRotatingFileHandler(
        filename=str(log_path),
        when="midnight",
        interval=1,
        backupCount=backup_count,
        encoding="utf-8",
        utc=False,
    )
    handler.suffix = "%Y-%m-%d"
    handler.namer = _gz_namer
    handler.rotator = _gzip_rotator
    handler.setLevel(level)
    handler.setFormatter(logging.Formatter(_FMT_FILE, datefmt=_DATE_FMT))
    return handler


def setup_logging(
    *,
    log_dir: str | None = None,
    level: str | None = None,
    app_backup_count: int | None = None,
    error_backup_count: int | None = None,
    enable_console: bool = True,
) -> None:
    """初始化日志配置，进程生命周期内只执行一次。

    参数优先级：显式传参 > 环境变量 > 默认值。

    环境变量：
        DATACLOUD_LOG_DIR           日志目录，默认 ./logs
        DATACLOUD_LOG_LEVEL         日志级别，默认 INFO
        DATACLOUD_LOG_APP_KEEP      app.log 保留天数，默认 30
        DATACLOUD_LOG_ERROR_KEEP    error.log 保留天数，默认 90
    """
    global _SETUP_DONE
    if _SETUP_DONE:
        return
    _SETUP_DONE = True

    # ── 参数解析（用 is not None 避免 0 被 or 短路）────────────────────────
    _log_dir = Path(
        log_dir if log_dir is not None
        else os.environ.get("DATACLOUD_LOG_DIR", "logs")
    )
    _level_str = (
        level if level is not None
        else os.environ.get("DATACLOUD_LOG_LEVEL", "INFO")
    ).upper()
    _level = getattr(logging, _level_str, logging.INFO)
    _app_keep = int(
        app_backup_count if app_backup_count is not None
        else os.environ.get("DATACLOUD_LOG_APP_KEEP", "30")
    )
    _error_keep = int(
        error_backup_count if error_backup_count is not None
        else os.environ.get("DATACLOUD_LOG_ERROR_KEEP", "90")
    )

    _log_dir.mkdir(parents=True, exist_ok=True)

    # ── 共享 Handler（两个命名空间复用同一对文件）──────────────────────────
    app_handler = _make_timed_handler(_log_dir / "app.log",   _level,        _app_keep)
    err_handler = _make_timed_handler(_log_dir / "error.log", logging.ERROR, _error_keep)

    console_handler: logging.StreamHandler | None = None
    if enable_console:
        console_handler = logging.StreamHandler()
        console_handler.setLevel(_level)
        console_handler.setFormatter(logging.Formatter(_FMT_CONSOLE, datefmt=_DATE_FMT))

    # ── 按命名空间配置，不触碰 root logger ──────────────────────────────────
    for ns in _MANAGED_NAMESPACES:
        ns_logger = logging.getLogger(ns)
        ns_logger.setLevel(_level)
        ns_logger.handlers.clear()
        if console_handler is not None:
            ns_logger.addHandler(console_handler)
        ns_logger.addHandler(app_handler)
        ns_logger.addHandler(err_handler)
        ns_logger.propagate = False   # 不向 root 传播，避免与 by_framework 重复打印

    # ── 降噪：仅针对真正高噪音的子模块 ─────────────────────────────────────
    for noisy in _NOISY_LOGGERS:
        logging.getLogger(noisy).setLevel(logging.WARNING)

    logging.getLogger(__name__).info(
        "Logging initialized: level=%s log_dir=%s app_keep=%dd error_keep=%dd",
        _level_str, _log_dir.resolve(), _app_keep, _error_keep,
    )


def reset_logging() -> None:
    """重置初始化标志，仅供测试使用。

    允许在不同测试用例中以不同参数重新调用 setup_logging()，
    防止全局状态污染测试隔离。生产代码不应调用此函数。
    """
    global _SETUP_DONE
    _SETUP_DONE = False
    for ns in _MANAGED_NAMESPACES:
        ns_logger = logging.getLogger(ns)
        ns_logger.handlers.clear()
        ns_logger.propagate = True
```

## 6. 接入方式

### 6.1 byclaw-data / worker.py（主入口）

在文件顶部 import 区加入，并在 `on_start()` 首行调用：

```python
# worker.py — 顶部 import 区
from datacloud_analysis.logging_setup import setup_logging

# ...

class DataCloudWorker:
    async def on_start(self) -> None:
        setup_logging()   # ← 首行，此后所有 getLogger(__name__) 自动受控
        ...
```

> `by_framework` 的 root logger 配置不受影响，两套日志互不干扰。

### 6.2 datacloud-analysis / agent.py

删除现有 `basicConfig`（被 `setup_logging` 的命名空间配置替代）：

```python
# 删除以下四行（agent.py 约第 20-23 行）：
# logging.basicConfig(
#     level=logging.INFO,
#     format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
# )
```

### 6.3 单元测试

现有测试无需改动。需要验证日志内容时使用 pytest 内置 `caplog`；
需要重新初始化日志配置时，在 fixture 中调用 `reset_logging()`：

```python
import pytest
from datacloud_analysis.logging_setup import reset_logging, setup_logging

@pytest.fixture(autouse=True)
def reset_log():
    reset_logging()
    yield
    reset_logging()

def test_something(caplog):
    setup_logging(level="DEBUG", enable_console=False)
    with caplog.at_level(logging.INFO, logger="datacloud_analysis"):
        run_something()
    assert "expected message" in caplog.text
```

## 8. 日志文件生命周期

```
Day 0  00:00  app.log 开始写入
Day 1  00:00  app.log 轮转
              ├─ app.log（新文件，Day 1 开始写入）
              └─ app.log.2026-04-20.gz（Day 0 压缩归档，约原始体积 10%）
...
Day 31 00:00  最旧的 .gz 自动删除（backupCount=30 触发 getFilesToDelete）
```

**`backupCount` 清理能正确工作的前提**：`_gz_namer` 必须返回磁盘上的真实文件名（含 `.gz`），handler 才能在 `getFilesToDelete()` 中匹配到旧文件并删除。v1.0 方案中 namer 返回不带 `.gz` 的名字，导致清理永远不触发——v1.1 已修复。

**磁盘占用估算：**

| 指标 | 数值 |
|------|------|
| 原始日志体积/天 | ~100 MB |
| gzip 后体积/天 | ~8–12 MB |
| 30 天 app.log 归档总量 | ~250–360 MB |
| 90 天 error.log 归档总量 | <50 MB |

## 10. v1.0 → v1.1 变更说明

| 问题 | v1.0 | v1.1 |
|------|------|------|
| `_gz_namer` bug：backupCount 清理失效 | namer 返回无 `.gz` 名，rotator 自己追加 `.gz`，handler 找不到旧文件 | namer 返回 `name + ".gz"`，rotator 直接写到 `dest`，handler 能正确匹配清理 |
| 干扰 `by_framework` | `root.handlers.clear()` 清空所有 handler | 只配置 `datacloud_analysis` / `byclaw_data` 命名空间，`propagate=False` |
| `0` 值被 `or` 短路 | `app_backup_count or env` 在传 `0` 时走默认值 | 改用 `is not None` 判断 |
| 测试隔离 | 无重置接口，全局 `_SETUP_DONE` 污染测试 | 新增 `reset_logging()` |
| 降噪范围过宽 | `langchain_core` 整体提升为 WARNING | 仅针对 `tracers` / `callbacks` 子模块 |
| 环境变量默认值类型 | `os.environ.get("...", 30)` 混用 int | 统一改为字符串 `"30"` |

## 11. 不纳入范围（可后续扩展）

- **结构化 JSON 日志**（structlog / python-json-logger）：现阶段消费方为人工排查，文本格式足够；如需接入 ELK/Loki，在 `_FMT_FILE` 替换为 JSON formatter 即可
- **分布式 Trace ID**：需与网关层（by_framework）协同，留待后续
- **日志采样**：高并发场景下对 DEBUG 日志按比例降频，现阶段不适用
