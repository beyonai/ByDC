## 1. 背景与目标

用户输入自然语言问题后，大模型可能决定调用以下三类数据查询工具：

| 工具类型 | 旧接口（paradigm wrapper） | 新接口（v3.0） | 内部执行链 |
|---|---|---|---|
| `query_{code}` | 五阶段参数（query_target/group_by/filter_condition 等） | 结构化 OQL 参数（中文字段名：select/where/group_by/order_by） | OqlRouter → SQL → DB（无 LLM） |
| `data_query_{code}` | 五阶段参数 → OWL 解析 → 拼回自然语言 → plan_generator LLM | `query: str` + `contextKnowledge: str` → plan_generator LLM | Object.query(question, knowledge_context) → SQL → DB |
| `compute_{code}` | 结构化参数（中文字段名）→ ComputeExecutor | **不变**，结构化参数（中文字段名）→ ComputeExecutor | 中文字段名 → field_code → SQL → DB（无 LLM） |

**核心问题（旧架构）**：

1. **层 B 注入静默失效**：paradigm wrapper 将工具接口改为五阶段参数，`before hook` 尝试 patch `tool_params["question"]`，但 `tool_params` 中根本没有 `question` 字段，patch 无声无息地失效
2. **`data_query_*` 双重 LLM 往返**：ReAct LLM 生成五阶段参数 → OWL 解析 → `five_stage_to_legacy_question()` 拼回自然语言 → plan_generator LLM 重新解析。结构化 → 文本 → 结构化，完全冗余
3. **重复调用 `analyze_query_clarification`**：intend_node 调用一次，before_hook 再调一次，性能浪费

**目标（v3.0）**：

1. **移除 paradigm wrapper**，工具接口回归简洁
2. **有歧义** → `analyze_query_clarification` 在 intend_node **预先分析**并缓存结果；用户实际看到追问表单的时机是 LLM 决定调用数据工具后（before hook 中触发 `interrupt()`）
3. **无歧义但有知识** → **A+B 双层注入**（按工具类型细分）：
   - **层 A（早）**：ReAct loop 启动前注入 system prompt，让 ReAct LLM 生成参数时已具备字段映射知识（三类工具均适用）
   - **层 B（晚）**：工具调用前 patch `contextKnowledge` 参数，让 SDK 内部 plan_generator LLM 也得到同等知识（**仅 `data_query_*` 适用**，因为只有它有第二个 LLM）
4. **其他** → 透传

> **注意**：`analyze_query_clarification` 在 intend_node 中仅对数据查询意图调用；若 intend_node 判断用户意图为非数据类（如操作指令、闲聊），应跳过该调用以避免无效 API 消耗。

## 2. 核心 API

### 2.1 analyze_query_clarification

```python
from datacloud_knowledge.intent import analyze_query_clarification

result = analyze_query_clarification("高效益、中效益、低效益网格的营收、利润、亩产")
# ClarificationResult(
#   query="高效益网格的营收、中效益网格的营收...",
#   needs_clarification=False,
#   knowledge='{"paradigmList":[...]}'   # 结构化知识（字段映射）
# )

result2 = analyze_query_clarification("信息技术、汽车各链的上游、下游的龙头、骨干企业数")
# ClarificationResult(
#   query="信息技术链的上游龙头企业数...",
#   needs_clarification=True,
#   form='{"paradigmList":[...]}'         # 需要追问的表单
# )

result3 = analyze_query_clarification("查询所有客户")
# ClarificationResult(query="查询所有客户")  # 透传，无增强
```

| 字段 | 类型 | 含义 |
|---|---|---|
| `query` | str | 规范化后的问题文本 |
| `needs_clarification` | bool | 是否需要向用户追问 |
| `form` | str (JSON) | 追问表单定义（含 `paradigmList`），`needs_clarification=True` 时有值 |
| `knowledge` | str (JSON) | 增强知识（字段/术语对照），`knowledge` 非空时直接注入 |

> **重要**：`needs_clarification` 与 `knowledge` **并非互斥**——API 可能同时返回追问表单（`form` 非空）和已知字段知识（`knowledge` 非空）。两者均应写入 state，不可因 `needs_clarification=True` 就丢弃 `knowledge`。

### 2.2 Object.query / View.query（SDK 层 B 注入点）

`Object` 和 `View` 均提供 `knowledge_context` 参数，这是**层 B 知识注入的底层锚点**。

```python
# datacloud_data_sdk/object.py — 单对象查询
async def query(
    self,
    question: str,
    include_plan: bool = True,
    knowledge_context: str | None = None,   # 层 B 注入点：字段映射知识
) -> dict[str, object]: ...

# datacloud_data_sdk/view.py — 跨对象视图查询
async def query(
    self,
    question: str,
    include_plan: bool = True,
    knowledge_context: str | None = None,   # 层 B 注入点：同上
) -> dict[str, object]: ...
```

**参数流转路径**：

```
before_call_back()（层 B）
  └── 无条件覆盖 tool_params["contextKnowledge"] = knowledge_str
        ↓
data_query_{code} 工具函数
  └── object.query(question=query, knowledge_context=contextKnowledge)
  或  view.query(question=query, knowledge_context=contextKnowledge)
        ↓
plan_generator LLM
  └── 接收 knowledge_context，生成 SQL 时参考字段映射
```

| 层级 | 参数名 | 说明 |
|---|---|---|
| 工具接口（ReAct LLM 调用） | `contextKnowledge: str` | before hook **无条件覆盖**注入；工具 schema 中说明"由系统注入，请勿填写" |
| SDK 方法（Object / View） | `knowledge_context: str \| None` | 透传给 plan_generator LLM |

> `data_query_*` 对应 `Object.query`；若存在跨对象查询工具（如 `data_view_*`），对应 `View.query`，接口与注入方式相同。两者均已支持 `knowledge_context`，无需修改 SDK 代码，仅需在工具函数中正确透传。

## 3. 现状分析

### 3.1 query_{code} 现状

#### 3.1.1 query_{code}（paradigm wrapper ON，structured_executor 可用）

```
ReAct LLM
  └─ 调用: query_grid(query_target=["营收"], group_by=["效益等级"], agg_function=["sum"])
           ← 五阶段结构化参数（无 question 字段）
  ↓
dispatch_tool() → before hook
  └─ _patch_question() 尝试 patch tool_params["question"]
     ← tool_params 里根本没有 question，patch 静默失效 ❌（层 B 完全无效）
  ↓
paradigm wrapper: _build_query_tool() [init_agent_conf.py]
  ├─ normalize_five_stage_args()
  ├─ resolve_five_stage_query()        ← OWL 词条解析 + 可能 interrupt 追问
  ├─ has_structured_terms? → YES
  ├─ five_stage_to_structured_payload() → OQL dict
  └─ _structured_executor(OQL dict)
       ↓
       DynamicQueryToolGenerator.generate_ontology_action()
       Action.execute(payload)
       OqlRouter.route(oql_params)    ← 策略 A/B，translate → SQL
       executor.run() → DB
```

**问题**：层 B 完全失效；OWL 词条解析（`resolve_five_stage_query`）使整个链路过长。

#### 3.1.2 query_{code}（has_structured_terms = NO，回退 legacy 路径）

```
query_{code}（has_structured_terms = NO，回退 legacy 路径）

  ... paradigm wrapper ...
  ├─ has_structured_terms? → NO（OWL 未匹配）
  ├─ five_stage_to_legacy_question(resolved) → "高效益网格的营收汇总"
  │   ← 五阶段结构化参数 → 拼回自然语言字符串
  └─ _legacy_executor("高效益网格的营收汇总")
       ↓
       loader.get_object(code).query(question)   [object.py:175]
         ↓ plan_generator LLM：自然语言 → 查询计划
         ↓ Executor.run() → SQL → DB
```

**问题**：OWL 未匹配时 fallback 到 `plan_generator LLM`，等价于 `data_query_*` 的 legacy 路径，但接口仍是五阶段参数，造成双重语义转换。

### 3.2 data_query_{code} 现状

#### 3.2.1 data_query_{code}（paradigm wrapper ON，structured_executor=None）

```
ReAct LLM
  └─ 调用: data_query_grid(query_target=["营收"], group_by=["效益等级"])
           ← 五阶段参数（与 query_ 相同接口）
  ↓
dispatch_tool() → before hook（同上，patch 同样静默失效 ❌）
  ↓
paradigm wrapper: _build_query_tool() [structured_executor=None]
  ├─ normalize + resolve_five_stage_query()   ← OWL 解析
  ├─ structured_executor=None → 强制走 legacy 路径
  ├─ five_stage_to_legacy_question(resolved) → "高效益网格的营收汇总"
  └─ _legacy_executor("高效益网格的营收汇总")
       ↓
       loader.get_object(code).query(question)  [object.py:175]
         ↓ plan_generator LLM：自然语言 → 查询计划
         ↓ Executor.run() → SQL → DB
```

#### 3.2.2 data_query_{code} 核心问题：双重 LLM 往返

```
问题链路：
  ReAct LLM 生成五阶段参数
    → OWL 解析（resolve_five_stage_query）
    → five_stage_to_legacy_question() 拼回自然语言
    → plan_generator LLM 重新生成查询计划

结构化 → 文本 → 结构化，完全冗余。且 before hook 注入的知识
在 paradigm wrapper 转换后丢失，plan_generator LLM 拿到的
只是拼回的自然语言，无任何字段上下文。
```

### 3.3 compute_{code} 现状

```
ReAct LLM
  └─ 调用: compute_grid(dimensions=[{field:"效益等级"}], metrics=[{field:"营收",agg:"sum"}])
           ← 直接结构化参数（中文字段名）
  ↓
dispatch_tool() → before hook（无 question 字段，patch 失效 ❌）
  ↓
Action.execute(payload)         ← 虚拟动作，无 paradigm wrapper
  ↓
ComputeExecutor.execute(object_code, arguments)  [compute_executor.py]
  ├─ _translate_arguments()    ← 中文字段名 → field_code，纯规则映射
  ├─ 直接拼 SQL（无 LLM！）
  └─ connector.execute(sql) → DB
```

**分析**：`compute_*` 是完全无 LLM 的确定性执行，无往返问题，接口设计合理。

> **v3.0 决策**：`compute_*` 接口保持不变，无需修改工具定义。仅通过层 A（system prompt）让 ReAct LLM 正确使用中文字段名即可。

### 3.4 追问后恢复现状

**byclaw-data/worker.py 恢复机制：**

```
前端提交 AskAgentCommand(ext_params.humanInput.paradigmList)
  ↓ worker.py:748-759
_paradigm_resume_value = {"paradigmList": [...]}
  ↓ 转为 Command(resume=_paradigm_resume_value)
  ↓ graph.astream(Command(resume=...), config)
  ↓ LangGraph 重放中断节点
  ↓ interrupt() 返回 resume 值（不再 raise）
  ↓ build_five_stage_from_frontend_reply() → resolved 查询 → 工具执行（旧路径）
```

**关键点**：LangGraph 在图恢复时重放中断节点，`interrupt()` 第二次调用返回 resume 值而非 raise。

> **v3.0 变更**：`build_five_stage_from_frontend_reply()` 依赖五阶段参数格式，随 paradigm wrapper 一起**删除**。v3.0 恢复路径改由 `_apply_resume_to_params()` 按工具类型构造对应参数（见 5.4、6.1.1）。

## 4. 架构设计

### 4.1 A+B 双层注入架构（v3.0 细化版）

| 层 | 时机 | 作用节点 | 目标 LLM | 适用工具 |
|---|---|---|---|---|
| 层 A | ReAct loop 启动**前** | `execution_node` 构造 system prompt | ReAct LLM（主推理） | 全部三类 |
| 层 B | 工具调用**前** | `before_call_back()` patch `contextKnowledge` | plan_generator LLM（SDK 内部） | 仅 `data_query_*` |

层 A 确保 ReAct LLM 生成正确参数（中文字段名 / 正确自然语言 / 正确 OQL 字段）；  
层 B 确保 SDK 内部 plan_generator LLM 在生成 SQL 时也具备字段知识（仅 `data_query_*` 有第二个 LLM）。

> **knowledge 注入格式要求**：写入 `knowledge_snippets`（供层 A）的内容应为**人类可读的字段映射文本**（如"营收 → 企业总营收（万元）；利润 → 企业总利润（万元）"），而非直接嵌入原始 `{"paradigmList":[...]}` JSON。原始 JSON 结构复杂、篇幅长，会显著增加每轮 ReAct think 的 token 消耗。intend_node 写入前需通过 `_format_knowledge_for_prompt()` 提取并格式化。`knowledge_payload` 中保留原始 JSON，供层 B 直接使用。

### 4.2 知识增强场景（needs_clarification=False）

```
用户发送问题（如"高效益、中效益、低效益网格的营收、利润、亩产"）
  ↓
intend_node
  ├── analyze_query_clarification(user_query)  ← 只调用一次（仅数据查询意图触发）
  │   └── knowledge 非空，needs_clarification=False
  ├── knowledge_snippets 写入 state（格式化后的可读文本，供层 A）
  └── knowledge_payload 写入 state（原始 JSON，供层 B 读取缓存）
  ↓
execution_node
  ├── 读取 state.knowledge_snippets
  ├── system_prompt += "\n## 数据查询知识增强\n营收→企业总营收（万元）..."
  └── run_react_loop(system_prompt=带知识的 prompt, ...)

  【层 A 完成：ReAct LLM 已知字段映射，生成正确参数】

┌─────────────────────────────────────────────────────────┐
│  分支 B.1：query_{code}（OQL 结构化参数，无第二 LLM）    │
│                                                         │
│  LLM → tool_call: query_grid_benefit(                   │
│    select=["企业总营收（万元）","物理网格亩产效益（万元/亩）"],│
│    group_by=["企业经济效益等级（高、中、低）"],            │
│    where=[]                                             │
│  )                                                      │
│  ↓                                                      │
│  dispatch_tool() → before_call_back()                   │
│    ├── needs_clarification=False，无追问                 │
│    └── tool_type ≠ data_query_*，跳过层 B patch         │
│  ↓                                                      │
│  工具函数：中文字段名 → field_code（含容错模糊匹配，见5.1）│
│  OqlRouter.route(params) → SQL → DB                     │
│                                                         │
│  【层 B 不适用：query_* 无第二 LLM】                     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  分支 B.2：data_query_{code}（自然语言 + contextKnowledge）│
│                                                         │
│  LLM → tool_call: data_query_grid(                      │
│    query="高效益、中效益、低效益网格的营收利润亩产汇总",    │
│    contextKnowledge=""  ← LLM 留空，由层 B 无条件覆盖    │
│  )                                                      │
│  ↓                                                      │
│  dispatch_tool() → before_call_back()                   │
│    ├── 读取 ctx.knowledge_payload（缓存，无需重复调用 API）│
│    └── 无条件覆盖 tool_params["contextKnowledge"] = knowledge │
│  ↓                                                      │
│  object.query(question=query,                           │
│               knowledge_context=contextKnowledge)       │
│    ↓ plan_generator LLM：已知字段映射，SQL 字段命中准确   │
│    ↓ Executor.run() → DB                                │
│                                                         │
│  【层 B 完成：SDK 内部 LLM 也知道字段映射】               │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  分支 B.3：compute_{code}（纯 SQL，无任何 LLM）          │
│                                                         │
│  LLM → tool_call: compute_grid(                         │
│    dimensions=[{"field":"企业经济效益等级（高、中、低）"}],│
│    metrics=[{"field":"企业总营收（万元）","agg":"sum"}]   │
│  )                                                      │
│  ↓                                                      │
│  dispatch_tool() → before_call_back()                   │
│    ├── needs_clarification=False，无追问                 │
│    └── compute_*，跳过层 B patch                        │
│  ↓                                                      │
│  ComputeExecutor: 中文字段名 → field_code → SQL → DB    │
│                                                         │
│  【层 B 不适用：compute_* 无 LLM】                       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  B.0：透传场景（knowledge 为空，三类工具均适用）          │
│                                                         │
│  knowledge_snippets 未写入 state → system_prompt 无增强  │
│  before_call_back()：                                    │
│    needs_clarification=False → 不追问                   │
│    knowledge_str=""    → 不触发层 B patch               │
│  → 三类工具直接执行，行为与引入知识增强前完全一致         │
└─────────────────────────────────────────────────────────┘
  ↓
respond_node → 返回结果
```

### 4.3 追问场景（needs_clarification=True）

```
用户发送问题（如"信息技术、汽车各链的上游、下游的龙头、骨干企业数"）
  ↓
intend_node
  ├── analyze_query_clarification() → needs_clarification=True
  ├── 若 knowledge 非空 → knowledge_snippets 同样写入 state（两者不互斥）
  └── knowledge_payload 写入 state（含 form 和 knowledge）
  ↓
execution_node → 若有 knowledge_snippets，system_prompt 含知识增强段落
  ↓
LLM 决定调用 query_{industry_chain}(...) / data_query_{...}(...) / compute_{...}(...)
  ↓
dispatch_tool() → before_call_back()
  ├── 读取 ctx.knowledge_payload → needs_clarification=True
  └── interrupt({"paradigmList":[...歧义表单...]})
       ↓ 【首次调用】raise GraphBubbleUp，图暂停，前端渲染追问表单
       ↓ 用户填写并提交
       ↓ 【Resume】LangGraph 重放，interrupt() 返回用户选择
       ↓ _apply_resume_to_params() 按工具类型构造参数：
         - query_*      → select/where/group_by/order_by（OQL 结构化）
         - data_query_* → query str（来自用户消歧后的问题）
                        + contextKnowledge（从选中的 paradigm 条目提取字段映射）
         - compute_*    → dimensions/metrics（中文字段名）
       ↓ 继续执行工具
```

> **contextKnowledge 来源说明**：追问场景下，`analyze_query_clarification` 返回的 `knowledge` 可能为空。resume 后，`_apply_resume_to_params` 从 `resume_value["paradigmList"]` 中提取用户选中的条目，读取字段映射信息，拼装为与层 B 相同格式的字符串写入 `tool_params["contextKnowledge"]`——这是 **Layer B 在 resume 路径的等效机制**（见 §6.9）。
>
> ⚠️ **外部假设**：上述机制依赖 `paradigmList` 每个条目携带字段元数据（字段名、分组信息等）。`analyze_query_clarification` 返回的 `form.paradigmList` 必须满足此结构，否则 `_apply_resume_to_params` 无法提取 `contextKnowledge`。实现前需确认 form API 的 paradigmList 条目格式（见 §6.9 注）。
>
> ⚠️ **Layer A 在 resume 路径的行为**：LangGraph resume 时只重放中断节点（before_call_back），**execution_node 不重跑**，Layer A 的 system_prompt 注入是一次性的（首轮 ReAct think 前完成，图暂停时 ReAct LLM 已消费了知识）。resume 后 `data_query_*` 的知识注入完全依赖 `_apply_resume_to_params` 写入 `contextKnowledge`；`query_*` 和 `compute_*` 的参数同样由 `_apply_resume_to_params` 直接从 paradigm 选择重建，无需 Layer A 重新注入。

### 5.4 中断&恢复调用链路

```
【第一轮：图暂停】

  ├── ctx.knowledge_payload.needs_clarification = True
  └── interrupt({"paradigmList": [...歧义表单...]})

       ↓ 图暂停，前端渲染表单

【前端交互】

前端渲染追问表单（paradigmList 中的选项）
用户选择并提交 AskAgentCommand(ext_params.humanInput.paradigmList)

【第二轮：图恢复】

worker.py
  ├── _paradigm_resume_value = {"paradigmList": [...用户选择...]}
  └── graph.astream(Command(resume=_paradigm_resume_value), config)
       ↓ LangGraph 重放中断节点（before_call_back 从头开始执行）
       ↓ interrupt() 第二次调用，直接返回 resume 值，不再 raise

before_call_back()（重放，interrupt 返回值后继续向下执行）
  │  【注意】此时 ctx["tool_params"] 包含原始 LLM 生成的参数（可能含歧义字段），
  │          _apply_resume_to_params 将对其执行整体替换（非增量更新）
  ├── resume_value = {"paradigmList": [...用户选择...]}
  ├── 根据工具类型构造参数（_apply_resume_to_params，详见 §6.9）：
  │   query_{code}      → select/where/group_by/order_by（OQL 结构化）
  │   data_query_{code} → query str + contextKnowledge（从选中条目提取字段映射）
  │                       ← Layer B 在 resume 路径的等效机制
  │   compute_{code}    → dimensions/metrics（中文字段名）
  └── ctx["tool_params"] 整体替换为新参数，原始 LLM 参数被丢弃
  ↓
工具正常执行
```

本方案涉及三个层次，各层职责严格分离：

┌───────────────────────────▼──────────────────────────────────┐

### 6.7 QueryClarificationPlugin（before_hook）

**文件**：`datacloud-analysis/src/datacloud_analysis/tool_hook_plugins/builtin/query_clarification_plugin.py`（新建）

**核心逻辑**：

```python
# 数据工具前缀白名单
# ⚠️ 部署注意：startswith 匹配会命中所有 query_* 前缀的工具（包括 query_intent 等非数据工具）。
# 部署时务必确保非数据工具不使用 query_/data_query_/compute_ 前缀。
_DATA_TOOL_PREFIXES: frozenset[str] = frozenset({
    "query_", "data_query_", "compute_",
})

def _is_data_tool(tool_name: str) -> bool:
    return any(tool_name.startswith(p) for p in _DATA_TOOL_PREFIXES)

def _is_data_query_tool(tool_name: str) -> bool:
    return tool_name.startswith("data_query_")


async def before_call_back(ctx: HookContext) -> HookDecision | None:
    """模块级函数（无 self），返回 HookDecision 字典而非 ctx 对象。"""
    tool_name = ctx.get("tool_name", "")

    # 仅对数据查询工具生效
    if not _is_data_tool(tool_name):
        return None

    # 优先读缓存（intend_node 已缓存，无需重复调用 API）
    payload = ctx.get("knowledge_payload") or {}
    if not payload:
        # 兜底：缓存不存在时重新调用（正常不走此分支）
        # 注意：analyze_query_clarification 为 async 函数，必须 await
        result = await _call_fallback_enhancer(ctx.get("user_query", ""))
        payload = {
            "needs_clarification": result.needs_clarification,
            "form": result.form,
            "knowledge": result.knowledge,
        }

    needs_clarification = payload.get("needs_clarification", False)
    knowledge_str = payload.get("knowledge", "")

    # 追问分支
    if needs_clarification:
        form_str = payload.get("form", "")
        # 【重要】interrupt() 两阶段行为：
        #   第一次调用：raise GraphBubbleUp，图暂停，下面代码不执行
        #   LangGraph resume 后重放此函数，第二次调用返回 resume_value，后续代码才执行
        try:
            paradigm_list = json.loads(form_str).get("paradigmList", [])
        except Exception:
            paradigm_list = []
        resume_value = interrupt({"paradigmList": paradigm_list})
        # resume 后根据工具类型构造参数（含 contextKnowledge 提取）
        _apply_resume_to_params(ctx, tool_name, resume_value, payload)
        # 返回 HookDecision 字典（patch 动作）
        return {"action": "patch", "patch": {"tool_params": dict(ctx["tool_params"])}}

    # 层 B 注入（仅 data_query_* 有第二个 LLM）
    # 无条件覆盖：无论 LLM 是否填写了 contextKnowledge，均以系统注入值为准
    if knowledge_str and _is_data_query_tool(tool_name):
        return {"action": "patch", "patch": {"tool_params": {"contextKnowledge": knowledge_str}}}

    return None
```

### 6.8 tool_hook_plugins/manager.py（GraphBubbleUp 修复）

**文件**：`datacloud-analysis/src/datacloud_analysis/tool_hook_plugins/manager.py`

**问题**：`_run_one_callback` 中 `except Exception` 会吞掉 LangGraph 的 `GraphBubbleUp`（中断信号），导致 `interrupt()` 无法正常暂停图执行。

**修复**：

```python
try:
    from langgraph.errors import GraphBubbleUp as _GraphBubbleUp
except ImportError:
    _GraphBubbleUp = type(None)


async def _run_one_callback(self, *, callback, plugin_id, context):
    try:
        return await _invoke_callback(callback, context)
    except _GraphBubbleUp:          # 必须先捕获并透传，不可被 except Exception 吞掉
        raise
    except Exception as exc:
        logger.warning("plugin %s callback failed: %s", plugin_id, exc)
        return None
```

### 6.9 _apply_resume_to_params（resume 参数重建）

**文件**：`datacloud-analysis/src/datacloud_analysis/tool_hook_plugins/builtin/query_clarification_plugin.py`

**说明**：追问 resume 后，将用户选中的 paradigm 条目转换为各工具类型所需的参数，**整体替换** `ctx["tool_params"]`——原始 LLM 生成的、可能含歧义字段的参数被完全丢弃。对 `data_query_*` 而言，提取的 `contextKnowledge` 即为 **Layer B 在 resume 路径的等效机制**。

> ⚠️ **外部结构假设**：本函数依赖 `resume_value["paradigmList"]` 每个条目携带字段元数据（字段名、聚合方式、分组信息等）。此结构由 `analyze_query_clarification` 返回的 `form.paradigmList` 保证。实现前须与 `datacloud_data_service` 侧确认 paradigmList 条目的具体字段定义，辅助函数（`_extract_knowledge_from_paradigm` 等）依赖该格式实现。

```python
def _apply_resume_to_params(
    ctx: HookContext,
    tool_name: str,
    resume_value: dict,
    payload: dict,
) -> None:
    """resume 后按工具类型重建 tool_params，整体替换原始 LLM 生成参数。

    Args:
        ctx:          HookContext，ctx["tool_params"] 将被整体替换
        tool_name:    当前工具名（用于判断工具类型）
        resume_value: interrupt() 返回值，{"paradigmList": [...用户选择...]}
        payload:      knowledge_payload 缓存，含 query（规范化问题）
    """
    selected: list[dict] = resume_value.get("paradigmList", [])

    if _is_data_query_tool(tool_name):
        # data_query_* ：自然语言 + contextKnowledge
        # query  : 取 payload["query"]（analyze_query_clarification 规范化后的问题）
        #          fallback 到 user_query（兜底）
        # contextKnowledge : 从用户选中的 paradigm 条目提取字段映射
        #                    与层 B 正常路径的注入格式一致（Layer B 等效）
        ctx["tool_params"] = {
            "query": payload.get("query") or ctx["user_query"],
            "contextKnowledge": _extract_knowledge_from_paradigm(selected),
        }

    elif tool_name.startswith("query_"):
        # query_* ：OQL 结构化参数（中文字段名）
        ctx["tool_params"] = _build_oql_params_from_paradigm(selected)

    elif tool_name.startswith("compute_"):
        # compute_* ：中文字段名结构化参数
        ctx["tool_params"] = _build_compute_params_from_paradigm(selected)

    # 辅助函数说明（具体实现依赖 paradigmList 条目格式，由 datacloud_data_service 定义）：
    # _extract_knowledge_from_paradigm(selected)
    #   → 从选中条目提取字段映射文本，格式同 knowledge_snippets（"营收→企业总营收（万元）"）
    # _build_oql_params_from_paradigm(selected)
    #   → 返回 {"select": [...], "where": [], "group_by": [...], "order_by": []}
    # _build_compute_params_from_paradigm(selected)
    #   → 返回 {"dimensions": [...], "metrics": [...]}
```
