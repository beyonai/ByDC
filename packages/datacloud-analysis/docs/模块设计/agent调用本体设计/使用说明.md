# Agent调用本体 - 使用说明

## 概述

本实现为DataCloud Agent提供了三种本体加载模式，通过环境变量配置即可切换，无需修改代码。

## 三种模式

### 模式1：MCP模式（远程服务模式）

**适用场景**：
- 本体数据由独立的datacloud-data服务管理
- 需要动态更新本体定义
- 多个Agent实例共享同一本体服务

**配置方式**：
```bash
export ONTOLOGY_LOAD_MODE=mcp
export ONTOLOGY_MCP_ENDPOINT=http://localhost:8080/api/v1/mcp/
```

**工作原理**：
- Agent启动时通过HTTP连接到MCP服务
- 调用`list_tools`获取可用工具列表
- 工具调用时通过`call_tool`转发到MCP服务
- 支持通过`x-object-ids`请求头过滤对象

**优点**：
- 本体定义集中管理
- 支持热更新
- Agent轻量化

**缺点**：
- 依赖网络连接
- 增加调用延迟
- 需要额外的MCP服务

---

### 模式2：动态Tool加载模式

**适用场景**：
- 本体定义相对稳定
- 需要离线运行
- 希望减少网络调用

**配置方式**：
```bash
export ONTOLOGY_LOAD_MODE=dynamic_tool
export ONTOLOGY_SCENE_PATH=/app/ontology/owl
export ONTOLOGY_AUTO_REGISTER=true
```

**工作原理**：
- Agent启动时解析OWL文件
- 为每个对象生成专用的查询工具（如`Order_query`）
- 为每个动作生成专用的动作工具（如`Order_create`）
- 工具内部仍调用`query_objects`和`execute_action`

**优点**：
- 无需外部服务
- 工具名称更明确
- LLM更容易理解

**缺点**：
- 启动时间增加
- 内存占用增加
- 工具数量多（对象数 × (1+动作数)）

---

### 模式3：统一接口模式（默认）

**适用场景**：
- 快速开发和测试
- 本体对象数量较多
- 希望最小化工具数量

**配置方式**：
```bash
export ONTOLOGY_LOAD_MODE=unified_interface
# 或者不设置，默认就是此模式
```

**工作原理**：
- 只注册两个通用工具：`query_objects`和`execute_action`
- 通过`object_type`参数指定要操作的对象
- 通过`action_type`参数指定要执行的动作

**优点**：
- 工具数量最少（固定2个）
- 启动最快
- 内存占用最小

**缺点**：
- LLM需要记住对象名称
- 参数较多，容易出错

---

## 快速开始

### 1. 安装依赖

```bash
cd packages/datacloud-analysis
pip install -e .
```

### 2. 选择模式并配置

**使用默认模式（推荐）**：
```bash
# 不需要额外配置，直接使用
```

**使用MCP模式**：
```bash
export ONTOLOGY_LOAD_MODE=mcp
export ONTOLOGY_MCP_ENDPOINT=http://localhost:8080/api/v1/mcp/
```

**使用动态Tool模式**：
```bash
export ONTOLOGY_LOAD_MODE=dynamic_tool
export ONTOLOGY_SCENE_PATH=/path/to/owl/files
```

### 3. 启动Agent

```python
from datacloud_analysis.agent import create_agent

agent = create_agent(
    model="gpt-4",
    mounted_objects=["Order", "Customer", "Product"],
)

# Agent会自动根据环境变量选择加载模式
```

## 故障排查

### 问题1：MCP模式连接失败

**症状**：
```
MCPLoader: failed to load MCP tools: Connection refused
MCPLoader: falling back to unified interface mode
```

**解决方案**：
1. 检查MCP服务是否启动：`curl http://localhost:8080/api/v1/mcp/`
2. 检查端点配置是否正确
3. 检查网络连接

### 问题2：动态Tool模式找不到OWL文件

**症状**：
```
DynamicToolLoader: scene path does not exist: /app/ontology/owl
DynamicToolLoader: falling back to unified interface mode
```

**解决方案**：
1. 检查路径是否正确：`ls /app/ontology/owl`
2. 检查文件权限
3. 确保路径中有`.owl`文件

### 问题3：环境变量未生效

**症状**：
Agent仍然使用默认模式

**解决方案**：
1. 确认环境变量已设置：`echo $ONTOLOGY_LOAD_MODE`
2. 重启Agent进程
3. 检查日志中的模式信息：`grep "load_mode" agent.log`

## 性能对比

| 模式 | 启动时间 | 内存占用 | 调用延迟 | 工具数量 |
|------|---------|---------|---------|---------|
| 统一接口 | 最快 | 最小 | 最低 | 2个 |
| MCP | 快 | 小 | 中等 | 动态 |
| 动态Tool | 慢 | 大 | 低 | 对象数×(1+动作数) |

**建议**：
- 开发环境：使用统一接口模式
- 生产环境（小规模）：使用动态Tool模式
- 生产环境（大规模）：使用MCP模式

## 扩展开发

### 添加新的加载模式

1. 在`tools/ontology_loader.py`中创建新的Loader类：

```python
class MyCustomLoader(OntologyLoader):
    def load_tools(self, mounted_objects: list[str] | None = None) -> list[Any]:
        # 实现你的加载逻辑
        return tools
```

2. 在`create_ontology_loader`工厂函数中添加分支：

```python
def create_ontology_loader(...):
    if load_mode == "my_custom":
        return MyCustomLoader(...)
    # ...
```

3. 更新配置类支持新模式：

```python
class OntologySettings(BaseSettings):
    load_mode: str = Field(
        default="unified_interface",
        description="... | my_custom",
    )
```
