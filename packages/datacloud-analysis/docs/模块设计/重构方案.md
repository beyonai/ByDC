### 3.3 callback 在工具调用链中的位置

`execution/tool_wrapper.py` 的 `dispatch_tool()` 负责串联 hook：

```
LLM tool_call
    │
    ▼
run_before(HookContext)          ← before_call_back（所有匹配插件按优先级升序执行）
    │  action=patch   → 修改 tool_params 继续
    │  action=short_circuit → 跳过工具调用，直接返回 hook 提供的结果
    │  action=fail    → 抛出错误
    ▼
actual_tool.invoke(patched_params)
    │
    ▼
run_after(HookContext+tool_output) ← after_call_back（按优先级降序执行）
    │  action=patch   → 修改输出继续
    │  action=recover → 替换工具输出
    │  action=fail    → 将工具输出标记为错误
    ▼
返回给 ReAct 消息链（ToolMessage）
```

### 3.4 tool_wrapper.py 核心实现

```python
async def dispatch_tool(
    tool_call: ToolCall,
    tools_map: dict[str, BaseTool],
    hook_manager: ToolHookPluginManager,
    state: AgentState,
) -> tuple[str, Any]:
    """调用工具，串联 before/after hook，记录 reason 日志。"""
    tool_name = tool_call["name"]
    raw_params = dict(tool_call["args"])

    # 1. 提取并记录 reason（大模型选择该工具的理由）
    reason = raw_params.pop("reason", "")
    logger.info("[tool_call] tool=%s reason=%s params=%s",
                tool_name, reason, _sanitize(raw_params))

    # 2. 构建 HookContext
    ctx: HookContext = {
        "tool_name":  tool_name,
        "tool_params": raw_params,
        "session_id": ...,
        "user_query": state.get("user_query") or "",
        "workspace_dir": state.get("workspace_dir"),
        "knowledge_snippets": state.get("knowledge_snippets") or [],
        "term_context": state.get("confirmed_terms") or [],
    }

    # 3. before hook
    ctx, before_decision = await hook_manager.run_before(ctx)
    if before_decision:
        action = before_decision.get("action")
        if action == "short_circuit":
            return tool_call["id"], before_decision["result"]["tool_output"]
        if action == "fail":
            raise ToolHookError(before_decision)

    # 4. 实际调用工具
    tool = tools_map[tool_name]
    try:
        output = await tool.ainvoke(ctx["tool_params"])
        ctx["tool_output"] = output
        ctx["tool_error"] = None
    except Exception as e:
        ctx["tool_output"] = None
        ctx["tool_error"] = {"error_type": type(e).__name__, "message": str(e)}

    # 5. after hook
    ctx, after_decision = await hook_manager.run_after(ctx)
    if after_decision:
        action = after_decision.get("action")
        if action == "recover":
            output = after_decision["result"]["tool_output"]
        if action == "fail":
            raise ToolHookError(after_decision)

    return tool_call["id"], ctx.get("tool_output")
```

工具注册时统一注入 `reason` 字段到 schema（不改变工具本身实现）：

```python
def inject_reason_field(t: BaseTool) -> BaseTool:
    """往工具的 args_schema 注入 reason: str 字段（必填）。"""
    schema = t.args_schema.schema()
    schema["properties"] = {"reason": {"type": "string", "description": "选择本工具的理由"}, **schema["properties"]}
    schema.setdefault("required", []).insert(0, "reason")
    return t
```

### 4.2 intend 节点（`orchestration/intend/`）

**职责：** 接收报文，判断是命令还是需要 ReAct 处理的任务。

```python
async def intend_node(state: AgentState, config: RunnableConfig) -> dict:
    user_query = state["messages"][-1].content

    # 1. 先走命令路由（低延迟、确定性）
    cmd_result = await command_router.try_dispatch(user_query, state, config)
    if cmd_result.handled:
        return {"intent": "command", "command_result": cmd_result.payload,
                "execution_status": "command_done"}

    # 2. LLM 意图分类（react / chitchat / clarify）
    intent = await intent_classifier.classify(user_query, state)
    return {"intent": intent, "execution_status": "execution"}
```

- `command_router.py`：封装现有 `CommandPluginManager`，内置优先级 < 扩展
- `intent_classifier.py`：轻量 LLM 调用，结构化输出意图标签

### 4.3 execution 节点（`orchestration/execution/`）

**职责：** 基于 tools + skills，通过 function call 执行 ReAct 循环。

ReAct 停止信号分三层（见第五章详细分析）：

```python
# react_loop.py 主循环骨架
for round_idx in range(MAX_ROUNDS):
    ai_msg = await llm_with_tools.ainvoke(messages)
    messages.append(ai_msg)

    if not ai_msg.tool_calls:
        # 【停止信号 L2】LLM 不再调用工具，直接文字回答
        state["react_final"] = {"result_type": "text", "answer": ai_msg.content,
                                 "stop_reason": "no_tool_call"}
        return

    for tc in ai_msg.tool_calls:
        tool_id, result = await dispatch_tool(tc, tools_map, hook_manager, state)
        if isinstance(result, dict) and result.get("__finish__"):
            # 【停止信号 L1】LLM 主动调用 finish_react
            state["react_final"] = {**result, "stop_reason": "finish_tool"}
            return
        messages.append(ToolMessage(content=str(result), tool_call_id=tool_id))

# 【停止信号 L3】超出最大轮数，强制停止
state["react_final"] = {"result_type": "text",
                         "answer": _summarize_last_output(messages),
                         "stop_reason": "max_rounds"}
```

**职责：** 读取 `state["react_final"]`，按结果类型格式化输出，与 execution 完全解耦。

## 五、ReAct 停止检测：三层信号机制

### 5.1 能否清楚识别停止？

**直接回答：靠单一手段不能，靠三层信号组合可以可靠识别。**

大模型在 function call 模式下，停止时有三种可能的表现：

| 表现 | 可靠性 | 原因 |
|------|--------|------|
| 不再产生 tool_calls，直接输出文字 | 中（~85%）| 多数模型默认行为，但部分模型会产生空 tool_calls |
| 主动调用 `finish_react` 工具 | 高（~95%）| 需要 prompt 明确指示，强制携带结构化元数据 |
| MAX_ROUNDS 超限 | 100% | 兜底保证，不依赖模型行为 |

三者组合后，**识别准确率接近 100%**，核心在于优先级顺序：

```
L1（finish_react 工具）> L2（无 tool_call）> L3（MAX_ROUNDS）
```

### 5.2 finish_react 工具定义

```python
@tool("finish_react")
async def finish_react(
    reason: str,
    answer: str,
    result_type: Literal["text", "csv_file"],
    csv_file_path: str = "",
) -> dict:
    """
    ReAct 分析完毕时必须调用本工具。禁止直接输出最终答案。
    - reason:        结束原因（用于审计）
    - answer:        文字类结论（result_type=text 时必填）
    - result_type:   'text' | 'csv_file'
    - csv_file_path: CSV 文件路径（result_type=csv_file 时必填）
    """
    return {"__finish__": True, "answer": answer,
            "result_type": result_type, "csv_file_path": csv_file_path}
```

**finish_react 相比"检测无 tool_call"的优势：**

1. **携带结构化元数据**：result_type 和 csv_file_path 直接传给 respond 节点，无需从消息历史推断
2. **强制显式声明**：LLM 不能"悄悄停止"，必须给出结论
3. **reason 可审计**：记录大模型为何认为任务完成

### 5.3 Prompt 约束（execution system prompt 必须包含）

```
你拥有以下工具：{tool_list}

规则：
1. 每次调用工具，必须在 reason 参数中填写选择该工具的理由。
2. 当分析任务完成时，必须调用 finish_react 工具声明结束，禁止直接输出最终答案。
3. 若结果是 CSV 文件，使用 result_type=csv_file 并填写文件路径。
4. 若结果是文字说明，使用 result_type=text 并在 answer 中填写结论。
```

### 5.4 是否需要独立的 respond 节点？

**需要，但理由不是"帮助识别停止"，而是职责分离：**

- execution 节点：只关心"循环执行工具直到停止"，停止信号已由三层机制解决
- respond 节点：只关心"将 react_final 格式化输出"，不涉及任何停止判断

两者独立后，格式化逻辑（chunk 6001 / 普通文字）可独立演化，不污染 ReAct 循环代码。

## 八、中断与恢复机制（HITL）

### 8.1 完整链路概览

```
ask_user 工具调用
    │
    ▼  langgraph.types.interrupt(payload)      ← 图在此处挂起，写 checkpoint
    │
    ▼  astream_events 循环结束（无 on_chain_end finish）
    │
    ▼  worker._stream_graph() 调用 aget_state() 读快照
    │
    ▼  snapshot.interrupts 非空 → 走中断分支
    │
    ▼  context.ask_user(AskUserEvent)           ← 网关推送问题给前端
    │
    ▼  return {"status": "waiting"}             ← worker 挂起，不 flush_to_history
    │
    (用户在前端作答)
    │
    ▼  网关收到 ResumeCommand
    │
    ▼  worker.process_command() → Command(resume=answer)
    │
    ▼  graph.astream_events(Command(resume=answer), config)
    │   config 中必须携带：
    │     thread_id      （同原 thread_id）
    │     checkpoint_id  （snapshot.config 中读取，由网关随 AskUserEvent metadata 回传）
    │     checkpoint_ns  （同上）
    │
    ▼  LangGraph 从 interrupt() 处恢复，ask_user 工具返回用户答案
    │
    ▼  ReAct 循环继续
```

### 8.2 ask_user 工具实现

`ask_user` 工具不通过 `gateway_context` 直接推送消息，**推送由 worker 层完成**。
工具本身只需调用 `interrupt()` 挂起图，并把问题内容放入 payload：

```python
# tools/ask_user.py
from langgraph.types import interrupt
from langchain_core.tools import tool

@tool("ask_user")
async def ask_user(question: str, reason: str = "") -> str:
    """向用户提问，等待用户作答后 ReAct 恢复。

    工具调用后图立即挂起（checkpoint 已写入），
    worker 层检测到 snapshot.interrupts 后通过网关推送问题给前端。
    ResumeCommand 到达后，本工具返回用户的回答字符串。
    """
    answer = interrupt({
        "question": question,
        "reason": reason,
        "reason_code": "ASK_USER",      # worker 层用于区分中断类型
    })
    return str(answer)
```

**关键约束：**
- `ask_user` 不需要也不应该注入 `inject_reason_field`（reason 已内置）
- `ask_user` 的调用结果不进入 `__finish__` 判断，`dispatch_tool` 中需跳过 `finish_react` 检测

### 8.3 worker 层中断检测（`_stream_graph` 中已有，重构后保持不变）

`worker._stream_graph()` 在 `astream_events` 结束后执行：

```python
# 已有逻辑，重构后原样保留
snapshot = await target_graph.aget_state(snapshot_config)

if snapshot is not None and snapshot.interrupts:
    first = snapshot.interrupts[0]
    interrupt_value = first.value          # = ask_user 工具传入 interrupt() 的 payload
    prompt = interrupt_value.get("question", str(interrupt_value))
    interrupt_reason = interrupt_value.get("reason_code", "interrupt")

    checkpoint_id = snapshot.config["configurable"]["checkpoint_id"]
    checkpoint_ns = snapshot.config["configurable"]["checkpoint_ns"]

    if interrupt_reason == "AGENT_DELEGATE_WAIT":
        return {"status": "waiting"}       # 内部代理等待，不向前端推送

    # 向网关推送问题
    await context.ask_user(
        AskUserEvent(
            prompt=prompt,
            metadata={
                "thread_id":      config["configurable"]["thread_id"],
                "checkpoint_id":  checkpoint_id,
                "checkpoint_ns":  checkpoint_ns,
                "agent_id":       by_agent_id,
                "conf_hash":      conf_hash,
                "interrupt_reason": interrupt_reason,
            },
        )
    )
    return {"status": "waiting"}           # 不调用 flush_to_history
```

**`aget_state` 必须使用不带 `checkpoint_id` 的 config**（读最新快照），
但 `AskUserEvent.metadata` 里必须带上快照里读到的 `checkpoint_id` + `checkpoint_ns`，
让网关在 `ResumeCommand` 时能精确定位。

### 8.4 恢复流程（ResumeCommand → 图继续）

```python
# worker.process_command() 中已有逻辑，重构后保持不变
if isinstance(command, ResumeCommand):
    resume_value = command.reply_data if command.reply_data is not None else command.content
    graph_input = Command(resume=resume_value)   # LangGraph 恢复命令

# config 中必须恢复 checkpoint_id / checkpoint_ns（从 header.metadata 中读取）
ckpt_id, ckpt_ns = self._resolve_resume_checkpoint_target(header_metadata=header_metadata)
if ckpt_id:
    config["configurable"]["checkpoint_id"] = ckpt_id
    config["configurable"]["checkpoint_ns"] = ckpt_ns
```

前端在收到 `AskUserEvent` 后，应将 `metadata` 中的
`thread_id` / `checkpoint_id` / `checkpoint_ns` / `conf_hash` 原样带回
`ResumeCommand.header.metadata`，worker 才能正确定位 checkpoint。

### 8.5 幂等保护（已有，重构后保持不变）

`DataCloudWorker` 已实现两层幂等：

| 层 | 机制 | 说明 |
|----|------|------|
| 缓存 | `_resume_result_cache`（LRU 256）| 相同 `session_id + checkpoint_id + resume_value` 直接返回缓存结果 |
| inflight | `_resume_inflight`（Future）| 并发重试时后续请求等待第一个完成后共享结果 |

去重 key = `sha1(session_id + checkpoint_id + checkpoint_ns + resume_payload)`

### 8.6 checkpoint 依赖说明

| 条件 | 结果 |
|------|------|
| `bootstrap.setup()` 已完成，PG checkpointer 已注册 | `create_agent()` 编译时注入 checkpointer，HITL 正常工作 |
| `bootstrap.setup()` 未完成（测试/开发场景）| 无 checkpointer，`aget_state` 跳过，`snapshot=None`，HITL 功能禁用，仅日志告警 |

**重构要求：`create_agent()` 和 `agent.py` 的逻辑不变**，仍在 `try/except RuntimeError` 中
尝试注入 checkpointer，保证两种场景均可运行。

### 8.7 `ask_user` 与 `finish_react` 的区分

`dispatch_tool()` 需区分两种特殊工具：

```python
async def dispatch_tool(tc, tools_map, hook_manager, state):
    tool_name = tc["name"]

    if tool_name == "finish_react":

    # 普通工具：走完整 before → invoke → after hook 链
```
