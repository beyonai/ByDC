# 知识增强模块设计

## 1. 背景

当前 `knowledge_enhance_node` 的主流程是：

```text
user_query
  -> search_knowledge(query=整句问题)
  -> 返回 term_matches / fuzzy_term_matches / term_subgraphs
  -> 提取 term_hints
  -> 生成 enriched_query
```

这条链路具备基础知识增强能力，但仍存在以下问题：

1. **术语抽取粒度不足**

   当前直接以整句问题发起知识检索，没有显式先提取“需要理解的业务术语”。

1. **术语确权链路不完整**

   虽然系统已有 `confirmed_terms` / `ambiguous_terms` 字段以及候选检索、LLM 消歧能力，但当前 `knowledge_enhance_node` 没有真正填充这些字段。

1. **知识增强产物分层不清**

   当前同时存在 `term_hints`、`knowledge_snippets`、`knowledge_payload`、`knowledge_preview` 等字段，但它们的职责边界未被清晰定义，容易造成文档理解和后续实现偏差。

---

## 2. 设计目标

1. 先由大模型从用户问题中抽取需要理解的业务术语。

1. 对抽取出的术语执行候选检索与消歧确权。

1. 基于确权术语加载更高质量的知识上下文。

1. 将知识增强结果分为：

   - 核心业务产物
   - 兼容/派生产物
   - 调试产物

1. 与现有 planning / execution / end 节点保持兼容，避免一次性破坏现有链路。

---

## 3. 总体方案

### 3.1 推荐方案

将术语提取、候选搜索、LLM 消歧、知识加载统一前移到 `knowledge_enhance_node`，由该节点一次性生成后续节点可复用的增强结果。

### 3.2 总体调用链路

```text
knowledge_enhance_node
  │
  ├─ 1. LLM 提取概念术语列表
  │      输入：user_query
  │      输出：concept_terms: list[str]
  │
  ├─ 2. search_all_candidates(concept_terms)
  │      输入：concept_terms
  │      输出：candidates_map: dict[str, list[CandidateDict]]
  │
  ├─ 3. disambiguate_candidates(candidates_map, original_question, llm=...)
  │      输入：candidates_map + 原始问题
  │      输出：
  │        - confirmed_terms: list[dict]
  │        - ambiguous_terms: list[dict]
  │
  ├─ 4. load_knowledge_for_confirmed(confirmed_terms)
  │      输入：confirmed_terms
  │      输出：knowledge_payload
  │
  └─ 5. 派生产物生成
         - enriched_query
         - term_hints
         - knowledge_snippets
        - 删除 `knowledge_preview`
```

---

## 4. 当前代码现状

### 4.1 当前 `knowledge_enhance_node` 已有能力

当前节点已经实现：

- 从 `messages` 中提取 `user_query`
- 调用 `search_knowledge.ainvoke({"query": user_query})`
- 从 `payload["term_matches"]` 中提取 `term_hints`
- 基于 `term_hints` 改写 `enriched_query`
- 返回：
  - `user_query`
  - `enriched_query`
  - `term_hints`
  - `knowledge_snippets`
  - `knowledge_payload`
  - `knowledge_preview`（现状字段，后续删除）

### 4.2 当前缺失能力

当前节点尚未实现：

- LLM 概念术语提取
- `search_all_candidates(...)` 主链接入
- `disambiguate_candidates(...)` 主链接入
- `confirmed_terms` / `ambiguous_terms` 的正式产出
- 基于确权术语加载知识的独立步骤

---

## 5. 详细设计

### 5.1 步骤 1：LLM 提取概念术语

在 `knowledge_enhance/node.py` 中新增 `_extract_concept_terms()`：

```python
async def _extract_concept_terms(
    user_query: str,
    *,
    llm: Any,
) -> list[str]:
    """用 LLM 从用户问题中提取需要知识库查询的概念术语列表。"""
```

#### Prompt 要求

- 只提取需要进入知识库做定义、映射、确权的业务术语
- 不提取时间词、数量词、通用动作词
- 输出严格 JSON 数组，例如：

```json
["GMV", "活跃用户", "品类"]
```

#### 输入来源

- 输入：`user_query`
- LLM 配置与 planning 阶段保持一致

---

### 5.2 步骤 2：候选检索

直接复用 `tools/knowledge.py` 中已有函数：

```python
candidates_map = await search_all_candidates(
    concept_terms,
    top_k=5,
)
```

#### 返回结构

```python
dict[str, list[CandidateDict]]
```

其中：

- key：原始术语 mention
- value：候选术语列表

#### `CandidateDict` 字段结构

```python
{
    "term_id": str,
    "term_name": str,
    "term_type_code": str,
    "match_type": str,
    "confidence": float,
    "score": float,
    "name_id": str | None,
}
```

说明：

- `CandidateDict` 在 `datacloud-knowledge` 中类型别名定义为 `dict[str, Any]`
- 具体字段由 `_candidate_to_dict()` 实际构造决定

---

### 5.3 步骤 3：术语消歧与确权

直接复用：

```python
confirmed_terms, ambiguous_terms = await disambiguate_candidates(
    candidates_map,
    original_question=user_query,
    llm=llm,
)
```

#### 消歧逻辑

1. 单候选/高置信度直接确权
1. 多候选由 LLM 结合上下文选择
1. 无法判断的进入 `ambiguous_terms`

#### `confirmed_terms` 结构

```python
{
    "mention": "GMV",
    "term_id": "t001",
    "term_name": "GMV",
    "term_type_code": "VIEW",
    "confidence": 0.98,
    "name_id": "n001",
}
```

#### `ambiguous_terms` 结构

```python
{
    "mention": "活跃用户",
    "candidates": [
        {
            "term_id": "t010",
            "term_name": "DAU",
            "term_type_code": "METRIC",
            "match_type": "bm25",
            "confidence": 0.76,
            "score": 0.76,
            "name_id": "n010"
        },
        {
            "term_id": "t011",
            "term_name": "MAU",
            "term_type_code": "METRIC",
            "match_type": "bm25",
            "confidence": 0.73,
            "score": 0.73,
            "name_id": "n011"
        }
    ]
}
```

---

### 5.4 步骤 4：确权术语知识加载

建议在 `knowledge_enhance/node.py` 中新增：

```python
async def _load_knowledge_for_confirmed(
    confirmed_terms: list[dict[str, Any]],
) -> dict[str, Any]:
    """基于确权术语加载结构化知识载荷。"""
```

#### 设计目标

- 不再以整句 query 为主做知识加载
- 改为以 `confirmed_terms` 为核心输入
- 输出统一 `knowledge_payload`

#### 建议输出内容

- 术语基础信息
- 术语类型信息
- 术语对应知识片段
- 子图/关系信息（按需要裁剪）

---

### 5.5 步骤 5：派生产物生成与字段清理

在完成确权和知识加载后，再生成兼容字段：

#### `enriched_query`

- 基于 `confirmed_terms` 对 `user_query` 进行标准化改写
- 作为 planning / execution 优先消费的 query

#### `term_hints`

- 由 `confirmed_terms` 派生
- 保持与旧 planning 输入兼容
- 长期目标可收敛，但短期不能删除

#### `knowledge_snippets`

- 从 `knowledge_payload` 中提取轻量摘录
- 供 execution / tool-hook 使用

#### `knowledge_preview`

- 该字段无实际业务价值
- 不作为正式状态契约
- 在后续开发中应删除，不再继续保留

---

## 6. 字段分层设计

### 6.1 核心业务产物

这些字段是知识增强阶段真正的主输出：

- `concept_terms`
- `confirmed_terms`
- `ambiguous_terms`
- `enriched_query`
- `knowledge_payload`

### 6.2 兼容/派生产物

这些字段保留是为了兼容现有调用链：

- `term_hints`
- `knowledge_snippets`

### 6.3 待删除产物

这些字段不应成为后续节点正式依赖，且应纳入后续开发清理范围：

- `knowledge_preview`

---

## 7. 字段评审结论

### 7.1 `enriched_query`

- **保留**
- 当前 planning / execution 已实际消费
- 属于主链核心字段，不冗余

### 7.2 `term_hints`

- **保留**
- 当前 planning 仍依赖
- 但定位应调整为“兼容字段”，而不是主语义字段

### 7.3 `knowledge_snippets`

- **保留**
- 当前 execution / tool-hook 已消费
- 定位为“执行侧轻量知识摘录”

### 7.4 `knowledge_preview`

- **删除**
- 当前没有核心节点依赖它
- 方案中明确后续开发应移除该字段及其相关生成逻辑

## 9. 与后续节点的关系

### 9.1 planning 节点

planning 建议优先使用：

1. `confirmed_terms`
2. `term_hints`
3. `enriched_query`

其中：

- `confirmed_terms` 作为高质量术语上下文来源
- `term_hints` 作为兼容补充
- `ambiguous_terms` 非空时可触发澄清判断

### 9.2 execution 节点

execution 继续消费：

- `enriched_query`
- `term_hints`
- `knowledge_snippets`

tool-hook 机制仍可基于 `knowledge_snippets` 做参数增强。

### 9.3 end 节点

`end` 节点不负责术语理解主流程，只消费：

- `confirmed_terms`
- `knowledge_payload`
- `enriched_query`

用于增强最终总结解释。

---

## 10. 降级策略

| 场景 | 处理方式 |
| ---- | -------- |
| LLM 提取术语失败 | 降级到原有 `search_knowledge(user_query)` 链路 |
| `search_all_candidates` 失败 | `confirmed_terms=[]`，`ambiguous_terms=[]`，继续主流程 |
| `disambiguate_candidates` 失败 | 单候选项直接确权，多候选项进入 `ambiguous_terms` |
| 确权术语知识加载失败 | 保留 `confirmed_terms`，`knowledge_payload={}` |
| 全部失败 | 回退到原有字段输出，不阻断主流程 |

---

## 11. 涉及文件

| 文件 | 改动类型 |
| ---- | -------- |
| `orchestration/knowledge_enhance/node.py` | 新增术语抽取、确权知识加载、改造主流程 |
| `tools/knowledge.py` | 复用已有候选检索/消歧能力，必要时补统一门面 |
| `orchestration/planning/node.py` | 优先使用 `confirmed_terms`（可选优化） |
| `orchestration/end/node.py` | 只消费增强结果，不承担术语理解主职责 |

---

## 12. 验收标准

1. 含明确业务术语的问题可以稳定产出 `confirmed_terms`。
2. 含歧义术语的问题可以稳定产出 `ambiguous_terms`。
3. `enriched_query` 成为 planning / execution 的优先输入。
4. `term_hints` 格式保持兼容，旧 planning 逻辑不被破坏。
5. `knowledge_snippets` 继续可供 execution / tool-hook 使用。
6. `knowledge_preview` 在后续开发中被删除，且删除后不影响主流程。
7. 任一步骤失败时，系统可降级回原链路，不阻断请求。
8. 术语提取 + 消歧增加的总耗时控制在可接受范围内。
