# 知识增强模块设计

## 背景

当前 `knowledge_enhance_node` 仅调用 `search_knowledge` 工具做一次模糊匹配，返回 `term_hints` 和 `enriched_query`。这条链路存在以下问题：

1. **术语歧义未处理**：`fuzzy_term_matches` 中存在多个候选项时，直接丢弃，不做消歧。
2. **confirmed_terms 未生成**：`AgentState` 中已有 `confirmed_terms` / `ambiguous_terms` 字段，但当前节点不填充，planning 只能依赖低质量的 `term_hints`。
3. **知识加载不完整**：`term_subgraphs`（术语子图）未被充分利用，planning 拿到的上下文偏薄。

---

## 方案 A：全量前移到 knowledge_enhance（推荐）

### 核心思路

将术语提取、候选搜索、LLM 消歧、知识加载全部在 `knowledge_enhance_node` 内完成，输出 `confirmed_terms`、`ambiguous_terms`、`knowledge_payload`、`term_hints` 四类标准产物，供 planning / execution / end 直接复用。

### 新增调用链路

```
knowledge_enhance_node
  │
  ├─ 1. LLM 提取概念术语列表
  │      输入：user_query
  │      输出：concept_terms: list[str]
  │
  ├─ 2. search_all_candidates(concept_terms)
  │      输入：concept_terms
  │      输出：candidates_map: dict[str, list[CandidateDict]]
  │             key = mention，value = 候选术语列表（strict→bm25→vector 漏斗）
  │
  ├─ 3. disambiguate_candidates(candidates_map, original_question, llm=...)
  │      输入：candidates_map + 原始问题
  │      输出：confirmed_terms: list[dict]   ← 高置信度，直接使用
  │             ambiguous_terms: list[dict]  ← 低置信度，需用户确认或降级处理
  │
  ├─ 4. 对 confirmed_terms 逐个加载知识子图
  │      调用：search_knowledge(term_name) 或直接从 candidates_map 取 term_subgraphs
  │      输出：knowledge_payload（含 term_subgraphs）
  │
  └─ 5. 生成标准产物写回 AgentState
         confirmed_terms, ambiguous_terms, concept_terms,
         knowledge_payload, term_hints, knowledge_snippets,
         enriched_query, knowledge_preview
```

---

## 详细设计

### 步骤 1：LLM 提取概念术语

在 `knowledge_enhance/node.py` 中新增 `_extract_concept_terms()` 函数：

```python
async def _extract_concept_terms(
    user_query: str,
    *,
    llm: Any,
) -> list[str]:
    """用 LLM 从用户问题中提取需要知识库查询的概念术语列表。"""
```

**Prompt 设计要点：**
- 只提取需要在知识库中查找定义/映射的业务术语（如"GMV"、"活跃用户"、"品类"）
- 不提取时间词（"上个月"）、数量词（"前10"）、通用动词（"查询"、"统计"）
- 输出 JSON 数组，如 `["GMV", "活跃用户", "品类"]`

**LLM 初始化：** 从 `state` 中读取 `model`、`model_provider`、`api_key`、`base_url`，与 planning 节点保持一致。

---

### 步骤 2：search_all_candidates

直接调用已有函数（`datacloud_analysis/tools/knowledge.py`）：

```python
from datacloud_analysis.tools.knowledge import search_all_candidates

candidates_map = await search_all_candidates(
    concept_terms,
    top_k=5,
)
```

**返回结构：**
```python
{
    "GMV": [
        {"term_id": "t001", "term_name": "GMV", "confidence": 0.98, ...},
        {"term_id": "t002", "term_name": "成交总额", "confidence": 0.85, ...},
    ],
    "活跃用户": [...],
}
```

---

### 步骤 3：disambiguate_candidates

直接调用已有函数：

```python
from datacloud_analysis.tools.knowledge import disambiguate_candidates

confirmed_terms, ambiguous_terms = await disambiguate_candidates(
    candidates_map,
    original_question=user_query,
    llm=llm,
)
```

**三层消歧逻辑（已在函数内实现）：**
1. 单候选且置信度 ≥ 阈值 → 直接 confirmed
2. 多候选 → LLM 根据问题上下文选择最合适的
3. LLM 无法判断 → 放入 ambiguous_terms

**confirmed_terms 结构：**
```python
{
    "mention": "GMV",
    "term_id": "t001",
    "term_name": "GMV",
    "term_type_code": "VIEW",
    "confidence": 0.98,
    "name_id": "n001",
}
```

**ambiguous_terms 结构：**
```python
{
    "mention": "活跃用户",
    "candidates": [
        {"term_id": "t010", "term_name": "DAU", ...},
        {"term_id": "t011", "term_name": "MAU", ...},
    ]
}
```

---

### 步骤 4：加载知识子图

对 `confirmed_terms` 中每个术语，从 `candidates_map` 中取已有的子图数据（避免重复请求），或补充调用 `search_knowledge`：

```python
async def _load_knowledge_for_confirmed(
    confirmed_terms: list[dict[str, Any]],
    candidates_map: dict[str, list[CandidateDict]],
) -> dict[str, Any]:
    """
    优先从 candidates_map 中取 term_subgraphs，
    若无则对 confirmed term_name 补充调用 search_knowledge。
    返回合并后的 knowledge_payload。
    """
```

---

### 步骤 5：生成标准产物

`knowledge_enhance_node` 最终返回：

```python
return {
    # 原有字段（保持兼容）
    "user_query": user_query,
    "enriched_query": enriched_query,
    "enriched_query_source": enriched_query_source,
    "enriched_query_confidence": enriched_query_confidence,
    "term_hints": term_hints,              # 从 confirmed_terms 生成，格式不变
    "knowledge_snippets": knowledge_snippets,
    "knowledge_payload": knowledge_payload,
    "knowledge_preview": preview,
    # 新增字段
    "concept_terms": concept_terms,        # LLM 提取的原始术语列表
    "confirmed_terms": confirmed_terms,    # 消歧后确认的术语
    "ambiguous_terms": ambiguous_terms,    # 有歧义的术语（供 planning 决策是否澄清）
}
```

---

## AgentState 字段变化

| 字段 | 原状态 | 新状态 | 说明 |
|------|--------|--------|------|
| `concept_terms` | 未填充 | ✅ 填充 | LLM 提取的术语列表 |
| `confirmed_terms` | 未填充 | ✅ 填充 | 消歧后高置信度术语 |
| `ambiguous_terms` | 未填充 | ✅ 填充 | 低置信度/多义术语 |
| `term_hints` | ✅ 已有 | ✅ 质量提升 | 从 confirmed_terms 生成，置信度更高 |
| `knowledge_payload` | ✅ 已有 | ✅ 内容更丰富 | 包含 term_subgraphs |
| `enriched_query` | ✅ 已有 | ✅ 质量提升 | 基于 confirmed_terms 改写 |

---

## planning 节点的变化

planning 节点无需修改逻辑，但可以利用新增字段做更好的决策：

- `confirmed_terms` 非空 → 直接用于构建 `term_context`，替代低质量的 `term_hints`
- `ambiguous_terms` 非空 → planning 可判断是否需要 `clarify_needed = True`，向用户确认术语

---

## 降级策略

| 场景 | 处理方式 |
|------|---------|
| LLM 提取术语失败 | 降级：直接用 `search_knowledge(user_query)` 走原有链路 |
| `search_all_candidates` 失败 | 降级：`confirmed_terms = []`，`ambiguous_terms = []`，继续 |
| `disambiguate_candidates` 失败 | 降级：将所有单候选项直接 confirmed，多候选项放入 ambiguous |
| 全部失败 | 降级：返回原有字段结构，新增字段为空，不阻断主流程 |

## 验收标准

1. 对含业务术语的问题（如"查询上个月各品类GMV"），`confirmed_terms` 非空，包含正确的术语映射。
2. 对含歧义术语的问题（如"活跃用户"可能是 DAU/MAU），`ambiguous_terms` 非空，包含候选列表。
3. `term_hints` 字段格式与原有保持兼容，planning 节点无需修改即可正常运行。
4. LLM 提取术语或消歧任一步骤失败时，系统降级到原有链路，不报错、不阻断。
5. `knowledge_payload` 中包含 `term_subgraphs`，planning 可利用子图信息生成更精准的任务。
6. 全链路耗时增量 ≤ 2 秒（LLM 提取术语 + 消歧合并为 1-2 次 LLM 调用）。
