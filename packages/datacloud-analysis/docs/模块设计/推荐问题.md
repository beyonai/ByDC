# 推荐问题方案

## 1. 背景

当前 `examples/e_commerce_demo/backend/datacloud_service/worker.py` 在回答结束时通过：

- `metadata={"relatedResources": ["推荐问题11"]}`

写死返回推荐问题，尚未根据用户提问、知识增强结果、执行结果动态生成推荐内容。

本方案目标是设计一套可落地的推荐问题能力，满足以下要求：

1. 推荐问题必须与当前用户提问强相关。
2. 推荐问题必须在当前 agent 能力边界内可回答。
3. 推荐问题需要综合对象、字段、关系、实例等知识生成。
4. 推荐生成不能增加本次主回答等待时间，主链路保持零等待。
5. 推荐结果在大模型失败时可降级，不能因为推荐失败影响主回答。
6. 每条推荐必须附带可溯源证据，证据缺失则丢弃。
7. 首版若无满足条件的推荐，允许直接返回空列表，不强凑。

## 2. 设计原则

### 2.1 核心原则

1. **先召回，后生成**：先用规则从“可回答知识池”中召回候选问题，再由大模型做筛选、排序、改写。
2. **回答能力优先于语言自然度**：宁可推荐保守，也不能推荐当前 agent 无法回答的问题。
3. **对外仅输出可直接回答推荐**：用户可见列表只允许 `direct`，任何 `requires_followup_query` 只保留在内部调试信息，不对外展示。
4. **首版宁缺毋滥**：首版没有合格推荐时直接返回空列表，不使用宽松补位策略。
5. **异步不阻塞主链路**：在知识理解完成后尽早启动推荐任务，与 planning / execution 并行，但主回答收尾不等待推荐任务。
6. **可观测、可降级**：推荐链路需要保留候选、过滤原因、模型输出、降级原因等日志。

### 2.2 推荐策略选型

经过对比，最终选择：

- **方案 C：规则召回 + 大模型重排生成**

不采用其他方案的原因：

- 纯规则方案：可控但推荐语言生硬、泛化差。
- 纯大模型方案：成本高、时延高、越界风险大，难以保证“本 agent 能回答出来”。

方案 C 的优势：

1. 规则层负责约束能力边界，避免幻觉和越权推荐。
2. 大模型层只在候选集上做排序与润色，成本可控。
3. 容易做异步化、缓存和失败降级。
4. 最符合当前场景“推荐必须可回答”的业务要求。

## 3. 推荐问题的类型梳理

推荐问题统一要求：**每一个推荐问题都必须能映射到明确的回答依据**，包括对象、字段、关系、实例或分析能力。

### 3.1 字段扩展型

适用场景：

- 用户问了某个对象的某个字段、指标或属性。

推荐方向：

1. 同对象下的关联字段。
2. 与当前字段属于同一分析主题的字段。
3. 能解释该字段的上游/下游字段。

示例：

- 用户问“企业的注册资本是多少”
- 可推荐“该企业的实缴资本是多少”
- 可推荐“该企业的成立时间是什么时候”
- 可推荐“该企业所属行业是什么”

约束：

1. 字段必须存在于 ontology / schema / 可查询数据源中。
2. 推荐关系必须来自元数据或明确配置，不能靠大模型臆测。
3. 若当前未锁定实例，则推荐应保持对象级表达，避免伪造实例问题。

### 3.2 元数据关系型

适用场景：

- 用户问了对象 A，需要引导到与 A 有明确关系的对象 B。

推荐方向：

1. 对象间主外键关系。
2. ontology view 中定义的关联路径。
3. 已配置的业务对象映射关系。

示例：

- 用户问“某企业的情况”
- 可推荐“该企业所属网格是什么”
- 可推荐“该企业所在产业链是什么”
- 可推荐“该网格下还有哪些重点企业”

约束：

1. 必须存在明确对象关系，不允许只因语义接近就推荐。
2. 若关系需要中间跳转，也必须在 agent 当前工具链可执行范围内。

### 3.3 实例邻近型

适用场景：

- 当前轮已经识别出具体实例，如企业、网格、产业链等具体对象。

推荐方向：

1. 该实例关联的其它对象。
2. 同层级、同分组、同关系下的其它实例。
3. 同实例下尚未追问的其它关键信息。

示例：

- 用户问“北京某企业的情况”
- 可推荐“该企业所在网格还有哪些企业”
- 可推荐“与该企业同产业链的其它企业有哪些”

约束：

1. 必须已识别并锁定实例锚点，或已从执行结果中拿到实例主键/唯一标识。
2. 若实例未锁定，只能退化为对象级推荐，不能生成“该企业/这个网格”类问题。
3. **首版默认关闭**。该类型通常依赖执行结果或稳定实例锚点，难以同时满足“零等待”和“本轮一定可答”，放到二期。

### 3.4 分析延展型

适用场景：

- 用户问题天然对应下一步分析动作，如趋势、对比、排序、异常、分布等。

推荐方向：

1. 时间趋势分析。
2. 同对象横向对比。
3. 所属群体中的相对位置。
4. 从单值查询延展到解释性分析。

示例：

- 用户问“这家企业今年营收是多少”
- 可推荐“这家企业近三年营收趋势如何”
- 可推荐“这家企业在所属行业中处于什么水平”

约束：

1. 必须在当前 planner / executor / tool 能力支持范围内。
2. 若该类候选仅能达到 `requires_followup_query`，则不进入用户可见推荐列表。
3. **首版默认关闭**。趋势、对比、排行等问题通常需要新增查询或计算，不满足首版“直接可答”硬约束。

## 4. 可回答性策略

推荐链路内部可保留多层判定，但对外严格单层输出。

### 4.1 对外输出层：仅 `direct`

定义：

- 当前轮经过知识增强、规划、执行后，已经具备足够上下文，推荐问题可直接由本 agent 在当前能力边界内回答。
- 不需要新增外部检索、跨系统调用或额外澄清。
- 不允许依赖“推荐生成后再补一次查询”来完成回答。

典型来源：

1. 已识别对象 + 已识别字段。
2. 已锁定实例 + 已知关系。
3. 当前轮知识快照中已确认的同一对象其它关键字段。

输出策略：

- 推荐结果最多返回 3 条，全部为 `direct`。
- `direct` 不足时，宁缺毋滥，允许返回空列表。

### 4.2 内部判定层：`requires_followup_query` 仅用于诊断

定义：

- 候选在能力边界内，但当前上下文证据不足，需追加查询后才可能回答。
- 该层候选仅用于调试、评估和离线分析，不进入用户可见输出。

典型来源：

1. 基于元数据关系可继续查到的对象问题。
2. 基于实例锚点可继续扩展的关联实例问题。
3. 基于已知对象和字段可进一步执行的趋势/对比分析。

处理策略：

- 记录到 `recommendation_debug.filtered_out_requires_followup_query`。
- 不参与前端 `relatedResources` 输出。

### 4.3 首版范围约束

为确保方案真正满足 4 条硬约束，首版只允许以下两类用户可见推荐：

1. **同对象字段扩展型**
   - 当前问题已识别对象 A。
   - 当前问题或当前轮知识快照已确认字段 AA。
   - 推荐字段 BB 与 A 同属一个对象，且存在明确字段分组、字段邻接或预置规则。
2. **知识快照内的一步关系扩展型**
   - 当前轮知识快照中已经存在对象关系线索。
   - 推荐仅允许沿已确认关系路径做一步扩展。

首版明确不纳入用户可见输出的类型：

1. 依赖 execution 新结果才能成立的结果补全型推荐。
2. 依赖新增查询的实例邻近型推荐。
3. 趋势、对比、排行、异常等分析延展型推荐。
4. 依赖历史缓存直接回填给本轮的推荐。

### 4.4 禁止推荐的情况

以下问题即使看起来合理，也不能推荐：

1. 当前 agent 不支持的分析动作。
2. 当前知识库/数据源不存在的对象或字段。
3. 缺少实例锚点却假定实例存在的问题。
4. 需要跨系统、跨权限、跨库调用但当前链路未接入的问题。
5. 与当前问题几乎等价、只是换说法的重复问题。
6. 证据不完整的问题（缺失对象/字段/关系/能力映射中的任一必需项）。

## 5. 需要查询的知识梳理

推荐问题不是从用户问题直接“脑补”出来，而是从“可回答知识池”中生成。建议将知识分为 5 层。

### 5.1 当前轮问题理解结果

来源：

- `user_query`
- `enriched_query`
- `term_hints`
- `confirmed_terms`
- planning 输入上下文

作用：

1. 判断当前问题锚点是对象、字段、实例、关系还是分析意图。
2. 提取推荐起点，如“企业”“注册资本”“某网格”“营收趋势”。

### 5.2 元数据知识

来源：

- ontology objects
- ontology fields
- views / relations
- schema / DDL
- 字段分组、对象映射、主外键关系

作用：

1. 支撑字段扩展型推荐。
2. 支撑元数据关系型推荐。
3. 保证推荐问题来自真实存在的结构，而不是语言推断。

### 5.3 实例上下文知识

来源：

- 当前执行结果中的实例 ID、实例名、过滤条件
- 当前轮锁定的对象实例
- 历史轮次已确认的对象实例

作用：

1. 支撑实例邻近型推荐。
2. 保证“该企业”“这个网格”“这条产业链”类问题有明确指向。

首版处理：

- 该层知识只用于内部评估和后续扩展，不作为首版用户可见推荐的硬依赖输入。

### 5.4 能力边界知识

来源：

- planner 当前支持的分析类型
- executor 可执行的 tool / capability
- 数据查询范围
- 对象级、字段级、实例级、分析级能力映射

作用：

1. 过滤当前 agent 无法回答的问题。
2. 作为可回答性判定的最终裁决依据。

补充约束：

- 首版能力映射必须是白名单，而不是“模型判断大概率能答”。
- 未进入能力白名单的问题一律丢弃。

### 5.5 会话历史知识

来源：

- 历史对话已确认对象
- 历史已确认实例
- 历史过滤条件
- 已推荐问题与已追问问题

作用：

1. 避免重复推荐。
2. 支撑连续多轮追问。
3. 在当前轮信息不足时补充上下文锚点。

## 6. 总体方案设计

### 6.1 核心架构

推荐链路分为四步：

1. **问题锚点抽取**
2. **规则候选召回**
3. **可回答性过滤**
4. **大模型重排与改写**

其中：

- 前三步决定“能不能推荐”。
- 第四步决定“推荐得好不好看”。

### 6.2 推荐输入与输出

推荐输入建议包含：

- `user_query`
- `enriched_query`
- 当前识别出的对象/字段/关系锚点
- 元数据关系摘要
- agent 能力摘要
- 最近已推荐问题列表

首版输入收敛：

1. 用户可见推荐的生成，**只依赖当前轮在 `knowledge_enhance` 结束时已稳定可用的上下文**。
2. `execution` 阶段返回的实例/结果上下文不作为首版硬依赖输入。
3. 历史会话只用于去重，不用于补齐本轮候选。

推荐输出建议结构：

```json
[
  {
    "question": "该企业所属行业是什么",
    "recommendation_type": "field_expand",
    "answerability_level": "direct",
    "target_object": "enterprise",
    "target_field": "industry",
    "target_instance": "enterprise_id=xxx",
    "confidence": 0.92,
    "source": "rule+llm",
    "evidence": {
      "trace_rule": "R1_FIELD_EXPAND",
      "capability_id": "enterprise_query",
      "anchor_source": "query",
      "source_object_id": "enterprise",
      "source_field_code": "industry_name",
      "source_relation_id": "",
      "evidence_snapshot_id": "knowledge_snapshot_xxx",
      "relation_path": ["enterprise", "industry"],
      "anchor": "enterprise"
    }
  }
]
```

其中关键字段说明：

- `recommendation_type`：字段扩展 / 元数据关系 / 实例邻近 / 分析延展。
- `answerability_level`：对外只允许 `direct`。
- `evidence`：必填，最小字段为 `trace_rule`、`capability_id`、`anchor_source`、`source_object_id`、`evidence_snapshot_id`，缺失即丢弃。

## 7. 规则召回设计

### 7.1 召回目标

规则层的目标不是直接输出最终推荐，而是从“可回答知识池”里生成一批高质量候选问题。

建议每轮初始召回 10~20 个候选，再做过滤与重排。

### 7.2 候选召回规则

#### 规则 R1：同对象字段扩展

输入：

- 当前识别出的对象 A
- 当前命中的字段 AA

输出：

- A 对象下与 AA 同主题、同分组、同分析链路的字段问题

示例：

- 问“企业注册资本”
- 召回“企业实缴资本是多少”
- 召回“企业成立时间是什么时候”

#### 规则 R2：对象关系扩展

输入：

- 当前识别出的对象 A

输出：

- 与 A 存在 view / relation / foreign key 关系的对象 B 相关问题

示例：

- 问“企业情况”
- 召回“该企业所属网格是什么”
- 召回“该企业所在产业链是什么”

#### 规则 R3：实例邻近扩展（首版关闭）

输入：

- 当前已锁定实例 A1

输出：

- 与 A1 同关系、同层级、同分组的其它实例相关问题

示例：

- 问“某企业”
- 召回“该企业所在网格下还有哪些企业”

#### 规则 R4：分析意图延展（首版关闭）

输入：

- 当前识别出的分析意图，如趋势、对比、排行、异常

输出：

- 同对象或同实例上的下一步分析问题

示例：

- 问“企业营收是多少”
- 召回“企业近三年营收趋势如何”

#### 规则 R5：结果补全扩展（首版关闭）

输入：

- 当前执行结果已返回的字段、对象、实例摘要

输出：

- 与结果内容直接邻接的问题

示例：

- 当前结果已经回答了企业名称、行业、营收
- 可召回“该企业所属产业链是什么”

### 7.3 候选去重

在规则层先做一次去重：

1. 相同对象 + 相同字段 + 相同实例的问题合并。
2. 语义过近的问题只保留置信度更高者。
3. 已在最近 N 轮推荐过且用户未点击的问题降低优先级。

首版补充：

4. 若候选无法落到 `R1` 或“知识快照内已确认的 `R2` 一步关系扩展”，直接丢弃。

## 8. 可回答性过滤设计

### 8.1 过滤目标

过滤器的核心职责是回答一个问题：

- **这条推荐问题，当前 agent 是否真的能回答？**

### 8.2 判定维度

每个候选问题至少从以下维度校验：

1. **对象存在性**：对象是否存在于 ontology / schema 中。
2. **字段存在性**：字段是否真实存在且可查询。
3. **关系可达性**：对象跳转路径是否存在且可执行。
4. **实例锚点完整性**：若问题使用“该企业/这个网格”，是否已锁定实例。
5. **能力支持性**：planner / executor 是否支持该问题对应的分析动作。
6. **上下文充分性**：是否达到 `direct`，凡是仅 `requires_followup_query` 一律过滤。
7. **重复与越界检查**：是否与当前问题重复，是否超出能力边界。
8. **结构相关性硬门槛**：必须命中当前轮锚点对象、字段、关系路径中的至少一项。
9. **焦点一致性硬门槛**：必须与本次提问焦点或本次回答焦点一致，不允许只因“同对象”而泛化推荐。
10. **可溯源硬门槛**：`evidence` 必须包含 `trace_rule`、`capability_id`、`anchor_source`、`source_object_id`，任一缺失则丢弃。
11. **证据快照硬门槛**：必须能指向本轮知识快照或回答快照，缺少 `evidence_snapshot_id` 则丢弃。

### 8.3 过滤结果

过滤后每个候选问题输出：

- 保留 / 丢弃
- 丢弃原因
- 可回答级别
- 相关性与溯源证据

建议记录结构：

```json
{
  "question_draft": "该企业在所属行业中处于什么水平",
  "status": "drop",
  "answerability_level": "requires_followup_query",
  "required_capability": "aggregation_compare",
  "focus_alignment": false,
  "reason": "not direct and focus misaligned"
}
```

### 8.4 可溯源证据最小结构

每条对外推荐必须绑定最小证据结构：

```json
{
  "trace_rule": "R1_FIELD_EXPAND",
  "capability_id": "enterprise_field_read",
  "anchor_source": "query",
  "source_object_id": "enterprise",
  "source_field_code": "registered_capital",
  "source_relation_id": "",
  "evidence_snapshot_id": "knowledge_snapshot_xxx"
}
```

说明：

1. `anchor_source` 可取 `query` 或 `answer`，但首版用户可见推荐仅允许 `query`；`answer` 留作二期扩展。
2. `source_field_code` 用于字段扩展型追溯。
3. `source_relation_id` 用于关系扩展型追溯。
4. `evidence_snapshot_id` 必须指向本轮生成时刻的快照。

## 9. 大模型重排与改写设计

### 9.1 模型职责边界

大模型在本方案中只负责：

1. 对候选问题排序。
2. 删除仍显重复或表达拗口的问题。
3. 将候选改写成自然、简洁、可追问的问句。

大模型不负责：

1. 凭空发明新对象、新字段、新关系。
2. 推断不存在的实例锚点。
3. 绕过规则过滤输出能力边界外的问题。
4. 新增规则层未召回的候选。

### 9.2 模型输入控制

为了节约性能和时延，模型输入只保留最小必要上下文：

1. 当前用户问题摘要。
2. 当前问题锚点摘要。
3. Top-K 候选问题。
4. 每个候选的类型与可回答级别。
5. 最近已推荐问题摘要。

不要输入：

1. 全量 `knowledge_payload`
2. 全量执行结果
3. 大段历史对话原文

### 9.3 模型输出约束

模型输出建议约束为：

1. 最多返回 3 个推荐问题。
2. 仅允许输出 `direct` 候选，禁止补位输出 `requires_followup_query`。
3. 问句长度尽量控制在 12~30 字。
4. 禁止输出“你还可以问”“是否想了解”等解释性前缀，只输出问题文本和结构化属性。
5. 每条输出必须携带 `evidence` 的最小字段（`trace_rule`、`capability_id`、`anchor_source`、`source_object_id`、`evidence_snapshot_id`）。
6. 若候选经改写后失去原始证据映射关系，必须丢弃，不得强行输出。

## 10. 异步执行与性能优化

### 10.1 启动时机

推荐任务建议在 `knowledge_enhance` 完成后立即启动，而不是等主回答结束后再开始。

原因：

1. 此时已完成用户问题理解和知识锚点识别。
2. 推荐需要的关键输入已经具备。
3. 可与 planning / execution 并行，降低用户感知时延。

### 10.2 异步链路建议

建议采用如下流程：

1. `knowledge_enhance` 结束后，基于当时已稳定可用的知识快照组装 `recommendation_context`
2. `asyncio.create_task(recommend_questions(...))`
3. planning / execution 继续主链路执行
4. 回答收尾时仅做一次非阻塞读取（`task.done()` 判定）
5. 未完成则直接返回空推荐，不做任何等待
6. 推荐任务结果只用于日志、离线评估或后续轮次候选预热，不直接回填本轮

### 10.3 性能控制策略

1. **早启动**：知识理解完成立即启动推荐。
2. **小上下文**：只给模型喂候选和摘要，不喂全量原始知识。
3. **两段式生成**：先规则召回，再模型重排，不从零生成。
4. **零等待收尾**：主回答收尾阶段禁止等待推荐任务。
5. **缓存仅用于候选预热**：缓存不能直接作为本轮用户可见推荐结果。

推荐缓存 key 建议：

- `session_id + object_anchor + field_anchor + instance_anchor + query_signature`

缓存约束：

1. 缓存命中只能缩短候选召回，不可跳过本轮 `relevance + traceability + direct` 校验。
2. 若无法证明与本次提问焦点一致，则缓存结果一律丢弃。
3. 首版默认允许关闭缓存，保证逻辑更清晰。

### 10.5 运行预算与限流（新增）

为避免推荐异步任务挤占主链路资源，首版增加明确预算：

1. 单请求候选上限：`max_candidates_before_filter = 12`。
2. 单请求大模型重排输入上限：`top_k_to_llm <= 6`。
3. 单请求推荐异步任务内部超时：`recommend_task_timeout_ms = 1200`（仅影响后台任务，不影响主回答收尾）。
4. 单 worker 推荐并发上限：`recommendation_concurrency <= 2`。
5. 单请求推荐模型 token 预算：`input_tokens + output_tokens <= 1200`。
6. 触发限流时优先降级为“规则过滤结果”，再降级为空列表。

### 10.4 降级策略

推荐不能影响主回答，因此必须有降级机制：

1. 大模型超时：直接返回规则过滤后的合格结果；若无合格结果则空列表。
2. 大模型失败：直接返回规则过滤后的合格结果；若无合格结果则空列表。
3. 若规则结果不满足 `direct + focus_alignment + traceability` 三重门槛：返回空列表，不强凑数量。
4. 无可回答候选：返回空列表，不返回虚假推荐。

## 11. 与现有链路的集成建议

### 11.1 建议接入位置

建议引入一个独立的“推荐问题服务”或“推荐问题模块”，避免把推荐逻辑直接堆在 `worker.py` 中。

推荐上下游位置：

1. 输入主要来自 `knowledge_enhance`
2. 可补充 `planning` 的能力判断信息
3. 首版不接入 `execution` 新结果作为用户可见推荐输入（仅可用于离线评估）
4. 最终由 `worker.py` 在回答完成 SSE 中带出（不等待推荐任务）

### 11.2 对 `worker.py` 的落点

当前写死逻辑位于回答完成处：

- `metadata={"relatedResources": ["推荐问题11"]}`

后续建议替换为：

1. 回答收尾阶段仅做非阻塞读取。
2. 若无可用结果则返回空数组，不等待、不补假数据。
3. 异步任务完成结果用于日志、评估与后续轮次候选预热，不反向修改本轮已发送回答。

### 11.3 状态建议

建议在状态中引入：

- `recommendation_context`
- `recommendation_candidates`
- `recommendation_result`
- `recommendation_status`
- `recommendation_debug`
- `recommendation_snapshot_id`

其中：

- `recommendation_context`：推荐所需的最小输入。
- `recommendation_candidates`：规则召回后的候选。
- `recommendation_result`：最终输出给前端的问题列表。
- `recommendation_status`：`pending/rule_ready/llm_ready/fallback/empty`。
- `recommendation_debug`：过滤与降级原因，仅用于日志和调试。
- `recommendation_snapshot_id`：本轮推荐所绑定的知识快照/回答快照 ID。

## 13. 验收标准

### 13.1 功能验收

1. 推荐问题不再写死，而是根据当前用户问题动态生成。
2. 首版用户可见推荐仅覆盖同对象字段扩展型（R1）与受限一步关系扩展型（R2）；实例邻近型与分析延展型在二期上线。
3. 每个推荐问题都能映射到明确的对象/字段/关系/实例依据。
4. 用户可见推荐仅包含 `answerability_level=direct`。
5. 对于当前 agent 无法回答的问题，不能出现在推荐列表中。
6. 任一推荐缺失必填证据字段时必须被过滤。
7. 首版允许返回空列表，空列表视为合格输出。

### 13.2 性能验收

1. 推荐生成不能增加主回答等待时间（主链路零等待，收尾等待耗时 = 0ms）。
2. 推荐任务能在知识理解后异步启动。
3. 大模型超时或失败时，主回答仍正常返回。
4. 回答收尾阶段不允许出现“等待推荐任务”逻辑。
5. 缓存命中与否都不能改变主回答等待时间。
6. 推荐异步任务必须满足运行预算：候选数、并发数、token 预算、后台超时均有日志可观测。

### 13.3 质量验收

1. 推荐问题与当前问题语义相关，不明显跑题。
2. 推荐问题之间无明显重复。
3. 推荐问题表述自然、简洁，便于用户直接点击追问。
4. 推荐问题不伪造对象、字段、实例或关系。
5. 推荐问题必须可由当前 agent 直接回答，不依赖额外检索。
6. 每条推荐都能追溯到本轮知识快照中的对象/字段/关系证据（包含 `evidence_snapshot_id`）。

### 13.4 可观测性验收

1. 日志中可看到推荐候选、过滤原因、模型输出、降级原因。
2. 日志中可看到 `focus_alignment` 与 `traceability` 校验结果。
3. 推荐失败原因可追踪，不是静默失败。
4. 日志中可看到 `evidence_snapshot_id` 与 `trace_rule`。

## 14. 风险与边界

### 14.1 主要风险

1. 元数据关系不完整时，规则召回覆盖面不足。
2. 实例锚点识别不稳定时，实例邻近型推荐容易缺失。
3. 分析延展型推荐若能力映射不清，容易误推不可回答问题。
4. 若把太多上下文喂给模型，会抬高时延和成本。

### 14.2 边界约束

1. 本期不追求“开放式智能推荐”，而追求“能力边界内的高可信推荐”。
2. 本期优先保证同对象字段扩展和受限关系扩展，实例型与分析型推荐分阶段放量。
3. 若会话上下文不足，允许推荐数量减少或为空，不强行补满。
4. 本期不允许使用历史缓存直接回填本轮用户可见推荐。

## 15. 自我评审

### 15.1 合理性评审

本方案合理性较高，原因如下：

1. 直接围绕“推荐的问题必须本 agent 能回答出来”这一核心约束展开。
2. 将“生成”拆为“规则召回 + 可回答性过滤 + 大模型润色”，符合当前场景的控制需求。
3. 首版主动收紧范围，只开放最容易证明“可答、相关、可追溯、零等待”的两类推荐。

需要注意的点：

1. 若底层 ontology / relation 配置不完整，推荐质量上限会受影响。
2. 由于首版主动关闭实例型和分析型推荐，初期命中率可能偏低，但这是为了换取严格正确性。

### 15.2 性能评审

本方案性能风险可控，主要因为：

1. 大模型不从零生成，只对小候选集重排和改写。
2. 推荐在 `knowledge_enhance` 后异步启动，与主回答并行。
3. 主回答收尾零等待，推荐结果未就绪时直接返回空列表，不使用缓存兜底本轮展示。

性能风险点：

1. 若候选召回依赖太多外部查询，仍可能拖慢异步任务，因此首版必须限制为本地规则和本轮快照。
2. 由于首版允许空列表，性能闭环比覆盖率更优先。

### 15.3 开发任务评审

本方案开发拆分较清晰，适合逐步落地：

1. 先做规则召回 + 过滤 + 动态输出，即可替换写死值。
2. 再接入大模型重排，提升语言质量，但不能扩大候选边界。
3. 最后补日志、快照绑定、缓存预热和更完整测试。

建议实施顺序：

1. 先落 `REC-01 ~ REC-04`
2. 再做 `REC-10 / REC-11 / REC-07`
3. 再做 `REC-05 / REC-06 / REC-09 / REC-08`

### 15.4 验收标准评审

当前验收标准较完整，已覆盖：

1. 功能正确性
2. 性能影响
3. 推荐质量
4. 可观测性

仍可补充的验收项：

1. 不同类型问题的样例集回归测试。
2. 空列表场景的回归测试。
3. 推荐点击后的二轮追问闭环验证。

### 15.5 总结结论

本方案经过收紧后，更适合作为满足“可答、相关、可溯源、零等待”四条硬约束的首版正式设计。

推荐结论：

- **按“规则召回 + 大模型重排生成”推进，但首版严格收敛候选边界。**
- **首版允许空列表，不用缓存或宽松规则冒险补位。**
