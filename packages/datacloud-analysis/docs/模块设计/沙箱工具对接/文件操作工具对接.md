## 一、现状梳理

### 1.1 Gateway 提供的文件操作能力

Gateway（`by-framework-python`）通过注入 `AgentContext` 提供统一的文件操作能力，访问路径如下：

```
gateway_context
  └── .agent_runtime_state
        └── .session_manager
              └── .file_manager          ← FileManager（agent_private scope）
                    ├── read_file(filename, encoding, offset, limit)
                    ├── write_file(filename, content, encoding)
                    ├── delete_file(filename)
                    ├── list_files(directory)
                    ├── glob_files(pattern)
                    ├── grep_files(pattern, glob_pattern, output_mode)
                    └── edit_file(filename, old_string, new_string, replace_all)
```

`FileManager` 已内置：
- **路径隔离**：通过 `session_id` + `tenant_id` + `agent_id` 做多租户隔离，所有路径由 `RuntimePathMapper` 翻译，外部无法越界
- **权限控制**：`FilePermissionPolicy` 拦截非法操作
- **大结果分页**：超出 `tool_result_max_chars` 的结果自动落盘，返回文件引用

`gateway_context` 在 Worker 处理每条消息时由框架自动构建并注入，`datacloud-analysis` 的工具层通过 `_context` 参数接收它（`tool_wrapper.py` 中 `_invoke_tool_with_runtime_context` 机制自动完成注入，检测函数签名中的 `_context` 或 `gateway_context` 参数名）。

### 1.2 datacloud-analysis 当前文件工具现状

#### 实际挂载的工具（`orchestration/execution/node.py`）

```python
_BUILTIN_TOOLS = [ask_user, read_file, write_file, write_code, execute_code]
```

| 工具名 | 来源 | 当前实现方式 | 问题 |
|---|---|---|---|
| `read_file` | `tools/file_io.py` | 读 `os.environ["DATACLOUD_ACTIVE_WORKSPACE"]` 定位路径，直接读本地文件 | 自己管路径，无法受益于 Gateway 的隔离和权限控制 |
| `write_file` | `tools/file_io.py` | 同上，直接写本地文件 | 同上 |
| `write_code` | `tools/code_exec.py` | 同上，写 `.py` 文件到本地 | 同上 |
| `execute_code` | `tools/code_exec.py` | 读本地 `.py` 文件后 `exec()` 执行 | 文件读取部分未走 Gateway |

#### 未挂载的废弃工具

| 工具名 | 来源 | 状态 |
|---|---|---|
| `sbx_read_file` | `tools/sandbox.py` | 未挂载，历史遗留，应废弃 |
| `sbx_write_file` | `tools/sandbox.py` | 未挂载，历史遗留，应废弃 |

#### 额外问题

`execution_node` 在每次 ReAct 循环前，将 `state["workspace_dir"]` 写入进程级环境变量（`node.py` 第 162 行）：

```python
os.environ["DATACLOUD_ACTIVE_WORKSPACE"] = str(workspace_root)  # 进程级，并发不安全
```

改用 Gateway `FileManager` 后，此行不再需要，应一并删除。

## 二、设计方案

### 2.1 核心思路

**将 `read_file`、`write_file`、`write_code` 的文件 I/O 实现，从"自管本地路径"改为"委托 `gateway_context.agent_runtime_state.session_manager.file_manager`"**。

`execute_code` 的代码执行逻辑保持不变（`exec()` 是本地运算，不是文件操作），仅将其中读取源文件的部分改为走 `file_manager.read_file`。

`_context` 为 `None` 时（本地开发、单测场景）降级使用原有本地路径逻辑，保持向后兼容。

### 2.2 数据流（改造后）

```
Gateway 框架（by-framework-python，不可修改）
  └── _handle_message()
        ├── 构建 AgentContext（含 FileManager，session/tenant 隔离已就绪）
        └── 注入 gateway_context 到 LangGraph configurable

execution_node（orchestration/execution/node.py）
  └── 删除 os.environ["DATACLOUD_ACTIVE_WORKSPACE"] = ... 这一行

tool_wrapper._invoke_tool_with_runtime_context()
  └── 检测工具签名中的 _context 参数
        └── 自动将 gateway_context 注入为 _context 实参

read_file / write_file / write_code（tools/file_io.py、tools/code_exec.py）
  └── _context.agent_runtime_state.session_manager.file_manager
        └── read_file() / write_file()    ← 路径隔离、权限控制由 FileManager 保证

execute_code（tools/code_exec.py）
  └── file_manager.read_file(filename)    ← 读源文件走 Gateway
        └── exec(code, namespace)         ← 本地执行，逻辑不变
```

### 2.3 具体改动

#### 改动 1：`tools/file_io.py`

增加 `_context` 参数，优先走 `file_manager`，无 `_context` 时降级本地实现：

async def write_file(
    path: str,
    content: str,
    encoding: str = "utf-8",
    _context: Any = None,
) -> dict:
    """将内容写入 workspace 内指定文件（自动创建父目录）。

## 三、验收用例

### 用例 1：read_file 委托 gateway FileManager 读取文件

**前置条件**：
- `file_manager.read_file("output/result.txt")` mock 返回 `{"success": True, "data": {"content": "hello"}}`

**执行**：
```python
mock_context = Mock()
mock_context.agent_runtime_state.session_manager.file_manager.read_file = AsyncMock(
    return_value={"success": True, "data": {"content": "hello"}}
)
result = await read_file.ainvoke({"path": "output/result.txt", "_context": mock_context})
```

**期望**：
- `result == "hello"`
- `file_manager.read_file` 被调用一次，入参 `filename="output/result.txt"`
- 未访问 `os.environ["DATACLOUD_ACTIVE_WORKSPACE"]`
