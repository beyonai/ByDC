# 数据文件落盘与综合读取方案设计

## 1. 方案概述
为了解决大批量数据查询结果在 Agent 对话中难以承载、难以流转和分页读取的问题，本方案制定了一套从工具落盘、Agent 上下文注入到前端分页拉取端到端的架构设计。

核心处理流程：
1. **工具落盘 (`data.py`)**：负责拦截服务底层大体量数据，将其保存为对分页友好的 `.jsonl` 格式记录，仅将精简后的元数据与摘要信息抛入任务上下文中。
2. **预加载插值 (`insight.py`)**：在调度最终的大模型进行推理时，代码层代替大模型自动巡检所有子任务的数据摘要，将其转换为 **Markdown 表格** 呈现形式后“预先投喂”给 LLM。
3. **独立检索 (`workspace_api.py`)**：提供固定的流式高并发接口，让前端应用可以直接对工作空间内落盘的文件进行毫秒级分页检索和文件流级下载。

通过预加载机制，常规情况下无需大模型配备读写文件的 Tool (工具)；而在面对极其复杂的**万级文件互查与运算场景**时，将完美联动系统架构中的 `sbx_run_code` （沙箱代码算子），以此实现从“查询呈现”到“深度挖掘”的架构闭环。

## 2. 工具层改造：`datacloud_analysis/tools/data.py`

### 2.1 JSONL 混合结构落盘策略
当 `data_query` 执行完毕，检查响应字段中 `code == 0` 且 `resultType == 'normal'` 且具有有效 `records` 数据时触发持久化。

**文件存储结构 (`data_query_{UUID}.jsonl`)**
利用“首行元数据 + 其余行具体数据”的文件结构解决表头不丢失问题，且天然支持分段式（Pagination）随机高速读取：
* **首行（Meta）：** 记录所有字段列结构和分页、统计信息。
    ```json
    {"_type": "meta", "columns": [...], "total": 241, "download_url": "http://..."}
    ```
* **第二行至第N行（Records）：**
    ```json
    {"enterpriseId": "911...", "enterpriseName": "李宁...", "riskLevel": "HIGH"}
    {"enterpriseId": "911...", "enterpriseName": "北京...", "riskLevel": "HIGH"}
    ```

### 2.2 给到 LLM 的返回参数截断
工具处理完毕后，禁止将动辄上万字符的极长完整 `records` 直接 `return`。向 Graph 返回的是一份“压缩说明包”：
```python
return {
    "status": "success",
    "file_path": str(saved_file_path),      # 落盘后的绝对或相对路径
    "total": total,                         # 总计条数，如 241
    "columns": columns,                     # 表头信息
    "preview": records[:5],                 # 固定切片取首部 5 条数据做上下文试看
    "original_download_url": download_url,  # 保留供下载的原地址
}
```

## 3. 编排层改造：`datacloud_analysis/orchestration/insight.py`

### 3.1 基于多个结果的智能还原（Markdown 表格转化）
由于允许通过“多个数据查询”的任务合并（如查询风险企业和正常企业两份清单），`insight` 节点会接收到一个列表 `results` 包含若干次不同类型请求的运行结果。`insight.py` 内部使用 for 循环结构，对所有含有 `file_path` 的结果并行分析。

**在将数据塞给 LLM Reasoning 之前，预制为 Markdown 文本：**
依靠代码逻辑将每个结果对象中含有的 `preview`（如果大于5条的数据）或者全部（少于5条的小数据量）与 `columns` 表头强行绑定，生成 Markdown 表格：

```markdown
【任务 1：高风险企业查询结果】
| 企业ID | 企业名称 | 行业名称 | 风险等级 |
|---|---|---|---|
| 91110000600000788P | 李宁（中国）体育用品有限公司 | 计算机... | HIGH |
...
*【重要】数据量较大（共 241 条），此处仅展示前 5 条。详细完整数据请查询对应 API*

【任务 2：正常企业查询结果】
| 企业ID | 企业名称 | 行业名称 | 风险等级 |
|---|---|---|---|
| 9111010200000000XX | 某正常企业 | 计算机... | LOW |
*共 1 条，已全量展示*
```

### 3.2 提示词增强（Sys Prompt）
为了强迫大模型照本宣科，在组装提示词发送给 LLM 前增加特定的规约约束：
> “你是一个高级数据分析师。以下为您提供的数据明细已经为您转换为标准的 Markdown 表格格式。在您的最终回复中进行信息汇总或对比时，**请务必原样保留或直接使用我们提供的 Markdown 表格格式进行列表式的数据展示**。”

## 4. 独立文件分页/下载 API：`datacloud_service/workspace_api.py`

在 Backend 中单独新建 `api` 文件建立固定路由，专供前台用户按页获取大容量文件以规避网络阻塞。

### 接口规范
`GET /api/v1/workspace/files`

**入参请求参数 (Query)**
* `session_id` (`str`): 用于锁定挂载沙箱（Workspace）。
* `relative_path` (`str`): 工作空间的相对路径（由 `data_query` 的 `file_path` 提供）。
* `start_line` (`int`, 选填): 用于 JSONL 内部控制分页（如 0）。
* `end_line` (`int`, 选填): 用于 JSONL 内部控制分页（如 50）。
* `download` (`bool`, 选填): 当标识为 True 时返回应用/附件流直接下载，忽略行号。

**功能实现重点：游标机制切片**
通过内置模块 `itertools.islice(file_obj, start_line, end_line)` 进行迭代器的无害偏移处理，只拿出第一行的 Meta 参数并立刻提取指定游标距离行的内容即可组成类似 `{ "meta": {...}, "records": [...] }` 的响应格式返回前端。即使应对动辄几十兆以上的文件系统仍然不需要耗用系统内存吞肚式加读。

---

## 5. 常规问题：是否需要提供文件读写的 Tool 挂给 Agent？
**常规情况（非强计算类）不需要额外提供。**
当前方案采用的是 **“预读取插值注入上下文”**（Pre-injection Context）的形式工作。这意味着 Agent 在组织最终回答时，所有前置任务的文件结果已经被系统代码自动整合成了漂亮的 Markdown 表格。大模型**无须**挂载额外的 File Search 或 Read Tool，省去了多余调用思考并杜绝了因上下文溢出带来的幻觉崩溃。

---

## 6. 进阶突围：依托沙箱工具的大规模数据关联运算
虽然通常大模型不需要手工翻表，但在某些特定的企业级极致场景中（**例如：1 万条全量园区企业名单 减去 8000 条在招工企业，剩余 2000 条未招工企业，再以这 2000 个名单为源进行网格追溯**），让大模型看 5 条预览显然无济于事，这时就必须激活《系统功能架构图》底座中的核心工具：**沙箱代码执行算子 (`sbx_run_code`)**。

### 业务闭环链路 (ReAct Loop)
1. **任务 1 (落盘 A)**：Agent 调用 `data_query("园区全量清单")`，引擎返回落盘全量路径 `file_A.jsonl`，返回上下文包含总量 1 万与文件相对路径。
2. **任务 2 (落盘 B)**：Agent 调用 `data_query("在招工企业清单")`，引擎返回落盘路径 `file_B.jsonl`，总量 8000 与文件路径。
3. **任务 3 (沙箱对撞)**：大模型推理由目标（求剩余2000个），激活工具库的 `T_SBX_RUN` (执行运算)，下达 Python 代码或自然语言指令让 Pandas 分析这俩大文件获取差集的企业 ID。该算子在极安全的本地隔离沙箱运作，将差集2000条再次落盘 `file_C.jsonl`，并返回给大模型新路径及 ID 数组。
4. **任务 4 (级联查询)**：大模型获得 2000 条过滤干净的企业 ID 后，将它们压入 `data_query` 的 `object_ids` 参数再查一次终态网格数据。

该设计无缝对齐了我们在《3.1 功能架构》中勾勒的蓝图工具 `T_SBX_RUN` / `T_SBX_READ`，在保留 Agent 判断自由度的同时彻底化解了超大规模数据本地运算（交并差）的死缓问题。
