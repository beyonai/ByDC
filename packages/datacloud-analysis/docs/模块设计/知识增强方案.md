# 知识增强方案

## 1. 背景

当前 5 节点主链路依赖 knowledge_enhance 节点产出结构化上下文，再驱动 planning → execution → end。近期在排查“复合问题只返回部分答案”与“歧义术语阻断”时，发现文档描述与真实代码不符，难以指导开发。现对真实实现重新梳理，并在此基础上提出可落地的改进方案。

## 2. 现状梳理（基于 2026-04-01 代码）

### 2.1 knowledge_enhance/node.py

- 调用 search_knowledge.ainvoke({"query": user_query})，原样持久化返回 JSON 为 knowledge_payload，并截取 500 字写入 knowledge_preview。
- _extract_term_hints 仅遍历 payload.term_matches，按 	erm_hint_confidence_threshold（默认 0.8）过滤，构造 	erm_hints，字段仅含 mention/normalized_term/term_id/confidence 等。
- _build_knowledge_snippets 只是把 	erm_matches、uzzy_term_matches 打包成简单列表，不做截断或去重。
- _rewrite_enriched_query 会遍历 term_hints，若 mention 被命中且 confidence >= 0.8，直接在 user_query 做字符串替换；没有“原始问题 + 知识摘要”的模板，也不会引用 knowledge_payload 中的正文。
- 输出状态中仍包含 knowledge_preview 字段，且 enriched_query_source 只会是 knowledge_rewrite 或 allback_user_query。
- 歧义术语（payload 中的 mbiguous_terms 或 fuzzy 匹配）不会被特殊处理，日志中也看不到详细提示。

### 2.2 planning/node.py

- 规划输入取值顺序为 state.intent > state.enriched_query > state.user_query。
- 状态或日志中没有 planning_input_source 字段，排查时无法知道最终实际使用了哪个文本。
- 	erm_context 来源为 confirmed_terms + term_hints，但 knowledge_enhance 并不会写 confirmed_terms，因此大多数场景都仅依赖 term_hints。

### 2.3 execution & end

- eact_runtime 与 sandbox_executor 已透传 enriched_query、	erm_hints、knowledge_snippets，对 Markdown/多段文本没有额外假设。
- knowledge_payload 在后续节点完全未被读取，仅用于日志或调试。
- end 节点只基于 esults、enriched_query（原样展示）生成总结。

## 3. 现存问题（以现状为依据）

1. **增强信息不充实**：enriched_query 最终只是术语替换版问题，未真正附带知识摘要，后续节点看不到补充信息。
2. **歧义无可观测性**：mbiguous_terms 没有写入状态/日志，用户和开发者都难以判断为什么某些术语缺失。
3. **planning 覆盖增强结果**：intent 仍优先级更高，经常覆盖 enriched_query，导致前面做的替换也无意义。
4. **冗余字段**：knowledge_preview 只是调试用截断文本，却混在状态契约里，且与 knowledge_payload 重复。
5. **snippets/摘要未区分**：目前 snippets 就是原始数组，和 enriched_query 不存在结构差异，也没有长度、去重控制，prompt 成本不可控。

## 4. 设计目标（基于现状可落地的改动）

1. 继续依赖 search_knowledge 输出，但在节点内部生成“原始问题 + 已确权知识摘要”的 Markdown 文本，替换 _rewrite_enriched_query 的纯替换逻辑。
2. 保留 knowledge_payload 作为完整快照，同时删除 knowledge_preview，把调试需求转移到日志。
3. 记录歧义项并输出到日志/SSE，但不阻断主链路；若无确权知识，则 enriched_query 直接回退到 user_query 并附上提示。
4. 将 planning 输入优先级调整为 enriched_query > intent > user_query，并把结果写入 planning_input_source 便于追踪。
5. knowledge_snippets 专注于工具上下文（结构化/字段/SQL 片段），与摘要分流；必要时做截断和重复度限制。

> 以上目标全部以“当前缺失→补齐”的形式描述，不假设已有实现。

## 5. 方案概要

### 5.1 摘要生成

- 新增 _build_confirmed_knowledge_summaries(payload)：
  - 仅遍历 payload.term_matches 中 match_score >= threshold 的项；
  - 截断 description 至 80~150 字；
  - 输出结构 [{"term_name": ..., "summary": ...}]，供 enriched_query 与 snippets 共享。
- 新增 _compose_enriched_query(user_query, summaries)：
  - 无摘要时直接返回原始问题；
  - 有摘要时使用标准模板：
    `	ext
    原始问题：
    {user_query}
    
    补充知识：
    - {term}：{summary}
    `
  - enriched_query_source 取值 confirmed_terms 或 allback_user_query；置信度取 summaries 中最大匹配分。

### 5.2 歧义记录

- 解析 payload.fuzzy_term_matches / candidate 列表，写入 mbiguities = [{"mention": ..., "candidates": [...]}]。
- 通过 logger.info 与 SSE “思考过程”输出，无需中断。
- 若 term_hints 为空，附加“无确权术语”提示到日志。

### 5.3 planning 输入

- 在 planning.node.planning_node 中：
  `python
  query_input = state.get("enriched_query") or state.get("intent") or state.get("user_query") or ""
  state["planning_input_source"] = "enriched_query" / "intent" / "user_query"
  logger.info("planning_node: using %s as planning input", planning_input_source)
  `
- 兼容旧状态；若 enriched_query 为空仍可回退。

### 5.4 snippets 去重

- _build_knowledge_snippets：保留 term_matches/fuzzy_term_matches 原始结构，但新增：
  - 每块数据限制 300 字；
  - 与摘要文本重复度（简单相似度）>0.8 时剔除；
  - 输出字段含 source、data、char_len，供后续监控。

## 7. 验收标准

1. knowledge_enhance 日志能看到“原始问题/歧义项/知识摘要/增强后问题”。
2. 状态中 enriched_query 为 Markdown 模板；enriched_query_source 标记为 confirmed_terms 或 allback_user_query。
3. planning_input_source 正确反映实际使用文本，并在日志中可见。
4. knowledge_payload 继续存完整 JSON，knowledge_preview 移除；knowledge_snippets 附带长度信息且与摘要去重。
5. uv run pytest packages/datacloud-analysis/tests/dca/unit/test_knowledge_enhance.py test_planning_node.py 通过。
6. 文档与实现保持一致，后续修改需同步更新此文档。
