# 任务规划及返回

## 1. 背景与问题
当前多目标问题在执行后只返回了最后一个任务的数据（如仅返回企业 Top10），前置任务结果（如网格 Top10）在最终输出中丢失。

典型输入：
- 帮我找出效益最低的 10 个网格，并找出这 10 个网格中排名最低的 10 个企业。

期望输出：
- 同时返回“网格结果”和“企业结果”，并支持综合说明两者关联关系。

## 2. 设计目标
1. 支持单次提问拆分为多个任务（Multi-task）。
2. 支持任务间依赖（DAG），后置任务可引用前置任务结果。
3. 每个任务均产出独立文件与结构化结果，禁止覆盖。
4. 最终返回支持“分别返回”与“综合返回”。
5. 任一子任务失败时支持部分可用返回，不因单点失败整体丢失。

## 3. 规划层（Planning）方案

### 3.1 任务拆分
将复合问题拆成 DAG：
- `t1`: 查询效益最低 10 个网格。
- `t2`: 查询 `t1` 网格集合内排名最低 10 个企业。

### 3.2 依赖表达
在任务定义中显式标注：
- `depends_on`: 依赖任务列表
- `inputs_from`: 从依赖任务提取输入字段

示例：
```json
{
  "tasks": [
    {
      "todo_id": "t1",
      "goal": "查询效益最低10个网格",
      "required_tools": ["grid_query"],
      "depends_on": []
    },
    {
      "todo_id": "t2",
      "goal": "查询这10个网格中排名最低10个企业",
      "required_tools": ["enterprise_query"],
      "depends_on": ["t1"],
      "inputs_from": {
        "grid_ids": "t1.result_meta.grid_ids"
      }
    }
  ]
}
```

## 4. 执行层（Execution）方案

### 4.1 任务执行顺序
- 按 DAG 拓扑顺序执行。
- `t2` 执行前解析 `t1` 的 `grid_ids` 作为过滤参数。

### 4.2 每任务产物契约
每个任务必须生成两类产物：
1. 文件产物（便于空间内追溯/下载）
2. 结构化结果（便于后续聚合与二次计算）

示例：
- `t1`:
  - `artifacts/t1_bottom10_grids.csv`
  - `result_meta`: `{"grid_ids": [...], "count": 10}`
- `t2`:
  - `artifacts/t2_bottom10_enterprises_in_grids.csv`
  - `result_meta`: `{"enterprise_ids": [...], "count": 10, "from_task": "t1"}`

### 4.3 状态写入原则（防覆盖）
统一使用追加策略：
- `state.results[]`：按任务 append
- `state.artifact_refs[]`：按任务 append

禁止：
- 用单一 key（如 `state.result`）反复覆盖。

## 5. 返回层（End/Insight）方案

### 5.1 返回模式
支持三种模式：
1. 分别返回：先网格结果，再企业结果。
2. 综合返回：网格 + 企业 + 关联说明。
3. 文件返回：列出两个文件路径/引用，供空间内查看或下载。

### 5.2 聚合规则
- 从 `state.results[]` 按 `todo_id` 重建任务结果集合。
- 若任务存在依赖，输出中明确“企业结果来自哪批网格”。

### 5.3 部分失败策略
- 若 `t1` 成功、`t2` 失败：返回 `t1` 结果 + `t2` 错误原因。
- 若 `t1` 失败：`t2` 标记为 `blocked_by_dependency`。

## 6. 日志与可观测性
建议最小日志：
- 规划：`planned_tasks=[t1,t2]`、`dag_edges=[t1->t2]`
- 执行：`task_start/task_end`、依赖解析结果（如 `resolved_grid_ids=10`）
- 返回：`returned_tasks=[t1,t2]`、`artifact_count=2`

## 7. 验收标准
1. 单次请求可看到两个任务均执行，并有明确先后关系（`t1 -> t2`）。
2. 工作空间内存在两个独立文件（网格文件、企业文件）。
3. 最终回复包含两个任务结果，不再只显示最后一个任务。
4. 综合返回中包含“企业结果来源于 t1 网格集合”的说明。
5. 子任务失败时可返回部分结果并附失败原因。

## 8. 后续实现建议
1. 优先改规划契约：补 `depends_on/inputs_from`。
2. 再改执行器：支持依赖输入注入与 append 写入。
3. 最后改结束节点：按 `results[]` 聚合输出与文件引用。
