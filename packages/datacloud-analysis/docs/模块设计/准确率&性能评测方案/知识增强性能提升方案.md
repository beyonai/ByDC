## 1. 背景与问题定义

当前系统存在两个性能/体验缺陷，本文档给出完整的分析与改造方案。

---

### 1.1 问题 A：`analyze_query_clarification` 无条件全量执行

**现象**：用户无论提什么问题，每次请求都会触发一次完整的知识增强流程（`analyze_query_clarification`），包括：
- LLM#1：expand_query（查询扩展）
- 召回：BM25 / 向量 / RRF 全库 term 召回
- LLM#2：llm_confirm（置信度过滤）

即使用户问的是一个参数完全无歧义的简单查询（如"查过去 7 天的营收总额"），也会走完整的两次 LLM 调用，造成 **TTFB（首字节延迟）大幅增加**，通常在 2~4 秒左右。

**根因定位**：

```
worker.py  _build_graph()
  └─ knowledge_enhancer = _async_knowledge_enhancer   ← 无条件注入
       └─ intend_node() 每次必然调用 knowledge_enhancer
            └─ analyze_query_clarification(query, gateway_context, metadata)
                 └─ LLM#1 expand_query + 召回 + LLM#2 llm_confirm
```

`intend_node`（`datacloud-analysis/orchestration/intend/node.py`）的逻辑如下：

```python
# intend_node.py 第 137~159 行（简化）
if knowledge_enhancer is not None:
    try:
        result = await _invoke_knowledge_enhancer(
            knowledge_enhancer, user_query=user_query, ...
        )
        knowledge_payload = {
            "needs_clarification": result.needs_clarification,
            "form": result.form,
            "knowledge": result.knowledge,
            ...
        }
        updates["knowledge_payload"] = knowledge_payload
    except Exception:
        logger.exception("[intend_node] knowledge_enhancer failed, skipping")
```

**只要 `knowledge_enhancer` 不为 `None`，就无条件执行**，没有任何预判断逻辑。

### 1.2 问题 B：ReAct 推送的"思考过程"是假的

**现象**：前端思考过程面板展示的内容实际上是：
- 工具调用的 `reason` 字段（LLM 自动填写的"我调用这个工具是因为..."）
- 工具入参 JSON
- 工具出参摘要

这些都是**工具执行层面的过程信息**，而非模型在真正思考时生成的自然语言推理过程（"thinking"）。

**根因定位**：

```python
# react_loop.py _stream_llm_call 第 78~99 行
if gateway_context is not None:
    # 注意：不在此处推送 LLM content 文字 token。
    # 部分 LLM 在生成 tool_calls 的同时也输出 content 文字
    # （如"我来查询..."），若在此处推送则会出现不必要的客套话。
    
    # 目前只推送 finish_react.answer 参数的内容
    for tcc in (getattr(chunk, "tool_call_chunks", None) or []):
        if tcc.get("name") == "finish_react":
            ...
```

当 LLM 在决策阶段（调用工具之前）产生的 content 文字（即真实推理过程）**被代码故意丢弃了**。

`tool_wrapper.py` 中当前所谓的"思考"只是 `reason` 字段：

```python
# tool_wrapper.py dispatch_tool 内
reason = raw_params.pop("reason", "")
if reason:
    await _emit_child_think(gateway_context, reason)  ← 这是工具选择理由，不是 LLM 真实 thinking
```

## 2. 整体调用链路（当前）

```
用户提问
  │
  ▼
intend_node
  ├─ CommandRouter.try_dispatch()        ← 命令路由（不变）
  └─ knowledge_enhancer(query, gw_ctx)   ← 【问题 A】无条件调用，2~4s LLM 开销
       ├─ LLM#1: expand_query
       ├─ 召回 (BM25/向量/RRF)
       └─ LLM#2: llm_confirm
            └─ ClarificationResult {
                 needs_clarification: bool,
                 form: dict,
                 knowledge: str,
               }
  └─→ knowledge_payload 写入 AgentState
  └─→ knowledge_snippets 写入 system_prompt（层 A）
  
  ▼
execution_node → ReAct循环
  ├─ LLM 调用 (stream)
  │   └─ content 文字（真实 thinking）【问题 B】被丢弃
  │   └─ tool_calls → dispatch_tool
  │         └─ sub_step(tool_name)
  │               ├─ emit reason（工具选择理由，"假"thinking）
  │               ├─ emit 工具入参
  │               └─ emit 工具返回
  └─ 若 tool=finish_react → 流式推送 answer
```

## 3. 方案设计

### 3.1 方案 A：LLM 自判歧义 + 按需调用知识增强（解决问题 A）

#### 3.1.1 核心思路

**放弃外部规则预判断，转而让 LLM 在调用工具时自报歧义**。

在 `query_{code}`、`compute_{code}`、`data_query_{code}` 三类工具的参数 Schema 中增加三个字段：

| 字段名 | 类型 | 用途 |
|--------|------|------|
| `intent_reason` | `str` | LLM 对用户意图的理解描述，**用于运营日志** |
| `extraction_confidence` | `float` (0.0~1.0) | LLM 对自身参数抽取准确度的自评分，**用于运营日志** |
| `ambiguous_params` | `list[str]` | LLM 认为存在歧义/不确定的参数名列表，**非空时触发知识增强** |

**触发逻辑**：
- `ambiguous_params` 为空列表 → **跳过**知识增强，直接执行工具（0 额外 LLM 调用）
- `ambiguous_params` 非空 → **触发** `analyze_query_clarification`，根据结果决定澄清追问或注入知识后继续

**根本优势**：判断者是 LLM 本身——它在调用工具的那一刻，已经完整理解了 Prompt 和上下文，比任何外部规则都能更精准地识别自己的不确定点。

#### 3.1.2 新旧架构对比

**当前架构（问题所在）**：
```
用户提问
  ▼
intend_node
  └─ knowledge_enhancer()   ← ❌ 无条件调用，每次 2~4s
       └─ analyze_query_clarification (LLM#1 + 召回 + LLM#2)
  ▼
execution_node → ReAct → 工具调用
```

**新架构（方案 A）**：
```
用户提问
  ▼
intend_node
  └─ knowledge_enhancer = None   ← ✅ 不再注入，intend_node 直接返回

  ▼
execution_node → ReAct
  └─ LLM 调用工具，同时填写：
       intent_reason: "用户查询当季营收"
       extraction_confidence: 0.65
       ambiguous_params: ["time_range"]   ← LLM 自报：time_range 这个参数我没把握
  ▼
query_clarification_plugin (before_call_back)
  ├─ ambiguous_params 为空 → 直接执行工具（✅ 0 额外开销）
  └─ ambiguous_params 非空 → 调用 analyze_query_clarification
        ├─ needs_clarification=True  → interrupt，展示追问表单
        └─ needs_clarification=False → 注入 knowledge 后继续执行工具
```

#### 3.1.3 工具 Schema 变更

三类数据工具（`query_*`、`compute_*`、`data_query_*`）的 Schema 新增三个字段。

**改造位置**：框架层 `datacloud-analysis` 中负责构建动态工具的地方（`node.py` 的 `_build_tools_list`），对所有 `_DATA_TOOL_PREFIXES` 工具注入这三个字段，与现有 `reason` 字段注入方式相同。

新增字段定义（Pydantic，追加到工具 Schema）：

```python
from pydantic import BaseModel, Field
from typing import Annotated

# 注入到所有 query_* / compute_* / data_query_* 工具的额外字段
intent_reason: str = Field(
    default="",
    description=(
        "你对用户本次查询意图的完整理解描述，"
        "例如：'用户希望查看当季各区域的营收合计，按区域分组'。"
        "用于运营分析，请如实填写。"
    ),
)
extraction_confidence: float = Field(
    default=1.0,
    ge=0.0,
    le=1.0,
    description=(
        "你对本次参数抽取准确度的自评分（0.0=完全没把握，1.0=完全确定）。"
        "综合考虑时间范围、分组维度、指标口径等参数的确定性，"
        "例如：参数较明确时填 0.9，存在一个模糊维度时填 0.6。"
        "用于运营质量监控，请客观评估。"
    ),
)
ambiguous_params: list[str] = Field(
    default_factory=list,
    description=(
        "你认为存在歧义或没有把握的参数名列表。"
        "若所有参数均已明确，填空列表 []。"
        "若某个参数语义不清（如时间范围不明确、指标口径有多种理解），"
        "将该参数名填入此列表，例如：['time_range', 'group_by']。"
        "【重要】此列表非空时，系统会自动触发知识增强流程以消除歧义。"
    ),
)
```

**注入机制**：在 `node.py` 的 `_build_tools_list` 中，为数据类工具调用新的 `inject_ambiguity_fields` 函数（类似现有 `inject_reason_field`），在工具 Schema 上追加这三个字段，并在底层 coroutine 调用前将它们剥除（不透传给原始工具实现）。

```python
# node.py _build_tools_list 中（伪代码）
def _is_data_tool_name(name: str) -> bool:
    return any(name.startswith(p) for p in ("query_", "compute_", "data_query_"))

for name, callable_or_tool in default_tools.items():
    tool = inject_reason_field(t)
    if _is_data_tool_name(name):
        tool = inject_ambiguity_fields(tool)   # ← 新增
    tools.append(tool)
```

#### 3.1.4 query_clarification_plugin 改造

**当前逻辑（第 60~109 行）**：
```python
# 从 intend_node 预先写入的 knowledge_payload 缓存中读取
payload: dict = ctx.get("knowledge_payload") or {}
if not payload:
    # 兜底：重新调用（正常流程不走此分支）
    result = await _call_fallback_enhancer(user_query)
needs_clarification: bool = payload.get("needs_clarification", False)
```

**改造后逻辑**：

```python
async def before_call_back(ctx: HookContext) -> HookDecision | None:
    tool_name = ctx.get("tool_name", "")
    if not _is_data_tool(tool_name):
        return None

    tool_params: dict = ctx.get("tool_params") or {}

    # ── Step 1：读取 LLM 自报的歧义信息 ──────────────────────────────────
    intent_reason: str = tool_params.pop("intent_reason", "")
    extraction_confidence: float = float(tool_params.pop("extraction_confidence", 1.0))
    ambiguous_params: list[str] = list(tool_params.pop("ambiguous_params", []) or [])

    # ── Step 2：记录日志 ──────────────────────────────────────────────
    logger.info(
        "[tool_intent] tool=%s | confidence=%.2f | ambiguous=%s | triggered=%s\n"
        "  intent : %s\n"
        "  params : %s",
        tool_name,
        extraction_confidence,
        ambiguous_params,
        bool(ambiguous_params),
        intent_reason,
        tool_params,   # 剔除元字段后的实际工具入参，含完整参数抽取结果
    )

    # ── Step 3：无歧义 → 直接跳过，执行工具 ──────────────────────────────
    if not ambiguous_params:
        return None   # ← 0 额外 LLM 调用，直接执行工具

    # ── Step 4：有歧义 → 触发知识增强 ────────────────────────────────────
    user_query = ctx.get("user_query", "")
    gateway_context = ctx.get("gateway_context")

    try:
        result = await _call_query_clarification(user_query, gateway_context)
    except Exception:
        logger.warning("[query_clarification_plugin] analyze_query_clarification failed, skipping")
        return None

    payload = {
        "needs_clarification": result.needs_clarification,
        "form": result.form,
        "knowledge": result.knowledge,
        "query": getattr(result, "query", user_query),
    }

    # ── Step 5：需要追问 → interrupt ──────────────────────────────────────
    if result.needs_clarification:
        from langgraph.types import interrupt
        try:
            paradigm_list = json.loads(result.form).get("paradigmList", [])
        except Exception:
            paradigm_list = []
        resume_value = interrupt({
            "prompt": "查询条件存在歧义，请确认查询维度",
            "reason_code": "PARADIGM_CLARIFICATION",
            "ask_user_payload": {"paradigmList": paradigm_list},
        })
        payload["needs_clarification"] = False
        ctx["knowledge_payload"] = payload
        _apply_resume_to_params(ctx, tool_name, resume_value, payload)
        return {"action": "patch", "patch": {"tool_params": dict(ctx["tool_params"])}}

    # ── Step 6：无需追问，但有知识 → 注入 contextKnowledge 后继续 ────────
    knowledge_str: str = result.knowledge or ""
    if knowledge_str and _is_data_query_tool(tool_name):
        return {"action": "patch", "patch": {"tool_params": {"contextKnowledge": knowledge_str}}}

    return None


async def _call_query_clarification(user_query: str, gateway_context: Any) -> Any:
    """调用 analyze_query_clarification，兼容 gateway 流式推送版本。"""
    try:
        # 优先使用 byclaw_data.paradigm 的 stream 版本（含 gateway SSE 推送）
        from byclaw_data.paradigm import (  # type: ignore[import]
            analyze_query_clarification as _stream_version,
        )
        return await _stream_version(user_query, gateway_context)
    except ImportError:
        pass
    # 兜底：使用 datacloud_knowledge 原生版本（无 SSE 推送）
    from datacloud_knowledge.intent import analyze_query_clarification  # type: ignore[import]
    import asyncio
    return await asyncio.to_thread(analyze_query_clarification, user_query, None)
```

#### 3.1.6 数据流变更总结

| 环节 | 当前 | 改造后 |
|------|------|--------|
| `intend_node` | 调用 `analyze_query_clarification`（2~4s） | 直接返回（< 1ms） |
| `execution_node` 启动 | knowledge 已在 state 中 | knowledge 尚未获取 |
| LLM 调用工具时 | 不感知歧义 | 填写 `ambiguous_params`（在同一次 LLM 推理中完成） |
| `query_clarification_plugin` | 读 `knowledge_payload` 缓存 | 读 `ambiguous_params`，按需触发 |

| 有歧义查询 | intend_node 阶段触发 | 工具调用前触发（时机稍后，但准确度更高） |

### 3.2 方案 B：推送 LLM 真实 Thinking 内容（解决问题 B）

#### 3.2.1 核心思路

LLM 在 ReAct 每一轮中，调用工具前可能产生自然语言的推理文字（存放在 `chunk.content`）。当前代码显式注释说明了丢弃原因：

```python
# 注意：不在此处推送 LLM content 文字 token。
# 部分 LLM 在生成 tool_calls 的同时也输出 content 文字（如"我来查询..."），
# 若在此处推送则会出现不必要的客套话。
```

**问题关键**：LLM 输出的 content 文字分两类：
1. **有价值的推理**："用户问的是当季营收，我需要查 Q1-Q3 数据，先用 data_query 获取原始记录..."
2. **无价值的客套话**："好的，我来帮您查询一下。" / "根据您的需求..."

目前的处理方式是两类都丢弃。需要改为：**保留推理，过滤客套**。

---

#### 3.2.2 `chunk.content` 的类型分析（不开 thinking 时是否安全）

这是改造的核心安全性问题。`chunk.content` 的实际类型取决于模型和配置：

| 模型 / 配置 | `chunk.content` 类型 | `chunk.tool_call_chunks` | 说明 |
|------------|---------------------|--------------------------|------|
| 普通 OpenAI 兼容模型（Qwen、GPT-4o 等） | `str`（可能为空字符串） | 有或无 | 最常见场景，无 thinking |
| Anthropic Claude（**未开** extended_thinking） | `str`（可能为空字符串） | 有或无 | 与 OpenAI 兼容协议行为相同 |
| Anthropic Claude（**开启** extended_thinking） | `list[ThinkingBlock \| TextBlock]` | 有或无 | content 变为 block 列表 |
| LLM 调用异常 / provider 差异 | `None` 或缺失属性 | 可能异常 | 需用 `getattr` + 兜底 |

**结论：不开 thinking 时 `chunk.content` 始终是普通字符串（或空字符串），不会出现 `list` 类型，也不会报错。**

但代码必须对所有情况做防御，原因：
1. `chunk.content` 可能是 `None`（部分 provider 的 chunk 不保证非 None）
2. 开了 extended_thinking 时 `content` 变成 `list`，直接做字符串操作会 `TypeError`
3. `emit_chunk` 失败不能中断主流程（`_stream_llm_call` 里任何异常都会触发 fallback 到 `ainvoke`）

#### 3.2.3 推送策略

**何时推送 thinking**：`chunk` 同时满足以下条件时，将 content 以 thinking 推送：

| 条件 | 原因 |
|------|------|
| `chunk.tool_call_chunks` 非空 | 说明这是中间推理轮次（正在调用工具），而非最终答案轮 |
| content 提取后为非空字符串 | 有实质内容才推送 |
| `_is_meaningful_thinking(text)` 返回 True | 过滤掉无价值的客套话 |

> **不在 `chunk.tool_call_chunks` 为空时推送**：无 tool_calls 的轮次是最终答案，其 content 已由 `finish_react.answer` 流式推送（或 formatter 一次性推送），不应重复推。

**客套话过滤**：

```python
_FILLER_PREFIXES = ("好的", "当然", "没问题", "根据您", "我来帮您", "让我来")

def _is_meaningful_thinking(text: str) -> bool:
    """判断 LLM content 是否有实质推理价值，过滤无意义客套话。"""
    text = text.strip()
    if len(text) < 10:          # 过短，无信息量
        return False
    if any(text.startswith(p) for p in _FILLER_PREFIXES) and len(text) < 30:
        return False            # 短句客套话
    return True
```

```python
_FILLER_PREFIXES = ("好的", "当然", "没问题", "根据您", "我来帮您", "让我来")


def _is_meaningful_thinking(text: str) -> bool:
    """判断 content 文字是否有推理价值，过滤无意义客套话。"""
    text = text.strip()
    if len(text) < 10:
        return False
    if any(text.startswith(p) for p in _FILLER_PREFIXES) and len(text) < 30:
        return False
    return True


def _extract_content_text(content: Any) -> str:
    """安全地从 chunk.content 中提取纯文本。

    兼容三种情况：
    1. 普通模型（未开 thinking）：content 是 str 或 None
    2. Claude extended_thinking 开启：content 是 list[ThinkingBlock | TextBlock]
       - ThinkingBlock: {"type": "thinking", "thinking": "..."}
       - TextBlock:     {"type": "text", "text": "..."}
    3. 其他 provider 差异：content 缺失或类型意外

    返回值：纯文本字符串（不含 thinking block 内容，只取 text block）。
    thinking block 单独由 _extract_thinking_text 处理。
    """
    if content is None:
        return ""
    if isinstance(content, str):
        return content
    if isinstance(content, list):
        # extended_thinking 模式：拼接所有 TextBlock（text 类型）的内容
        parts: list[str] = []
        for block in content:
            if isinstance(block, dict):
                block_type = block.get("type", "")
            else:
                block_type = getattr(block, "type", "")
            if block_type == "text":
                text = block.get("text", "") if isinstance(block, dict) else getattr(block, "text", "")
                if text:
                    parts.append(str(text))
        return "".join(parts)
    # 兜底：强转字符串
    return str(content)


def _extract_thinking_text(content: Any) -> str:
    """从 Claude extended_thinking 的 content list 中提取 ThinkingBlock 文本。

    普通模型（content 为 str）时返回空字符串，不报错。
    """
    if not isinstance(content, list):
        return ""
    parts: list[str] = []
    for block in content:
        if isinstance(block, dict):
            block_type = block.get("type", "")
        else:
            block_type = getattr(block, "type", "")
        if block_type == "thinking":
            text = block.get("thinking", "") if isinstance(block, dict) else getattr(block, "thinking", "")
            if text:
                parts.append(str(text))
    return "".join(parts)


async def _emit_thinking_token(gateway_context: Any, token: str) -> None:
    """向 gateway_context 以 reasoningLogDelta 类型推送 thinking token。

    任何异常均静默降级（debug 日志），绝不中断主流程。
    """
    if not gateway_context or not token:
        return
    try:
        from by_framework import StreamChunkEvent          # type: ignore
        from datacloud_data_sdk.stream_text import coerce_stream_chunk_text  # type: ignore
        await gateway_context.emit_chunk(
            StreamChunkEvent(content=coerce_stream_chunk_text(token)),
            event_type="reasoningLogDelta",
            content_type="1002",
        )
    except Exception as exc:
        logger.debug("[react_loop] thinking_token emit failed: %s", exc)
```

**`_stream_llm_call` 内 chunk 循环中的改造**（替换第 78~83 行的注释块）：

    raw_content = getattr(chunk, "content", None)

#### 3.2.5 安全性保障

| 场景 | 处理方式 | 是否会报错 |
|------|----------|-----------|
| 普通模型，未开 thinking，`content` 为 `str` | `_extract_content_text` 直接返回；`_extract_thinking_text` 因非 list 返回 `""` | ✅ 不报错 |
| 普通模型，`content` 为 `None` 或缺失 | `getattr(chunk, "content", None)` → `None`；两个 extract 函数均返回 `""` | ✅ 不报错 |
| Claude，未开 extended_thinking，`content` 为 `str` | 同普通模型路径 | ✅ 不报错 |
| Claude，开启 extended_thinking，`content` 为 `list` | `_extract_thinking_text` 提取 ThinkingBlock；`_extract_content_text` 提取 TextBlock（通常为空） | ✅ 不报错 |
| `emit_chunk` 网络异常 / gateway 不支持该事件类型 | `_emit_thinking_token` 内 `try/except` 静默降级，`logger.debug` 记录 | ✅ 不中断主流程 |
| `content` 类型完全意外（如 `int`、`dict`） | `_extract_content_text` 兜底 `str(content)` 转换 | ✅ 不报错（极端情况顶多推出奇怪内容，被 `_is_meaningful_thinking` 长度过滤） |

#### 3.2.6 与 tool_wrapper 的层级关系

改造前（"假" thinking）：
```
任务执行 (execution_node sub_step)
  └─ data_query_xxx (tool sub_step)
       ├─ reason: "我需要查询营收数据"   ← 工具选择理由（tool_wrapper 推送）
       ├─ 工具入参: {...}
       └─ 工具返回: {...}
```

改造后（"真" thinking）：
```
任务执行 (execution_node sub_step)
  ├─ [思考过程] LLM content 流式推送    ← 新增：真实推理 token（reasoningLogDelta）
  │    "用户问的是当季营收，需要区分按自然季还是财年季..."
  └─ data_query_xxx (tool sub_step)
       ├─ reason: "需要查询 Q1-Q3 营收"  ← 工具选择理由（更精准）
       ├─ 工具入参: {...}
       └─ 工具返回: {...}
```

> `did_stream_text` 标志**不因推送 thinking token 而置为 True**（thinking 不是最终答案），仅 `finish_react.answer` 推送后才置 True，保持原有 TTFB 语义不变。
